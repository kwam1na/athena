Manager elevation for Athena store-day access

Athena should keep POS terminals easy to use during the store day without turning every POS account into an admin account. The proposed approach is temporary manager elevation on the terminal: the base session stays POS-scoped, and a manager can unlock relevant operating surfaces with fresh staff credentials when the business needs it.

Problem Frame

Today, the pos_only membership keeps a terminal focused on point of sale and product work. That is useful for limiting unnecessary exposure, but it creates friction during a normal store day. A manager may need to review operations, cash position, stock issues, orders, or other decision surfaces from the same terminal where sales are happening.

Requiring the manager to sign out the terminal user and sign into a separate full-admin web account interrupts the floor workflow. The system needs a way to recognize fresh manager presence without broadening the permanent account role attached to the terminal.

Current Model

Account membership

full_admin and pos_only decide what the signed-in Athena account can generally access.

Operational staff identity

Roles like manager and cashier represent who is physically acting inside store workflows.

Command approval

Protected actions can require server-validated manager proof before mutation, so UI access is not treated as authority.

Recommendation

Recommended

Keep the terminal signed in as POS, then allow temporary manager elevation.

A manager authenticates with staff credentials or PIN from the terminal. Athena records a short-lived elevated operating context for that store and terminal session. While active, the sidebar and protected routes expose selected manager-relevant store-day surfaces. High-risk actions still require command-level approval proof before they can mutate protected state.

Decision principle

Navigation can be elevated. Authority stays at the command boundary.

The app can make the right surfaces visible after manager elevation, but protected business actions must still be enforced server-side. The browser should not become the source of truth for whether a manager approved something.

Alternatives

Alternative A: Broaden pos_only

Permanently allow POS users into more surfaces.

Pros: simplest user experience and least new product machinery.

Cons: weakens the role permanently, makes every terminal user more privileged, and blurs accountability.

Use only for: narrow read-only facts that are genuinely safe for any POS operator.

Alternative B: Temporary manager elevation

Keep POS as the base session, then unlock relevant surfaces after fresh manager authentication.

Pros: matches store-floor behavior, keeps account roles clean, supports auditability, and reuses Athena's approval direction.

Cons: adds an elevated-session concept and requires careful expiry, visibility, and audit behavior.

Best fit: the first implementation.

Alternative C: Store-day manager mode

A manager unlocks a terminal for the whole store day or until explicitly ended.

Pros: lower repeated-auth friction once the terminal is supervised.

Cons: higher risk because elevation can outlive the manager's physical presence.

Consider later: only if short-lived elevation proves too disruptive.

Proposed Scope

Access tier Surfaces Rationale
Base POS session Point of sale, product lookup, product details needed to sell Keeps the terminal focused and safe for normal cashier use.
Manager-elevated terminal Daily operations, cash controls, register open/close context, same-day stock adjustment review, selected order/customer context, operational queue signals Gives a manager the information needed to make store-day decisions without account switching.
Full admin account only Organization members, role management, configuration, storefront settings, destructive/global administration, deep analytics, vendor/procurement administration These are business-owner or back-office controls, not store-floor interruptions.

Behavior Requirements

  1. A POS terminal can request manager elevation without signing out the current Athena account.
  2. Manager elevation requires fresh operational staff authentication from an active staff profile with a manager role.
  3. Elevation is scoped to the current store and terminal session, and expires automatically.
  4. The UI clearly indicates that the terminal is operating with manager elevation and offers a direct way to end it.
  5. Elevated access can reveal approved store-day surfaces, but protected mutations still enforce their own command-level approval or role requirements server-side.
  6. Manager elevation creates an audit trail identifying who elevated the terminal, when it started, when it ended, and what protected actions relied on it.

Guardrails

  • Do not make pos_only equivalent to admin access.
  • Do not trust a browser-held manager identity as proof for protected mutations.
  • Do not create separate manager-only command paths that bypass the normal business workflow.
  • Do not expose organization or configuration surfaces just because a terminal is elevated.
  • Do not let elevation survive indefinitely without an explicit manager presence model.

Open Signoff Decisions

Decision 1

Should the first version unlock read-only decision surfaces only, or include a small set of low-risk operational mutations?

Decision 2

Should elevation expire after a short idle/windowed period, at register close, at daily close, or only by explicit manager exit?

Decision 3

Which surfaces are essential for first release: daily operations, cash controls, orders, inventory adjustments, procurement, or analytics?

Decision 4

Should the elevated state be visible only in the header/sidebar, or should every protected surface show the active manager context?

Suggested Signoff

Approve Temporary Manager Elevation as the product direction. Start with manager-authenticated access to selected store-day decision surfaces from a POS terminal, keep admin configuration behind full-admin account access, and keep protected mutations enforced through Athena's command approval boundary.
