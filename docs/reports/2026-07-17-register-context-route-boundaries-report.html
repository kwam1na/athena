A delivery-candidate guide to the register identity contract, legacy trace fallback, route boundaries, and operator-facing close workflows.

Executive Summary

This candidate keeps register identity visible as an operator moves between Cash Controls activity and Workflow Trace history. Convex now returns a small, normalized register-session header contract, and both screens render that contract through one shared presentation component.

The same delivery also clarifies Daily Close and Daily Operations status presentation and moves route implementations out of TanStack route declaration files. The route split removes Fast Refresh export warnings without changing route URLs or authentication boundaries.

Why It Mattered

Before this change, an operator could open activity or trace history and lose the drawer identity that made the page meaningful. Generic history headings forced manual reconstruction, while route modules mixed route declarations with component exports and triggered Fast Refresh warnings.

  • Register number now travels with activity responses and, when a valid same-store register session resolves, register-session trace responses; terminal name remains optional and same-store only.
  • Legacy traces remain readable when their primary subject is stale but their distinct legacy lookup still identifies a valid same-store session.
  • Malformed cross-store references never expose another store's terminal label.

Mental Model

Treat register identity as a compact label attached to an authorized operational record, not as a new source of financial truth. Convex resolves the session inside the already-scoped store query, returns only ID, register number, and an optional same-store terminal name, and the UI formats that label consistently.

authorized activity or trace
  -> resolve same-store register session
  -> resolve terminal only when terminal.storeId matches
  -> return { _id, registerNumber, terminalName }
  -> shared RegisterSessionIdentity header

Before and After

  • Before: activity and trace pages could show generic context. After: both receive the same normalized register identity shape.
  • Before: Workflow Trace exposed a back button, summary, status and health badges, plus primary lookup diagnostics in its header. After: the header is intentionally reduced to register identity or generic title plus / History; those diagnostics were removed rather than relocated, while event ordering and timeline content remain available below.
  • Before: a stale primary trace subject could prevent a valid legacy lookup from being attempted. After: primary and distinct legacy candidates are tried in order, with store validation for each.
  • Before: a referenced terminal name could be read without confirming store ownership. After: terminal labels are suppressed unless the terminal belongs to the trace or session store.
  • Before: route declaration files also exported page implementations. After: hyphen-prefixed companion modules hold implementations while route files retain route declarations.
  • Before: two Fast Refresh warnings remained. After: non-component helpers and route implementations live in dedicated modules, leaving the changed frontend lint set clean.

Layer-by-Layer Mechanics

  • Cash Controls query: attaches the authorized session header to read-model, sync-evidence, and empty activity pages, so pagination state never determines whether identity is present.
  • Workflow Trace query: enriches register-session traces after access validation, prefers the primary subject, then tries a distinct legacy lookup, and rejects missing or cross-store sessions.
  • Shared presentation: formats Register <number> with an optional / <terminal> suffix and supplies restrained generic fallback titles.
  • Daily Close presentation: reduces visual panel weight, refines closeout/reopen states, and changes the transaction report to four columns by placing amount beside the accessible payment-method icon. Rows, redaction, calculations, and closeout authority are unchanged.
  • Daily Operations presentation: constrains status-band width, adjusts borders and spacing, and only renders the open-register divider when automation content precedes it. Automation decisions and register state are unchanged.
  • Route files: retain the same TanStack paths and loaders while importing implementation components from non-route companion modules.

Failure Boundaries and Intentionally Unchanged Behavior

  • Invalid, missing, or cross-store trace session candidates produce no register identity enrichment; the trace still renders its generic title.
  • A valid register session with no terminal, or with a foreign terminal reference, renders the register number without a terminal name.
  • Malformed, missing, or cross-store trace candidates leave registerSession null and render the existing generic trace title plus / History; they do not fail the trace read.
  • Trace header summary, status, health, primary lookup diagnostics, and the header back button were intentionally removed. Timeline events and chronological rendering remain the operational evidence surface.
  • The activity query keeps its existing access control and store/session checks; the new header is derived only after those checks pass.
  • No schema migration, route URL change, financial calculation change, or new authorization role is introduced.
  • The bun lockfile was reserialized as part of the transferred dirty set; package manifests were not changed.

Next-Time Guidance

  • Add identity fields at the authorized response boundary, not by issuing a second unscoped browser query.
  • For compatibility lookups, try distinct candidates in priority order and validate ownership on every candidate.
  • Never display a related entity's name until its tenant or store scope matches the authoritative parent record.
  • Keep route declaration modules limited to route exports; move page implementations and reusable helpers into companion modules.
  • Cover populated, empty, generic, stale-primary, legacy-fallback, and cross-store cases whenever the identity contract changes.

Quiz: Pass Required

1. Where is register identity resolved for these operational pages?
Identity is enriched inside the authorized store/session query so the browser receives only scoped presentation data.
2. What is the shared identity contract intended to standardize?
It standardizes register presentation when identity resolves. Generic context remains for unenriched traces, and financial or persistence authority does not move.
3. What happens when a trace's primary session subject is stale but a distinct legacy lookup is valid?
Candidates are tried in priority order, so a stale primary subject does not suppress a valid distinct legacy lookup.
4. When may a terminal name appear in the register header?
Tenant/store ownership is checked before terminal display text is returned.
5. A register-session trace contains malformed and cross-store identity candidates. What should the operator see?
Identity enrichment degrades to null without weakening the authorized trace read. A foreign label would violate store isolation, and identity failure does not invalidate the timeline.
6. Why were route implementations moved into hyphen-prefixed companion files?
The split preserves routing behavior while removing component/non-component export mixing.
7. Which behavior intentionally did not change?
The change does not introduce new URLs, authorization roles, or authentication boundaries.
8. What specific review scenarios were added to the focused test set?
Those cases directly prove the compatibility and data-isolation boundaries found during adversarial review.
9. What should an extension of this identity contract test?
The important risks are missing context, compatibility fallback, and tenant isolation—not incidental generated counts.
10. An operator can access Workflow History through active terminal manager elevation but opens the POS activity route for the same session. What should happen?
The identity component does not merge authorization policies. Activity remains full-admin scoped, while traces retain their existing manager-elevation and limited shared-demo paths.
