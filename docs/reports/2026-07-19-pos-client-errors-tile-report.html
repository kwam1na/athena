POS Client-Errors Metric Tile: No-Swap Loading and Sheet Drill-In

A follow-up to the POS observability delivery: the posClientEvent telemetry stream now has a product surface — a metric tile on Terminal Health that opens a two-level sheet, with a loading approach chosen specifically to avoid the flash a skeleton would still cause.

Delivery candidateBranch: codex/pos-client-errors-panelBase: origin/mainMerge: pending

Executive Summary

  • A "Client errors" tile joins the existing metrics row on the POS Terminal Health page (Terminals / Healthy / Pending sync / Needs review / Stale), danger-toned when there are recent errors.
  • Clicking it opens a sheet with a filterable list (All / Errors / Warnings) that drills into a per-event detail view (message, error, stack trace, metadata, terminal fingerprint) with a back button.
  • The placement iterated twice in review: an initial panel at the bottom of the page was rejected as invisible once the terminal roster is realistically sized; the metrics-row tile fixed that.
  • The loading state iterated three times: a sized skeleton still produced a visible swap-flash; a label-fade approach was rejected for the same reason; the final approach has no loading placeholder at all — the count renders its true current value (0 while unresolved) and updates in place.

Problem And Context

The prior POS observability delivery added a durable client-error telemetry pipeline (posClientEvent table, offline-buffered ingest) but no product surface — the data was API-only. This delivery closes that gap on the existing POS Terminal Health page.

Two design questions dominated the work, both settled by iterating against actual rendering behavior rather than assumption: where does a store-level signal belong on a page dominated by a per-terminal card list, and how do you show a Convex-query-backed metric without a loading-state flash.

What Intentionally Did Not Change

  • No new Convex functions or schema. The panel consumes the existing pos/public/telemetry:listClientEvents query from the prior delivery unchanged.
  • No loading skeleton. Considered and rejected twice — see Next-Time Guidance.
  • Tile count follows the sheet's active filter rather than always showing the unfiltered total, to avoid a second query subscription just for the tile.

Next-Time Guidance

  • Layout-shift and swap-flash are different problems. Sizing a skeleton to match final content prevents the page from jumping, but the skeleton element is still unmounted and replaced — that unmount/mount is what reads as a flash, independent of size matching.
  • For advisory, non-blocking metrics on a live-query UI, the simplest fix is often to stop treating "loading" as a state that needs its own visual: render the real (if temporarily default) value immediately and let it update in place when the query resolves.
  • When a new page signal is store-level rather than per-item, put it in the page's existing summary region (a metrics row, a header stat) rather than appending a new section after an unbounded list of item cards.

Quiz: Pass Required

You must score at least 4 out of 5 to pass.

1. Why was a panel at the bottom of the Terminal Health page rejected as the initial placement?
The roster grows with terminal count, so anything appended after it scales poorly. The metrics row, by contrast, is a fixed-height summary region regardless of how many terminals exist.
2. Why did a skeleton sized to match the final content still produce a visible flash?
Layout shift and swap flash are distinct problems with distinct causes. This delivery solves the flash by never mounting a placeholder in the first place.
3. What does the tile show while its query is still loading?
No loading treatment: isLoading maps to an empty events array, so the same count element renders 0 and later renders the resolved count — nothing is swapped.
4. In the sheet, how does the level filter (All / Errors / Warnings) affect the Convex query?
The filter drives which indexed query runs server-side. The two-index nuance was surfaced by independent verification and is worth knowing if extending the query later.
5. How does the sheet return from the error-detail view to the list?
The sheet is a single component with two render modes toggled by whether an event is selected, not two separate sheets.
