Executive Summary

This candidate turns the current app and shared demo work into one coherent experience: a visitor enters a real Athena owner workspace for Osu Studio, explores useful operational surfaces, and remains inside a server-enforced effect boundary.

The same delivery closes the difficult seams around that experience. Admission proves the configured tenant is the current synthetic foundation, restore status and epoch fence both cloud writes and browser-local POS work, and register authority converges without sacrificing local-first operation.

Why This Work Mattered

A convincing demo cannot be a disconnected mock. It has to use the same product, order, service, cash, and POS paths that operators use, while ensuring that anonymous visitors cannot inherit ordinary full-administrator effects.

The earlier pieces also left cross-layer failure modes: browser bootstrap could outlive a restore epoch, old POS authority could disagree with cloud state, public service mutations could bypass disabled controls, and unknown baseline versions could reset or fail without an explicit migration decision.

Mental Model: One Synthetic Store, Three Fences

Think of the demo as one real synthetic store protected by three nested fences.

  • Admission fence: the feature flag and supported stage must be enabled, and the configured store must match a current ready provision-owned baseline before a short-lived ticket is issued.
  • Effect fence: a server-derived demo principal and central capability policy allow safe internal exploration while denying payments, refunds, exports, integrations, identity changes, and other external or destructive effects.
  • Epoch fence: every restore publishes status and epoch. Server writes verify readiness transactionally; browser work gates children synchronously and cancels stale local jobs before they touch IndexedDB.
visitor -> admission ticket -> shared-demo principal -> synthetic Osu Studio store
                                              |
                                              +-> capability policy -> allowed internal workflow
                                              |                    -> typed denial
                                              +-> restore {status, epoch}
                                                   -> cloud write fence
                                                   -> browser/POS epoch fence

Before and After

  • Admission moved from generic configured relationships to proof that the target is the current ready synthetic foundation, before rate budget or ordinary tenant reads.
  • Scattered or UI-only restrictions became a server-derived default-deny capability catalog with store and restore clamps on public mutations.
  • Duplicated fixtures became one shared story for Osu Studio identities, eight canonical products, imagery, order, customer, staff, and register 01.
  • Loosely coupled restore and browser bootstrap became a status-plus-epoch protocol with whole-app overlay, synchronous render gating, per-epoch coordination, and stale-work cancellation.
  • Implicit older-baseline behavior became an explicit migration registry: versions 11-15 reset operational state, 16 preserves continuity into 17, and unknown paths fail closed.
  • Generic POS repair became precise authority outcomes for missing mappings, same-ID legacy bootstrap, stale cloud subjects, exact tombstones, and cloud-session loss. Reconciliation is bounded to 100 rows with scheduled continuation.
  • The demo UI now favors quiet entry and free exploration. Search, filters, selection, pagination, rows, and details remain interactive; effect-producing controls are hidden, disabled, or denied consistently.

Failure Boundaries

  • Misconfigured tenant IDs fail admission unless they point at the current ready provision-owned baseline. Production support remains deliberate behind ATHENA_SHARED_DEMO_ENABLED and STAGE; no deployment-ID allowlist was added.
  • A restore that exceeds the 500-row per-table safety bound fails instead of publishing a partial baseline. The overlay exposes failure and retry state.
  • A newer restoring or failed epoch invalidates the previous local coordinator immediately. Children are not rendered ready for epoch N+1 until that epoch is applied locally.
  • Visitor registers use 02-99. Only the machine-readable register-number collision retries; persistent or generic conflicts return immediately, and namespace exhaustion is explicit.
  • Mapping-pending POS activity does not drain an unbounded backlog on the report hot path. Direct-ID recovery processes 100 rows and schedules continuation.
  • Concurrent visitors can observe shared state by design. The hourly restore returns the synthetic tenant to its baseline; this is not tenant isolation.

What Intentionally Did Not Change

  • Normal authenticated tenants continue through their existing handlers and authority model.
  • The demo uses real Athena workflows rather than a parallel mock application.
  • Production demo support remains available only when explicitly configured; the earlier deployment allowlist proposal remains rejected.
  • No production deployment is included. The finish line is PR merge plus a clean local root fast-forward to origin/main.

How to Extend or Debug This Safely

  • Add new demo effects to the central capability catalog and exported-mutation coverage. Do not rely on disabled UI controls as authorization.
  • Treat each baseline bump as an explicit migration entry. Choose continuity or operational reset deliberately; never default unknown versions.
  • When changing restore behavior, test server transaction fencing, browser render gating, coordinator cancellation, and local applied-epoch persistence together.
  • When changing register recovery, preserve exact store, terminal, register, session, mapping, and epoch fingerprints. Keep reconciliation work bounded.
  • Regenerate harness and Graphify artifacts after relevant source/test changes, and refresh solution/report fingerprints after the final diff is frozen.
