Daily Operations cash-closeout continuity

How one operating-day view now routes unresolved work without replacing the systems that own it.

Branch: fix/daily-operations-cash-controlsCommit: 96d86638PR pendingProduction deploy not runRoot alignment pending

Executive Summary

Daily Operations now makes unresolved register sessions and related review work easier to find, while Cash Controls and Open Work remain the authoritative places to close, approve, and resolve it.

The change adds session routing for the operating date, a protected register sales summary, and calmer grouped inventory-review cards. It does not introduce a new settlement engine or loosen cash-data permissions.

Mental model

Think of Daily Operations as the day board, Cash Controls as the register ledger, and Open Work as the resolution queue. The board points an operator to the right ledger or queue; it does not duplicate their authority.

For the current day, Daily Operations uses its attention items. For a historical day, it asks a bounded, authorized query for sessions opened on that operating date and still in an open, active, or closing state.

Before and after

  • Before: an operator could see operating-day signals without a dedicated, labeled list that routes an unresolved session to its existing Cash Controls closeout.
  • After: Daily Operations shows terminal and register labels and deep-links to the authoritative register-session view.
  • Before: repeated synced-sale inventory reviews made the queue noisy.
  • After: reviews for the same SKU are grouped for display, while every underlying work item remains durable and is still resolved individually.

Layer-by-layer mechanics

  • Convex daily-operations reads open sessions by store, status, and operating date, then returns safe terminal/register labels.
  • Daily Operations renders those results as navigation links to Cash Controls. Carryovers are not represented as sessions opened for a different selected date.
  • Cash Controls derives total sales, completed-sale count, average sale, and payment mix from completed cloud transactions linked to the register session. Financial values keep the existing manager/full-admin redaction boundary.
  • Open Work adds a URL-backed work-type filter, pages display groups, and submits all underlying inventory-review ids through the existing resolver.

Failure boundaries and intentional non-changes

  • The sales summary is not a cash count, settlement balance, or guarantee that every offline sale has arrived. Voided sales are excluded from its completed-transaction query.
  • Daily Operations does not provide a global register inventory. Historical results are limited to the requested operating date and open, active, or closing sessions.
  • Grouped Open Work is presentation-only. It does not discard evidence, adjust inventory automatically, or create a new bulk backend mutation.
  • The grouped UI resolves items sequentially and stops on an error, so it must not be described as an all-or-nothing transaction.

Next-time guidance

  • Extend the existing Cash Controls workflow rather than adding a second closeout action to Daily Operations.
  • Test current and historical operating dates independently whenever surfacing register sessions.
  • Keep manager/full-admin financial redaction tests paired with any new session financial data.
  • When changing grouped operational work, test both the displayed group count and every original id sent to the resolver.
