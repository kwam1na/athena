Executive Summary

This PR moves the operation admission rail from a single proving mutation to the V26-1096 shared-demo reachable write slice: POS sale/corrections/sync, operations, cash-control, inventory adjustment, staff authentication/messages, storefront fulfillment, return/exchange, and protected-effect declaration writes.

The important boundary is that migrated shared-demo writes no longer depend on a generic helper option to smuggle write capability through ordinary auth. Migrated public mutations declare their operation definition, resolve the actor through operation admission, and leave only remaining unmigrated public writes as exact legacy exemptions.

  • V26-1096 migrates the next demo-reachable writes through operation admission definitions and adapters.
  • V26-1097 removes the migrated shared-demo write paths from generic auth helper admission and legacy policy inventories while preserving the explicit read bridge.
  • The candidate is open and mergeable at PR #689; merge and production deployment remain out of scope.

Why This Mattered

Before this candidate, Athena had two write-admission stories. The new operation rail handled one proving operation, while shared-demo write compatibility still lived in helper options and broad policy inventories. That made it too easy for new or migrated writes to bypass the operation-definition catalog that is supposed to become the platform-owned source of truth.

The operational risk was not that every legacy write was wrong. The risk was that migrated writes could remain partly admitted by old shared-demo paths, leaving reviewers without a crisp answer to which catalog owns a write boundary.

Mental Model

Think of operation admission as the front desk for migrated public writes. A migrated public mutation does not ask a generic auth helper to infer whether a demo actor may write. It presents a named operation definition, the rail validates that definition against the actor and scope, and the command handler receives the admitted actor context.

The legacy exemption inventory is now a deliberate map of what has not yet moved through the front desk. Every migrated write should leave that inventory, otherwise the system is claiming two owners for the same permission boundary.

  • Operation definitions describe the function, capability, readiness, and scope.
  • Adapters translate normal and shared-demo principals into operation-admission actors.
  • Public mutations use admitPublicMutation or a narrow wrapper around it before command logic runs.
  • The structural checker counts definitions and remaining legacy exemptions so drift is visible.

Before And After

Before: the proving path covered only a narrow write slice while many shared-demo reachable writes still relied on legacy helper or policy paths. Generic Athena user auth could receive shared-demo write capability options, and the shared-demo public-function inventory still listed writes that were intended to migrate.

After: the V26-1096 write groups have operation definitions. The public handlers route through operation admission, shared-demo denial is normalized into command-style user errors where the command surface expects that shape, and generic auth keeps only the explicit shared-demo read bridge. 

Before: helper option -> shared-demo write capability -> command
After: operation definition -> operation admission actor -> command

Layer Breakdown

The operationAdmission layer declares and validates the authoritative write catalog. The operations layer wraps public mutations with admission before running command logic. The sharedDemo layer supplies actor and policy support for admitted demo operations, but no longer owns the migrated write inventory. The checker layer proves that migrated functions are represented by definitions and that only remaining unmigrated public mutations stay in the legacy exemption list.

  • approvalRequests now derives the reviewer from ctx.operationAdmission.actor when admission succeeds, then falls back to ordinary auth without shared-demo write options only for the normal path.
  • POS, cash-control, stock operation, staff, daily-opening, terminal registration, and storefront fulfillment/refund-capable writes now wrap public mutation handlers with shared-demo operation admission.
  • sharedDemo/public lifecycle mutations route through operation admission definitions instead of policy-only legacy inventory entries.
  • policy and checker coverage were tightened so operationAdmission owns structural proof for migrated shared-demo capabilities.

What Intentionally Did Not Change

This PR does not migrate every public mutation. It keeps the remaining unmigrated writes as exact legacy exemptions and reports the current count through the checker. Public queries, public actions, and read paths remain outside this write-admission slice unless already part of the existing foundation.

The protected effect gateway is present in the rail foundation, but this V26-1096/V26-1097 slice does not turn every migrated operation into an effect-gateway consumer. That is a later hardening boundary, not a hidden claim in this candidate.

Operational Guidance

When the next public write migrates, start by adding the operation definition and test coverage, then wrap the public handler with operation admission, and only then remove the function from the legacy exemption inventory. If a command surface expects user_error results instead of thrown Convex errors, normalize admission denial at that boundary and test the shape explicitly.

Do not reintroduce generic shared-demo write capability options into Athena user auth. That helper is no longer the write-admission switch for migrated operations.
