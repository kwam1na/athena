Executive Summary

This delivery deepens the landing page's fixed one-day story (Wednesday 2026-07-15) that the shared demo store narrates. It adds a dark ('charcoal') visual variant for every hero/product screenshot, gives the hero section anime.js-driven entrance motion, adds a new read-only RegisterSessionScene that renders the real register-session detail view inline, wires the demo store's cash-controls config so the story's GH5 drawer variance actually gates into manager-approval-pending, recasts the demo cashier/manager names, and restyles the Bulk Operations pages and RegisterSessionView around the shared PageLevelHeader/PageWorkspace components.

It builds directly on PR #685 (operations screenshot fixtures + POS gate-flash fixes) and commit 14a22923 (one-day story coherence pass), both already on main, and keeps live scenes (demoDayFixtures.ts) and static screenshot fixtures (src/stories/operations/*Fixtures.ts) reconciled to the same narrated numbers.

Why It Mattered

The landing page was previously forced fully light regardless of visitor preference, had no motion, and — despite the copy implying a drawer variance needs a manager's signoff — the demo store's Convex seed never actually set the config field that gate reads, so the closeout would have silently auto-cleared instead of demonstrating the product's actual behavior.

The demo cashier/manager names also needed a one-time recast, which touches a Convex seed constant, a TypeScript story-constant module, and several test files independently, since there is no single indirection point for staff identity strings today.

The Mental Model

Two parallel sources must always agree on the same narrated day: demoDayFixtures.ts (what the live React scenes on /landing render) and src/stories/operations/*Fixtures.ts plus captured PNGs (what the static hero/product screenshots show). This delivery adds a third axis that both sources now share: theme (light vs. charcoal dark) and motion (animated on live scenes, already-settled on static shots).

  • Convex seed (provision.ts) -> shared/sharedDemoStory.ts constants -> demoDayFixtures.ts (live scenes) and src/stories/operations/*Fixtures.ts (static shot captures) -> both consumed by -index-route-view.tsx.
  • useLandingTheme decides light vs. charcoal-dark for the whole landing while it is mounted; LandingWorkspaceShot picks the light or -dark.png asset to match.
  • RegisterSessionScene is the real RegisterSessionViewContent component fed by fixture data with every command handler stubbed to a rejected 'read-only exhibit' result — not a mock, so it stays truthful to the real UI.

Before vs. After

  • Before: useForcedLightTheme unconditionally stripped the .dark class while mounted — the landing was always light, full stop, with no dark assets to show anyway.
  • After: useLandingTheme respects the visitor's light/dark preference and, only when dark is active, pins data-theme-variant to 'charcoal' (the variant every shot was captured in), restoring the prior variant on unmount.
  • Before: the demo store's cash-controls config never set requireManagerSignoffForAnyVariance, so the narrated GH5 shortage could auto-clear on closeout despite the story's copy implying otherwise.
  • After: provision.ts seeds config.operations.cashControls.requireManagerSignoffForAnyVariance = true, plugging into the existing registerSessionCloseoutGate.ts consumer — no new mechanism was needed, only the missing seed value.
  • Before: the hero and register-session detail rendered fully settled with no entrance choreography.
  • After: an anime.js timeline staggers the hero copy/glow/shot on mount (skipped entirely under prefers-reduced-motion), and RegisterSessionScene staggers its embedded rows in on scroll.

Layer-by-Layer Mechanics

Convex seed layer: convex/sharedDemo/provision.ts sets the store's cashControls policy and (via shared/sharedDemoStory.ts) the renamed staff identity at provisioning time.

Shared story-constants layer: shared/sharedDemoStory.ts is the single TypeScript source of truth for staff names/usernames and now exports sharedDemoStaffShortName() for the abbreviated display form used in activity feeds, so call sites stop hand-writing the abbreviation.

Fixture layer: demoDayFixtures.ts (live scenes) and src/stories/operations/*Fixtures.ts (static shot captures) both import from sharedDemoStory.ts so a rename only needs to happen once at the source; the fixtures were still updated for the new short-name helper and expanded activity events.

Presentation layer: useLandingTheme (theme/dark-variant pinning), the anime.js timelines in -index-route-view.tsx and RegisterSessionScene.tsx (entrance motion, reduced-motion-aware), and the restyled BulkOperationsPage/Filters/Preview and RegisterSessionView around PageLevelHeader/PageWorkspace.

Failure Boundaries

  • Reduced motion: -index-route-view.tsx HeroSection checks window.matchMedia('(prefers-reduced-motion: reduce)').matches and snaps every animated element straight to its resting state, skipping the anime.js timeline entirely.
  • RegisterSessionScene is read-only by construction: every command prop (onAuthenticateStaff, onRecordDeposit, etc.) is stubbed to resolve an 'unexpected_error' / 'Read-only exhibit' result, and the exhibit wrapper disables pointer events — a visitor cannot mutate anything from the marketing page even if a click somehow reached the component.
  • Theme unmount: useLandingTheme restores the prior data-theme-variant (or removes it) when the landing unmounts, so navigating away doesn't leave the rest of the app pinned to charcoal.
  • The cash-controls policy change is seed-only in this delivery — no closeout/gate domain logic changed, so behavior for real (non-demo) stores is unaffected; only the shared demo store's provisioned config changed.

Next-Time Guidance

  • When renaming a demo staff member, grep the old literal string across shared/*.ts, convex/sharedDemo/**, and src/components/**/*.test.tsx before renaming — there is no single indirection point today, so a partial rename leaves stale assertions.
  • Before adding a narrative beat to the landing story that implies a policy gate, verify the underlying Convex seed actually sets the config field a real consumer reads (grep the field name in convex/operations and convex/sharedDemo) — a plausible-sounding demo policy that isn't wired up will silently no-op.
  • New inert landing exhibits that embed a real production component should stub every command/mutation prop to a rejected 'read-only' result and disable pointer events on the wrapper, matching RegisterSessionScene.tsx, instead of mocking the component.
  • If debugging a POS/register-session gate flash surfaces again in this area, instrument the live view-model fields during the real interaction before theorizing from the code — see docs/solutions/workflow-issues/athena-pos-gate-flash-instrument-before-theorize-2026-07-19.md, which found three plausible code-reading theories wrong in a row on the immediately preceding delivery.
