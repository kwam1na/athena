POS Catalog Refreshes at Safe Idle Boundaries

How Athena closes remote product freshness gaps without making the full register catalog reactive or changing data beneath active work.

Executive Summary

An already-open POS register can now discover remote product and metadata changes without a route reload. The register subscribes to one small store revision, waits until operational work is idle, then performs one imperative full-snapshot read and adopts it only after durable local persistence.

Active sale, payment, drawer, checkout, and local-save work continues using the exact catalog revision it started with. One restrained bottom-left app-message toast communicates waiting, offline, refreshing, retry, and authorization-paused states.

Why the Gap Mattered

POS search is intentionally local-first. That keeps checkout fast and available offline, but the old one-shot mount refresh meant a product created elsewhere stayed invisible for as long as the register route remained open.

Making the entire catalog a live Convex subscription would solve freshness by repeatedly reading and transferring a large snapshot. The delivery instead separates cheap change detection from expensive snapshot transfer.

Mental Model: Doorbell, Delivery, and Receipt

The revision is a doorbell, not the package. It tells the register that catalog metadata changed without carrying catalog rows. The revisioned snapshot query is the delivery. IndexedDB staging is the receipt proving the package arrived safely. Promotion makes that durable version the idle default, while runtime pins keep active work on its original package.

A fetched response is never trusted directly. Athena exposes only a locally selected version that survived staging, scope checks, idle checks, and any active runtime pin.

                                  v

Before and After

Before: route mount or an explicit refresh key triggered a full snapshot. Remote changes had no signal while the route stayed mounted, and nested product entry could create a second catalog owner.

After: snapshot-affecting mutations advance one monotonic store revision. The POS view model alone observes that tiny signal, coalesces newer revisions, and supplies one trusted row set to ProductEntry. Expense, prewarm, and generic consumers keep their existing one-shot lifecycle and do not mount the POS coordinator.

  • Idle register: fetches and adopts the latest durable snapshot automatically.
  • Busy or offline register: retains the prior catalog, performs no full read, and keeps one pending message.
  • Availability-only churn: does not advance metadata revision.
  • Multiple tabs: monotonically share staged/active versions while owner-scoped pins prevent one tab from releasing another tab's active revision.

Layer-by-Layer Mechanics

Convex owns the metadata truth. Effective product, SKU, taxonomy, import, provisional, pending-checkout, repair, and deletion paths advance exactly once when their register-visible projection changes. Sales evidence, quantities, stock, holds, and availability overlays stay outside the metadata signal.

The register coordinator owns orchestration. It observes only the revision, checks online and operational-idle state, runs one scope-owned attempt, stages the revisioned envelope, promotes monotonically, and reselects the owner's durable pin before exposing rows.

The local store owns durability. It retains active, staged, legacy-compatible, and pinned versions. Command entry synchronously captures the selected version; the first durable operational event stores its owner pin atomically. Leases, heartbeats, per-command guard reference counts, unique request identity, and BFCache-aware document claims cover remount and multi-tab races.

Presentation remains quiet. A stable priority-50 app message uses the existing bottom-left toast host with no cashier action. It clears only when the current runtime adopts the latest durable catalog.

Failure and Concurrency Boundaries

  • Server or IndexedDB failure keeps the last trusted rows; fetched but unpersisted rows never reach search.
  • Transient retries use 1, 2, 4, and 8 second attempts, then 60 second quiet periods. Offline and authorization states do not hot-loop.
  • Store, terminal, auth, owner, and unique attempt identity reject stale completions, including A to B to A navigation.
  • A command beginning during promotion synchronously guards exposure and then persists the prior owner pin with its event.
  • Overlapping commands use reference-counted, idempotent guard settlement; one command cannot clear another's guard.
  • Owner pins use bounded leases and a non-idle heartbeat. Expired abandoned owners are reclaimed while active owners remain retained.
  • Document claims rotate copied session owners in duplicated tabs and preserve ownership across BFCache restoration.
  • Authorization lookup on the hot revision path is indexed-only and tenant-scoped; it cannot fall back to collecting the Athena user table.

What Intentionally Did Not Change

  • The full catalog is still imperative rather than a reactive row subscription.
  • Availability retains its existing bounded live and full offline snapshot contracts.
  • Active cart lines are not repriced, removed, or rewritten by catalog refresh.
  • The legacy row-array endpoint and local compatibility snapshot remain available.
  • Expense and generic catalog consumers retain one-shot behavior.
  • There is no cashier apply, retry, dismiss, or refresh action and no cross-tab refresh leader election.

Next-Time Operational Guidance

  • When adding a snapshot-visible field or membership rule, update the effective-change revision predicate and add a no-op counterexample.
  • Keep sales and availability churn outside the metadata revision unless the full catalog projection truly changes.
  • Never expose server rows before durable local selection; treat staging, promotion, pin selection, and UI exposure as separate boundaries.
  • When debugging stale catalogs, inspect observed revision, applied revision, idle blockers, owner pin, retry/auth state, and selected local version in that order.
  • When adding a new catalog consumer, decide explicitly whether it is the POS owner or a one-shot generic consumer. Do not mount both paths.
