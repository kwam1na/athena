Executive summary

The in-store POS runs local-first and drains events to Convex. Phase 2 closes the client→server trust boundary for the online completion paths, which previously trusted terminal-supplied prices, retries, clocks, and money units. Four coordinated units land together:

  • U7 — the server re-prices every sale line against catalog authority; an unauthorized deviation cannot complete, an authorized one needs a manager proof and writes an append-only audit.
  • U8 — a client-stable idempotency token dedups the transaction mint and guards the drawer cash, so a retry can't mint a second sale or double-charge.
  • U9 — the server derives the business timestamp and operating date from its own timezone authority; terminal clock skew is clamped/flagged without breaking sync retry-matching.
  • U10 — a deterministic, idempotent pesewas migration + completion marker for the never-migrated POS money tables. The schema constraint-flip is deliberately deferred to a gated production cutover (V26-1066).

The problem, in plain language

A local-first cash register is only as trustworthy as the values the terminal sends when it finally syncs. The audit found the online completion paths would:

  • Trust the price. Totals were recomputed from item.price and only checked for internal arithmetic — the catalog price was never compared. A stale-catalog or tampered terminal could sell at any price, with no record.
  • Trust the retry. No idempotency token; every submit minted a fresh transaction number. A network retry after a committed sale created a second sale, a second stock decrement, and — because the drawer-cash call omitted the key the void path uses — double cash.
  • Trust the clock. occurredAt and the operating-day window came straight from the terminal. A skewed clock booked sales into the wrong operating day and skewed closeout variance.
  • Trust the units. POS money is unconstrained v.number() and, unlike the online-store tables, was never migrated to pesewas — it still stores cedis.

The offline projection path already re-priced and deduped. Phase 2 treats it as the template and makes the online paths inherit the same discipline.

The mental model

Client proposes, server decides. For every money-relevant value the terminal sends, the server derives the authoritative version from its own state and treats the client value as a proposal it must validate:

ValueClient proposesServer derives / enforces
Line priceitem.priceimportedPrice ?? netPrice ?? price; match, or manager-audited override, or reject
Sale identityidempotency tokendedup the mint before minting; return the original on replay
Business time / dayoccurredAt, operatingDateclamp vs server time; derive operating date from timezone authority
Money unita numberinteger pesewas (migration now; validator flip later)

Before / after — the completion path

Before

completeTransaction(items, client totals)
  totals = sum(client item.price)   // client-trusted
  if totals internally consistent → OK
  mint new txn number (always)
  record drawer cash (no idem key)
  store item.price verbatim

After

completeTransaction(items, token?, proof?)
  if token && existing → return original  // U8
  derive basis per line from catalog       // U7
    match → record basis
    deviate + manager proof → record + audit
    deviate + no proof → approval_required
  mint once, keyed by token                // U8
  record drawer cash guarded by token      // U8
  reporting *Minor = derived price         // U7

Ingest (U9): the terminal occurredAt/payload are left untouched (they drive the isSameLocalEvent retry-match); the clamped business time and server operating date are stored in additive record fields computed once at first ingest.

What changed — and what deliberately did not

Changed

  • Online completion now re-prices, dedups, and audits overrides on both paths.
  • Drawer cash is idempotent at every sale call site (incl. completeSession).
  • Ingest stores server-derived business time + operating date; store-day opens under the server date.
  • POS money migration mechanism + completion marker exist and are tested.

Intentionally unchanged

  • The schema money validators stay v.number() — flipping them before every legacy cedis row is migrated would reject those rows. Gated to V26-1066.
  • The event-sourcing core, sync-contract event shapes, and the isSameLocalEvent equality (occurredAt + payload) are untouched.
  • Offline-first posture: no online-only fallback sell path was added.
  • Pending-checkout lines keep their operator-entered price (own review workflow); exempt from catalog re-pricing.

Next-time / operational guidance

  • Hardening an online path that has an offline twin? Mirror the offline projector's basis, dedup key, and conflict discipline instead of re-deriving.
  • Adding server-derived attribution to an event with an idempotency/equality check? Use additive fields and confirm what the equality function compares first.
  • Running the U10 cutover (V26-1066): convert writers to pesewas → deploy + run migratePosAmountsToPesewas until posAmountMigrationStatus reports complete for every table → only then flip the validators.
  • Touching a public Convex module with return validators? Touch a sibling assertConformsToExportedReturns proof in the same change, or lint:convex:changed blocks the push.
