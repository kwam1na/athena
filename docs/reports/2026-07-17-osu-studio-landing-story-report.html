Marketing landing page that walks a store's real workday using the product's own components as inert exhibits, backed by a re-storied shared demo store.

Executive Summary

This branch delivers two intertwined things. First, it re-stories the shared demo store from a single-product grocery seed into "Osu Studio - Atelier", an artisanal handmade-goods shop with an 8-SKU catalog, recast staff, and a pickup order. Second, it rewrites the public marketing landing page as a chronological walk through one operational day at that store - opening handoff, daily operations, a POS sale that completes offline, the sync bridge, cash-controls closeout, and end-of-day review.

The defining design decision: the landing page does not use hand-drawn mockups. Each scene renders the product's real presentational components (the same Cash Controls dashboard, store-pulse view, register/session views, and daily-close report the owner uses) fed by fixture data and wrapped in inert exhibit chrome. The store the page describes is the exact store the "Try the demo" button provisions, because both read one shared facts module.

This report is a delivery candidate: the branch is committed locally, the dev Convex backend has been deployed, but the PR is not yet merged and production is not yet deployed.

Why It Mattered

A marketing page built from screenshots or bespoke mock UI drifts from the product and ages with every change. Worse, if the demo store the page pitches differs from the store the demo actually opens, the first click contradicts the pitch.

Athena's answer is to make the marketing surface a thin presentation of real product components and a single shared story, so the page cannot silently diverge from either the product UI or the provisioned demo.

Mental Model

Think of each scene as a museum exhibit: a real artifact (the product component) behind glass (an inert, pointer-events-disabled wrapper), with a placard (the figure's aria-label) and a spotlight (entrance animation). The artifact is genuine; the glass stops visitors from operating it; the placard describes it for anyone who can't see the spotlight.

  • Artifact = real presentational component (CashControlsDashboardContent, StorePulseSummaryView, DailyCloseReadOnlyReport, register/session views).
  • Glass = WorkspaceExhibit (inert + pointer-events-none + select-none).
  • Frame = WorkspaceFrame / AppShellExhibit (browser + app-shell chrome, optional zoom).
  • Data = fixture snapshots in demoDayFixtures.ts, derived from shared story facts.
  • Spotlight = animejs entrance driven by IntersectionObserver, with a static final-frame fallback.

Before vs After

Before: a 577-line hand-built AthenaLandingPage component drew marketing UI by hand, and the demo store was a single-product grocery seed unrelated to the page copy.

After: AthenaLandingPage is deleted; the page is composed of day-ordered scenes rendering real product components from fixtures; the demo store is "Osu Studio - Atelier" whose identity, staff, and catalog live in shared/sharedDemoStory.ts and are consumed by both the scenes and the demo provisioner.

Layer-By-Layer Mechanics

Data flows from one shared facts module out to two consumers - the provisioned backend demo and the static landing scenes - so they cannot disagree.

  • shared/sharedDemoStory.ts + convex/sharedDemo/config.ts hold identity, staff, the Handmade category/subcategories, 8 SKUs, and SHARED_DEMO_BASELINE_VERSION = 11.
  • convex/sharedDemo/provision.ts migrates an existing demo from baseline 10 to 11: patches identity/staff/terminal/opening message, reconciles the catalog by wholesale delete+reinsert (SKU inventory fields are owned by the reporting pipeline, so in-place patching would trip the direct-write boundary), repoints DEMO-ORDER-001 to the Black Soap Bar, and rebuilds sessions/opening.
  • src/components/landing/story/* holds the scenes, SceneChrome (WorkspaceFrame / WorkspaceExhibit / AppShellExhibit), useSceneAnimation, and useForcedLightTheme.
  • demoDayFixtures.ts feeds each real component; demoDay.ts + demoDay.test.ts enforce that fixture numbers reconcile page-wide.
  • The demo CTA funnel is threaded client to backend: landingFunnelClient.ts emits demo_cta / page_view -> convex/http/domains/core/routes/landingFunnelEvents.ts ingress -> convex/marketing/landingFunnelEvents.ts mutation -> convex/schemas/marketing/landingFunnelEvent.ts.

Failure Boundaries

  • No live store on a public route: exhibits render from fixtures with isLoading=false; page-boundary hooks (useGetActiveStore, use-navigate-back) are stubbed so components never fetch or navigate.
  • No interactivity leaks: WorkspaceExhibit sets inert + pointer-events-none, removing embedded links/controls from focus order and the a11y tree.
  • Reduced motion / no JS / jsdom: scenes render their final frames statically; animation only adds entrance motion and is never required to read content.
  • Theme leakage: useForcedLightTheme pins the marketing palette instead of leaking app theme onto the public route.
  • Migration safety: provision throws on an unknown-future baseline (>11), and reporting-position reconcile throws if a SKU has more than one inventory position (ambiguous).

Validation

landing.test.tsx renders the whole page with no IntersectionObserver and asserts the static final frames, the day-ordered headings, the traced GH385 sale appearing at least twice, that demo is the sole CTA (3 links), and that the walkthrough/sign-in links are absent.

demoDay.test.ts asserts the fixture numbers reconcile across scenes; provision.test.ts, public.test.ts, landingFunnelClient.test.ts, and landingFunnelEvents.test.ts cover the demo migration and funnel plumbing.

Local delivery gates run before push: landed-report:check, delivery:documentation-check, and harness:inferential-review. CI re-runs everything authoritatively.

Next-Time Guidance

  • To show a product screen on a marketing/onboarding surface, wrap the real presentational component in WorkspaceExhibit/AppShellExhibit and feed it fixtures - do not fork the UI into a mock.
  • Keep demo-store facts only in shared/sharedDemoStory.ts; hard-coded catalog/identity strings in a scene are a bug waiting to diverge.
  • When changing the demo story, bump SHARED_DEMO_BASELINE_VERSION and add the migration branch; reconcile SKU-owned tables by wholesale replace, not in-place patch.
  • Removing a marketing link is a page-copy change: update landing.test.tsx, leave the route intact.
