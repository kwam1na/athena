POS Phase 1 — Liveness & Convergence Hardening

Four coordinated fixes at the seams around the local-first POS sync core: total projection, a bounded ledger, a truthful sync status, and a review resolution that actually converges with the server.

Branch: fix/pos-phase1-liveness-convergence Linear: V26-1058 / V26-1059 / V26-1060 / V26-1061 Follow-up: V26-1065 Status: delivery candidate Finish line: open mergeable PR No merge No production deploy

Executive Summary

A nine-pass audit found the event-sourced POS core sound (single Convex mutation, exactly-once ingest, durable-commit-before-ack, monotonic per-cursor ordering) but exposed clustered failures in the seams around it. Phase 1 closes four of them in one integration PR.

  • U3 (V26-1058): a sale whose store organization cannot be resolved wrote the transaction row and then threw, aborting the batch and wedging the register's sync stream on every retry. The check now runs before the first write and returns a review conflict instead.
  • U4 (V26-1059): the IndexedDB ledger grew unbounded until quota_exceeded blocked selling. An evidence-gated purge now reclaims settled, unreferenced, past-boundary events at a safe idle boundary.
  • U5 (V26-1060): the sync chip masked an outstanding review as "ready" and a held successor of a stuck review looped silently. The chip is now truthful and the held gap escalates.
  • U6 (V26-1061): clearing a review locally never told the server, so terminal and server diverged forever. Local resolve now round-trips through an authorized mutation and only marks settled after the server acks.

Mental Model: The Seam Invariant

Convex is the durable source of truth; IndexedDB is the terminal's operational log that drains into it. The event-sourcing core is not the problem — the validation, purge, status, and convergence seams around it are. The single cross-cutting rule that ties three of the four fixes together is:

A locally-cleared review is not "settled" until the server confirms it.

That rule is encoded once as a serverConfirmedAt stamp (and its derived serverConfirmedResolution fact) and honored identically by the status chip, the sync high-water mark, and the ledger purge — so no surface can quietly disagree with another. A destructive delete is the worst possible place to use a laxer definition of "settled" than the chip shows.

append -> [single Convex ingest mutation, NO partial rollback]
   |-- data-shaped failure --> validate BEFORE first write -> conflicted (U3)
   |-- genuine infra error  --> throw -> whole mutation aborts (retry)
settled + past boundary + server-confirmed -> purgeable (U4)
locally cleared review -----> needs_review until server ack (U5/U6)

Before vs After

U3 — projection totality

  • Before: validateSaleCompletedInputs checked the store organization only for service sales; a merchandise sale in an org-less store reached createTransaction, wrote the posTransaction row, then threw — aborting the batch (Convex has no partial rollback) and wedging the cursor forever.
  • After: the org check runs before the first ctx.db write for every sale and returns a needs_review conflict — zero committed rows for the poison event, the cursor advances, surrounding events commit. Genuine infra errors still abort.

U4 — bounded ledger

  • Before: assessPosLocalLedgerRetention was dead code; the ledger only grew.
  • After: an evidence-gated purgeSettledLedgerEvents deletes only settled, unreferenced, past-boundary events, refusing under active cashier presence and never touching an unsynced, drawer-authority-referenced, current-session, or server-unconfirmed event. Count surfaced, never a silent bulk delete.

U5 — truthful status + escalation

  • Before: an operable review rendered as success "ready"; an unconfirmed locally_resolved read as synced; a held successor looped silently in backoff.
  • After: the chip shows an actionable "needs review" (register still operable); an unconfirmed local resolution reads as outstanding; the scheduler surfaces heldWithoutProgress and the runtime drives one review-inclusive escalation drain.

U6 — convergence round-trip

  • Before: clearLocalReviewEvents marked locally_resolved with no server call; the posLocalSyncConflict stayed open forever.
  • After: an authorized resolveLocalSyncReview mutation resolves the server conflict (idempotent, attributable); the terminal marks locally_resolved with serverConfirmedAt only after ack, else stays needs_review.

Next-Time Guidance

  • Never catch-and-continue inside a Convex mutation — there is no partial rollback. Convert a post-write data-shaped throw into pre-write validation that returns conflicted, and assert zero committed rows for the poison event.
  • When a local action "resolves" server-owned state, round-trip it and mark settled only after ack. Encode "settled" once and reuse it across every surface (status, high-water, purge) so a destructive path can never use a laxer definition than the UI.
  • Never mask a review or needs-reconciliation state as success in an operator UI.
  • Residual post-write throws in the service-line / payment / posSession-mapping paths are tracked in V26-1065 for the same validate-before-write treatment.

Quiz: Pass Required

You must score at least 8 out of 10 to pass.

1. Why can't U3 fix the wedge by wrapping the projector in try/catch and continuing?
A single Convex mutation is one OCC transaction with no savepoints; a caught post-write throw does not undo prior writes, so it would commit a corrupted partial sale. The fix validates before the first write.
2. What is the actual reachable post-write data condition U3 moves before the first write?
createTransaction wrote the posTransaction row and then threw "organization could not be resolved"; validateSaleCompletedInputs now checks it for every sale before any write.
3. On a data-shaped failure, what must the ingest batch do?
Totality means the poison event becomes a zero-write conflict while good events still commit and the cursor advances — liveness preserved. A genuine infra error still aborts.
4. Which condition makes the U4 purge refuse the whole operation?
The purge reuses the cashier-presence safety gate but deliberately NOT the whole-DB events>0 or blanket authority>0 refusals, which would always block a selective purge.
5. Why must the purge never delete an unconfirmed locally_resolved event?
This was the review-caught data-loss bug: a locally-cleared review the server has not confirmed has not converged; deleting it loses the record needed to reconcile. The classifier now returns unsettled_sync for it.
6. What does the U5 chip now render for an operable outstanding review?
The old code masked an operable review as success "ready"; it now surfaces the review truthfully and actionably while keeping the register operable.
7. What single marker distinguishes a converged local resolution from an unconverged one?
serverConfirmedAt (and its derived serverConfirmedResolution fact) is the one marker honored by the chip, the high-water mark, and the purge — so no surface diverges.
8. How does the scheduler stop a held successor of a needs_review precursor from looping silently?
A no-forward-progress held batch is surfaced distinctly (not a generic transient failure) and escalated once with a review-inclusive drain so the precursor is re-projected; if it still needs manager action, the truthful chip shows it.
9. What authorization gates the U6 resolveLocalSyncReview mutation?
requireOrganizationMemberRoleWithCtx(["full_admin","pos_only"]) is a single static role set checked after the store lookup and before the resolution work — no runtime role branch.
10. On a U6 server-resolution failure, what happens to the events being cleared?
The clear round-trips first; only after the server acks does it mark locally_resolved with serverConfirmedAt. On failure it returns server_resolution_unavailable and leaves the events needs_review.
