Athena Primary Color Consolidation

The workflow blue is now the single authoritative app primary

Delivery candidateLocal main @ 300376b7Root aligned: completed locallyDeploy: not applicableNo compatibility aliases

Executive Summary

Athena now uses the established workflow blue as its canonical primary color in light, dark, and classic-dark themes. The primary token family also owns the existing soft and border treatments used for selected and contextual surfaces.

The former signal, action-workflow, and action-commit color families and button variants were removed rather than aliased. Existing surfaces across POS, operations, procurement, authentication, landing, navigation, and Storybook now consume the primary contract directly.

POS settings uses the tonal primary-soft treatment for its repeated section save actions, while links to Store Hours and terminal health use neutral outline buttons with an up-right arrow.

Why This Mattered

The product already treated workflow blue as the dominant interaction color, but the token system still described pink primary and commit roles plus separate workflow and signal roles. That made equivalent actions appear semantically different and encouraged new code to extend an obsolete distinction.

Mental Model

Primary is one family with two levels of emphasis. Solid primary is the default emphasized action and interactive accent. Primary soft is the quieter surface treatment for a selected control or contextual action area, paired with primary-border and primary text.

Status colors remain state-specific. Transaction-signal, success, warning, danger, comparison, and neutral action tokens are not substitutes for primary.

Solid: bg-primary text-primary-foreground
Soft: bg-primary-soft border-primary-border text-primary

Before and After

  • Before: primary was pink while workflow blue, signal, and commit each had overlapping action contracts.
  • After: primary is workflow blue, with primary-soft and primary-border preserving the established tonal treatment.
  • Before: Button exposed workflow, workflow-soft, and commit-soft variants.
  • After: Button defaults to primary and exposes only primary-soft for the tonal form.
  • Before: chart 1 inherited the old pink primary hierarchy.
  • After: chart 1 is blue and chart 2 retains pink as a data-only secondary hue.

Layer-by-Layer Mechanics

Theme variables establish the palette in every supported theme. Tailwind exposes primary soft and border subkeys and no longer exposes signal or workflow/commit action colors. UI primitives then inherit primary by default, while application surfaces use the same semantic classes directly.

Stories, tests, and agent design guidance were updated in the same change so examples and future implementation instructions cannot reintroduce the retired vocabulary.

Failure Boundaries and Intentional Non-Changes

  • No backward-compatibility aliases exist; an old workflow or commit variant should fail typechecking instead of silently working.
  • Primary-soft keeps its existing visual values and use cases; only its semantic ownership changed.
  • Transaction-signal remains intact because it describes a transaction-specific status surface, not the app action hierarchy.
  • Success, warning, danger, comparison, and neutral-action semantics were not collapsed into primary.
  • No production deployment, GitHub mutation, Linear mutation, commit, or staging action was requested or performed.

Next-Time Guidance

  • Start at the semantic contract: decide whether a color represents action, status, comparison, or data before editing component classes.
  • For a full token rename, sweep variables, Tailwind mappings, arbitrary CSS values, variant types, stories, tests, and agent docs.
  • Prefer removal over aliases when the old vocabulary must not remain supported.
  • Keep negative token assertions so retired namespaces cannot return unnoticed.
