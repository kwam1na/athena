Executive Summary

This branch turns the shared demo into a two-lane operational story: current-day demo state remains live and mutable, while historical operations, opening, closeout, and POS transaction history render from deterministic client fixtures.

The backend baseline moves to version 19 and preserves operational continuity while promoting staff identity, credentials, opening messages, product image versions, and Opening Handoff event links. The frontend adds shared-demo-aware fixture views and tests so visitors can inspect a complete history without creating backend transaction rows or enabling mutation controls.

Problem And Context

The previous shared demo leaned on live baseline state and current-day operational data. That worked for a sandbox, but it left historical workspaces thin or inconsistent: Daily Operations, Daily Opening, EOD Review, and POS transaction history could not tell the same complete store story across dates.

Prior fixture work established that prop-driven content fixtures are useful for screenshots, but full app chrome and live exploration still need real authenticated store context. This delivery applies that lesson inside the product surface instead of treating every demo scene as a screenshot-only artifact.

Mental Model

Think of the shared demo as a store with a live front desk and a curated archive. The live front desk is today: visitors can explore real connected state and existing operational workflows. The archive is history: deterministic fixtures fill prior days with coherent operations, closeout, opening, and transaction records without writing those records into Convex.

Server provisioning owns durable continuity. Client fixtures own historical presentation. The boundary keeps re-provisioning safe and makes historical pages predictable.

Before And After

  • Before: shared-demo baseline migrations covered narrower credential and image changes; historical operations surfaces were not a coherent reusable story.
  • After: baseline version 19 preserves continuity from versions 16-18, promotes staff story rows, reconciles product image paths, and links Opening Handoff through the daily_opening_acknowledged event shape.
  • Before: Daily Operations, Opening Handoff, EOD Review, and POS transaction history mostly depended on live data availability.
  • After: shared-demo historical dates route to fixture-backed read models, while current-day views keep live Convex behavior where needed.
  • Before: fixture transactions could not be safely opened as detail pages without looking like live mutable sales.
  • After: fixture transaction IDs are recognized, live queries are skipped, and mutation affordances such as update, void, and receipt printing are suppressed.

Layer Breakdown

Convex provisioning advances the durable baseline and migrates existing demo stores without resetting operational state. The fixture layer creates 21 days of shared-demo operational history. The operations and POS components decide at route time whether they should use live data, fixture data, or a controlled blend.

The list/detail boundary matters most in POS. The transaction list can merge historical fixtures with live current-day rows, but a fixture detail route must be read-only and must not call live transaction mutations.

Failure Boundaries

  • Fixture history is capped to the prior 21 days and excludes today for opening and close snapshots.
  • Current-day Daily Operations still queries live Convex but receives client week analytics to make the demo chart feel populated.
  • Fixture transaction IDs skip getTransactionById and render from client fixtures.
  • Fixture transaction details do not expose void, correction, receipt printing, or customer mutation controls.
  • Normal non-demo operations and POS paths still use live Convex queries and existing mutation policies.
  • Date/timezone behavior remains the main residual risk because many fixtures are relative to the local operating date.

Intentionally Not Changed

  • The fixture layer does not create backend POS transaction rows.
  • Current-day shared-demo operations remain live and mutable according to existing workflow rules.
  • Non-demo operations, approvals, voids, receipt printing, and close/reopen paths remain on live Convex behavior.
  • Local root alignment is deferred by explicit instruction.
  • Production deployment is skipped by explicit instruction; catalog image v2 asset readiness is therefore not claimed by this delivery report.

Next-Time Guidance

When extending shared-demo history, decide first whether the data belongs to durable server continuity or deterministic client presentation. Baseline version changes, staff credentials, product image versions, and operational events belong on the server. Historical read-only context can usually live in fixtures if it does not need to be mutated.

Keep timer-based fixture tests contained. One leaked fake timer in the transaction detail test caused the file to hang until the test restored real timers in a finally block.
