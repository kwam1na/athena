Shared demo polish: one safe story across seed data, policy, and UI

Delivery-candidate report for PR #693. This change tightens the Osu Studio demo so visitors can explore realistic order, refund, expense, and exit flows while external provider effects remain simulated.

Delivery candidate Branch codex/demo-experience-delivery-2 Head 8332f805 Base 3e622d80 Production deploy: not applicable Root alignment: pending merge

Executive summary

The branch turns a set of visible demo comments into one coherent cross-layer update. Demo entry and sign-out now return to the landing route, seeded story data looks like a real customer/order/expense day, and online-order actions are allowed for exploration through the shared demo policy.

The safety boundary stays explicit. Demo emails, refunds, and other provider-like effects are recorded as simulated decisions instead of live external calls. Browser validation and production deploys are intentionally outside this delivery.

Mental model

Think of the demo as a real synthetic tenant with a theater rail around it: the operator can touch the realistic controls, but the rails decide whether an action is internal, simulated, or denied.

Landing route
  └─ Try the demo in same tab
      └─ Shared demo workspace
          ├─ Story seed data: realistic order, customer, phone, expense, terminal state
          ├─ UI rules: exit/sign-out to landing, no pickup exception button in demo, single-item partial refund disabled
          └─ Policy rules: allowlisted online-order actions, simulated provider effects

Before and after

Before

The demo had placeholder customer details, an order that looked freshly created, live-looking provider affordances, and inconsistent exits that could send a visitor into login.

After

The seeded story has a realistic customer, phone number, four-hour-old order, already-sent order email, restored expense state, disabled demo heartbeats, and landing-route exits.
