Athena shared demo: real workflows with bounded authority

How one synthetic store can support shared writes without becoming an unrestricted administrator or a parallel demo application.

Executive summary

Athena now has a dev/QA-gated /demo entry that exchanges a short-lived opaque ticket for a per-admission demo principal. Every principal maps to the same synthetic owner, organization, and store, so the deployed design shares writes without sharing renewable authentication authority; cross-visitor visibility remains part of the pending browser proof.

The experience uses Athena's real POS, inventory, cash, fulfillment, staff-message, Daily Operations, and Reports surfaces. A thin owner-oriented layer labels the store as shared, warns against sensitive input, explains six workflows, and exposes a manual restore. Hourly and manual restore share one versioned store-scoped boundary.

Why this mattered

A normal full-admin account is too broad for a public demo: ordinary membership checks also permit identity, permissions, integrations, exports, payments, refunds, and destructive administration. A fake demo application would be safer but would not prove Athena's real operating workflows.

The design therefore separates authentication authority, business identity, the source-derived demo function inventory, provider effects, and store restoration. Demo auth users cannot resolve through Athena's ordinary email-based full-admin path; only explicitly classified adapters can map them to the shared business actor.

Mental model

Think of the shared store as one stage used by many short-lived badge holders. The badge identifies one admission, the store actor is shared, the capability usher permits only six real workflows, provider doors remain locked or simulated, and the stage manager restores the exact set after each hour.

/demo -> one-time ticket -> per-admission auth principal
       -> shared Athena owner/store
       -> real command + demo capability + restore epoch
       -> real store data
       -> provider effect simulated/denied
       -> hourly/manual baseline restore

Before and after

  • Before: no shared-demo principal, /demo entry, shared-store restore coordinator, or owner orientation existed on origin/main.
  • After: each admission has its own non-renewable server-clock authority while all admissions map to one synthetic Athena owner/store.
  • After: a recursive CI sensor detects exported public mutations/actions and provider-bearing modules for explicit review. Runtime denial remains owned by the demo actor, store-scoped adapters, foundation guards, and provider-effect guards.
  • After: provisioning creates coherent source and child rows, and restore replaces baseline rows, removes visitor-added rows, measures restored counts, and schedules existing Reports rematerialization.
  • After: staff communication is a genuine bounded internal store workflow, not a customer banner or navigation placeholder.

Failure boundaries

  • The feature requires the enable flag, development/QA environment, and an explicit deployment-ID allowlist; production and unknown deployments fail closed.
  • Tickets are opaque and hash-only, expire after one minute, are transactionally single-use, and mint/exchange paths are throttled.
  • Expired demo principals throw instead of falling through as normal actors. Cross-store authority comes from the principal, never client IDs.
  • Configured demo user, organization, and store IDs are protected by auth-independent foundation guards, so signing out cannot bypass destructive administration checks.
  • Catalog administration, global maintenance, store migration, cache invalidation, OTP, storage, media, payment, refund, export, and provider actions enforce ordinary non-demo authority or demo-effect denial before database changes, credential reads, uploads, deletes, or network dispatch.
  • Order fulfillment accepts only bounded status changes. Customer notifications are simulated for demo actors; normal actors retain live scheduling.
  • Every demo write reads the restore epoch in the same transaction. Verification or capacity failures escape the restore mutation so Convex rolls back every baseline change instead of committing a partial restore.
  • The 500-row per-table boundary is intentional fail-closed capacity. Supporting larger stores would require a cursor-based multi-transaction coordinator.

What intentionally did not change

  • No custom OIDC, JWKS, proof-of-possession, cookie proof, or browser key store was ported.
  • No per-visitor stores, generations, purge/retention system, isolated materialization, or generation-fenced POS storage was added.
  • Normal tenant authorization and application surfaces remain authoritative.
  • Reports stay read-only and claim only relationships Athena already materializes.
  • The isolated sandbox branch/worktree, production, and local root checkout remain untouched.

Next-time guidance

  • Add every new demo-writable public function and provider gateway to the classification inventory before exposing it.
  • Keep payload restrictions next to generic commands; capability permission alone is not enough for multi-purpose updates.
  • Register every new mutable source or child table in provisioning, baseline capture, measured restore, and tests.
  • Treat a new external effect as live for normal actors and explicitly simulated or denied for demo actors at the dispatch boundary.
  • Use two browser contexts during QA to prove shared visibility, restore recovery, and cross-visitor communication behavior.

Quiz: Pass Required

You must score at least 8 out of 10 to pass.

1. Why should a visitor never enter real personal, payment, or credential data?
Correct: the synthetic store is shared. Visitors must use synthetic content because writes may be visible until the shared baseline is restored.
2. What identifies the authoritative demo store?
Correct: the durable server-owned principal. Browser IDs, URL slugs, and navigation history are not authorization evidence.
3. What should another admitted visitor observe after you post a staff message?
Correct: staff communication is a real shared-store write, not a per-visitor session artifact or customer-facing effect.
4. What happens to an order notification triggered by demo fulfillment?
Correct: it is simulated without provider dispatch. The fulfillment state remains real, but a demo visitor cannot contact a customer.
5. Why must a demo mutation read the restore epoch in its own transaction?
Correct: the same-transaction restore-state read prevents a stale write from crossing a restore; it is unrelated to auth refresh, CSS, or chart generation.
6. What is the scope of the manual Restore demo action?
Correct: manual restore resets the shared synthetic store for every active visitor, which is why the UI asks for confirmation.
7. What happens if a restored table exceeds the current 500-row atomic budget?
Correct: the bounded read throws, so Convex rolls back the entire restore mutation. Rows are never silently ignored and a partial baseline is not committed.
8. What three checks enable the demo runtime?
Correct: all three checks must match. A copied feature flag cannot enable production or an unknown deployment ID.
9. When should Reports reflect a demo write?
Correct: Reports remain read-only and truthful to Athena's existing projections. The demo does not invent unsupported metrics.
10. When adding a new demo-writable feature, what is the complete extension posture?
Correct: classify function and effects, enforce payload/store/epoch boundaries, and register mutable tables. UI-only links or full-admin grants are insufficient.
