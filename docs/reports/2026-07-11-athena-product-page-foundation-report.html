Mental Model: Two Doors, One Durable Rail

Think of Athena as having two doors. The public door explains the product and accepts a walkthrough request. The app door restores authenticated operators to their organization. Behind the public door, one durable rail owns validation, admission budgets, idempotency, notification, privacy, retention, and measurement.

Public / -> walkthrough CTA -> bounded Hono ingress -> internal Convex acceptance
Authenticated /app -> login or organization dispatch -> existing operational app
Durable acceptance -> notification attempt (failure never erases the accepted request)

Before and After

  • Before: / inherited operational chrome and app-entry assumptions. After: / is anonymous, /app owns authenticated dispatch, and /landing redirects to /.
  • Before: no owned lead intake or acquisition funnel. After: two bounded Hono endpoints accept walkthrough requests and coarse anonymous events.
  • Before: no privacy lifecycle for acquisition data. After: email-control challenges gate export/redaction, with bounded raw and aggregate retention.
  • Before: crawler HTML had only a generic title. After: title, description, canonical, and Open Graph text are present before JavaScript runs.
  • Unchanged: protected POS, reporting, procurement, storefront, and organization APIs remain behind existing authorization boundaries.

Failure Boundaries

  • Wrong origin, media type, malformed JSON, or oversized streamed body receives a generic rejection; missing configuration returns a generic temporary-unavailable response.
  • Same-key retries remain idempotent. Newly observed equivalent keys consume bounded replay-alias capacity; exhaustion returns recoverable temporary unavailability without persistence.
  • Canonical hashing and persistence share control-character normalization, preventing digest drift.
  • Notification delivery uses leases, budgets, retry/recovery, and outcome-unknown handling. A notification failure never rolls back durable acceptance.
  • Funnel failures never block the prospect. Browser events contain only an allowlisted event, coarse device class, and unknown source. Raw events expire after 30 days; daily aggregates expire after 395 days.
  • Privacy export/redaction requires a 24-hour challenge sent to the stored email. If that address is unavailable, stop this proof path and escalate to a separately approved identity review; the system does not bypass proof of control. Request PII is redacted after the 180-day lifecycle and HMAC tombstones expire after 365 days.
  • Compile-time VITE_WALKTHROUGH_PRIVACY_CONTACT and runtime WALKTHROUGH_PRIVACY_CONTACT must match; empty or invalid configuration keeps intake closed.
