Store service principals: identity without implicit authority

A delivery-candidate guide to Athena's consumer-neutral machine identity foundation and its first POS implementation.

Executive summary

Athena's POS application authentication was anchored to one shared, human-shaped account even though recovery credentials and terminals were store-scoped. This candidate introduces the replacement architecture and application path through first-class store service principals: durable non-human identities that begin with no authority. Store migration and legacy retirement remain gated future operations.

Consumers grant authority explicitly from a closed catalog. POS is the first adapter and owns pos.application, recovery credentials, terminal proof, application bindings, reconnect behavior, and offline receipts. The generic foundation contains no POS semantics and can support future store-owned systems without repeating the shared-account pattern.

  • Registered browsers say Sign in to the terminal name; recovery no longer asks for or exposes a synthetic POS email.
  • Store scope comes from verified terminal registration, not a route, URL, email, or client-selected store.
  • Online revocation is checked on the next protected request; bounded offline continuation honors only an already-issued signed lease.
  • This is a delivery candidate. PR merge, root alignment, and browser validation are not claimed here.

The authority model

Identity, capability, transport, exact application session, consumer credential, device proof, and human operational authority are independent lanes. A valid lane never implies another.

The foundation creates one store-scoped principal and one immutable neutral Convex Auth transport binding. POS explicitly grants pos.application. A machine receives no Athena user, organization membership, email identity, or human role. Invalid service state cannot fall through to human membership or shared-demo authorization.

principal exists
  + current closed-catalog grant
  + exact Auth session and service session
  + current store and credential revisions
  + current terminal proof and binding
  = POS application authority

Staff, manager, drawer, register, and command authority remain additional gates.

Exact-session recovery

Recovery runs in an isolated, empty Auth namespace. The server validates the registered terminal, proof, store, principal, grant, credential, and revisions, then prepares an exchange for one exact Auth user/session pair. The credentials provider returns both identifiers, and only that authenticated exact session can activate the exchange.

Activation creates the service session and terminal application binding, superseding only the predecessor for the same principal, terminal, and consumer lineage. The browser promotes isolated Auth storage only after activation and exact-session verification. Its journal carries routing metadata, never authority or secrets.

  • An old or parallel Auth session for the same transport user cannot inherit a newly activated POS session.
  • Ambiguous failures preserve resumable pending state.
  • Authorized abort and bounded cleanup remove only prepared artifacts and do not disturb active lineages.

Current-state enforcement

Every protected POS business boundary reloads the exact Auth session, service session, immutable transport binding, current store, principal lifecycle and revision, pos.application grant and revision, recovery credential status and revision, terminal lifecycle and proof revisions, and terminal-specific binding. Resource IDs are clamped to the actor's store.

A source-derived inventory classifies callable POS surfaces as human administration, POS business operation, device control, or intentionally public. Unknown endpoints fail the sensor, which prevents new server functions from silently skipping the service-principal guard.

Terminal disconnect and reconnect experience

Disconnecting one checkout station revokes only that terminal's application binding and advances its lifecycle and proof revisions. The terminal row, cloud identity, local ledger, sync cursor, queued evidence, and sibling-terminal authority remain intact.

A revoked browser that still proves possession of the old terminal proof may obtain a five-minute opaque reconnect intent. The intent grants no POS authority and cannot validate the recovery code. A same-store full administrator must sign in on the affected browser, consume the single-use intent, reactivate the same terminal row, and rotate proof. Fresh POS recovery is still required afterward.

  • Online access fails on the next protected server request.
  • A disconnected terminal may continue locally only until its existing offline lease expires.
  • If terminal proof is lost, same-row reconnect is unavailable; the browser must be enrolled as a new terminal.

Local-first operation remains local-first

Server-side current-state reads happen only when the POS calls the server. Cashier commands remain IndexedDB-first; the design does not move the register's command durability boundary into Convex.

The server can issue a canonical ECDSA P-256 signed receipt scoped to the exact store, terminal, principal, service session, binding, capability, credential, and revisions. The browser can verify a finite lease but cannot mint, extend, refresh, or rescope it offline.

  • Valid within-lease evidence can ingest normally.
  • Missing, expired, or ambiguous evidence is durably routed to needs_review.
  • Forged, revoked-key, or copied-scope evidence is rejected.
  • Infrastructure failure and nondurable outcomes do not advance the local sync cursor.

Additive migration and retirement gates

Migration is bounded, preview-first, cursor-driven, fingerprinted, conflict-aware, and idempotent per store. It preserves terminal IDs, proofs, fingerprints, revisions, ledgers, and cursors. Neutral blank Auth users are created or reused; the legacy shared POS user is explicitly forbidden as a transport binding.

Cutover is explicit: compatibility, then shadow, then enforced. An enforced store never falls back to the legacy account. Retirement remains closed until every active store is enforced, terminal evidence is complete, conflicts and plaintext or rotation-required credentials are absent, and the rollback window is closed.

Security and recovery-secret hardening

New and rotated recovery credentials use 600,000-iteration PBKDF2-SHA256 with per-credential salt and versioned server-held peppers. Plaintext is returned once and is not persisted. Verification has bounded throttling, and legacy rows that cannot be safely upgraded require administrator rotation.

Runtime configuration fails closed when pepper or signing material is absent or malformed. Audit records carry operational identifiers, states, revisions, counts, and correlations without codes, salts, peppers, proofs, tokens, receipts, or private keys.

Production rollout hard stops

This candidate deliberately performs no production deploy, migration, environment mutation, key rotation, capability change, terminal reconnect, legacy retirement, or schema narrowing.

The browser offline public-key registry is intentionally empty, so production offline authority remains fail-closed. Migration-mode transitions and global retirement are code-level contracts, not approved operator-callable mutations. Missing operational evidence is Hold, never Pass.

  • POS is the first and only delivered consumer of the generic foundation.
  • Prepared recovery cleanup exists as a bounded internal operation but is not scheduled.
  • Terminal proof remains bearer proof; asymmetric browser-held device keys are deferred hardening.
  • Legacy data and compatibility paths remain until a later approved cutover and retirement.

What intentionally did not change

  • POS is the first and only delivered consumer; the generic foundation remains consumer-neutral.
  • Human full-admin, staff, manager, drawer, register-session, and command checks remain independent authority gates.
  • IndexedDB remains the cashier command durability boundary; current-state server reads do not make cloud availability a prerequisite for within-lease local work.
  • Terminal possession still uses rotated bearer proof; asymmetric browser-held device keys remain deferred hardening.
  • No production deploy, migration, key rollout, mode transition, reconnect, retirement, or schema narrowing occurs in this delivery.

Next consumer and rollout checklist

  • Declare each consumer capability in its own frozen closed catalog; do not add consumer names or imports to the foundation.
  • Bind service authority to exact Auth sessions, preserve mutually exclusive actor resolution, and inventory every protected callable boundary.
  • Begin rollout with bounded preview and conflict census; do not apply against stale fingerprints or incomplete terminal evidence.
  • Ship reviewed recovery peppers, signing keys, and browser JWKs before enabling the corresponding runtime paths.
  • Add approved operator-callable mode-transition and retirement controls before shadow, enforcement, or narrowing work.
  • Treat missing evidence as Hold and follow docs/runbooks/store-service-principal-operations.md; never patch tables to bypass a gate.
