Executive Summary

This candidate makes register closeout reporting outcome-aware. A newly projected closeout with a variance keeps the existing manager-review alert path, while an exact cash match sends an administrative completion report.

The same batch improves daily-manager comparisons, operator-facing email hierarchy, Daily Operations and Cash Controls context, POS status copy, and the ability to export every relevant open-work SKU to a client-side PDF.

Mental Model: One Closeout, Two Outcomes

The POS sync projection owns the decision to schedule a notification after a register_closed event. The event is not an invitation to resend arbitrary email; it is a single closeout outcome with a specific, deduped notification owner.

  • Variance: a fresh pending variance review receives the existing approval-request notification marker and variance alert.
  • Exact match: the register session receives the local closeout-event marker before the completion report is scheduled.
  • Both paths only schedule in production; templates and previews remain safe to render outside production.
register_closed sync
  variance + pending review -> approval request marker -> variance alert
  exact match -> register session event marker -> match completion report

What Changed Across the Operator Flow

Daily Close now retains prior completed-day summary data so the daily manager report can compare sales, expenses, voids, cash metrics, reviewed variances, and payment totals instead of presenting an isolated day.

Daily Operations keeps open-register context visible, gives past, present, and future store days appropriate activity empty states, and separates automation and historical workflow context. Cash Controls and POS receive smaller status and responsive-layout corrections.

Open Work can build an on-demand PDF from all logical inventory-review SKU groups, rather than only the current visible page. The export carries product names, barcodes, category, price, and variants; it does not mutate inventory.

Failure Boundaries

A match-report payload rejects a missing, non-closed, non-exact, or count-less register session. The dedupe marker is written before scheduling, so repeating the same sync event cannot resend an alert.

That deliberate dedupe also means a downstream send failure is not retried by replaying the same event; recovery needs a new closeout event, reopen path, or operational intervention. The report preserves this limitation rather than overstating reliability.

The PDF export is browser-side. Font, barcode, or jsPDF failures surface the existing generic export failure toast; invalid barcode rendering falls back to a missing-barcode label.

Next-Time Guidance

  • Keep notification scheduling near the POS projection boundary and use one marker per notification owner.
  • Keep production send gating outside templates so previews stay safe and useful.
  • When a closeout field is shown in email and operations UI, prefer the same persisted closeout facts and shared review-reason formatting.
  • For an export that represents a logical work set, derive rows from the full loaded identifier set rather than the current pagination window.
