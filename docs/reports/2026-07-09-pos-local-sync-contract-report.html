POS Local Sync Contract: What Changed and Why

This report explains PR #637, which made Athena's POS local sync event contract single-sourced across the browser replay path, Convex public upload validation, stored event literals, and ingestion.

Merged PR #637 Merge commit 4e4312ac Linear V26-966 No production deploy Root main aligned

Executive Summary

Before this change, the POS sync contract existed in too many places. The browser had local event names like transaction.completed, Convex had upload event names like sale_completed, public validators had their own event list, and ingestion had its own parser/type boundary. Those copies could drift.

This change makes shared/posLocalSyncContract.ts the canonical table for event identity. Browser upload code and Convex validators now derive from that table. Ingestion also fails closed on unsupported event types before anything is persisted.

Operational result: adding or changing a POS local sync event now has one obvious starting point: the shared contract. The public upload boundary is stricter, while stored historical rows remain compatible.

The Problem In Plain English

POS local sync is Athena's offline-first replay system. A terminal can perform work locally, store events in the browser, and upload them to Convex later. That only works safely if every layer agrees on the event contract.

The risk was not just duplicate code. The deeper risk was semantic drift: one layer could think an event was uploadable, another layer could validate a different payload shape, and ingestion could dispatch it differently.

Browser layer Stores local dot event names such as register.opened and expense.completed.
Upload/API layer Receives sync event names such as register_opened and expense_recorded.
Ingestion layer Projects validated events into cloud register, transaction, closeout, pending item, and expense records.

The Core Intuition

Think of the local sync contract as the dictionary between the browser's local event language and Convex's cloud sync language. This PR made that dictionary explicit and shared.

1. Local POS Browser records dot-name events, for example transaction.completed.
2. Shared Contract The canonical table maps local name, sync name, scope, and uploadability.
3. Browser Upload Upload code asks the shared contract which events can be uploaded and what sync name to use.
4. Public Convex The public mutation validates event-specific payloads using validators derived from the same contract.
5. Ingestion Server logic applies semantic business rules and rejects unknown event types before persistence.
Important distinction: the shared contract owns event identity and payload type boundaries. Ingestion still owns business semantics such as register/session checks, projections, conflicts, staff proof hashing, and retry behavior.

Canonical Event Table

The canonical table now lives in packages/athena-webapp/shared/posLocalSyncContract.ts. It records the sync event name, browser local event name, sync scope, and whether the browser replay path may upload that event.

Sync event Local event Scope Browser uploadable? Why it matters
register_opened register.opened POS Yes Creates or resumes the cloud register session from a local opening.
pending_checkout_item_defined pending_checkout_item.defined POS Yes Uploads offline-defined pending checkout item data.
sale_completed transaction.completed POS Yes Projects completed local sale transactions into cloud records.
register_closed register.closeout_started POS Yes Starts closeout projection and variance review flow.
register_reopened register.reopened POS No Server-supported, but intentionally not uploaded from normal browser replay.
sale_cleared cart.cleared POS Yes Records local sale clear events while avoiding later completed-sale conflicts.
expense_recorded expense.completed Expense Yes Uploads drawerless expense sync with a separate expense cursor identity.

What Changed By Layer

1. Shared contract owns event identity

POS_LOCAL_SYNC_EVENT_CONTRACT is now the event identity table. POS_LOCAL_SYNC_EVENT_TYPES is derived from that table rather than maintained as a separate hand-written list. 

export const POS_LOCAL_SYNC_EVENT_TYPES = eventTypesFromContract(
  POS_LOCAL_SYNC_EVENT_CONTRACT,
);

This removes one source of drift: the event literal list cannot accidentally diverge from the event metadata table.

2. Browser upload uses shared helpers

Browser upload conversion no longer owns an independent dot-to-underscore map. When it needs the cloud sync name for a local event, it asks the shared contract. 

eventType: syncEventTypeForLocalEvent(event.type, "sale_completed")

The fallback argument preserves the existing behavior if a path is structurally known, but the source of truth is now the shared contract. Uploadability also comes from the shared helper canUploadPosLocalSyncLocalEventType.

3. Convex public validators are derived from the contract

The public upload mutation now imports posLocalSyncUploadEventValidator from a dedicated schema module. That module maps over POS_LOCAL_SYNC_EVENT_CONTRACT and builds event-specific Convex validators. 

export const posLocalSyncUploadEventValidator = unionFromValidators(
  POS_LOCAL_SYNC_EVENT_CONTRACT.map((contract) => {
    const payload = posLocalSyncPayloadValidators[contract.eventType];
    return v.object({
      eventType: v.literal(contract.eventType),
      payload,
      ...
    });
  }),
);

This is stricter than a generic payload: record public boundary. A sale_completed event must carry a sale payload, not a register payload.

4. Stored schema compatibility is preserved

The stored posLocalSyncEvent schema still keeps payloads broad enough for historical rows and review surfaces. The change tightens event/status literals and public upload validation, not the stored payload record shape.

5. Ingestion now rejects unsupported event types before persistence

The ingestion service now checks the batch for unsupported event types before it starts projection or creates sync event records. The check is based on the found event object, not truthiness of the event type, so empty strings and missing discriminants cannot bypass it. 

const unsupportedEvent = batch.events.find(
  (event) => !isPosLocalSyncEventType(event.eventType),
);
if (unsupportedEvent !== undefined) {
  return userError({
    code: "validation_failed",
    message: `Unsupported POS sync event type: ${String(unsupportedEvent.eventType)}.`,
  });
}

The Most Important Bug Avoided

The highest-risk bug was parser fallthrough. An unknown event type could reach the parser and land in the final branch, which behaved like register_reopened. That is dangerous because an event the system does not understand could be interpreted as a valid register lifecycle event.

Reviewer catch: the first unknown-event guard checked the unsupported value by truthiness. That would miss eventType: "" or a missing eventType. The final fix checks whether an unsupported event object was found.

The regression test now covers three unsupported cases: "mystery_sync_event", "", and undefined. All must return validation_failed, and none may create a sync event or register session.

What Did Not Change

  • The public sync mutation still performs auth, role, terminal/store, active terminal, and sync secret checks before ingestion.
  • Business rule validation still belongs to server ingestion and projection code.
  • Browser payload conversion remains best-effort and normalizing. It did not become a new hard-stop for local POS operation.
  • Stored historical payload compatibility remains broad.
  • register.reopened remains intentionally non-uploadable from the normal browser replay path.
  • Production deploy was not performed for this change.

How To Modify This Area Next Time

  1. Start at shared/posLocalSyncContract.ts. Add or update the event identity there first.
  2. Decide whether the browser replay path should upload the event. Do not confuse server support with browser uploadability.
  3. Add or update the event-specific Convex public payload validator.
  4. Update browser conversion if this event is browser-uploaded.
  5. Keep ingestion semantic validation server-authoritative.
  6. Add negative tests for unsupported or mismatched event payloads.
  7. Run focused tests, typecheck, Graphify if code changes, and pr:athena.
