How Athena removes recurring read amplification across shell, Operations, POS runtime, storefront, and offline catalog refresh

Executive summary

Athena had consumed 29.4 GB of a 50 GB Convex Database I/O allocation while calls and compute remained low. The implementation attacks the repeated-read multipliers: global subscriptions returning oversized payloads, eager detail queries, broad recovery scans, duplicate runtime publishers, homepage hydrate-before-bound behavior, and concurrent one-shot catalog refreshes.

U1-U7 ship as one reviewed Convex plus Athena-client release. Results are observed through matching Convex Usage dashboard windows and bounded Convex Insights snapshots. No in-app monitoring route, telemetry table, recurring write, or continuous log tail was added.

  • V26-1047/U8 is deferred because the Dev backfill already ran; it earns no retrospective savings credit.
  • The July 13 screenshots establish urgency and function attribution, but not a reproducible time/build boundary. The signed predeploy baseline is therefore Hold.
  • Production has one heartbeat-producing terminal, M Supplies. It provides longitudinal evidence only; multi-context concurrency is proved in deterministic tests and Dev.

Read-boundary changes

  • U1: the authenticated shell reads the compact catalog summary instead of subscribing to products.getAll; untrustworthy summary state is hidden rather than displayed as a false zero.
  • U2: Daily Operations loads compact posture plus bounded week analytics, then selected-day command-grade detail only after explicit operator intent. Cached week metrics survive detail hydration.
  • U3: the POS opening guard consumes a compact server-owned Daily Close lifecycle gate rather than the full close snapshot.
  • U4: accepted heartbeats query the selective verification index with a fixed 20-row budget: five newest commands remain prompt while a persisted cursor rotates fifteen rows through the full ready backlog.
  • U5: runtime material classification is shared by client and server, and one cross-context publisher owns steady-state reporting where supported. Full follower material is retained across an in-flight leader publish.
  • U6: homepage candidates are bounded before expensive hydration. Regular featured and shop-look candidates have independent quotas, while best-seller compatibility fallthrough remains capped at 100 rows.
  • U7: catalog refresh ownership is store-keyed and coalesces in-flight/recent successful one-shot reads. Superseded consumers settle from the newer persisted generation instead of remaining stuck.

Correctness boundaries preserved

The containment queries remain read models. Checkout, stock, catalog, register, terminal-recovery, and Daily Close commands still revalidate live server truth. Local POS snapshots continue to support offline continuity but never authorize a sale.

Authorization and store keys remain at every server boundary. Homepage output preserves public visibility, money, guest/cookie, rank, exact banner-expiry, hidden-candidate fallthrough, and shop-look singleton contracts. Runtime coordination retains the existing terminal-health freshness threshold and server duplicate/no-op safety.

Race and high-cardinality defenses

  • Recovery verification never scans completed history and cannot starve a matching middle command behind permanently unmatched edge rows; repeated-heartbeat tests prove cursor progress.
  • Runtime tests cover Web Locks ownership, unsupported-browser fallback, stale owners, follower handoff, volatile storage, and follower-only material arriving during an in-flight publish.
  • Homepage tests cover a late shop-look behind 99 regular placements and a sellable best-seller sibling beyond row 20, proving both independent quota termination and the 100-row compatibility cap.
  • Catalog tests cover concurrent consumers, store switches, explicit generations, persistence failures, retry, stale completion, and superseded-consumer settlement.
