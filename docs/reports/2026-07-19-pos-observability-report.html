POS Observability v1: Offline-Durable Telemetry, Heartbeat Health, and Edge-Triggered Alerts

Before this branch, a failing POS terminal was invisible: errors died at the terminal's console, the server discarded every storage-health field terminals reported, and nobody got alerted until store staff called. This delivery adds remote observability in four phases — under a hard constraint that nothing may poll the database.

Executive Summary

  • Phase 1: POS client errors now buffer in a durable localStorage ring (survives reloads and offline periods) and drain to a new posClientEvent Convex table. mapThrownError still shows users a generic message but tees the raw error to telemetry, so "unexpected error" is finally diagnosable.
  • Phase 2: The terminal heartbeat's storage-health telemetry was being discarded at the server boundary despite the client building it, the schema storing it, and the UI having rendering branches for it. Reconnected, plus new sync-stall signals (backoffUntil, heldEventCount, heldWithoutProgress) and counters for silent-catch rails.
  • Phase 3: Edge-triggered health alerts (storage_critical, sync_stuck) ride the heartbeat write itself — zero additional reads — and email admins via MailerSend on entry into a degraded condition, with a 6-hour anti-flap cooldown.
  • Phase 4: Operational contract documented (docs/operations/pos-observability-v1.md), a Checkly readiness selector on the POS hub, console hygiene, and — from independent diff review — a redaction fix so client error text is scrubbed of secrets/PII before persisting.

Problem And Context

The POS app is local-first: an IndexedDB event ledger, a sync scheduler with exponential backoff, and cross-tab leader election keep sales flowing offline. Local observability was good — toasts, typed command results, a sync-status chip. Remote observability was absent: logger.ts ended at console.*, checkout errors were flattened to one generic message before anyone could see the cause, and ~30 deliberate silent catches guarded exactly the rails (storage probes, migrations, leader election) whose failure makes an offline register unsafe.

The binding constraint: the deployment is under an active database-IO containment effort (docs/operations/convex-io-containment-observation.md), which forbids recurring observation reads, polling crons, and read-heavy telemetry. Every signal in this delivery had to ride a write that was already happening.

Mental Model

Think of three rails, all push-based, none polling:

  1. The black box recorder (phase 1): everything bad that happens on a terminal is written to a small durable ring buffer on-device, and shipped to the cloud whenever connectivity allows. Offline failures are the ones you most need — so the transport must survive offline.
  2. The pulse (phase 2): the terminal already checks in every ~110 seconds. The check-in now carries the vital signs that used to be dropped on the floor: storage pressure, migration state, sync stall indicators, and counters from the silent-catch rails.
  3. The alarm (phase 3): the server compares each pulse to the previous one — which it was already reading — and only a transition into a severe condition rings the bell. Steady-state (healthy or degraded) costs nothing extra.
The recurring design move: never create a new read to observe something — find the write that already happens and attach the observation to it.

Before And After

Client error on the checkout path

Before: thrown error → mapThrownError → generic "unexpected error" toast → detail lost forever.
After: same toast, but the raw error (with operation name) lands in the durable buffer → drains to posClientEvent → queryable per store/level via listClientEvents.

Storage degradation

Before: client measured pressure/migration/persistence, sent it in every heartbeat, and stripRuntimeStatusInput deleted it; the terminal-health UI's storage branches never fired on fresh data.
After: persisted end-to-end; degradation state changes are "material," so they publish promptly; entering storage_critical emails admins once.

Stuck sync

Before: the drain scheduler's held-without-progress escalation existed only as local state.
After: it reaches the heartbeat, renders as a "Sync stuck" badge and "Sync retry" / "Oldest unsynced" rows in the fleet views, and triggers a sync_stuck alert edge.

What Intentionally Did Not Change

  • User-facing error messages stay generic. Only telemetry gets the raw detail.
  • Silent catches keep swallowing. They now increment counters, but behavior is unchanged — best-effort rails remain best-effort.
  • /health stays shallow. The production observability runbook contracts it as a zero-read liveness ping polled by Cloudflare every minute; deepening it would add recurring reads and break the documented contract.
  • No operationalEvent per held/conflicted sync event. Those outcomes already persist as posRegisterSessionActivity rows.
  • No offline-terminal cron. Absence detection requires recurring scans; staleness stays visible on demand in the terminal health views.
  • No server capture of invariant throws. A Convex throw rolls back the transaction — nothing can be written — but the error reaches the client, where phase-1 telemetry records it.

Risks And Review Notes

  • Persistent outages alert exactly once. Edge-triggering means a terminal stuck in storage_critical for a day never re-nags. Intended, but operationally significant.
  • Multi-tab dedupe correctness depends on FIFO-prefix draining. Concurrent tab drains serialize via Convex OCC on the dedupe index; a future change to buffer ordering or partial removal would break the single-read fast path.
  • Wider material signature → more heartbeat writes under churn. Storage-degradation oscillation now publishes per transition instead of coalescing into the freshness window.
  • Alert classification compares raw input vs the stored (cleaned) previous row. Benign today because cleaning doesn't touch alert-relevant enums; a latent trap if it ever does.
  • posClientEvent rows carry no terminalId yet — the drain host only wires storeId; device correlation is via fingerprint hash. The terminal-scoped index is ready when a terminal-aware drain lands.
  • No retention job on posClientEvent. If volume grows, attach retention to an existing daily cleanup cron rather than adding a new one.

Quiz: Pass Required

You must score at least 8 out of 10 to pass.

1. What happens to a POS client error that occurs while the terminal is offline?
The buffer survives reloads and offline periods (200-event ring, in-memory fallback). Offline is when the most diagnostic failures happen, so the transport had to be offline-durable.
2. How does the client-event ingest dedupe a fresh 50-event batch?
The client only removes events after an acked commit and always drains a FIFO prefix, so replay detection needs only the first event's id. Per-event reads happen only on a detected replay; concurrent tabs serialize via OCC.
3. After this change, what does a cashier see when a checkout throws an unrecognized error?
mapThrownError's user-facing behavior is intentionally unchanged; the fix was that the detail was previously destroyed entirely, not that users should see it.
4. Why were the terminal-health UI's storage-degradation branches dead before phase 2?
Client built it, schema accepted it, UI rendered it — the strip in convex/pos/public/terminals.ts was the single severed seam. Most of phase 2 was reconnection, not construction.
5. Why does health-alert transition detection cost zero additional database reads?
The previous row was always read for merge/coalescing decisions. Returning it made healthy→degraded comparison free, and per-condition alert timestamps live on the same row.
6. A terminal enters storage_critical and stays there for two days. How many alert emails go out?
Edge-triggered semantics: persistence is not a new edge. The 6h cooldown suppresses flapping (recover-then-degrade cycles), not steady state. Operators should know there is no re-nag.
7. Why does the alert path insert operationalEvent rows via raw buildOperationalEvent instead of recordOperationalEventWithCtx?
recordOperationalEventWithCtx does a .collect() of every prior event for the subject as idempotent-replay dedupe. Edge-triggering + cooldown already dedupe, so the raw insert (an existing repo precedent) avoids the read.
8. Why is there no server-side capture of invariant throws (e.g. in sync ingest)?
In-band capture is structurally impossible. The client-side rail is the invariant-throw channel: the thrown message crosses to the caller, hits mapThrownError, and lands in the buffer.
9. Why is there no cron detecting offline terminals?
You cannot ride a write that never arrives. Under the no-polling constraint, "no check-in for N minutes" renders when an operator opens terminal health, instead of a scheduled scan paying reads around the clock.
10. What did the independent delivered-diff review catch, and what was the fix?
The heartbeat path already scrubbed secrets/PII from diagnostic text, but the new ingest — which carries far more free-form error text — did not. The asymmetry was fixed in the final commit and captured as a Prevention rule in the solution note.
