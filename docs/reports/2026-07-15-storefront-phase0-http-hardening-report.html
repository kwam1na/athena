Storefront Phase 0 — Close HTTP API IDOR/PII & Forged-Webhook Surfaces

The customer storefront's HTTP API trusted client-supplied IDs, reflected any origin with credentials, and accepted unsigned payment webhooks. This change closes those authorization, cross-site, and webhook-forgery holes — the storefront counterpart to the POS Phase 0/1 hardening already merged.

1. Executive summary

Every customer-facing HTTP route under packages/athena-webapp/convex/http now derives the acting identity from session cookies and checks resource ownership before reading or mutating. Three structural fixes land together:

  • Authorization (IDOR/PII): arbitrary-id user reads/updates, order/bag/saved-bag owner reassignment, bag item mutations, review edit/delete, and reward grants are now bound to the authenticated actor.
  • Payments: Paystack webhook signature verification is restored (HMAC-SHA512 via Web Crypto), so an attacker can no longer forge a paid order.
  • Cross-site: reflect-any CORS is replaced with a first-party origin allowlist, so a logged-in customer's cookies can't be driven from a third-party page.

2. Why it mattered

The storefront backend issued SameSite=None session cookies and answered with Access-Control-Allow-Credentials: true while reflecting any Origin. Combined with routes that trusted a path/body userId, bagId, or ownerId directly, a customer's PII (name, email, phone, shipping/billing address), orders, cart, reviews, and loyalty points were reachable and tamperable — either cross-origin from any website, or directly by ID. Convex document IDs are not secret (they appear in API responses, and the public product-reviews endpoint returns reviewer IDs), so "unguessable" was never a real defense. Separately, the payment webhook accepted unsigned JSON: a crafted charge.success would place a "paid" order with no money collected.

3. The mental model

Think of each route as a door. Before, the door checked the label on the visitor's clipboard ("I'm here for user X") and let them act on X. Now the door checks who the visitor actually is — the identity in their session cookie — and only lets them act on resources that identity owns. The claim ("merge my guest cart into my account") is the one legitimate cross-identity action, so it requires a double proof: the target is your logged-in cookie and the source is your own guest cookie.

One helper, applied everywhere: isAuthorizedResourceOwner(resourceOwnerId, actorId) — exact match, returns false if either side is null. Ownership fails closed.

4. Before vs. after

Before

  • GET/PUT /users/:userId served/updated any id.
  • /owner routes trusted currentOwnerId/newOwnerId from the body.
  • Bag item add/update/delete acted on any bagId/itemId.
  • reviews.update/deleteReview had no actor.
  • rewards /order-points read any order; /award-guest-orders trusted body ids.
  • CORS: origin: (o) => o (reflect any) with credentials.
  • Paystack signature check commented out.

After

  • /users/:userId → 403 unless "me", keyed to the user_id cookie.
  • /owner requires target == user_id cookie and source == guest_id cookie.
  • Bag/saved-bag item ops call assertBagOwnership first.
  • Review mutations take requestedByStoreFrontUserId and throw on mismatch.
  • rewards routes enforce order ownership / cookie binding.
  • CORS: first-party allowlist (apex + *.wigclub.store; localhost only off-prod).
  • Webhook verifies HMAC-SHA512 signature, fails closed.

6. Failure boundaries

  • Missing cookie: read/PII routes fail soft (user.ts → 200 null); mutation/claim routes fail hard (401/403).
  • Ownership mismatch: 403 Forbidden across order/bag/saved-bag/rewards; review mismatch throws in the mutation (surfaced as 500).
  • Claim routes: require both cookies present and equal to the body ids — a request missing either cookie fails closed.
  • Webhook: any missing/invalid signature is rejected before order creation; replay is still covered by isDuplicateChargeSuccess.

7. What intentionally did not change (scope boundary)

Cookie integrity is out of scope — tracked as V26-952 (Phase 1). These checks trust the raw user_id/guest_id cookie values. A browser can't forge them cross-site (that vector is now closed by CORS), but a direct non-browser client (curl/Postman) can still set Cookie: user_id=<victimId>, and IDs are harvestable from public endpoints. Fully closing the platform-level IDOR requires signed cookies or opaque server-side session tokens plus CSRF — a cross-cutting change that makes the synchronous identity helper async across ~17 read sites and needs a coordinated, session-invalidating rollout. That is deliberately a separate workstream.

Webhook amount re-verification was not wired in. The available session.amount is subtotal-only (checkoutSession.ts:413, sessionAmount = pricedCheckout.subtotal, excluding delivery fee/discount), so comparing it to the charged total would false-positive on every order with a delivery fee and block legitimate checkouts. Signature verification alone closes the forgery hole; a correct amount reconciliation is a follow-up.

9. Next-time guidance

  • New storefront route touching a user-owned resource? Derive identity from the cookie (getStorefrontUserFromRequest / getCookie) and gate with isAuthorizedResourceOwner; don't trust path/body ids.
  • Cross-identity actions (claims/merges) need proof of both identities from cookies.
  • Push ownership into the Convex mutation (as reviews now do) so a second caller can't bypass a route-only check.
  • Any new webhook must verify its provider signature with Web Crypto (not node:crypto) in the Convex HTTP runtime, and fail closed.
  • When V26-952 lands signed/session cookies, these ownership checks stay — they are the authorization layer regardless of how identity is proven.

1. What is the core change in how storefront routes decide who you are?

Identity comes from the user_id/guest_id cookie and is compared to the resource owner via isAuthorizedResourceOwner.

2. Why does the /owner claim route require two cookie checks?

Without the source (guest) check, an attacker could absorb an arbitrary guest's orders/cart into their account.

3. Why is the Paystack signature computed with Web Crypto rather than node:crypto?

This is almost certainly why the original code was commented out; the fix uses crypto.subtle.importKey/sign.

All missing/invalid cases fail closed before any DB mutation — hence the deploy prerequisite that the secret be set.

5. Which origins does the new CORS allowlist accept?

The suffix check rejects look-alikes like wigclub.store.evil.com and notwigclub.store.

6. Where is review edit/delete ownership enforced?

Pushing the check into the mutation is defense-in-depth: any caller of the mutation is covered, not just the one route.

checkoutSession.ts:413 sets sessionAmount = pricedCheckout.subtotal. Signature verification alone closes the forgery hole; correct reconciliation is a follow-up.

8. What is explicitly out of scope, and where is it tracked?

The ownership checks are still necessary regardless; V26-952 hardens how identity itself is proven.
