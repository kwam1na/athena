POS Register Authority Replication

How Athena keeps remote drawer lifecycle authority local-first, heartbeat-independent, and safe under repair and rollout.

Branch: codex/pos-register-authority-replicationLinear: V26-997 through V26-1003Status: delivery validation in progressFinish line: open mergeable PRNo mergeNo production deploy

Executive Summary

Arc exposed a split-authority bug: a reactive Convex read knew the cloud drawer was closed, but paused heartbeat meant IndexedDB never received the closure. The UI showed an opening gate while the local command gateway still reused the old drawer.

Athena now replicates exact register lifecycle authority through a dedicated terminal-authenticated lane. The result is committed to IndexedDB before the local projection changes, so presentation and cashier commands make the drawer decision from the same durable source.

Mental Model: Replicate Authority, Keep Commands Local

Think of Convex as the source of external lifecycle evidence and IndexedDB as the cashier's operational authority. Convex does not execute drawer commands for the cashier. It publishes an exact, versioned observation; the browser validates and persists it; then both the view model and command gateway read the refreshed local projection.

Cash Controls -> lifecycle + mapping revisions -> terminal query
-> snapshot bijection -> IndexedDB compare-and-apply
-> local projection -> UI + local command gateway

Before vs After

  • Before: the view model gated from reactive Convex state while drawer commands gated from IndexedDB.
  • Before: runtime-status heartbeat was the only ordinary server-to-local closure directive path.
  • After: a heartbeat-independent query accepts only exact locally known candidates and returns redacted classifications.
  • After: mapping epoch is primary; lifecycle revision is secondary within the same exact cloud subject.
  • After: snapshot results must match requested candidates one-for-one before any local write.
  • After: IndexedDB atomically revalidates the complete mapping fingerprint and rejects stale or conflicting evidence.
  • After: the local projection exclusively controls drawer operability; Convex register state is passive enrichment.

Operator Behavior

  • Exact cloud closure with an empty sale shows a replacement-drawer form and creates a distinct local drawer identity.
  • Exact cloud closure with an old draft keeps cart, payments, and customer visible but read-only. Only Clear sale and Sign out remain available.
  • Clear sale stays attached to the old drawer. The replacement form appears only after the durable clear is reprojected.
  • An unmapped local drawer is normal pending outbound state. A mapped missing, ambiguous, or scope-invalid target becomes redacted repair authority.
  • Authority unknown, local persistence failure, terminal repair, and sync review remain distinct states with separate retry ownership.
  • Checkout-local payment editor state is cleared when the register becomes read-only, so it cannot leak into a replacement drawer.

Ordering, Races, and Failure Boundaries

  • Missing legacy revisions baseline deterministically to zero.
  • Production mapping creation, ambiguity, and repair advance a durable mapping epoch. Tombstone helpers and states exist for future deletion or supersession paths, but those transitions are not yet active production flows.
  • Versioned dedicated observations outrank unversioned legacy runtime directives.
  • Mapping changes between subscription and IndexedDB commit produce mapping_invalidated, not an arrival-order overwrite.
  • Partial, duplicate, foreign, or count-mismatched snapshots fail before any apply.
  • Offline, loading, and transport absence preserve last-known durable authority.
  • A wedged local IndexedDB readiness read times out into a local-store repair state instead of leaving POS indefinitely on a checking screen.
  • Semantic snapshot and candidate signatures prevent fresh-but-equivalent Convex results or local projections from retriggering persistence and refresh in a render loop.
  • Persistence failure latches drawer, sale, payment, and checkout mutations closed until reconciliation reaches a later durable ready state. The exact old-sale Clear sale command remains the recovery escape hatch.
  • Remote close never fabricates counted cash, variance, approval, deposit, or closeout history.

Rollout and Observability

  • disabled: stops new subscription and application without clearing durable authority.
  • shadow: observes and acknowledges without applying authority.
  • canary: applies only for exact configured terminal IDs.
  • broad: applies for every eligible provisioned terminal.
  • Development defaults broad; production defaults shadow until release configuration opts into canary or broad.
  • Acknowledgements are terminal-scoped, cursor-validated, redacted, coalesced, and advisory. They do not authorize cashier commands.

Intentionally Unchanged

  • Drawer, cart, payment, completion, and closeout commands remain local-first.
  • Fully offline operation continues from last-known IndexedDB authority.
  • Facts completed before the first reconnect snapshot remain immutable reconciliation history.
  • Existing carts, transactions, payments, mappings, and closeout history are never rebound to a replacement drawer.
  • Legacy runtime directives and active-session seeding remain during mixed-client rollout.
  • The new lane never reconstructs a missing local register.opened event from a cloud drawer.

Next-Time Guidance

  • When a local-first feature reacts to server state, replicate that state into the durable local authority model before using it operationally.
  • Keep diagnostic heartbeat and recovery channels out of ordinary correctness ownership.
  • Version identity repair separately from entity lifecycle; lifecycle ordering alone cannot defeat stale mappings.
  • Test both observation orders, restart with compacted history, metadata-only mapping races, disabled rollback, and non-empty in-progress work.
  • Treat canary acknowledgements as rollout evidence, never as command authority or sole compatibility-retirement proof.

1. What is the operational source of drawer authority after this change?
Both presentation and commands consume the refreshed IndexedDB-backed local projection.
2. Why did the original drawer gate reuse the closed drawer?
The UI saw cloud closure directly, but heartbeat-paused IndexedDB did not, so local idempotency returned the old drawer.
3. Which cursor component is compared first?
A newer mapping epoch outranks lifecycle observations from an older subject.
4. What happens when an exact cloud closure arrives with a non-empty old sale?
The old sale remains attached to the old drawer and must be durably cleared before replacement opening.
5. What does a remote closure create locally?
It is authority evidence only and never fabricates financial facts.
6. How does shadow mode behave?
Shadow mode gathers bounded rollout evidence but leaves operational IndexedDB authority unchanged.
7. What happens if the mapping changes after the query but before IndexedDB commit?
The transaction rechecks the complete mapping fingerprint and refuses a stale subject.
8. When may the persistence command guard clear after a write failure?
Retry and applying remain guarded; reconciliation may clear the latch only after durable ready state, including an already-current authoritative result.
