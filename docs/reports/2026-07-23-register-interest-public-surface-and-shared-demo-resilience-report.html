Executive Summary

This delivery bundles several threads that accumulated in one working tree. The largest is the register-interest public surface: the landing page's close was reworked, the "walkthrough" framing was reframed to interest capture across the walkthrough and privacy pages, a shared reveal-on-scroll footer was added to the public shell, the nav secondary links were replaced by the theme switcher on the interest and privacy pages, and the paper-grain texture was extracted into a reusable component.

The second thread is a shared-demo resilience fix: the seeded POS terminal — the row that gives the demo register its "Studio Front Counter" name — had no durable handle and could be lost permanently, leaving a nameless register that survived every hourly restore. Provisioning now heals it on every pass, and the prod deploy runs the restore step so a baseline bump no longer locks the demo out for up to an hour.

The third thread is delivery tooling: a global framer-motion test mock (so entrance-animated content stays visible in jsdom), a tsconfig exclude that stops tsc from parsing gitignored build artifacts and running out of memory, and a deploy-script default that makes the interest form deploy-ready in prod.

Why It Mattered

The public surface is the product's front door and its only market-interest signal; "walkthrough" implied a service the product does not yet offer to other businesses. The shared-demo terminal defect was live in prod — a visibly broken demo card — and self-perpetuating. The tooling fixes were not cosmetic: the merge gate runs tsc --noEmit and the full test suite, both of which the OOM and the framer/visibility interaction would have broken.

Thread 1 — The Interest Public Surface

The landing close was split so the demo invitation and an availability beat ("Built running a real store. Opening to more.") share one uninterrupted field, then blended with the section above it. A minimal footer was added to the shared PublicLayout: it is pinned beneath the page as a fixed layer, and the opaque page content scrolls up off it to reveal it — a drawer-like reveal with no scripting.

The walkthrough page became the interest page: eyebrow removed, copy reframed, the submit CTA humanized to "Send my details", the field grid bottom-aligned so inputs stay on one line when a neighbor's description wraps, and the submission-error status region restored after a prior edit had dropped it. The privacy page was aligned to match — same shell, texture, padding, and reframed copy — while preserving the two legal phrases its tests pin. Browser-tab titles were de-walkthroughed. The nav's "Register interest" and "Sign in" links were replaced by the theme switcher on both pages.

Thread 2 — Shared-Demo Seeded Terminal Self-Heal

The seeded terminal is identified only by fingerprintHash: "shared-demo-terminal", and posTerminal is deliberately excluded from the restore registry while registerSession is included. So a deleted seeded terminal left the session and its captured baseline document pointing at a dead id, which the hourly restore re-asserted forever. Three composable helpers now find-or-create the terminal by fingerprint and re-point any dangling session and baseline document — invoked on every provision including the kind: "existing" path, so an already-provisioned store heals without a version bump. The full rationale and verification are in the paired solution note.

The Framer Mock's One Sharp Edge

The first version of the mock returned a fresh forwardRef component on every motion.div access. Because JSX reads motion.div on every render, React saw a new component type each time and remounted the whole subtree — which manifested as dropped keystrokes in register-closeout inputs (typing "30" yielded "3"). Caching the component per tag fixed it. This was caught by running the full suite specifically to vet the global mock; the input remount was the only fallout.

Quiz: Pass Required

You must score at least 8 out of 10 to pass.

1. Why did the shared-demo register card render with no terminal name?
listTerminalNames does db.get on session.terminalId and drops the entry when the terminal is missing. Prod had no row with fingerprintHash "shared-demo-terminal", and the session pointed at the deleted id.
2. Why did the broken terminal reference survive every hourly restore?
posTerminal is excluded from SHARED_DEMO_MUTABLE_TABLES so device registration survives restores; registerSession is included, so its captured baseline document — carrying the frozen terminalId — is replayed every hour.
3. Why didn't re-running provisionSharedDemo fix the broken store on its own?
This is the recurring failure mode: a recovery guarded behind a version bump that has already passed. The fix runs the foundation check unconditionally, including on the existing path.
4. Why must the repair fix the captured baseline document and not just the live session?
Fixing only the live row leaves the captured document to re-assert the dangling id the next time the restore replays it. Both halves must be re-pointed.
5. What was the one sharp edge in the first global framer-motion mock?
JSX reads motion.div every render; a fresh forwardRef each time is a new component type, forcing a remount. Typing "30" yielded "3" until the mock cached one component per tag.
6. What actually caused the local tsc out-of-memory crash?
These directories are gitignored and absent in CI, which is why CI typecheck was clean while local OOM'd. An explicit exclude makes local match CI.
7. Why does adding exclude to the tsconfig require re-listing node_modules?
tsc's built-in exclude (node_modules, bower_components, jspm_packages, outDir) applies only when no explicit exclude is set; once you provide one, you must re-add node_modules.
8. Why was the interest-form CTA disabled, and what enables it?
The gate is a build-time frontend flag. The prod convex backend (HMAC, origins, contact, recipient, MailerSend) was already fully configured; the deploy script now defaults the frontend contact so the next prod build enables the form.
9. What did the reframe deliberately NOT change?
Only user-facing "walkthrough" framing was reframed to interest. Renaming the plumbing (funnel events, backend, routes, crons) would be a large, risky refactor with no user benefit.
10. How was the dev interest pipeline verified end to end?
Running the real path — POST to the dev endpoint, then reading back the stored request and its notification attempt — confirmed HMAC/origin/contact config and that the sending domain is verified.
