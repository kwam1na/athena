Executive Summary

This candidate packages several related operational-hardening slices from iterative feedback. It makes app-login admission explicit and server-enforced, keeps POS recovery terminal-scoped, gives the Operations sidebar cheap bounded attention counts, preserves historical Opening membership, narrows Daily Close deposit reads to the operating-day slice, and replaces broken product images with stable fallbacks.

These changes improve posture and presentation without moving command authority. Queue actions, register and staff authority, frozen Opening evidence, checkout validation, and Daily Close command checks stay on their existing server-owned paths. The 72-hour Convex containment checkpoint remains Hold because the required Usage artifacts were unavailable.

Why This Batch Mattered

  • Any syntactically valid email could previously enter the OTP provider path before Athena made an app-access decision.
  • Navigation lacked count-only Open Work and Approvals posture, while loading full queue evidence would be too expensive for a sidebar.
  • Daily Close capped store-wide allocation rows before filtering, so unrelated history could hide deposits from the selected operating day.
  • Legacy Opening records could present multiple acknowledgements for one logical same-SKU work group unless exact historical members were projected together.
  • Truthy but broken remote image URLs displayed browser failure chrome, and the POS shell did not react to online/offline events after mount.

Mental Model: Posture, Evidence, Authority

Think of the new summaries and UI states as an instrument panel. They tell the operator where attention exists and whether the observation is exact. They do not steer the system or authorize commands.

Posture reads are compact and bounded. Evidence views retain detailed source records. Commands revalidate current server truth. When a posture probe reaches its budget, Athena says N+ instead of pretending the lower bound is exact. 

indexed probe -> logical projection -> { count, completeness } -> N or N+

client preflight -> OTP provider send gate -> verification-time authorization

historical member ids -> logical carry-forward group -> one acknowledgement

Before and After

  • Login before: normalized email flowed directly to signIn. Login after: a server decision preflights the request, while provider delivery and token redemption independently enforce the same allowlist.
  • Operations before: no compact navigation metadata. Operations after: access-gated queries mount only while the Operations section is expanded and return explicit completeness.
  • Daily Close before: take 200 allocations by store, then filter in memory. Daily Close after: type, direction, status, and operating-day range are enforced by a compound index before a 201-row completeness probe.
  • Opening before: legacy carry-forward rows became separate items. Opening after: only the prior close's referenced rows enter the logical projection, preserving history while consolidating one operator decision.
  • POS images before: a failed URL remained an image element. POS images after: onError swaps the card or cart row to an accessible package placeholder.

Failure Boundaries

  • The client approval query is feedback, not security. An unapproved address is also blocked at delivery and verification, and the submitted normalized email must match the auth account identifier.
  • POS recovery is shown only when local terminal provisioning exists. Recovery still authenticates an app account; it does not grant staff, manager, drawer, or sale authority.
  • Open Work and Approval counts use global budgets. Nonzero incomplete observations render N+; an incomplete zero Approval observation currently renders no metadata.
  • The login preflight exposes a public Boolean membership decision even though the static address list stays out of the browser bundle. Updating the hard-coded allowlist requires a backend code deployment.
  • Frozen version-2 Opening membership remains authoritative. The compatibility path only consolidates exact legacy member ids and must not import later same-SKU work.
  • Browser online/offline state changes shell presentation and rehydration behavior only. It is not terminal integrity, permission, or proof that a sale may proceed.
  • A broken image falls back visually without disabling product selection, quantity, cart, or checkout behavior.

What Intentionally Did Not Change

  • The existing OTP provider id remains stable for existing auth-account records.
  • Raw operational work rows remain audit truth; logical groups are projection, count, and acknowledgement behavior.
  • The detailed queue query and its actions remain separate from sidebar summaries.
  • Daily Close keeps its bounded cap and fail-closed completeness contract; only selectivity before the cap changed.
  • Service principals are not part of this candidate; the branch is based on the main commit that reverted that rollout.
  • No production deploy will be run for this delivery. The requested finish line is merged PR and local root alignment.

Next-Time Guidance

  • Put expected access-policy denials before framework side effects, then retain server-side defense in depth at delivery and redemption.
  • For persistent navigation metadata, expose count-only queries with a shared budget and explicit completeness; do not mount detail companions eagerly.
  • Every capped downstream helper that feeds an exact-looking summary must propagate overflow evidence.
  • Apply the complete business predicate in the index before a cap, and probe limit plus one.
  • Historical workflows should reconstruct from historical member ids, never live equivalent-looking rows.
  • Test failed remote media at every real rendering boundary, not only URL-selection helpers.
