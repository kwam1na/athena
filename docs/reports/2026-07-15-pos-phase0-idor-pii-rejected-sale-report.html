Executive summary

A nine-pass audit found the event-sourced POS core sound but exposed active silent-failure paths in the seams around it. Phase 0 closes the three highest-severity ones:

  • V26-1055 (U1): register.getState had no authorization and openDrawer was authentication-only — two cross-tenant IDOR holes. Both now require org membership scoped to the store that owns the resource.
  • V26-1056 (U11): all 13 endpoints of customers.ts had zero authorization (full PII by customerId, unauthenticated cross-tenant writes). Every endpoint is now org-scoped.
  • V26-1057 (U2): a rejected money-bearing offline sale advanced the sync cursor and left no record. It now creates a manager-visible server_rejected conflict while preserving cursor liveness.

Why it mattered

This is an offline cash system running on in-store terminals. The failures that matter most are "another tenant can read/alter my POS state" and "money changed hands but the record is gone." The three gaps produced exactly those outcomes: a foreign-org user could read a store's cashier/drawer/held-cart state or open its drawer (corrupting cash-control attribution); anyone could read full customer PII by id or write customer records across tenants; and a validation-rejected cash sale could vanish with the drawer left short and nobody notified.

The mental model

Think of two trust boundaries around a sound core:

  1. Authorization boundary (U1/U11): every public Convex POS endpoint must resolve the store that owns the targeted resource and confirm the caller is a member of that store's organization before reading or writing. The template already existed: requirePosTransactionStoreAccess in transactions.ts.
  2. Data-durability boundary (U2): no money-bearing event may leave the sync pipeline without either a canonical transaction or a needs_review conflict. The conflict rail already existed; the rejected branch just wasn't using it.

Before vs after

Before

  • getState: any/no caller → full register state for any store.
  • openDrawer: authenticated-only → open a drawer on any org's terminal.
  • customers.ts: 13 endpoints, no auth → PII by id, cross-tenant writes.
  • Rejected money-bearing sale → cursor advances, no conflict, silent loss.

After

  • getState: resolve store → require org member (full_admin/pos_only).
  • openDrawer: same org check in the command before opening the session.
  • customers.ts: every endpoint org-scoped; id-keyed reads scoped to the resource's store.
  • Rejected money-bearing sale → one server_rejected conflict, cursor still advances.

Rejected-sale flow (U2)

pending --drain--> [single Convex ingest mutation] |-- valid ------------------> projected (canonical write + mapping) [unchanged] |-- business/perm/dup ------> conflicted (needs_review) [unchanged] |-- envelope/payload -------> rejected (cursor advances) | └── money-bearing (sale_completed + payments)? | ├── yes → ALSO create server_rejected conflict (needs_review) [NEW] | └── no → stay silent (no conflict spam) [unchanged]

Cursor advance is preserved in every branch (liveness). createConflict dedups on (store, terminal, localEventId, needs_review, sequence, conflictType, summary, details), so replays create no duplicate; the replay path re-lists the existing conflict via listConflictsForEvent.

What intentionally did NOT change

  • The event-sourcing core (single-mutation exactly-once, monotonic cursor, durable-commit-before-ack) and the sync-contract event shapes.
  • POS catalog.ts search/barcode exposure — deliberately deferred to V26-956.
  • terminalAppSessions.ts — a deliberate terminal-proof + validatePosAppAccount exception; no redundant guard added.
  • Sibling POS surfaces (terminals.ts, sync.ts, posRecoveryCodes.ts) were swept per-endpoint and found already guarded — no change needed.
  • Non-financial rejected events stay silent (no new conflict) to avoid manager-facing spam.

Failure boundaries

  • Cross-org caller: requireOrganizationMemberRoleWithCtx throws → the endpoint rejects; the underlying query/command never runs.
  • Missing store/customer: returns a clean null / empty / not_found without leaking existence beyond the pre-existing pattern; membership checks are skipped.
  • Foreign valid id: id-keyed customer reads resolve the target's store first, so a valid foreign customerId is still denied.
  • Malformed rejected payload: amount summarization is defensive (guards non-finite numbers), so the rejected branch cannot throw and abort the batch.
  • Replay: conflict dedup makes re-ingesting a rejected event idempotent (exactly one conflict).

Next-time guidance

  • Adding a POS public endpoint? Resolve the store (directly or via the targeted resource) and call requireOrganizationMemberRoleWithCtx before any read/write — copy requirePosTransactionStoreAccess.
  • Auditing auth coverage? Check per endpoint; an auth import in a file does not prove every export is guarded.
  • Touching sync ingest? Preserve cursor-advance on rejects, and never let a money-bearing event exit without a canonical row or a conflict.
