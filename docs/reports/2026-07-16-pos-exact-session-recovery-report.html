POS Exact-Session Recovery and Register Authority

How terminal recovery now handles expired handoffs, Convex mutation replay, and Auth-provider remounts safely—requesting a fresh code when needed without weakening staff accountability.

Delivery candidateBranch: codex/pos-terminal-app-session-gateBase: PR #666 / 2c4a4a5fReport snapshot: 2026-07-16Production deploy: out of scopeRoot checkout: clean on main

Executive Summary

Fullscreen register and expense routes now require the terminal's exact POS service-principal session. A human manager can administer POS, but cannot accidentally stand in for the checkout station's application authority.

Recovery is now resilient across three independent boundaries: an expiring browser/server handoff, deterministic execution inside a replayable Convex mutation, and the short interval while the root Auth provider remounts onto promoted service-session tokens. Register commands still require separate verified staff identity.

Operators now see “POS sign-in required” when a fullscreen terminal route lacks application authority. If the prepared exchange has expired, the flow clears the confirmed local attempt and returns to fresh recovery-code entry.

Problem and Context

The terminal and store were valid, but the browser had no active POS application-session binding. Human authentication allowed the fullscreen register to mount, while protected POS calls correctly rejected that human session. Recovery then exposed additional failures: stale handoffs, expired prepared exchanges, mutation-forbidden cryptographic randomness, and a human-only daily-close query after activation.

The safe response was to preserve the backend's exact-session checks and repair each protocol boundary, not loosen POS operations to accept a manager login.

Mental Model: Four Separate Authority Lanes

  • Terminal application authority: exact Auth session, service-principal session, store capability, terminal, and application binding.
  • Recovery continuity: bounded prepared exchange plus browser handoff journal; it can resume or expire but cannot grant authority by itself.
  • Offline continuity: server-signed, expiring receipt verified against the reviewed development trust anchor; production trust remains empty and fail-closed.
  • Personal accountability: verified cashier or manager staff proof for register and cash-control commands. Machine authority never supplies this identity.

Before and After Flow

Before: human manager auth could mount the fullscreen register; stale recovery state produced generic retry loops; receipt issuance called cryptographic APIs that Convex mutations reject; provider remount temporarily looked like a terminal failure.

After: fullscreen route enters POS recovery; an isolated Auth namespace prepares the exact pair; one mutation validates current authority, issues the service session and binding, signs the receipt deterministically, and records audit evidence; promoted tokens remount; bounded polling verifies the exact activated session; only then does the register mount.

Recovery and Failure Boundaries

  • An expired prepared exchange is exact-session-aborted and returns code_required; the browser clears only the confirmed-aborted handoff and asks for a fresh code.
  • A stale local lease can be reclaimed by its owner. A new attempt may discard only the abandoned local handoff journal and pending Auth namespace in prepared/auth-issued phases, never activated or promoted state. The server exchange expires or is cleaned through its own bounded path.
  • Mutation-time receipt issuance uses seeded Math.random bytes and deterministic P-256 signing through @noble/curves. Missing signing configuration aborts the whole transaction.
  • Immediately after promotion, getCurrentPosTerminalServiceSession may return unavailable while Auth remounts. The browser retries with bounded backoff and still requires an exact field-for-field activation match; persistent unavailability ultimately fails and may represent a real authority problem.
  • Stage detail is logged server-side; public activation failures retain the generic recovery message.
  • A mismatched development issuer, key version, or public JWK fails receipt verification. Production trust anchors remain empty.

Authorization After Activation

Only the daily-close lifecycle gate gained dual read authority: current POS application authority or the existing human store-membership lane. Broader daily-close snapshots and manager commands were not opened to the service principal.

Register closeout and opening-float correction accept the POS service lane for application authority, but fail closed without an independently verified staff profile. Non-fullscreen routes keep their existing human-session behavior.

What Intentionally Did Not Change

  • Exact principal, store, terminal, credential, capability, lifecycle, proof, and Auth-session validation remains fail-closed.
  • Only the predecessor binding for the same principal and terminal is superseded; sibling terminal sessions are untouched.
  • Production browser trust anchors stay empty and private signing keys remain environment-only.
  • Human access remains valid on non-fullscreen administration surfaces.
  • The bounded expired-artifact cleanup command was not scheduled or run; no schema migration or production cleanup is introduced.

Next-Time Operational Guidance

  • When recovery fails, use the server-side activation stage to identify the boundary while keeping the client message generic.
  • Treat prepared-exchange expiry, stale local ownership, and provider remount as different states with different cleanup rules.
  • Never add Web Crypto randomness or randomized signing to a Convex mutation; preserve deterministic transaction replay.
  • Rotate the development server key and compiled public anchor together. Keep production trust enablement as an explicit reviewed operation.
  • When extending a POS endpoint, decide separately whether it needs application authority, human membership, verified staff identity, or more than one lane.

Quiz: Pass Required

You must score at least 8 out of 10 to pass.

1. Why can a signed-in manager no longer directly mount the fullscreen register?
Human membership and terminal application authority are separate lanes; administration can remain human-authenticated while fullscreen execution requires the terminal session.
2. What should happen when a prepared recovery exchange expires before activation?
Expiry is a typed recoverable state: exact-session cleanup returns code_required and the browser starts a fresh attempt.
3. Why was Web Crypto signing replaced in the activation mutation?
Mutation replay requires stable outputs; both explicit crypto randomness and randomized ECDSA signing violate that boundary.
4. What does the unavailable current-session result mean immediately after token promotion?
Unavailable may be a transient provider-remount state immediately after promotion. Bounded polling continues with exact activation matching; persistent unavailability eventually fails and can indicate a real authority problem.
5. Which stale local handoff may a new recovery attempt discard?
Only local pre-activation state is discarded. Activated/promoted handoffs must be completed or safely resolved, while server exchanges expire or are cleaned through a separate bounded path.
6. What additional identity is required when a service principal submits a register closeout?
Machine authority permits the terminal application call, while staff proof supplies the person accountable for the cash action.
7. Which daily-close surface gained POS service-principal access?
The change is deliberately narrow: the register lifecycle gate supports dual authority, while broader manager operations remain unchanged.
8. What does this candidate do to production offline trust?
Only a development anchor is compiled under the development guard; production enablement remains a separate reviewed operation.
9. Who cleans up an abandoned stale pre-activation attempt?
Local ownership and server authority are separate. Browser cleanup cannot grant or delete server authority outside the exact-session/terminal-proof rules.
10. What must happen before the promoted recovery flow accepts the current session?
The remount window may be transient, but acceptance remains strict: the observed authority must match the activation's session, binding, terminal, store, expiry, and receipt.
