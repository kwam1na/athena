Make pr:athena the visible merge-ready gate

A delivery-policy correction that surfaces cheap prerequisites before expensive validation and prevents push-first discovery.

Executive Summary

Athena already had an authoritative PR validation command with the correct fail-fast order, but the most visible agent instructions and shipping skill allowed an agent to assemble and run the full package suite before consulting it.

This candidate makes pr:athena the explicit merge-ready authority at every relevant entry point and adds a source-level regression that fails if those instructions drift apart.

Problem and Context

A delivery agent declared its own release checklist, ran the full Vitest suite, build, audits, and freshness checks, then learned during git push that delivery documentation was required. The repository's pr:athena provider would have checked delivery documentation before coverage, saving the expensive run.

The failure was not in the sensor. It was an instruction-routing gap: execute documented the right rule, while root onboarding, package validation docs, and ce-commit-push-pr did not.

Mental Model

Treat pr:athena as the delivery controller and package commands as diagnostic instruments. Diagnostics prove a touched surface while work is changing; the controller decides merge-ready order, prerequisites, provider validation, review, and proof reuse.

iteration: focused test -> focused lint/typecheck
merge-ready: bun run pr:athena -> ordered prerequisites -> provider suite -> review -> proof
push: reuse exact-tree proof or fail closed

Before and After

  • Before: root guidance explained focused Vitest usage but omitted the merge-ready authority.
  • Before: package docs listed a full package test and other broad commands without warning that they were not the merge gate.
  • Before: ce-commit-push-pr moved directly from commit to push and handled validation only after a hook blocked.
  • After: all entry points name pr:athena first at merge-ready boundaries and explicitly reject independently assembled substitute suites.
  • After: the shipping skill validates before push, allowing cheap documentation and artifact prerequisites to fail before coverage.
  • After: one Vitest contract reads all instruction surfaces and prevents future wording drift from reopening the gap.

Failure Boundaries

  • Focused tests remain required during implementation; this change does not replace or delay them.
  • A pr:athena prerequisite failure blocks push and directs repair before expensive leaf suites.
  • A missing reusable proof triggers pr:athena now rather than deferring equivalent work to the pre-push hook.
  • The underlying pr:athena command order and CI behavior are unchanged; this candidate changes routing and regression coverage only.

Validation and Review Evidence

  • Regression observed red against the previous guidance, then passed after the instruction changes.
  • workflow:check passed for all 4 GitHub workflow files.
  • delivery:documentation-check and git diff --check passed before report generation.
  • Graphify regenerated successfully after the new TypeScript test.
  • Final pr:athena, GitHub checks, merge, and root alignment are pending at candidate-report time.
  • Production deployment is not applicable because the change only affects agent guidance, tests, durable learning, and generated documentation artifacts.

Next-Time Guidance

When a branch becomes merge-ready, do not narrate or invent a broad release checklist. Read the repository validation entry point and run pr:athena first. If it stops on documentation or generated artifacts, repair that prerequisite before spending the provider-suite cost.
