Shared-Demo Operation Admission: Rail Migration and Reconciliation

Migrating every resource-access site onto the operation admission rails, and reconciling the WIP that did it

Delivery candidateNot mergedNot pushedNo PR openedDeploy: not applicableRoot alignment: pendingBase: bf17e511

Executive Summary

This delivery migrates every remaining shared-demo resource-access site onto the operation admission rails, removes the legacy requireSharedDemoStoreReadIfApplicable helper, and grants the demo two new capabilities: pos.transaction.void and catalog.maintain. The user-visible result is that demo users can void completed POS sales, adjust transaction items, and repair catalog summaries instead of hitting shared_demo_action_denied.

The change is behavior-preserving at each migrated call site. Per-site shared-demo actor branches were consolidated into lib/athenaUserAuth.ts, which already short-circuits admitted-user resolution. The duplicated ternary at eleven call sites was deleted; the resolution it performed still happens, once, centrally.

A large share of the work in this branch was not writing the refactor but reconciling it. The change arrived as a ~90-file WIP stash reapplied onto a moved main. Eight files conflicted textually; eleven more had code removed with no conflict marker at all; and eighteen tests failed in ways that looked like the refactor was incoherent. Establishing that the deletions were deliberate — and that the failures were stale call-shape assertions rather than regressions — was the substance of the delivery.

Why It Mattered

The shared demo is the primary evaluation surface for prospective operators. Every action that returns shared_demo_action_denied is a dead end in that walkthrough. The originating session was opened against exactly such a dead end: a Convex error on inventory/products:repairCatalogSummary.

The deeper reason is architectural. Before this change, admitted-user resolution was duplicated at every access site as a hand-written ternary. Each copy was a place where a future capability could be admitted in one file and silently denied in another. Consolidating the resolution into one helper makes 'is this actor admitted, and who are they' answerable in a single place.

The Mental Model

Think of admission as a turnstile that stamps a badge onto the request before any handler runs. The wrapper (withOperationMutationAdmission / withOperationReadAdmission) resolves an actor and attaches it to the context as ctx.operationAdmission.actor.

Downstream code should never re-inspect the turnstile to figure out who walked through. It asks the identity helper, and the helper reads the badge. Before this change, eleven call sites each re-inspected the turnstile themselves. After it, they ask the helper.

// before — every call site re-derived identity from the actor
const athenaUser = demoActor !== null
  ? await ctx.db.get("athenaUser", demoActor.athenaUserId)
  : await requireAuthenticatedAthenaUserWithCtx(ctx);

// after — one call; the helper reads the badge
const athenaUser = await requireAuthenticatedAthenaUserWithCtx(ctx);

Before vs After

  • Before: admitted-user resolution duplicated as a ternary in 11 convex modules; the legacy requireSharedDemoStoreReadIfApplicable helper still in use; demo denied for voids, item adjustments, and catalog repair.
  • After: resolution flows requireAuthenticatedAthenaUserWithCtx -> getAuthenticatedAthenaUserWithCtx -> getOperationAdmissionActorUserId, which returns the admitted actor's row when a badge is present; legacy helper deleted; pos.transaction.void and catalog.maintain added to SHARED_DEMO_ALLOWED_CAPABILITIES.
  • Unchanged: actor-kind reads that do real work stayed. demoActor is still computed and still returned from storeMemberAccess, because capability checks and scope checks legitimately need to know the actor kind.
  • Unchanged: lib/athenaUserAuth.ts itself. It is byte-identical across the stash base, the WIP, and origin/main. The short-circuit already existed; this branch stopped duplicating it.

Layer-by-Layer Mechanics

  • Definition layer — operationAdmission/definitions.ts and readDefinitions.ts declare each operation's capability, scope, and which actor kinds are admitted. validateOperationDefinition fails closed on an unknown capability.
  • Adapter layer — adapters.ts and readAdapters.ts try the shared-demo adapter first. A non-demo caller returns not_applicable and falls through to the normal-user adapter. A recognized demo denial throws immediately and never falls through.
  • Context layer — the wrapper attaches ctx.operationAdmission = { actor, constraints, decision, provenance, operation } before the handler runs.
  • Identity layer — getOperationAdmissionActorUserId reads actor.athenaUserId off the context; getAuthenticatedAthenaUserWithCtx returns that user directly when present, otherwise performs full authentication.
  • Access layer — storeMemberAccess, requirePosTransactionAccess, cash controls, stock ops, and the rest call the identity helper and separately use demoActor for capability and scope decisions.

Failure Boundaries

  • Unauthenticated caller: both adapters return not_applicable; the handler throws 'Sign in again to continue.'
  • Demo actor, capability not granted: the shared-demo adapter produces actor_denied, surfaced through denySharedDemoAction as a thrown demo-policy error — not a userError.
  • Demo actor, wrong store or organization: scope_denied, same throw path.
  • Expired demo or demo gate disabled: getSharedDemoActorWithCtx throws; isRecognizedSharedDemoActorError converts it to a recognized denial that re-throws rather than falling through to normal-user auth. This is the property that keeps an expired demo from silently becoming an anonymous request.
  • Admission failures throw. Domain failures inside handlers still return userError with codes such as validation_failed, authentication_failed, or not_found. That split is preserved.

Reconciling the WIP: What Made This Hard

The branch arrived as a stash reapplied onto a moved main, and the central question — were the missing admission blocks deliberate? — was answered wrong three times before it was answered from evidence.

First theory: a stale base caused git to drop merged work cleanly. Disproved by one command; the stash's base already contained the admission commits. Second theory: an ordinary branch-versus-main preference. Disproved because applying 'prefer the branch' produced failures in five files the branch had never touched. Third theory: the WIP was internally incoherent, because a test its own author wrote asserted that requireAuthenticatedAthenaUserWithCtx was never called while the source plainly called it.

The third theory was the most persuasive and still wrong. That assertion pinned a call shape, not a behavior, and consolidation legitimately changes the call shape while preserving the behavior. The resolution came from reading the originating session transcript, where the operator's instruction was explicit: all sites need to be on the new rails, and the legacy function should be removed.

Why Eighteen Tests Failed Without a Regression

Every one of the eight affected test files calls vi.mock on lib/athenaUserAuth — the exact module the resolution was consolidated into. The mock replaced the function containing the short-circuit, so the admitted actor could never be resolved inside those tests. The failures were an artifact of mocking the boundary the refactor moved logic behind.

The fix moved the harness toward the refactor: an admission-aware fake that mirrors the real helper, and each call-shape assertion replaced by the guarantee it stood for — that the admitted actor reaches the helper, and that the resolved identity is the demo actor's rather than the fallback.

// mechanism — cannot survive consolidation
expect(requireAuthenticatedAthenaUserWithCtx).not.toHaveBeenCalled();

// behavior — what that line actually protected
expect(requireAuthenticatedAthenaUserWithCtx).toHaveBeenCalledWith(
  expect.objectContaining({ operationAdmission: expect.objectContaining({
    actor: expect.objectContaining({ athenaUserId: "demo-user-1" }) }) }));
expect(requireOrganizationMemberRoleWithCtx).not.toHaveBeenCalledWith(
  expect.anything(), expect.objectContaining({ userId: FALLBACK_AUTH_USER_ID }));

Validation

  • Full webapp suite: 6829 passing across 661 files, zero failures.
  • bunx tsc --noEmit -p packages/athena-webapp/tsconfig.json: exit 0.
  • lint:convex:changed: exit 0. lint:frontend:changed: exit 0 (14 pre-existing react-refresh warnings, non-blocking).
  • Every replaced assertion was falsified: admission was deliberately broken and the new assertion confirmed to fail. Telemetry was falsified two independent ways, catching at two different assertions.
  • Three replacements were initially vacuous because the demo actor and the normal-user fallback shared an id, making fallthrough undetectable by construction. Those fixtures were given distinct identities (dailyClose, staffCredentials) or additional assertions on the userId actually threaded through (terminals) until each failed under break.
  • Residual risk not covered by tests: no browser validation was performed in this session. The originating session also deferred browser validation.

What Intentionally Did Not Change

  • Manager approval for high-variance manual stock adjustments remains disabled. requiresStockAdjustmentApproval returns false unconditionally; this was confirmed as deliberate product intent, and the function now takes no arguments so no caller is misled into thinking type or variance affect the outcome. Variance is still measured and reported via hasHighStockAdjustmentVariance.
  • The synced-sale inventory review UI in OperationsQueueView remains removed, as the branch has it. Consequence, recorded deliberately: resolveSyncedSaleInventoryReviewGroup is now a live, admission-gated, fully-tested convex write path with no frontend entry point. It is the only symbol newly orphaned by this branch.
  • sharedDemo/provision.ts keeps main's varianceApprovalThreshold of 100 from #695, merged with the branch's contact and receipt fields.
  • storeFront/onlineOrder.ts keeps the branch's check ordering; the two checks are mutually exclusive by actor kind, so ordering is observationally neutral.

Next-Time Guidance

  • When a WIP's intent is load-bearing, read the author's session transcript before inferring intent from the diff. Extract only the human turns. Four rounds of rework here were spent inferring what one logged sentence stated outright.
  • Before claiming a merge lost something, verify the base: git rev-parse 'stash@{0}^' and git log --oneline <base>..origin/main. If the commit you think was lost is an ancestor of the base, the deletion was deliberate.
  • Never validate a cross-cutting refactor with a name-filtered test run. The originating session's checks were all -t filtered, which is precisely why six files' stale assertions were never exercised.
  • When a test fails after a refactor, classify before editing: does it assert behavior, or code arrangement? Only call-shape assertions may be rewritten, and only into behavioral ones — then falsify the replacement by breaking what it guards.
  • Reviewer follow-up worth tracking: publicQuery.ts attaches admission with Object.assign on the existing ctx rather than cloning. A nested admitted query reusing that ctx would short-circuit on the outer actor instead of re-deriving. Benign today because both resolve to the same user; it stops being benign if actor impersonation is ever introduced.
