EOD Automation: Safe Outcomes and Manual-Close Alerts

How Athena preserves automation evidence, tells admins when a day remains open, and avoids repeated cron email delivery.

Delivery candidateBranch codex/fix-eod-admin-alertsMerge pendingProduction deploy deferredLocal root alignment deferred

Executive Summary

This candidate fixes the EOD outcome ledger boundary and applies an explicit notification policy: every failed run and every skipped run except classification completed sends an action-required admin email. Applied and prepared reports keep their existing paths.

Action-required delivery is reserved per recipient and store-day before the email provider is called. Successful sends are terminal, while failed or expired reservations can retry without every cron cycle resending the same alert.

Why It Mattered

The production incident began as a safe refusal to close: source evidence was incomplete, so Athena correctly did not complete the day. A second ledger patch then supplied eventIds as undefined; Convex removed the required field during validation and the visible outcome became a failure instead of a trustworthy skip.

The operational consequence is the same whenever automation does not close the day: an operator must open EOD Review and finish manually. Silent skips leave admins without the instruction needed to close the operating day.

Mental Model

Treat EOD automation as a close attempt followed by an outcome router. The router trusts the recorded outcome and decision classification: failed and non-completed skipped outcomes require manual-close guidance. It does not independently query close state. A delivery ledger sits in front of the provider so retries are safe.

  • Applied: the day closed; send the completed daily report.
  • Prepared: manager review is expected; send the prepared report.
  • Skipped and not classified completed: policy requires action-required guidance.
  • Failed: policy requires action-required guidance and support context.
  • Completed classification: no email because the day is already closed.

Before and After

Before, only fresh applied and prepared results entered the manager-report path. Skips and failures could leave an open day without admin notification, and the safe-skip outcome patch could erase the required eventIds evidence.

After, optional patch fields are omitted unless a concrete value exists. The result router maps applied, prepared, skipped, and failed outcomes; every failure and every skip not classified completed carries the automation run ID into the action-required email path.

eventIds: undefined        -> omit the patch field
skipped + completed       -> silent
skipped + day still open -> action-required email
failed                    -> action-required email

Delivery and Retry Boundary

For each normalized admin email, Athena reserves an eod_action_required delivery keyed by domain, action, store, operating date, kind, and recipient. A pending reservation holds a five-minute lease. Sent is terminal; failed or expired reservations can be reclaimed and increment attemptCount.

  • Provider failure marks the reservation failed and throws so a later scheduler run can retry.
  • Recipients are processed sequentially; recipients already marked sent are suppressed on retry.
  • The design is intentionally at-least-once at the provider boundary: acceptance followed by a failed sent-marker mutation can permit a later duplicate after lease expiry.
  • Dedupe is once per recipient and store-day, so a later reason change that day does not send a second action-required email after success.

Operator Message

Skipped automation says Athena did not close the operating day and directs admins to open EOD Review, resolve remaining items, and complete the close.

Failed automation says Athena could not complete the automated check, directs admins to close manually, and advises contacting support if the workflow is unavailable. Raw backend error text is not exposed as the primary instruction.

What Did Not Change

  • EOD eligibility, review thresholds, and close-command mechanics are unchanged.
  • Applied and prepared daily-report behavior remains intact.
  • ADMIN_EMAILS remains the recipient source; store-scoped recipient management is not added.
  • Scheduled dispatch remains production-stage only.
  • This delivery does not rerun the affected production day, align local root main, or deploy production services.

Next-Time Guidance

  • Diagnose automation outcomes from the durable run and source-read evidence before treating a safe skip as a business-policy failure.
  • Never patch a required Convex field with undefined; conditionally omit optional patch inputs.
  • Route notifications by the operator action still required, not by a narrow list of backend classifications.
  • Reserve external notifications before provider I/O and make retry semantics explicit in tests.
  • When extending action-required mail, preserve the completed-classification silence and the per-recipient store-day dedupe boundary.

The router does not query close state; it suppresses a skipped run only when decision evidence is classified completed.
2. Why is eventIds omitted from the outcome patch when no new IDs exist?
Passing undefined can remove the required Convex field during validation; omission preserves the existing array.
3. Which skipped outcome remains silent?
A completed classification means no operator close action remains.

It reserves and records per-recipient action-required delivery state.
5. What happens after a provider send fails?
Failed reservations are retryable; successful sent reservations are terminal.
6. What does a skipped-action email ask the admin to do?
The message is operational: resolve remaining items and complete EOD Review.
7. Which behavior intentionally remains unchanged?
Existing applied and prepared reports remain; skip/failure action messaging is additive.
8. What residual duplicate edge remains?
After lease expiry, the system may retry because it cannot know the provider accepted the prior email.
9. Where should future notification routing logic focus?
The durable product boundary is whether someone must still close the day.
10. What tradeoff follows from deduping action-required mail once per recipient and store-day?
The dedupe key does not include the run or reason, so a successful alert suppresses later same-day action-required outcomes for that recipient.
