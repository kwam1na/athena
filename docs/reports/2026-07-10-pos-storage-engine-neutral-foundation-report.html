Athena POS Storage Engine-Neutral Foundation

How POS moved from IndexedDB embedded across the application to one durable semantic contract with IndexedDB as the first native engine.

Branch codex/v26-1004-pos-engine-neutral-foundationLinear V26-1004 through V26-1011Ready for PR validationNot mergedNo production deploy

Executive Summary

Athena now exposes local-first POS persistence as high-level application outcomes instead of IndexedDB mechanics. Cashier, register, expense, catalog, sync, authority, provisioning, recovery, and health paths use one stable runtime-selected durable engine generation.

IndexedDB remains the only production engine today. The change creates the seam, compatibility proofs, bounded operations, and failure behavior needed for a future SQLite or worker engine to use its native schema, queries, transactions, and coordination without emulating object stores.

Why It Mattered

Before this work, production consumers constructed IndexedDB stores, checked browser globals, inferred types from the concrete implementation, and could bypass asynchronous initialization. Physical layout, logical records, hot-path scans, maintenance, migration, and health were coupled to the first engine.

  • A local cashier command is successful only after non-reconstructable facts are durably committed.
  • A future engine must preserve POS semantics without inheriting IndexedDB object stores, cursors, callbacks, or Web Locks.
  • PR #642 authority is the baseline architecture for every terminal on the new build, not a rollout option.

Mental Model

Think of the ordinary application contract as a list of promises: append this event durably, apply this exact authority observation atomically, return this scoped page, replace this current mapping, and summarize this ledger. Safe-clear assessment and portable migration are separate lifecycle-foundation semantics; engine-specific coordination stays below composition.

Each engine chooses how to keep those promises. IndexedDB uses compound indexes, object-store transactions, and browser locks. SQLite could use SQL indexes, transactions, and worker coordination. The application never asks either engine for a cursor or transaction handle.

POS consumer -> stable runtime facade -> selected ready engine
                                      |
                                      +-- IndexedDB today
                                      +-- SQLite/worker later

Before and After

  • Before: multiple consumers selected or constructed IndexedDB. After: one application-root runtime selects and publishes one durable generation.
  • Before: concrete-store types crossed layers. After: posLocalStoreTypes.ts is the sole semantic type authority and a structural test rejects production concrete-store imports.
  • Before: upload, mapping, projection, and health work could enumerate broad history. After: exact IDs, scoped opaque pages, compound indexes, and a native count/oldest summary bound normal operations.
  • Before: physical and business versions moved together. After: IndexedDB physical v10, logical record v1, and portable envelope versions evolve independently.
  • Before: maintenance and failures lacked coherent operational evidence. After: readiness, durable failures, persistence, quota pressure, ledger pressure, migration, maintenance, and last commit are redacted and observable.

Failure Boundaries

  • Ephemeral engines are rejected in production composition; there is no implicit memory fallback.
  • Concurrent initialization shares one attempt. Failed attempts can retry with a new generation; late or disposed stores are never published.
  • Blocked IndexedDB upgrades settle after an actual blocked event, while healthy long upgrades are not timed as contention.
  • Failed durable writes publish no partial sequence, event, mapping, or authority state and keep readiness unavailable until a later durable commit succeeds.
  • Clear and migration fail closed when inspection, capacity, integrity, semantics, or activation is uncertain. Failed maintenance remains visible but does not deadlock ordinary operations.
  • Persistence denial and quota estimates are advisory. Diagnostics never grant cashier, drawer, staff, or terminal authority.

PR #642 Semantics Preserved

The storage refactor changes where durable outcomes live, not who owns register authority.

  • Convex still owns lifecycle and mapping revisions.
  • The exact current mapping is compared with terminal-authenticated evidence, then mapping revision and drawer authority commit all-or-neither.
  • Offline, loading, or unauthorized reads preserve last-known durable authority and expose authorization state without mutating it. Invalid candidates or snapshots, mapping invalidation, and persistence failure enter owned failure or repair outcomes.
  • Server authority and independent local-review authority remain separate.
  • Cashier commands continue to trust the refreshed durable local projection; heartbeat and health remain diagnostic only.
  • The authority refresh bridge remains stable across equivalent terminal objects.

Compatibility and Migration

  • A real fake-indexeddb v9 database populated across every logical section upgrades in place to v10 and proves keys and values survive.
  • Legacy logical transformation and the logical-version marker commit atomically and roll back together on failure.
  • Unknown future logical versions fail non-destructively.
  • Portable migration coordination is foundation and test coverage only; production alternate-engine activation and retained-source destruction are intentionally absent.
  • Integrity manifests detect incomplete or corrupted snapshots but do not claim authenticity.

How to Extend It Safely

  • Implement PosLocalStorePort outcomes directly in the new engine; do not reuse the IndexedDB adapter as the universal abstraction.
  • Use native exact lookups, scoped pagination, counts, indexes, transactions, and coordination while preserving semantic ordering and atomicity.
  • Add the new engine to runtime composition only after open, compatibility, migration, retry, dispose, failure, and conformance proofs are complete.
  • Keep physical, logical, and envelope versions independent.
  • Test compatibility with a populated previous real layout, not only mocked upgrade callbacks.
  • Keep health allowlisted and payload-free, and keep diagnostic evidence outside command authority.

Intentionally Not Changed

  • No SQLite schema, VFS, worker topology, RPC protocol, lock strategy, transaction strategy, or digest algorithm is prescribed.
  • No alternate-engine activation UI, automatic fallback, settled-history deletion, secure-erasure claim, or downloadable export was added.
  • No #642 rollout, reset-compatibility program, cloud authority redesign, or sync contract redesign was added.
  • IndexedDB remains the production-selected engine until a separately reviewed engine implementation exists.

Quiz: Pass Required

You must score at least 8 out of 10 to pass.

1. What is the cross-engine abstraction?
The application owns semantic outcomes. Each engine retains native storage and transaction mechanics.
2. What does the runtime publish to consumers?
Consumers dispatch only after one selected durable generation finishes open and logical initialization.
3. How should a future SQLite engine implement the port?
Engine neutrality preserves outcomes, not the first engine's mechanics.
4. What happens after a durable POS write fails?
Failed durable commits block the affected command and remain operationally visible; no partial state or implicit fallback is allowed.
5. Why is ledger pressure a native summary outcome?
Bounded health uses count and oldest timestamp only, with no payload enumeration.
6. What must happen when a clear attempt finds protected POS records?
Protected records block deletion and preserve evidence. Maintenance state remains observable, but only actively owned exclusive maintenance fences ordinary operations.
7. Which #642 rule remains authoritative?
The refactor preserves #642's exact mapping race protection and durable local authority boundary.
8. How was v9 compatibility proven?
The test exercises the actual IndexedDB upgrade path with representative records and indexes.
9. What should a developer do before selecting another production engine?
A new engine must earn selection through the same durable outcomes and lifecycle proofs.
10. What happens if alternate-engine activation outcome is ambiguous?
Unknown activation cannot authorize deletion or automatic fallback; both sides remain available for an owned recovery decision.
