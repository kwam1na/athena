Executive Summary

This delivery improves how Cash Controls explains POS register-session activity. The ledger remains a bounded, full-admin evidence surface; it now presents item, payment-method, and opening-float context in operator language.

The delivery is still a candidate: merge and root alignment are pending. Production deployment is outside this requested delivery.

Problem and Mental Model

A register session is a replayable evidence stream, not a sales or closeout mutation authority. Core sync answers settlement status; the activity ledger explains what the terminal did in local sequence.

Before this refinement, activity rows could say a cart item was added when quantity was zero, payment context was incomplete, and opening-float evidence was less useful to an operator reviewing a drawer.

local POS event -> safe metadata allowlist -> activity row -> full-admin session query -> Cash Controls review

What Changed

  • Activity summaries expose safe item, SKU, quantity, unit-price, opening-float, and deduplicated payment-method context.
  • Quantity-zero cart events are labeled as removals.
  • The activity table uses a smaller operational filter set, compact status dots, and no duplicate evidence-link chips.
  • An authorized historical closeout correction may reopen even when a later drawer is active; the reopened session remains in closing state and is not sale-usable.
  • Cash Controls and adjacent operations surfaces receive presentation-only clarity fixes, with focused regression coverage.

Failure Boundaries and Intentional Non-Changes

  • Raw payloads, PINs, proof tokens, customer data, and arbitrary payment internals remain excluded by positive metadata allowlists.
  • Activity reporting remains independent from core sync settlement and preserves mapping_pending uncertainty.
  • No public API, schema, transaction, terminal binding, or sales write path was added.
  • The detailed review remains session-scoped, bounded, and full-admin authorized.

Next-Time Guidance

  • Add a new activity detail only when it is derived from the explicit safe allowlist on both browser and Convex boundaries.
  • Treat zero quantity as an event semantic that must be reflected consistently in labels and tests.
  • Keep historical correction authority separate from active-drawer sale authority; prove the coexistence case with a lifecycle test.
