# VPS Production Runtime

This runbook captures the production VPS shape for `wigclub.store` and the steps needed to recreate it on any Ubuntu-based VPS.

## Topology

- Cloudflare Tunnel routes public hostnames to the VPS.
- nginx listens on port `80`.
- Static builds are served from versioned directories under `/root/athena`.
- Convex hosts the backend API and data layer.
- Valkey runs locally on the VPS.
- `valkey-proxy-server` runs under PM2 and binds to `127.0.0.1:3000`.

## Hostname Routing

| Hostname | VPS target |
| --- | --- |
| `wigclub.store` | nginx static storefront |
| `www.wigclub.store` | nginx static storefront |
| `athena.wigclub.store` | nginx static Athena admin app |
| `qa.wigclub.store` | reserved QA endpoint; nginx returns `204` |
| `athena-qa.wigclub.store` | nginx reverse proxy to Athena Vite dev server on `127.0.0.1:5175` |
| `api.wigclub.store` | nginx reverse proxy to prod Convex HTTP |
| `dev.wigclub.store` | nginx reverse proxy to dev Convex HTTP |
| `cache.wigclub.store` | Cloudflare Tunnel to `http://localhost:3000` |

Do not expose the cache proxy through the VPS public interface. The service has cache read, write, and invalidation endpoints, so the expected runtime boundary is loopback plus Cloudflare Tunnel.

## Bootstrap A New VPS

Run the setup script as `root` on a fresh Ubuntu VPS:

```bash
curl -fsSL https://raw.githubusercontent.com/kwam1na/athena/main/scripts/setup-production-vps.sh | bash
```

For a local checkout copied to the machine:

```bash
scripts/setup-production-vps.sh
```

The script installs git, nginx, rsync, Valkey, Node.js 22, PM2, writes `/etc/nginx/conf.d/wigclub.conf`, creates `/root/athena`, enables systemd services, and registers PM2 startup.

## nginx Configuration

The VPS has one Athena-managed nginx file:

```text
/etc/nginx/conf.d/wigclub.conf
```

Ubuntu's default `/etc/nginx/nginx.conf` includes `/etc/nginx/conf.d/*.conf`, so no `sites-enabled` symlink is required for Athena. The setup script removes `/etc/nginx/sites-enabled/default` and writes the full app config into `conf.d/wigclub.conf`. `/etc/nginx/sites-available/default` may still exist on disk as the package default, but it should not be enabled.

Do not hand-maintain separate app files under `/etc/nginx/sites-enabled`. If the hostnames or upstreams need to change, update `scripts/setup-production-vps.sh`, run it with the appropriate environment variables, then validate and reload nginx.

The generated server blocks are:

| nginx server_name | Behavior |
| --- | --- |
| `wigclub.store www.wigclub.store` | Serves `/root/athena/storefront/current` with SPA fallback |
| `athena.wigclub.store` | Serves `/root/athena/athena-webapp/current` with SPA fallback |
| `qa.wigclub.store` | Reserved endpoint that returns `204` |
| `athena-qa.wigclub.store` | Proxies to the QA Vite dev server at `127.0.0.1:5175`, including websocket upgrade headers |
| `api.wigclub.store` | Proxies to the production Convex HTTP site with CORS handling |
| `dev.wigclub.store` | Proxies to the dev Convex HTTP site with CORS handling |

nginx only listens on local HTTP port `80`. Public TLS is terminated at Cloudflare and traffic reaches nginx through Cloudflare Tunnel. Do not add certificate files to nginx for the tunnel-based deployment unless intentionally moving away from Cloudflare Tunnel.

After bootstrap, verify the config topology:

```bash
nginx -t
nginx -T | grep -E 'configuration file|server_name|root |proxy_pass'
ls -la /etc/nginx/conf.d /etc/nginx/sites-enabled
```

Expected results:

- `/etc/nginx/conf.d/wigclub.conf` exists.
- `/etc/nginx/sites-enabled` is empty or contains only intentionally added non-Athena config.
- `nginx -T` shows the Athena hostnames from `wigclub.conf`.

Verify each local nginx route before checking Cloudflare:

```bash
curl -sS -I -H 'Host: wigclub.store' http://127.0.0.1/
curl -sS -I -H 'Host: athena.wigclub.store' http://127.0.0.1/
curl -sS -I -H 'Host: qa.wigclub.store' http://127.0.0.1/
curl -sS -I -H 'Host: athena-qa.wigclub.store' http://127.0.0.1/
```

If the config changes manually during incident response, copy the working change back into `scripts/setup-production-vps.sh` so a new VPS can reproduce it.

## Cloudflare Tunnel Credentials

Cloudflare Tunnel credentials are intentionally not generated by the setup script because they are secrets. A tunnel credential is the JSON file that lets `cloudflared` run a specific Cloudflare Tunnel from this VPS.

There are three valid ways to provide the credential:

1. Reuse the existing production tunnel by securely copying its JSON credential file from the current VPS into `/etc/cloudflared/` on the new VPS. Keep the same `tunnel:` id and `credentials-file:` path in `config.yml`.
2. Create a new named tunnel from the new VPS. `cloudflared tunnel create <name>` will generate a new tunnel id and JSON credential file, usually under the invoking user's `.cloudflared` directory. Move that JSON file into `/etc/cloudflared/`, then reference it from `config.yml`.
3. Use a Cloudflare-provided tunnel token install command if the tunnel is managed from the Cloudflare dashboard. In that case, follow Cloudflare's generated command instead of using a `credentials-file:` entry.

For a locally managed tunnel, install `cloudflared`, create `/etc/cloudflared/config.yml`, and point `credentials-file:` at the credential JSON:

```yaml
tunnel: <tunnel-id>
credentials-file: /etc/cloudflared/<tunnel-id>.json

ingress:
  - hostname: wigclub.store
    service: http://localhost:80
  - hostname: www.wigclub.store
    service: http://localhost:80
  - hostname: athena.wigclub.store
    service: http://localhost:80
  - hostname: qa.wigclub.store
    service: http://localhost:80
  - hostname: athena-qa.wigclub.store
    service: http://localhost:80
  - hostname: api.wigclub.store
    service: http://localhost:80
  - hostname: dev.wigclub.store
    service: http://localhost:80
  - hostname: cache.wigclub.store
    service: http://localhost:3000
  - service: http_status:404
```

The credential JSON must not be committed to the repo, pasted into tickets, or printed in logs. It should be readable only by `root` or the service user that runs `cloudflared`.

Then enable the tunnel:

```bash
systemctl enable --now cloudflared
```

When adding a new hostname to an existing named tunnel, also create the Cloudflare DNS route:

```bash
cloudflared tunnel route dns <tunnel-id> qa.wigclub.store
cloudflared tunnel route dns <tunnel-id> athena-qa.wigclub.store
```

## Deploy Apps

`scripts/deploy-vps.sh` is the authoritative deploy entrypoint for VPS-hosted surfaces. It uses a shared Git checkout on the VPS at `/root/athena/repo`, builds on the VPS, writes static builds into timestamped version directories, and atomically moves the `current` symlink.

Before first use, add the VPS SSH key to GitHub with read access to this repo.

On the VPS, create or locate the public key:

```bash
test -f /root/.ssh/id_ed25519.pub || ssh-keygen -t ed25519 -C athena-vps -f /root/.ssh/id_ed25519
cat /root/.ssh/id_ed25519.pub
```

Add that public key to GitHub as a deploy key for `kwam1na/athena`, with read-only access unless the VPS needs to push. Then verify from the local checkout:

```bash
scripts/deploy-vps.sh check-git
```

Or verify directly on the VPS:

```bash
ssh -T git@github.com
```

Deploy commands:

```bash
scripts/deploy-vps.sh status
scripts/deploy-vps.sh versions athena
scripts/deploy-vps.sh versions storefront
scripts/deploy-vps.sh athena
scripts/deploy-vps.sh storefront
scripts/deploy-vps.sh valkey-proxy
scripts/deploy-vps.sh qa
scripts/deploy-vps.sh full-prod
scripts/deploy-vps.sh all
scripts/deploy-vps.sh check-git
```

Convex production deploys still run from the local checkout through the same entrypoint:

```bash
scripts/deploy-vps.sh convex-prod
```

The interactive helper delegates deploy choices to the same script:

```bash
./manage-athena-versions.sh
```

Choose `deploy`, then the target. Rollback and version deletion remain interactive because they are intentionally human-selected operations.

The active static paths are:

```text
/root/athena/athena-webapp/current
/root/athena/storefront/current
```

Each new static version includes deployment metadata:

```text
/root/athena/<app>/versions/<timestamp>/fun-name.txt
/root/athena/<app>/versions/<timestamp>/deploy.json
```

`deploy.json` records the app, package path, version timestamp, fun name, Git SHA, deploy ref, and UTC deployment time. Keep the timestamped directory name as the serving version id; use the Git SHA metadata for audit and diagnosis.

## Production Rollback

Static app rollback is first-class through `scripts/deploy-vps.sh`. It does not rebuild, redeploy Convex, or change Valkey. It only moves the selected app's `current` symlink back to an existing timestamped version under `/root/athena/<app>/versions`.

List deployed versions:

```bash
scripts/deploy-vps.sh versions athena
scripts/deploy-vps.sh versions storefront
```

Roll back to the most recent non-current version:

```bash
scripts/deploy-vps.sh rollback athena previous
scripts/deploy-vps.sh rollback storefront previous
```

Roll back to an explicit timestamp:

```bash
scripts/deploy-vps.sh rollback athena 20260502063412
scripts/deploy-vps.sh rollback storefront 20260502063412
```

Aliases are available for direct use:

```bash
scripts/deploy-vps.sh rollback-athena previous
scripts/deploy-vps.sh rollback-storefront previous
```

After rollback, verify the local nginx route before checking Cloudflare:

```bash
curl -sS -I -H 'Host: athena.wigclub.store' http://127.0.0.1/
curl -sS -I -H 'Host: wigclub.store' http://127.0.0.1/
```

`.github/workflows/athena-production-rollback.yml` exposes the same rollback path as a manual GitHub Actions workflow. It accepts `app` and `version` inputs, delegates to `scripts/deploy-vps.sh rollback`, and smoke-checks nginx on the VPS. The workflow targets the `production` GitHub Environment, so configure required reviewers there if production rollback should require human approval.

The production rollback workflow needs these GitHub environment or repository secrets:

| Secret | Purpose |
| --- | --- |
| `ATHENA_VPS_HOST` | VPS hostname or IP address reachable from GitHub Actions |
| `ATHENA_VPS_USER` | SSH user for the VPS; use `root` unless the VPS has a dedicated deploy user |
| `ATHENA_VPS_SSH_PRIVATE_KEY` | Private key that can SSH to the VPS user |

## QA Dev Server

`athena-qa.wigclub.store` is a remotely hosted Athena dev frontend pointed at the shared dev Convex deployment. `qa.wigclub.store` remains reserved for the broader QA endpoint namespace and should not serve the Athena dev server.

```text
athena-qa.wigclub.store
  -> Cloudflare Tunnel
  -> nginx
  -> 127.0.0.1:5175 Vite dev server
  -> https://jovial-wildebeest-179.convex.cloud / .site
```

Deploy or refresh QA:

```bash
scripts/deploy-vps.sh qa
```

The script refreshes `/root/athena/repo`, installs Bun if needed, runs `bun install --ignore-scripts`, and starts the Athena webapp through PM2 as `athena-qa`.

Expected PM2 command shape:

```bash
VITE_CONVEX_URL=https://jovial-wildebeest-179.convex.cloud \
VITE_API_GATEWAY_URL=https://jovial-wildebeest-179.convex.site \
VITE_STOREFRONT_URL=https://wigclub.store \
pm2 start bun --name athena-qa -- run dev -- --host 127.0.0.1 --port 5175 --strictPort
```

Because this exposes a Vite dev server through a public hostname, protect `athena-qa.wigclub.store` with Cloudflare Access or equivalent edge auth before sharing it broadly.

## Automatic QA Deploys

`.github/workflows/athena-qa-deploy.yml` deploys QA automatically on every push to `main`. In normal use that means merged PRs refresh `athena-qa.wigclub.store` without an operator running the deploy script locally.

The workflow intentionally delegates to the same authoritative deploy path:

```bash
DEPLOY_REF="$GITHUB_SHA" REMOTE=athena-qa-vps scripts/deploy-vps.sh qa
```

Configure these GitHub environment or repository secrets before relying on the workflow:

| Secret | Purpose |
| --- | --- |
| `ATHENA_QA_HOST` | VPS hostname or IP address reachable from GitHub Actions |
| `ATHENA_QA_USER` | SSH user for the VPS; use `root` unless the VPS has a dedicated deploy user |
| `ATHENA_QA_SSH_PRIVATE_KEY` | Private key that can SSH to the VPS user |

The workflow targets the `qa` GitHub Environment. Keep that environment free of required reviewers if QA should deploy without human approval after every merge.

The VPS still needs its own GitHub deploy key configured because the deploy script builds from the remote checkout at `/root/athena/repo`. The Actions key gets the workflow into the VPS; the VPS deploy key lets the VPS fetch `kwam1na/athena`.

The workflow smoke check runs against nginx on the VPS:

```bash
curl -fsS -I -H "Host: athena-qa.wigclub.store" http://127.0.0.1/
```

That keeps the deploy check independent of Cloudflare DNS propagation and Cloudflare Access policy.

## Hardening Checks

After setup or proxy deploy, verify the proxy is loopback-only:

```bash
ss -tulpn | grep ':3000'
curl -sS http://127.0.0.1:3000/health
curl -sS --max-time 3 http://<vps-public-ip>:3000/
```

Expected result:

- `ss` shows Node bound to `127.0.0.1:3000`.
- localhost health succeeds.
- public IP port `3000` does not respond.

Check service state:

```bash
systemctl is-active nginx cloudflared valkey-server
pm2 list
```

For QA, verify:

```bash
ss -tulpn | grep ':5175'
curl -sS -I -H 'Host: qa.wigclub.store' http://127.0.0.1/
curl -sS -I -H 'Host: athena-qa.wigclub.store' http://127.0.0.1/
pm2 describe athena-qa
```
