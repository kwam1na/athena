Problem Frame

Athena's intelligence layer has durable runs, snapshots, artifacts, visibility modes, and source refs. The missing foundation is upstream: surfaces can collect useful raw signals, but there is no shared way for those signals to become intelligence-ready context without coupling capabilities to one surface's raw analytics shape.

This plan builds new primitives only: no legacy analytics migration, no tracking-session read model, and no full Ask Athena UI.

Requirements

  • Define a shared context event envelope with surface, event id, schema version, scope, actor/session refs, source refs, visibility, origin, synthetic marker, and idempotency key.
  • Keep surface event definitions surface-owned, with adapter-specific payload validation and compiler behavior.
  • Persist new context events in a separate append-only Convex table; legacy analytics stays out of scope.
  • Make Convex/server validation authoritative.
  • Store minimized typed payloads and compact evidence, not broad raw data or sensitive material.
  • Compile bounded context bundles with source refs, data windows, snapshot hashes, freshness state, omitted-evidence markers, and redaction metadata.
  • Require storefront-derived intelligence to consume compiled bundles rather than raw storefront rows.
  • Enforce visibility and redaction at server compile/read boundaries.
  • Distinguish recorded, rejected, uncompiled, partial, stale, unavailable, and hidden-by-permission states.
  • Exclude synthetic monitor traffic from business/customer intelligence by default.
  • Use mutation-first Convex patterns and bounded public reads.
  • Treat client-supplied actor, session, customer, cart, checkout, order, and subject refs as claims until server-derived or corroborated.
  • Define v1 retention/deletion semantics before production writes: retention class, expiration/archive fields, and anonymization behavior.

Scope

  • No legacy storefront analytics normalization, migration, backfill, or compatibility taxonomy.
  • No generic tracking-session table or read model in v1.
  • No POS, inventory, operations, service, or automation adapter implementation beyond contract examples.
  • No autonomous domain mutations from intelligence output.
  • No provider SDK or model capability work beyond consuming the compiled bundle.

Key Decisions

  • Place browser-safe primitives and tracker helpers in packages/athena-webapp/shared/intelligence/.
  • Place Convex persistence and compilers in packages/athena-webapp/convex/contextTracking/; convex/intelligence/ consumes compiled bundles.
  • Create a separate append-only contextEvent table for new events.
  • Do not create trackingSession in v1; compilers group event streams for specific bundles.
  • For storefront v1, use a thin HTTP API wrapper and an Athena HTTP route that derives trusted scope before calling the append helper.
  • Make appendContextEvent the durable server helper, backed by registry validation, subject corroboration, and idempotency collision checks.
  • Keep indexes tenant/scope-first and tied to proven reads, including primary subject/session, status, occurred time, retention expiration, and idempotency.
  • Use source refs rich enough to explain surface/event version, subject refs, time window, redaction, omitted evidence, and synthetic marker.
  • Keep operational events, workflow traces, automation runs, and domain ledgers in their existing roles.
  • Use ephemeral ContextBundle compiler outputs copied into immutable intelligenceContextSnapshot rows in v1; do not add a durable bundle table yet.

Implementation Units

U1. Shared context primitive types and schema validators

Define common envelope, refs, visibility, status, retention fields, idempotency hashes, denormalized compiler keys, and bundle vocabulary without surface-specific semantics. Modify convex/schemas/intelligence.ts and convex/schema.ts; add shared tests.

U2. Surface definition registry and authoritative event append boundary

Add the storefront HTTP route, appendContextEvent, registry entries, idempotency collision semantics, subject corroboration, schema version handling, and normalized non-compilable rejection behavior. Public functions stay thin; internal functions compile/project.

U3. Context compiler and bundle contract

Compile append-only events into permission-aware bundles with source refs, hidden counts, omitted evidence, freshness, quality flags, and snapshot hash. Storefront-derived insights consume these bundles.

U4. Storefront first surface adapter

Define a small v1 storefront event set for browse, product intent, cart/bag, checkout milestone, adapter failure, and synthetic health events. Emit through the storefront context HTTP API wrapper, not legacy analytics.

U5. Intelligence snapshot and artifact integration

Copy compiled bundle fields into intelligenceContextSnapshot, update capabilities/actions.ts away from raw storefront analytics reads, and enforce artifact reads against recorded visibility and source refs.

U6. Adapter authoring documentation and gates

Document the envelope, registry responsibilities, compiler responsibilities, visibility, synthetic handling, event versioning, and test checklist for future surfaces.

Validation

  • Primitive validator and schema tests.
  • Registry version acceptance and unknown-event rejection tests.
  • Idempotent append tests.
  • Same-key/different-hash idempotency conflict tests.
  • Cross-customer/session/cart/checkout/order subject spoofing tests.
  • Retention class, expiration/archive, and deletion/anonymization tests.
  • Rejected or unauthorized rows are non-compilable and unauthorized payloads are not retained.
  • Compiler freshness, stale-state, and synthetic handling tests.
  • Permission/redaction and hidden-source tests.
  • Storefront browse, cart, checkout, failure, and synthetic fixture tests.
  • Non-storefront contract fixture proving the envelope is not storefront-shaped.
  • Integration test from storefront event append to compiled bundle to intelligenceContextSnapshot.
  • bun run graphify:rebuild and bun run graphify:check after code changes.

Risks

RiskMitigation
Primitive layer becomes a second business ledger.Store minimized event/evidence data and keep source ledgers authoritative.
Storefront taxonomy leaks into the base contract.Put taxonomy and payload validators in adapter files only.
Event volume creates expensive reads.Use bounded windows and tenant/scope-first indexes tied to proven reads.
Hidden customer, payment, staff, or support data leaks.Enforce server-side visibility in append, compile, and read paths.
Client-supplied subjects poison another customer or order's context.Treat refs as claims until server-derived or corroborated and test spoofing paths.
Append-only customer/session behavior becomes an indefinite ledger.Add retention class, expiration/archive fields, deletion/anonymization semantics, and a retention index before production writes.
Bad client versions poison context.Server registry rejects unsupported versions and payloads.
Stale context looks current.Record windows, hashes, stale triggers, TTLs, and artifact stale/superseded state.
