feat: Move staff PIN collection to four digits

Created 2026-05-15 | Type: feat | Status: active | Source: markdown plan
Athena will require four-digit staff PIN entry in organization members, staff authentication, and manager elevation/approval flows. The plan centralizes the staff PIN policy in one reusable component and adds credential-level throttling at the server authentication boundary.

Scope

  • Accept and save exactly four digits for staff PIN setup and reset.
  • Collect exactly four digits in shared staff authentication and inherited manager auth surfaces.
  • Throttle repeated failed staff PIN authentication attempts on the credential record.
  • Do not migrate existing credential hashes or local verifier records.
  • Do not change roles, approval policy, audit behavior, or local staff authority proof semantics.

Key Decisions

  • Add a staff-specific PIN component that wraps the existing visual PinInput.
  • Use that component in both StaffAuthenticationDialog and organization members PIN setup/reset.
  • Keep pinHash, localPinVerifier, manager elevation, and approval proof contracts unchanged.
  • Add failed-attempt and lockout metadata to staff credentials, and require full-admin access before public create/update mutations can reset PIN or lockout state.

Implementation Units

U1. Create shared four-digit staff PIN collection

Add packages/athena-webapp/src/components/staff-auth/StaffPinInput.tsx to centralize numeric sanitization, four-digit length, and slot rendering.

Tests: authentication dialog coverage for four-digit submit, incomplete PIN, and pasted mixed input.

U2. Apply four-digit collection to staff and manager authentication

Update StaffAuthenticationDialog to remove six-digit assumptions so cashier, staff, command approval, and manager elevation surfaces inherit the new rule.

Tests: staff auth four-digit submission, four-digit error copy, manager elevation credential forwarding.

U3. Apply four-digit setup/reset in organization members

Update StaffManagement.tsx setup/reset dialog to collect and confirm four digits using the shared component.

Tests: save 1234 to both server hash and local verifier paths, mismatch handling, incomplete PIN error.

U4. Protect four-digit staff authentication with credential throttling

Update staffCredentials.ts and the staff credential schema so failed PIN attempts lock the active credential temporarily, authorized resets clear the lockout, and public authentication mutations require authenticated store access.

Tests: repeated wrong attempts, success reset, authorized reset clear, unauthorized reset block, unauthorized public auth block, and manager approval proof lockout.

Risks

RiskMitigation
Existing six-digit PIN users cannot enter old PINs after deploy.Accept as part of the policy change; reset affected staff PINs through organization members.
A caller keeps a six-digit assumption.Search for hard-coded six-digit PIN checks before final validation.
Four-digit PINs increase brute-force feasibility.Enforce credential-level failed-attempt lockout across staff authentication and manager approval proof paths.
PIN reset clears lockout without authorization.Require full-admin organization membership before public staff credential create/update mutations can create, reset, or clear credential state.
Public authentication creates denial-of-service lockouts or out-of-band approval proofs.Require authenticated store access before public staff authentication and approval-proof mutations can compare PIN hashes.
Four-slot layout looks awkward.Keep the low-level visual primitive responsive and browser-check affected dialogs when possible.
