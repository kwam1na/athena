Active Plan · 2026-06-12

Complete Remote Assist Live Session Flow

Follow-on delivery for the foundation plan. POS terminals are the first adapter; Remote Assist remains a generic browser-client capability.

featruntime claimprovider neutral

Problem

Terminal Health can start a Remote Assist support request, but the support UI stops at connecting. The runtime still needs to claim the session, support needs reload-safe hydration, and both sides need visible live assist state.

Delivery Goals

Hydrate

Support reloads show the current backend session instead of resetting local UI.

Claim

POS runtime claims only through existing terminal sync-proof status submission.

Surface

Terminal Health renders active live assist state and an end-session action.

Runtime

The POS browser shows a visible banner and local disconnect control.

Flow

1. Support starts
Terminal Health creates a session with reason and policy context.
2. Runtime checks in
POS status submission validates store, terminal, and sync secret.
3. Session activates
The Remote Assist domain records runtime claim and support hydration.
4. Both sides can end
Support end and local disconnect are idempotent and audited.

Scope Boundaries

  • No unattended browser screen capture.
  • No desktop, shell, devtools, storage mutation, raw network replay, or clipboard scraping.
  • No bypass of terminal integrity, drawer authority, staff authority, manager proof, Cash Controls, Operations review, payments, inventory, closeout, or reconciliation.
  • No non-POS adapters in this batch.

Implementation Units

  1. Remote Assist session hydration: current-session repository/query support and public support-safe API.
  2. POS runtime claim integration: claim matching unattended sessions from successful terminal runtime status submission.
  3. Runtime banner wiring: active/waiting Remote Assist state reaches the POS runtime UI and local disconnect.
  4. Terminal Health live panel: durable session card, active state, end action, and empty reason placeholder.
  5. Live assist contract: provider-neutral co-browse/control DTOs and guardrail helpers.
  6. Audit and artifacts: lifecycle assertions, solution notes, generated artifacts, graph rebuild.
