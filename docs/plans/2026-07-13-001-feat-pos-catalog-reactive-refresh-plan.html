Delivery plan · Feature · Active

Refresh the POS catalog at safe idle boundaries

Use a constant-size store revision to wake an open register, then coalesce and apply one full local catalog refresh only after the register is operationally idle and the snapshot is durably persisted.

Created July 13, 2026 · Canonical source: Markdown plan · Origin: requirements

Problem and delivery boundary

Gap

The register searches a local metadata snapshot. An already-open terminal currently misses products created or edited elsewhere until the route remounts.

Outcome

Remote metadata becomes searchable at the first safe idle boundary, without a route reload, a live full-catalog subscription, or interference with active sale work.

Included

  • Store revision and atomic snapshot envelope
  • Monotonic IndexedDB persistence
  • Idle-gated single-flight coordinator
  • Silent background retry and recovery

Excluded

  • Full-catalog live query or fixed polling
  • Per-SKU delta feed
  • Availability-triggered metadata refresh
  • Manual apply/dismiss controls
  • Cross-tab leader election

Technical design

The current trusted snapshot remains active throughout the flow. Every non-idle edge pins its revision and rows in memory; the first restorable busy command atomically verifies or re-materializes that version before committing its durable pin. The idle default may advance without changing busy runtimes, and its rows are mirrored for offline rollback compatibility.

Key decisions

DecisionWhy
Dedicated monotonic store revisionStock-derived summaries and boolean latches do not represent repeatable metadata-only changes.
Effective snapshot lifecycle coverageThe snapshot includes projections, provisional imports, and pending-checkout alias/suppression state.
Additive atomic {revision, rows} envelopePreserves the legacy row-array endpoint while preventing acknowledgment of a revision whose rows came from another database snapshot.
Immediate and command-atomic runtime pinsEvery transient blocker pins revision and rows in memory; durable upgrade verifies or re-materializes the version before committing restorable busy state.
Shared idle authority with repeated checksA sale can begin during fetch or persistence; activation must still wait.
Register-specific orchestrationExpense, prewarm, and generic consumers keep their current one-shot lifecycle.
Background-only coordinationWaiting and automatic recovery do not require cashier decisions or compete with actionable register messages.

Signal and snapshot divergeBump in the same mutation after effective writes; return revision and rows from one additive query transaction.

Verification strategy

Backend contract

  • Eligible metadata changes advance exactly when effective output changes.
  • Availability-only changes never advance.
  • Snapshot rows and revision correspond atomically.
  • High-cardinality projection read counts stay bounded.

Durability and races

  • Write failure exposes only previous trusted rows.
  • Transient pins carry rows; durable upgrade re-materializes a concurrently pruned version.
  • Busy transitions, other tabs, and remounts select the runtime-pinned revision.
  • Previous readers can load the mirrored default offline.

Register behavior

  • Every agreed blocker independently prevents refresh.
  • Burst during a sale causes zero full reads until idle and one after.
  • Remote create becomes searchable without reload.
  • ProductEntry cannot trigger a parallel register refresh.

Background recovery

  • Offline pauses; reconnect resumes.
  • Auth failure does not hot-loop.
  • Busy, offline, running, delayed, and auth-paused states remain internal.
  • No catalog-refresh app message or cashier control is registered.

Rollout notes

  • No server backfill: a missing server revision is zero, while unversioned local data uses a distinct legacy sentinel so the first canonical envelope cannot be skipped.
  • Add payload-free transition diagnostics for observed, target, applied, retry, and failure states.
  • Production verification should exercise both an idle already-open register and a register holding active sale work.
  • Revision graph records are additive; each promotion also updates the exact legacy catalog record so the previous one-shot reader remains usable offline after an idle-safe rollback.
