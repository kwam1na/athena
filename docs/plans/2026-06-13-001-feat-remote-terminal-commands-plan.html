feat: Enable Remote Terminal Recovery Commands

Type: feat Status: active Date: 2026-06-13 Markdown: 2026-06-13-001-feat-remote-terminal-commands-plan.md
Summary Requirements Decisions Units Risks Verification

Summary

Extend the existing POS terminal recovery rails so support can issue audited remote commands from Terminal Health and the matching production terminal can run local repair commands without depending on an interactive signed-in Athena user session or an active POS app session.

The plan keeps the current command queue, browser-local executors, and runtime verification model. The main change is hardening terminal authentication, enabling support actions, and making terminal-local repair repeatable for any eligible active terminal.

Problem Frame

The M Supplies evidence showed an active online terminal with fresh check-ins, terminal integrity requiring repair, drawer authority blocked by a stale cloud-closed session, local review backlog, and no queued recovery commands.

The missing work is not a remote shell. It is a narrow support-to-terminal command lane where full admins issue allow-listed commands, the active terminal authenticates by terminal identity and sync-secret proof, executes preconditioned local helpers, and proves completion through fresh runtime status.

Requirements Digest

  • Full-admin support users can issue allow-listed terminal recovery commands to any eligible active store terminal, with each command scoped to exactly one target terminal.
  • The terminal can list, claim, acknowledge, and verify commands using terminal identity plus sync-secret proof without requiring an Athena user session, cashier session, open register session, or active POS app session.
  • Payloads and acknowledgements remain non-secret and redacted.
  • The browser runtime executes only supported local commands with existing precondition checks.
  • Terminal Health exposes safe actions, duplicate suppression, lifecycle status, and runtime verification state.
  • Terminal-integrity and stale drawer-authority blockers are expressible through reusable repair_terminal_seed and clear_stale_drawer_authority commands.

Key Decisions

DecisionRationale
Extend the existing command queue.It already models command status, audit, expected evidence, local execution, and runtime verification.
Split support auth from terminal auth.Issuing commands remains full-admin only; runtime list/claim/ack/status uses active terminal sync-secret proof.
Keep completion two-stage.Acknowledgement records local execution; runtime verification proves the blocker cleared.
Enable Terminal Health controls.The UI already has a support recovery panel and disabled buttons; binding them avoids a parallel workspace.
Do not special-case M Supplies.The repair should be ordinary typed commands with expected evidence, not one-off production code.

Implementation Units

Dependency flow: U1 terminal-runtime auth → U2 issue contract and dedupe → U3 UI actions; U4 runtime hardening depends on U1 and U2; U5 docs follows U3 and U4.

U1. Separate terminal-runtime auth from user auth

Allow active terminal sync-secret proof for list, claim, acknowledge, and runtime status without requiring an app session. Keep command issue full-admin only.

Primary files: convex/pos/public/terminals.ts, public terminal tests.

U2. Harden command issuance and expected evidence

Add duplicate suppression or supersede behavior, preserve redaction, and make targeted expected evidence safe to issue from UI.

Primary files: terminal command service, schema, types, public terminal tests.

U3. Enable Terminal Health safe action controls

Bind cloud repair and terminal command buttons to existing mutations, with disabled states for unsafe, duplicate, or already-complete actions.

Primary files: Terminal detail view and terminal health presentation tests.

U4. Harden runtime execution and verification loop

Keep local execution preconditioned, acknowledge failures safely, and force fresh runtime status so verification can complete unattended.

Primary files: local sync runtime, terminal recovery command executor, runtime status tests.

U5. Document operational use and examples

Update the terminal recovery solution note with support boundaries, generic command examples, the M Supplies case as an acceptance fixture, and what remains manual review.

Risks

Verification Strategy

  • Convex public terminal tests prove terminal-runtime auth and full-admin issue boundaries.
  • Terminal command service tests prove dedupe, expiry, redaction, acknowledgement, and runtime verification states.
  • Local runtime tests prove claim, execute, acknowledge, and fresh status publication.
  • Terminal recovery command tests prove terminal seed repair and stale drawer authority clearing preserve preconditions.
  • Terminal detail/presentation tests prove safe actions are enabled and manual-review blockers are not repairable.
M Supplies should not be marked fully healthy if local review backlog or manager-rejected business evidence remains unresolved. These commands repair terminal-integrity and drawer-authority blockers while preserving manual review.
