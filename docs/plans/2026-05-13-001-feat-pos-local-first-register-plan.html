POS local-first register runtime

Make Athena POS usable first from local terminal state for already provisioned businesses. Register actions are durably recorded in the browser before the cashier sees completion, then synced later through an ordered and idempotent Convex ingestion boundary.

POS only Offline-first by default No new payment confirmations Closeout and reopen preserved Non-POS offline states deferred

Plan Summary

Product promise

POS should boot from local state first and keep selling when field connectivity is unreliable.

Operating boundary

This applies only to provisioned POS terminals for already claimed businesses and stores.

Sync model

Local POS events are uploaded in stable order and projected into existing Athena cloud records after acceptance.

Scope Boundary

In Scope

  • Provisioned terminal local boot state.
  • Local register open, selling, checkout, receipt, cash movement, closeout, and permitted reopen.
  • All configured POS payment methods using the existing method and amount model.
  • Local event log, sync outbox, id mapping, and reconciliation records.
  • POS app-shell/offline-readiness for already provisioned terminals.

Out of Scope

  • Offline state handling for non-POS surfaces.
  • Brand-new business creation without prior online claim.
  • Peer-to-peer sync between disconnected devices.
  • Real-time offline stock coordination across terminals.
  • Payment-provider-specific offline authorization.
  • Automatic conflict resolution or a full manager reconciliation workbench.
Explicit deferral: R4 remains only as a boundary. This plan must not design or implement stale, unavailable, read-only, or offline UI states for non-POS Athena workspaces.

Key Technical Decisions

IndexedDB local store

Use transactional structured browser storage for terminal seed, catalog snapshot, register state, event outbox, sync cursors, and id mappings. Do not use localStorage for the POS ledger.

Event-log sync

Append ordered local events first. Sync through a dedicated ingestion boundary instead of replaying existing Convex mutations from the browser.

Projection over replay

Server ingestion validates terminal, store, staff, order, and idempotency, then projects accepted history into current POS, cash-control, inventory, payment, and trace rails.

Foreground scheduler first

Sync is driven by app/POS entry, online hints, visibility regain, local event append, manual retry, foreground interval, and high-priority closeout attempts.

App shell separate from sync

POS needs local runtime assets to reopen offline. Background Sync is not the v1 guarantee for delivery of the event outbox.

Reconciliation over rollback

Completed customer-facing local history stays visible. Inventory, payment, or permission conflicts become manager-review work.

Sync Strategy

Sync is ordered per terminal and local register session. The cashier never waits for cloud sync to complete a locally allowed checkout. The cloud side deduplicates accepted events and returns mappings, cursors, and reconciliation outcomes.

1. Local action Cashier opens register, edits cart, checks out, closes, or reopens.
2. Durable append POS writes an ordered IndexedDB event before showing success.
3. Sync hint Scheduler collapses triggers and reads the next ordered batch.
4. Server ingest Convex validates identity, order, and idempotency.
5. Projection Accepted history updates cloud records and local mappings.

Sync Triggers

POS route entry App boot Browser online event Visibility regain Manual retry Debounced local event append Foreground interval with backoff High-priority closeout attempt

Closeout and Reopen

Offline closeout preserves the current Athena behavior: closing pauses selling on the register session, but the existing permitted reopen-for-correction path remains available.

Closeout: append a local closeout event, pause sale controls, preserve full local timeline, and schedule a high-priority sync attempt.
Reopen: if the current workflow permits it, append a local reopen event before sale controls resume.
Sync order: project sale events -> closeout -> reopen -> later sale events exactly in timeline order.

Implementation Units

U1

Define the local POS event and storage foundation

Create the IndexedDB-backed store for provisioned terminal state, catalog snapshots, local register/session state, ordered events, sync cursors, and cloud id mappings.

Key proof: event append ordering, failed-write behavior, schema-version handling, and local sync status derivation.

U2

Capture online provisioning into the local POS seed

Persist the store, terminal, staff, permissions, register config, catalog snapshot, price/tax context, and inventory snapshot required for local POS boot.

Staff auth storage must be scoped verifier material only. No plaintext PINs or reusable server credentials.

U3

Implement the local-first POS command gateway

Add local command paths for register open, session start/resume, cart changes, checkout, receipt creation, cash movements, closeout, and permitted reopen.

Preserve existing payment semantics: method, amount, timestamp, and staff context. No extra payment confirmation states.

U4

Wire POS register UI to local-first runtime

Make the active register render from local state by default, show synced/syncing/pending/needs-review states, and preserve closeout and reopen controls.

Do not change routed non-POS workspaces for offline behavior.

U5

Build the sync scheduler and trigger model

Centralize sync triggering, batching, ordering, retry/backoff, and concurrency control for local POS events.

Checkout and local closeout stay independent from sync completion.

U6

Add Convex sync ingestion, idempotency, and mappings

Create a dedicated sync boundary that accepts ordered local batches, validates terminal/store/staff context, dedupes accepted events, and stores local-to-cloud mappings.

U7

Project accepted events into existing Athena rails

Convert accepted local history into register sessions, POS sessions, transactions, transaction items, payment allocations, inventory effects, cash controls, and workflow traces.

Inventory and payment conflicts become reconciliation records rather than silent rewrites of completed sales.

U8

Surface pending sync and reconciliation

Keep pending sync distinct from conflicts requiring action in POS and minimal cash-control visibility.

U9

Make the provisioned POS app shell offline-ready

Ensure a provisioned terminal can reopen the POS runtime from locally available browser assets without relying on a live network fetch.

U10

Refresh validation, harness coverage, and docs

Add validation-map coverage, browser-facing verification where needed, generated-doc refreshes, and a durable solution note for the POS event-log sync pattern.

Risks and Mitigations

Risk Mitigation
Local browser storage is lost before sync. Use IndexedDB, surface local storage health, and sync opportunistically whenever connectivity exists.
Retries duplicate sales, payments, receipts, or closeouts. Add server idempotency records and tests for repeated accepted batches.
Closeout or reopen projects out of order. Enforce strict per-register-session ordering and treat closeout/reopen as timeline checkpoints.
Local staff permissions drift from cloud policy. Validate on sync and create permission reconciliation items without rewriting local history.
Offline POS scope spreads to non-POS workspaces. Keep local runtime under POS infrastructure and do not edit non-POS screens for offline behavior.
Runtime assets are unavailable when the terminal reopens offline. Add POS app-shell/offline-readiness validation separately from sync scheduling.

Validation Expectations

Core tests

  • Local store ordering, schema versioning, failed writes, and id mappings.
  • Local command durability before UI success.
  • Payment-model preservation across all configured POS methods.
  • Closeout pause, permitted reopen, and ordered projection.
  • Sync scheduler trigger collapse, backoff, and no concurrent upload for the same queue.

Integration tests

  • Convex ingestion identity validation, idempotency, and mapping persistence.
  • Projection into transactions, register sessions, cash controls, inventory, payment allocation, and traces.
  • Pending sync versus needs-review visibility.
  • POS app shell and local seed load without live backend availability.
