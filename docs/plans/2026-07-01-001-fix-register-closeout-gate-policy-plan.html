fix: Collapse POS and Cash Controls closeout gating

Register-session closeout variance should pass through one server-owned gate. Cash Controls and POS local-sync projection will use the same cash-controls threshold, manager-signoff flags, and closeout-hold precedence before deciding whether to close, wait, or request approval.

Type: fix Status: active Date: 2026-07-01 Worktree: .worktrees/codex/closeout-gate-plan Source: Markdown plan

Problem

Current POS Path

POS closeout sync treats every non-zero variance as register_closeout_variance review unless an approved replay flag is present.

Current Cash Controls Path

Cash Controls applies store configuration: threshold, any-variance signoff, over signoff, and short signoff.

Target

The source path no longer determines manager-review policy. The shared gate decides once; each caller adapts the same result to its workflow.

Requirements Digest

  • Cash Controls and POS projection evaluate variance with the same server-side policy and defaults.
  • Under-threshold POS variance closes during sync when no closeout holds remain.
  • Approval-required POS variance returns projected, creates a closeout mapping, and routes through one Cash Controls-owned review semantic.
  • Cash-affecting holds stay higher priority than variance approval.
  • Legacy register_closeout_variance conflicts remain resolvable.
  • Duplicate or ambiguous closeout uploads keep their existing conflict/reject behavior.
  • POS local-first continuity remains intact for unresolved local closeouts and replacement drawers.
  • Manager review evidence stays session-targeted and operator-readable.

Shared Gate Outcome

DecisionMeaningPOS sync behaviorCash Controls behavior
blocked_by_holds Submitted facts exist, but cash-impacting work can still change expected cash. Keep session closing; preserve submitted evidence; do not append final closeout records. Return submitted/unresolved state or block finalization with hold copy.
approval_required Variance requires manager approval by threshold or signoff flags. Persist or reuse one Cash Controls review owner, patch the session to closing with managerApprovalRequestId, return projected, create a closeout mapping, and create no POS sync conflict. Create/reuse approval request or require valid manager proof.
close_allowed No holds and no policy-required approval. Project as closed, map local event, and record closeout evidence. Close or finalize immediately.
projection_conflict Duplicate, ambiguous, stale, or identity-mismatched sync event. Use existing sync conflict/reject behavior. Not applicable to direct Cash Controls closeout.

Approval Persistence Contract

  • Reuse an existing pending variance_review for the same registerSessionId and identical closeout facts.
  • Never insert a second approval request when the same localEventId retries.
  • Conflict or reject changed closeout facts for the same register session before policy approval.
  • Patch the register session to status: "closing" with counted cash, variance, notes, managerApprovalRequestId, closeoutOwnedAt, closeoutOwnershipSource: "approval_request", and closeout operating-date fields.
  • Store review metadata for counted cash, expected cash, variance, notes, local event id, local register-session id, terminal id, closeout time, syncOrigin: "local_sync", and the gate decision/reason.
  • allowRegisterCloseoutVarianceProjection bypasses only the variance-approval decision for approved legacy conflicts; identity, duplicate, stale-session, and hold checks still apply.
