Implementation Plan

feat: Add POS local staff authority

Add a POS-only local staff authority layer so registered staff can sign into an already-provisioned terminal without a live Convex connection.

Date: 2026-05-14 Type: feat Status: active Canonical markdown: plan.md
Problem Requirements Decisions Design Flow Units Risks Verification

Problem Frame

POS can now render from local readiness work, but cashier authentication still depends on the live Convex terminal staff-auth mutation. A provisioned terminal needs enough scoped local authority to verify registered staff while offline, without turning browser storage into a global login bypass.

This plan separates local login authority from posLocalStaffProof. The proof remains sync/event evidence; local sign-in needs verifier material proving the entered PIN matches a registered staff credential.

Requirements Digest

Offline staff sign-in

Registered active POS staff can sign into a provisioned terminal without a live connection when local authority is fresh and scoped.

Scoped verifier material

Local records include store, terminal, staff, credential, role, version, expiry, and slow verifier metadata. They never include plaintext PINs.

Server reconciliation

Sync validates authority drift and preserves completed local register work while creating review items when permissions changed.

Key Decisions

  • Use a dedicated POS local staff authority store/module instead of trusting posLocalStaffProof as login proof.
  • Use slow salted local verifier metadata for offline PIN checks; keep the current online hash path as compatibility unless migrated deliberately.
  • Refresh local authority after successful online staff auth and terminal authority refresh.
  • Keep local cashier auth separate from manager elevation and command approval proofs.
  • Fail closed when authority is missing, expired, unsupported, or scoped to another terminal/store.

Design Flow

  1. Manager assigns staff credentials online in Athena.
  2. Provisioned POS terminal refreshes staff authority for its store and terminal.
  3. Client writes scoped verifier records to IndexedDB.
  4. Cashier signs in offline with username and PIN.
  5. POS verifies against local authority and signs cashier into the local register.
  6. Local POS events carry staff context.
  7. Sync later validates staff/role/credential state and records permission conflicts when needed.

Implementation Units

U1. Model local staff authority in the POS local store

Add durable IndexedDB storage for staff authority records keyed by store, terminal, and normalized username.

Primary files: posLocalStore.ts, localStaffAuthority.ts, and tests.

U2. Introduce a slow local PIN verifier

Create Web Crypto-compatible verifier generation and matching with per-record salt and explicit algorithm metadata.

U3. Export online staff authority snapshots from Convex

Project active POS-authorized staff authority from Convex, filtered by terminal/store scope and credential status.

U4. Refresh terminal-local staff authority in the client

Persist refreshed local authority after online staff authentication and terminal provisioning/readiness refresh.

U5. Authenticate cashier sign-in from local authority while offline

Replace the offline fail-fast guard with local verification when authority exists, while keeping fail-closed states explicit.

U6. Revalidate local staff authority during sync projection

Extend sync evidence and projection tests so permission drift creates reconciliation records without deleting local sales.

U7. Expose operator-safe readiness and diagnostics

Show useful authority readiness states without leaking verifier, token, PIN hash, or raw credential material.

Risk Summary

Risk Mitigation
IndexedDB snapshot is copied and attacked offline. Store only scoped, expiring, salted slow verifiers; never raw PINs or global credentials.
Staff is revoked while the terminal is offline. Expire local authority and create permission-drift reconciliation during sync.
Cashier auth is mistaken for manager approval. Keep command approval proof minting and consumption server-bound and action-scoped.
Verifier work factor hurts low-end terminal responsiveness. Choose concrete parameters during implementation with Web Crypto and focused performance-sensitive tests.

Verification Strategy

  • React tests prove offline cashier sign-in succeeds from local authority and never calls Convex.
  • Local store tests prove authority refresh, expiry, scoping, and no raw PIN persistence.
  • Convex tests prove server authority snapshot filtering and permission-drift sync behavior.
  • Register view-model tests prove local staff identity reaches local POS events after offline sign-in.
Command approval and manager elevation remain unchanged authority boundaries. Local cashier auth must not satisfy protected command approvals.
