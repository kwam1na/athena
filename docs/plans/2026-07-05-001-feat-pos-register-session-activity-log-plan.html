feat: Add POS register session activity log

type: feat status: active date: 2026-07-05 scope: register session
Core decision: build a POS-owned session activity channel that uploads sanitized summaries of local POS events. Existing core sync remains authoritative for business projection; the activity log is investigation evidence.

Problem Frame

A cashier reported closing out a register session, but the cloud session still appeared open. That incident is one use case for a broader gap: Athena has the evidence in pieces, including local events, sync rows, mappings, conflicts, workflow traces, operational events, closeout records, and Cash Controls review. The feature pulls those pieces into one foundational register-session timeline for full-admin Cash Controls investigators so they can answer what POS did, what the cloud received, and what projection or review did with it.

This is not a store-day daily log. Store-day and Daily Close status can appear as context, but the replay axis is one register session and its terminal evidence.

Requirements

  • Show a register-session-scoped activity log independent of Daily Operations timeline ordering.
  • Represent all POS activities captured for the session, including local-only activity types that current core sync does not upload.
  • Preserve local event id, local sequence, terminal id, local/cloud register-session identity, upload/projection timestamps, and related cloud links where available.
  • Distinguish recorded, uploaded, accepted, projected, held, conflicted, rejected, reviewed, and repaired states.
  • State clearly when cloud cannot know about later local activity from an offline or unreported terminal.
  • Show activity coverage and attention summary above the rows: reported-through sequence/time, uncertainty, category counts, and held/conflicted/rejected/mapping-pending counts.
  • Keep activity upload auxiliary and non-blocking for cashier work.
  • Use bounded, indexed, session-scoped reads and field-aware redaction.

Key Decisions

DecisionReason
Build a POS-owned read modelThe evidence is register-session and terminal scoped, not store-day scoped.
Add sanitized activity uploadCurrent core sync rows cannot reconstruct all cart, payment, service, session, and reopen actions.
Keep core sync authoritativeThe log explains outcomes; it does not mutate sales, closeouts, payments, inventory, or drawer authority.
Persist mapping-pending rowsActivity may upload before cloud register-session mapping exists; server-side pending rows keep retries idempotent.
Add activity checkpointsCore sync cursors do not cover all local POS events, so empty-vs-unreported states need their own high-watermark.
Order by local sequenceOffline upload, retry, and projection timestamps can disagree; local sequence is the replay spine.
Show uncertainty explicitlyThe cloud can prove what it received, not what a disconnected terminal has not uploaded.
Redact by positive allowlistLocal event payloads can carry proof, customer, and payment internals that do not belong in operator UI.
Isolate activity patch failuresActivity visibility must not roll back authoritative sale, closeout, cursor, or manager-review outcomes.

Risks

RiskMitigation
Cloud cannot prove unuploaded offline workShow last reported sequence/time and limitation copy.
Activity upload blocks cashier flowsSeparate activity state from core sync state; make it retryable and non-blocking.
Activity volume grows quicklyUse compact metadata, indexes, pagination, and category filters.
Sensitive local payload leaksUse client sanitizer, server positive allowlist validation, role-aware DTOs, sanitized logs, and redaction tests.
Retries duplicate rowsUpsert by local event keys and server-originated review/closeout decision keys.
Compromised terminal fabricates rowsVerify store/terminal/session binding and label terminal-reported rows until server projection proves outcomes.
Activity patch fails in projection pathUse best-effort helpers so projection/review still succeeds and sanitized patch failures are observable.

Acceptance

  • Online sessions replay open, cart/payment, cash/expense, sale, review, and optional closeout activity with applicable links.
  • Offline activity shows uncertain coverage before reconnect and session rows after upload.
  • Any mismatch between reported terminal activity and cloud-visible state can be investigated through normal activity categories.
  • Out-of-order and duplicate events are labeled without overstating business outcomes.
  • Resolved manager decisions stay visible after active review clears.
  • Full-admin readers with valid store/register-session scope receive activity rows; POS-only, wrong-store, and unauthorized users receive no data.
  • No role receives raw payloads, customer contact, proof tokens, PINs, secrets, local notes, or payment internals.
  • The activity section loads from register-session activity, not Daily Operations timeline data.
Important limitation: The log can be definitive about cloud-reported activity. It cannot definitively know about local-only activity on a terminal that has not uploaded yet.
