Add money-settled POS item adjustments

Build a completed-transaction adjustment workflow for POS item and quantity corrections. The original sale remains immutable, while a linked adjustment records corrected line state, inventory deltas, settlement, one complete-bundle manager approval, and visible transaction history.

Recommendation

Selected: adjustment ledger workflow. Keep the original completed transaction immutable and record a linked adjustment that carries item, inventory, payment, approval, and audit effects.

Why not direct patching?

Direct edits make old reports and receipts shift underneath operators and weaken the ledger-safe correction pattern.

Why not void and recreate?

It is too disruptive for partial line mistakes and creates noisy receipt/history behavior for small quantity corrections.

Approval rule

Operators can revise freely. Manager approval is required once for the complete submitted adjustment bundle.

Repeat adjustment rule

V1 allows one active adjustment at a time. Additional corrections must compute from the latest effective state or be blocked for manager/admin review.

Scope

  • Support completed POS transaction item and quantity corrections.
  • Require refund when corrected total is lower, collection when corrected total is higher, and no payment movement when totals are equal.
  • Record adjustment records, inventory movement, SKU activity, payment allocation, register expected-cash effect for cash, and operational event history.
  • Defer discount-only corrections, arbitrary total overrides, tax overrides, original receipt rewrites, and POS local-first adjustment projection.

Requirements Digest

  • Original sale rows remain historical truth.
  • Adjustment preview must show original total, corrected total, refund/balance/no-payment state, and per-SKU inventory deltas.
  • Approval requests must have a concrete resolver path and must not mutate before approval.
  • Read models must distinguish original totals from adjusted/net totals.
  • UI must follow existing command-result and command-approval presentation patterns.

Design Shape

The markdown plan contains the canonical Mermaid diagram. The operational path is:

1. Open completed transaction detail and choose Items or quantities.
2. Build adjustment draft and preview corrected totals and deltas.
3. Add refund or collection settlement method when money moves.
4. Submit complete bundle for manager approval.
5. Apply after matching inline proof or async approval resolution.
6. Show adjusted summary and update history without rewriting original sale.

Implementation Units

U1. Add POS adjustment schema and read helpers

Create parent/line adjustment records and transaction-scoped repository helpers.

U2. Build deterministic item-adjustment planner

Compute corrected lines, totals, inventory deltas, settlement direction, and validation failures.

U3. Add approval-backed adjustment commands

Preview, request approval, and apply one complete adjustment bundle through existing command approval rails.

U4. Apply inventory, payment, drawer, and audit effects

Write inventory movements, SKU activity, payment allocations, register cash impact, and operational events idempotently.

U5. Expose adjustment state in transaction read models

Return adjusted/net fields and adjustment summaries while preserving original totals.

U6. Build the transaction detail adjustment workflow UI

Replace static guided-route copy with an operator workflow and operations queue approval cards.

U7. Harden reporting, validation, and browser coverage

Confirm cash-control, daily operations, harness, graphify, and browser validation coverage.

Key Risks

RiskMitigation
Partial writes across inventory, payment, and audit effectsUse one command boundary, validate before writes, idempotent source ids, and mark applied only after required effects are recorded.
Approval payload driftBind approval to a payload fingerprint and recompute before apply.
Silent report semantic changesPreserve original transaction fields and expose explicit adjusted/net fields.
Closing or reconciled drawer cash settlementBlock immediate cash settlement or route through a separate admin path without partial apply.

Verification Strategy

  • Focused planner, command, approval, payment-allocation, SKU activity, cash-control, and transaction read-model tests.
  • Transaction detail and operations queue UI tests for preview, settlement requirements, approval states, and history.
  • Convex audit/lint checks, typecheck/build, harness coverage updates, graphify rebuild, and browser validation on the transaction detail route after implementation.
