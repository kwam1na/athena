Scope

Core direction: contextEvent becomes the sole durable evidence source for storefront intelligence. Legacy analytics remains out of the live runtime/compiler path; valuable historical rows are used only after bounded conversion into contextEvent evidence.

The markdown plan is the authoritative implementation artifact. This HTML is a browser-readable digest.

  • Emit route, product, cart, and checkout context events through /tracking-events.
  • Derive trust-sensitive fields server-side: store, organization, actor, subject, synthetic, visibility, and retention. Browser session refs remain claims unless server-corroborated.
  • Import safe historical analytics through an internal/admin-only, dry-run-first, idempotent conversion path with reviewed mapping approval, batch lineage, omitted-evidence counts, freshness metadata, and quarantine support.
  • Compile store and customer readout bundles from contextEvent rows only.
  • Keep shopping flows non-blocking when telemetry fails.

Key Decisions

  • Context-event only: no dual-write or legacy fallback for intelligence context.
  • Historical value preserved: safe legacy analytics rows are imported into contextEvent instead of read directly by compilers.
  • Enum-coded checkout evidence: checkout state and blocker values are server-allowlisted codes, not raw error or payment/provider text.
  • Narrow v1 event catalog: route viewed, product viewed, cart changed, checkout state changed.
  • Server authority: the backend derives actor, subject, source, visibility, retention, and synthetic fields.
  • Origin is not auth: the write route must verify server-issued storefront context or mutually consistent trusted cookies/session records.
  • Diagnostics, not blockers: storefront events are evidence collection and cannot block customer workflows.
  • Synthetic exclusion: diagnostics may record synthetic events, but store/customer intelligence excludes them by default.

System Impact

The rollout crosses the storefront SPA, the Hono tracking route, historical analytics import, Convex context-event persistence, context-bundle compilers, intelligence run/snapshot/artifact records, and readout UI trust metadata.

Unchanged: provider invocation, artifact lifecycle, dismiss/rerun behavior, and approval/apply boundaries remain unchanged. Legacy analytics endpoints stay outside the live intelligence path.

Risks

RiskMitigation
Limited evidence immediately after switching sources.Surface limited-evidence state honestly; do not fallback to analytics.
Historical analytics value is lost.Import safe historic rows into contextEvent with lineage, idempotency, freshness, omitted-evidence metadata, and an import run/batch id.
Historical import pollutes the new model.Treat rows as untrusted source material and map only registered, allowlisted event shapes.
Historical import runs too broadly or maps incorrectly.Require dry-run discovery, reviewed mapping approval, explicit write enablement, store/window batches, stop thresholds, and quarantine/revoke support.
Sensitive browser-controlled text reaches snapshots.Use strict server allowlists and scalar-only payloads.
Rerenders duplicate route/product events.Use stable idempotency inputs and repeated-render tests.
Synthetic monitor traffic pollutes readouts.Derive synthetic server-side and exclude by default in compilers.

Verification Strategy

  • Storefront builder/API tests for context envelopes and non-blocking emission failures.
  • Backend route/registry/append tests for origin policy, server-derived metadata, enum-coded checkout/blocker values, payload allowlists, synthetic classification, and idempotency conflicts.
  • Historical import tests for internal/admin authorization, discovery reports, reviewed mapping approval, dry-run/reporting, resumable idempotency, import run/batch lineage, quarantine behavior, synthetic handling, and omitted unsafe fields.
  • Context bundle and intelligence capability tests proving readouts use contextEvent evidence only, including imported historical context, with no direct analytics compiler refs, analytics table queries, or legacy quality flags.
  • Readout UI tests for limited-evidence and sanitized debug metadata.
  • Cross-package integration checkpoint: storefront emitter to Hono route to appendContextEvent to context bundle to intelligence snapshot.
  • Checkout route/payment/verification tests for telemetry-failure non-blocking behavior.
  • Graphify rebuild after code changes so package navigation reflects the new path.
