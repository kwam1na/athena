Athena implementation plan

Make completed POS sales authoritative for Reports

Core invariant: A completed POS sale can be flagged for source-integrity uncertainty, but it cannot be excluded because operating-hours evidence is missing or says the store was closed.

Problem and scope

Reporting currently bundles financial eligibility, local-date attribution, and expected operating hours. That allowed valid completed Wigclub sales to disappear when a historical policy ended before live ingress began, and still lets closed or missing schedules suppress commerce.

The redesign separates those responsibilities: completed POS status establishes eligibility; immutable occurrence plus an effective-dated store timezone establishes reporting date; Store Schedule supplies optional expected-hours context and comparison windows.

Explicit early-development purge: this supersedes the prior schedule-gated Reports design. A dev-only operator script empties the enumerated generated reporting tables while preserving authoritative POS and operational sources, then Athena rebuilds only the new contract. Reports may show rebuilding until the first verified bundle activates.

Target flow

  1. Deploy lifecycle journaling, then seal a POS-only stabilized baseline and durable terminal cursor.
  2. Run the dev-only generated-reporting purge, then allow one authenticated store full admin to trigger the idempotent pipeline with immutable server-derived actor evidence.
  3. Normalize original sales and linked refund, void, adjustment, and settlement effects.
  4. Resolve immutable reporting dates from occurrence plus timezone; attach schedule context when available.
  5. Seal preview dispositions and apply idempotently.
  6. Prove source-to-fact counts, identities, signed money, units, dates, currencies, and lifecycle categories.
  7. Rebuild and reconcile projections behind the active generation.
  8. Activate verified generations and materialize Reports without requiring hours.

Implementation units

U1 — Store-time authority

Introduce effective-dated timezone versions with trusted provenance and widen-first compatibility.

U2 — Financial date contract

Separate recognition, local date, timezone lineage, and optional schedule context in facts and projections.

U3 — Live canonicalization

Remove schedule-derived quarantine from valid new POS commerce while preserving source-integrity checks.

U4 — Dev purge and full-store backfill

Empty generated reporting tables with a dev-only operator command, verify that precondition, then run sealed preview/apply across all POS history and lifecycle effects.

U5 — Source-to-fact proof

Add bounded reconciliation that makes unexplained POS omissions an activation blocker.

U6 — Projection set

Rebuild and verify a new-contract generation set from purged-and-backfilled facts while Reports remains explicitly rebuilding.

U7 — Reports cutover

Materialize the projection set, seal the complete bundle, and atomically switch one public bundle pointer.

U8 — Wigclub migration

Exercise the U1–U7 capabilities once in dev and capture sanitized post-activation and rollback evidence.

Verification strategy

  • Every valid Wigclub POS event through the watermark has one canonical identity or linked lifecycle effect.
  • Source-to-fact and fact-to-projection reconciliation have zero unexplained differences.
  • A second full-store run creates no duplicate facts.
  • Sales during hours, outside hours, on closed days, and without hours all appear in Reports.
  • Missing cost and item attribution remain partial without suppressing trustworthy money.
  • The rebuilding window starts when the new-only reader deploys and ends at first verified bundle activation; later rebuilds preserve the prior new-design bundle.

Production deployment is deliberately excluded pending separate approval after dev evidence review.
