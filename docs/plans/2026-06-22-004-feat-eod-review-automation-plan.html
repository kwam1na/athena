Summary

Add a policy-gated EOD Review completion path that lets Athena complete clean store days and narrowly-approved low-risk review days as an explicit automation actor.

Manual EOD completion keeps the manager approval proof contract. Automation completion records policy evidence and Athena attribution instead of pretending a manager approved the close.

Problem Frame

Daily Operations already prepares EOD Review through eod.prepare, but completion still requires a human manager proof even when the store day is clean or the only review items fall inside a configured low-risk policy. Athena needs a truthful automation completion boundary with durable evidence.

Requirements

  • Add a distinct Daily Operations automation action for EOD completion.
  • Complete clean EOD Reviews only when store policy is enabled, the local EOD completion window is open, and command-time snapshot checks pass.
  • Complete low-risk EOD Reviews only when every review item is policy-approved and within thresholds.
  • Keep blockers, carry-forward items, unknown review categories, invalid dates, reopened/superseded lifecycle conflicts, and pending approvals as hard stops.
  • Persist automation attribution on the completed close/report evidence and operational event.
  • Persist structured dry-run/skipped/applied decision evidence, including timing, threshold, item-key, and carry-forward status.
  • Preserve existing financial-detail redaction when returning policy-reviewed evidence to Daily Close, Daily Operations, and history views.
  • Expose full-admin policy controls and calm operator-facing attribution copy.

Key Decisions

DecisionRationale
eod.auto_complete is separate from eod.prepareCompletion mutates lifecycle state and needs a distinct audit boundary.
Automation authorization is not manager approvalManual proof semantics stay intact while Athena records policy evidence.
Store-local completion window gates actionAthena may evaluate earlier, but it cannot close before the configured EOD window opens.
Structured run evidence is requiredDry-run, skipped, and applied decisions need auditable item keys, thresholds, timing status, and carry-forward status.
Low-risk categories are narrowInitial automation can cover clean days, thresholded cash variance, and thresholded voided sales.
Restricted evidence stays permissionedAthena attribution may be visible broadly, but cash amounts, thresholds, item snapshots, and source evidence keep existing read-side redaction.
Carry-forward stays human-only in v1Any existing carry-forward item or non-zero carry-forward count skips automation completion.
Snapshot revalidation happens at apply timePrevents races from closing a day that became blocked after eligibility evaluation.
Lifecycle state remains primary in UIAutomation evidence explains attribution; it does not replace Daily Close state.

Implementation Units

U1. Add EOD Completion Policy Shape

Add typed store-scoped configuration for EOD completion, local completion-window settings, and structured run decision evidence without reusing Opening-specific policy fields.

Core files: convex/schemas/automation.ts, convex/automation/runLedger.ts, convex/operations/dailyOperationsAutomation.ts.

U2. Add EOD Auto-Complete Eligibility Decision

Register a distinct action and define deterministic clean/low-risk eligibility from the server Daily Close snapshot, local EOD completion window, and carry-forward skip rules.

Key tests: full hard-blocker matrix, local timezone/window boundaries, unknown review categories, and carry-forward skip.

U3. Add Automation-Safe Daily Close Completion

Share completion persistence while keeping human manager proof required and adding an internal automation path with policy evidence, command-time window validation, and carry-forward refusal.

Key tests: duplicate scheduler replay, already-applied runs, and retry after close/event persistence before run patching.

U4. Extend Daily Operations And Daily Close Read Models

Expose automation completion attribution and permission-redacted policy-reviewed evidence from backend snapshots.

U5. Add Operator UI And Settings For EOD Automation

Add full-admin EOD policy settings and completed-by-Athena copy in Daily Close, Daily Operations, and history.

U6. Round Out Validation, Generated Artifacts, And Solution Note

Refresh generated artifacts, Graphify, tests, and reusable architecture learning after implementation.

System-Wide Impact

  • Interaction graph: scheduled automation, policies, run ledger, Daily Close completion, operational events, snapshots, settings, and UI attribution.
  • Error propagation: safe automation outcomes and normalized UI copy; raw errors stay server-side.
  • State lifecycle risks: command-time races, early completion, carry-forward ambiguity, duplicate retries, already-completed days, reopened/superseded closes, and partial run patching.
  • Read-side access: widened automation evidence preserves existing financial-detail redaction.
  • Unchanged invariants: manual manager proof, hard blockers, immutable completed snapshots, and server-owned Daily Operations state.
