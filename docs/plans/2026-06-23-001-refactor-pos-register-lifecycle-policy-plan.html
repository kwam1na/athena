Problem Frame

The current POS register patch encodes drawer lifecycle invariants in local read-model projection, local command gating, runtime status, Convex sync projection, repository conflict lookup, and register review presentation. That can drift when a cashier closes one drawer and opens another while browser-local POS and server projection reconcile at different times.

Requirements

  • Define shared lifecycle policy once for browser and Convex consumers.
  • Keep open and active sale-usable; keep closing lifecycle-visible but not sale-usable.
  • Allow safe replacement drawers after settled closeout, submitted closeout review, or cloud_closed drawer authority.
  • Close the direct-cloud-id reviewed-closeout sale projection bypass.
  • Define scoped session as same store, terminal, and register-session scope.
  • Preserve truthful Cash Controls and Daily Close review surfaces.
  • Refresh Graphify after code changes.

Key Decisions

  • Create a pure shared lifecycle policy module under packages/athena-webapp/shared/.
  • Expose separate decisions for sale usability, lifecycle visibility, replacement-open eligibility for later new register-session scopes, reviewed-closeout supersedability, and runtime evidence.
  • Keep repository data access outside the policy; repositories return facts and ordering evidence to application-layer policy callers.
  • Integrate Convex projection before frontend cleanup because projection guards stale clients.
  • Land as one coordinated integration PR to avoid repeated generated-artifact churn.

Implementation Units

U1. Shared Lifecycle Policy Foundation

test-firstCreate shared/registerSessionLifecyclePolicy.ts and focused tests for status, authority, closeout, and review decisions.

U2. Browser-Local POS Policy Integration

Replace duplicated predicates in the local read model, command gateway, sync runtime, runtime status, and register view model while preserving local-first command behavior.

U3. Convex Projection and Repository Policy Integration

Move pure interpretation into shared helpers, keep DB lookup in repositories, and fix direct-cloud-id sale projection against reviewed closeouts.

U4. Review Surface and Daily Close Alignment

Keep closeout review and mixed review queues visible and actionable even when POS can open a later drawer.

U5. Harness, Generated Artifacts, and Delivery Documentation

Regenerate Graphify, update validation-map coverage if needed, and document the new lifecycle policy boundary.
