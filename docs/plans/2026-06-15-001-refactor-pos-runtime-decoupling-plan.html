refactor: Decouple the POS Runtime

Focused plan for splitting the current POS browser runtime and Terminal Health recovery paths while preserving cashier continuity and existing runtime contracts.

type: refactor status: active date: 2026-06-15 worktree: codex/pos-runtime-decoupling

What changes

The public runtime facade stays stable, while readiness sensing, sync draining, drawer-authority reconciliation, runtime status publication, recovery command execution, staff authority refresh, runtime-status side effects, and Terminal Health presentation move into explicit modules.

The plan also splits detail-only Terminal Health work out of roster rows without splitting the truth operators see.

What stays fixed

  • Local-first POS sales continue through check-in and support failures.
  • Runtime check-ins stay latest-wins serialized.
  • Recovery commands verify through fresh runtime evidence.
  • Remote Assist presence does not grant POS authority.

Core Boundary

Sales readiness, support recovery, and diagnostic evidence remain separate. Terminal Health should treat TerminalRecoveryPreview as the primary recovery source of truth, while raw fallback paths exist only for older or partial responses.

Dependency Shape

Implementation Units

U1 Readiness sensing

Extract local readiness observation from usePosLocalSyncRuntime.ts into runtimeReadiness.ts, without upload or publish side effects.

U2 Sync drain coordinator

Own upload scheduling, retry timing, status-only behavior, and append-triggered uploads behind a focused coordinator.

U3 Runtime status publisher

Preserve serialized latest-wins publication, payload signatures, and best-effort failure handling in a tested publisher module.

U4 Commands + staff authority

Extract recovery command orchestration and centralize redacted staff-authority refresh/write behavior by context.

U5 Server side effects

Name and test runtime-status side effects: command verification, Remote Assist presence, and session claims after accepted check-in.

U6 Terminal Health split

Make roster cheaper while sharing recovery/readiness classification rules with detail views.

U7 Preview-first presentation

Make TerminalRecoveryPreview primary in frontend presentation, with legacy fallback only when preview is absent.

U8 Facade + validation

Reassemble the facade, run focused tests, rebuild graph knowledge, and use bun run pr:athena as the final integration gate.

Operator Contract

Terminal Health list and detail must preserve the same terminal identity, runtime freshness, recovery readiness, current safe action category, latest command state, and local sync evidence classification.

Command acknowledgement remains an intermediate state: operators see that the terminal ran the helper, but the terminal is not called recovered until fresh runtime proof verifies the expected evidence.

Roster optimization is limited to avoiding detail-only command/action work in rows. List-visible recovery fields still include readiness, action category, duplicate-disable command state, verification state, and sync classification.

Post-status server side effects are isolated: accepted runtime status stays accepted, command verification still runs, and Remote Assist failures remain diagnostics.

High-Risk Checks

  • Runtime check-in conflicts must not return.
  • Status-only pages must not start upload drains.
  • Command acknowledgement must not masquerade as verified health.
  • Roster optimization must not contradict detail recovery state.
  • Staff authority helper must classify refresh results before changing local clearing behavior.
  • Staff authority helper must not leak proof/PIN material or clear authority on generic failures.
  • Remote Assist runtime host drain ownership must stay explicit so extraction does not create duplicate or missing upload schedulers.

Validation

Focused coverage should center on runtime extraction tests, terminal command runtime tests, staff authority classifier/helper tests, drawer-authority reconciliation tests, Terminal Health server/presentation parity tests, Remote Assist side-effect tests, and existing facade/consumer tests.

After code changes, run bun run graphify:rebuild. Before merge or PR handoff, run bun run pr:athena.
