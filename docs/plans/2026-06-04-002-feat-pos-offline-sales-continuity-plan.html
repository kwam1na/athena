feat: Keep provisioned POS terminals selling offline

Type: feat Status: active Date: 2026-06-04 Markdown: 2026-06-04-002-feat-pos-offline-sales-continuity-plan.md

Summary

This plan extends Athena's route-scoped POS continuity into local sales continuity: a returning provisioned terminal with local sale authority can keep starting and completing sales when app-session validation is unavailable, including when Convex is totally unreachable. The implementation updates the POS route shell, audits sale blockers, converts cloud-validation uncertainty into reconciliation metadata, and verifies the full no-network register path.

Requirements Digest

Route continuity

Returning provisioned terminals should enter POS without fresh app sign-in when network, Convex, and app-session validation are unavailable.

Sales continuity

New sale start and active cart/payment completion continue locally when local POS authority is sufficient.

Blocker audit

Every sale-affecting blocker is classified as local recording impossible, locally known unsafe, or cloud-validation uncertainty.

Reconciliation

Cloud-validation uncertainty becomes internal audit/reconciliation context; clean history syncs normally and mismatches route to review.

Scope Boundaries

  • Fresh app login, fresh POS recovery-code verification, first-time terminal provisioning, and broad non-POS offline access remain out of scope.
  • Offline manager approval for protected manager-only commands remains out of scope unless planned separately.
  • Terminal repair offline is out of scope unless it can be completed entirely from already trusted local state.
  • Payment-provider-specific offline authorization behavior is unchanged; the plan preserves current POS payment recording.
  • Follow-up work may broaden offline continuity to non-register POS subroutes or add fleet-level reporting.

Context & Research

The plan reuses existing POS local-first boundaries rather than adding a new offline auth model.

  • src/routes/_authed.tsx owns route-scoped POS shell and app-session recovery behavior.
  • usePosTerminalAppSessionRecovery.ts publishes recovery states including waiting_for_network.
  • registerReadModel.ts and localCommandGateway.ts compute and enforce sale authority.
  • usePosLocalSyncRuntime.ts, syncScheduler.ts, and terminalRuntimeStatus.ts own sync/review diagnostics.
  • The offline route access Playwright spec provides the no-network browser regression pattern.

Key Technical Decisions

Decision Rationale
Keep app-session continuity POS-scoped Prevents register reliability work from becoming generic offline Athena access.
Turn waiting_for_network into local sale continuation Network absence, including total Convex unreachability, should delay cloud validation and sync, not local sale recording.
Add a sale-blocker policy boundary Stops product decisions from staying scattered across route, read-model, command, and UI code.
Keep cashier UI normal Operators need confidence that sales proceed; review housekeeping belongs behind the scenes.
Do not bypass authenticated cloud sync boundaries Offline recording can proceed, but upload waits for supported recovery, POS-scoped validation, or existing authenticated sync context.

Design Shape

POS sale authority remains grounded in local terminal, staff, drawer/register, readiness, and command evidence. App-session uncertainty is metadata for reconciliation, not the sale gate.
Blocker class Meaning Cashier behavior
Local recording impossible The terminal cannot create durable local sale evidence. Hard block.
Locally known unsafe Local evidence says the terminal or drawer cannot safely append sale-affecting work. Hard block with recovery path.
Cloud-validation uncertainty The cloud cannot validate now, but local recording remains possible. Continue sale locally and reconcile later.

Implementation Units

U1. Allow POS route continuity while recovery waits for network

Change the route shell so trusted POS terminals mount the register while app-session recovery is waiting for network. Keep blocked recovery and non-POS redirects unchanged.

Primary tests: src/routes/_authed.test.tsx, usePosTerminalAppSessionRecovery.test.ts.

U2. Create the POS sale-blocker audit policy

Encode the preliminary audit table as a shared decision boundary for the read model and command gateway. Any newly discovered sale stop must be classified before it remains a hard block.

Primary tests: saleBlockerPolicy.test.ts, registerReadModel.test.ts, localCommandGateway.test.ts.

U3. Let local read model and command gateway continue sales under cloud uncertainty

Apply the blocker policy to sale-affecting local commands so new sales and active sale completion proceed locally when only cloud validation is missing.

Primary tests: local read-model, command-gateway, and register view-model tests.

U4. Carry app-session uncertainty into sync and reconciliation metadata

Preserve support-safe evidence for events recorded while app-session validation was unavailable, then accept clean history or route mismatches to review on reconnect.

Primary tests: local store, sync contract, sync scheduler, runtime status, and Convex sync tests.

U5. Keep cashier presentation normal while preserving support visibility

Remove normal-flow app-session/review interruption from the cashier path while preserving hard-block recovery copy and terminal-health diagnostics.

Primary tests: POS register, opening guard, drawer gate, terminal health, and readiness tests.

U6. Add no-network sales-continuity regression coverage

Prove the browser case end to end: no network and no reachable Convex backend, missing app session, local cashier sign-in, new sale start, active payment/completion, and local evidence persistence without Convex reads, mutations, or sync.

Primary tests: src/tests/pos/offlineSalesContinuity.spec.ts.

U7. Document the final blocker policy and validation map

Add solution guidance and harness mapping so future POS changes do not reintroduce cloud-validation sale stops.

System-Wide Impact

  • Interaction graph: route classification feeds recovery context, register view-model, sync runtime, terminal health, and browser no-network behavior.
  • Error propagation: cloud failures become support-safe reconciliation categories; local impossibility and unsafe evidence remain hard blockers.
  • State lifecycle: app-session-unverified local events must survive reload and sync retry without storing reusable credentials.
  • Unchanged invariants: non-POS routes still require app auth; local staff authority remains separate from app-session recovery.

Risks

Risk Mitigation
Converting too many blockers could allow sales under locally unsafe state. Use the explicit taxonomy and retain locally unsafe or impossible states as hard-block candidates.
Quiet cashier UI could hide support work. Keep reconciliation in terminal health, sync diagnostics, and manager/support surfaces.
No-network browser tests may be brittle in CI. Reuse existing offline route access patterns and assert DOM/local evidence rather than network idle.
Sync may still be blocked after reconnect if the app session is missing. Keep local sales durable and defer upload until supported recovery or POS-scoped validation succeeds.
