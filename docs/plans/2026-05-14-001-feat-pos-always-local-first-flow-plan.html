Athena plan May 14, 2026 POS local-first

Make the POS register local-first even when internet is available

This follow-up plan inverts the landed offline fallback: cashier actions should write to local POS state first, render from that local state, and let cloud sync project the register history in the background.

Problem Frame

The current implementation has the local store and sync rails, but normal online register actions still try Convex first and only fall back locally after unavailable or unknown failures. That does not satisfy the origin requirement that POS loads and operates from local state first every time.

The plan keeps the existing POS-only boundary. It does not make analytics, procurement, admin, staff management, or cash-control review offline-first.

Requirements

Cashier Path

  • Append local events before cashier success, online or offline.
  • Render active register, sale, cart, checkout, and closeout from local state.
  • Do not wait for cloud sync to make the sale complete.
  • Preserve current payment methods and payment semantics.

Cloud Path

  • Keep Convex sync ingestion as the cloud acceptance boundary.
  • Project local history into register, transaction, inventory, payment, and trace records.
  • Return mappings and conflicts to update local state.
  • Separate pending sync from manager review work.

Provisioned Terminal

  • Use local seed, staff proof, and catalog snapshot after online provisioning.
  • Support reload and temporary network loss without blocking POS.
  • Fail closed for missing, expired, or scope-mismatched staff proof.

Explicit Non-Goals

  • No offline new-business creation or first-time provisioning.
  • No peer-to-peer disconnected sync.
  • No real-time cross-terminal stock coordination.
  • No new payment-provider authorization behavior.

Key Technical Decisions

Primary boundary

Local POS command gateway becomes the normal register command path.

Read model

Local register state is explicit and replayable from IndexedDB events.

Sync

Existing Convex ingestion and projection remain the cloud acceptance path.

Auth

Server-issued POS staff proof is persisted locally with expiry and scope.

Catalog

Register search uses the latest local snapshot first; online refresh is opportunistic.

Closeout

Local closeout pauses selling until a permitted local reopen event is recorded.

Design Shape

The implementation should make local storage the first durable write, then treat sync as background projection.

1. Cashier action Open, add item, pay, complete, closeout, or reopen.
2. Local command Validate local authority and append event.
3. Local render Update register read model and show success.
4. Background sync Upload ordered pending events when possible.
5. Projection Cloud records, mappings, and conflicts catch up.

Implementation Units

U1

Create the local register read model

Reconstruct active register, active sale, cart, payments, closeout, receipt identity, mappings, and sync state from local seed plus ordered local events.

Core files: posLocalStore.ts, registerReadModel.ts, syncStatus.ts

U2

Add the primary local POS command gateway

Implement local-first register open, sale start, cart edits, payment capture, transaction completion, closeout, and reopen.

Core files: ports.ts, localCommandGateway.ts, syncContract.ts

U3

Wire the register view model to local state first

Replace online-primary branches in useRegisterViewModel with the local read model and command gateway.

Core files: useRegisterViewModel.ts, registerUiState.ts, POSRegisterView.tsx

U4

Persist and refresh local staff authority

Store scoped staff proof material locally so provisioned terminals can sign into POS during temporary network loss.

Core files: staffCredentials.ts, CashierAuthDialog.tsx, localStaffAuthority.ts

U5

Make catalog and availability local-first

Hydrate register search from a local catalog snapshot first, while online catalog rows refresh the snapshot opportunistically.

Core files: localCatalogSnapshot.ts, catalogGateway.ts, useRegisterCatalogIndex.ts

U6

Keep sync projection aligned with richer local events

Ensure always-local register history still syncs idempotently into cloud records and returns mappings/conflicts to local state.

Core files: syncContract.ts, ingestLocalEvents.ts, projectLocalEvents.ts

U7

Add app-shell and reload readiness

Distinguish provisioned-ready, needs-provisioning, proof-expired, catalog-missing, local-store-unavailable, and schema-unsupported states.

Core files: offlineReadiness.ts, pos/register.index.tsx, POSRegisterOpeningGuard.tsx

U8

Update tests, docs, and validation coverage

Make future online-first regressions visible in tests, validation maps, generated docs, and solution notes.

Core files: validation-map.json, docs/solutions/architecture/..., harness registry files if needed.

Risks & Mitigations

Risk Mitigation
Local store corruption blocks register operation. Schema handling, explicit unsupported states, and failed-write tests before cashier success.
Staff proof storage weakens security. Persist scoped proof tokens only; enforce expiry, terminal, store, and role scope; validate again during sync.
Online refresh overwrites active local sale state. Treat Convex reads as seed/snapshot refresh only; local event history remains active source.
Stale catalog creates inventory or price drift. Preserve completed receipts and route drift to reconciliation instead of rewriting local history.
Cloud projection duplicates records. Keep local event ids, strict sequence, mappings, and retry equality checks in server ingestion.
