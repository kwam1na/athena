feat: Make POS route accessible with no network

This plan closes the remaining route-access gap in Athena's local-first POS work: an already-provisioned terminal should be able to open or hard reload the POS route after total network loss.

Scope

The minimum offline route contract is /pos plus /pos/register. The service worker provides the static app shell; POS business state remains in the existing IndexedDB local stores.
  • Already-provisioned terminals only; the terminal must load POS online once after this feature is deployed so the browser can install the service worker and cache the static app shell.
  • POS remains the only offline-first Athena workflow.
  • Cache Storage must not hold staff proofs, sync secrets, payments, customer data, or POS event payloads.
  • Expense POS and offline-friendly POS history/report subroutes are deferred unless later requirements include them.

Requirements

  • Previously provisioned POS terminals can load POS with no active network after POS has loaded online once, giving the browser a chance to install the service worker and cache the static app shell.
  • /pos and /pos/register render from route slugs and local terminal seed when live reads are unavailable.
  • Offline route continuity does not widen POS app-session recovery into generic Athena access.
  • Register behavior continues through local staff authority, local readiness, local command gateway, catalog snapshots, and sync status.
  • Browser-level verification proves hard reload with network blocked reaches the local register/sign-in path.

Key Decisions

Focused service worker

Add POS app-shell caching rather than making every Athena route offline-first.

Static shell only

Cache HTML, JS, CSS, and static assets; keep business data in POS IndexedDB.

Route-scoped continuity

POS app-session recovery remains limited to POS hub/register routes.

Browser proof

Hard-reload no-network coverage is required because unit tests cannot prove app-shell delivery.

Implementation Units

U1. Add POS app-shell service worker infrastructure

Add build/runtime service-worker support, route policy helpers, and startup registration. Test POS navigation classification and excluded request types.

U2. Make POS hub and register route entry local-first under offline shell

Use local entry context for POS-critical links and preserve non-POS route protection. Test hub/register render paths without live org/store reads.

U3. Surface offline readiness for app shell and POS local prerequisites

Expose app-shell, terminal seed, staff authority, catalog, and availability readiness without leaking sensitive diagnostics.

U4. Harden register reload continuity around local staff and drawer state

Characterize and protect the offline reload path through local staff sign-in, drawer state, register readiness, and local command gates.

U5. Add no-network browser regression coverage for POS route accessibility

Prepare online, block network, hard reload POS, and assert the local register/sign-in shell renders without server access.

U6. Document the POS offline route boundary and validation map

Record the service-worker app-shell versus IndexedDB business-state boundary and wire future validation to the new test slice.

Risks

Risk Mitigation
Service worker caches sensitive data. Restrict caching to static app-shell assets and exclude API/Convex/fetch payloads.
Stale cached chunks break POS after deploy. Use build/precache manifests and clean outdated caches coherently.
Offline shell is mistaken for sale authority. Keep app shell separate from terminal, drawer, staff proof, and local command gates.
POS continuity leaks to non-POS routes. Preserve route-scope tests in _authed.test.tsx.

Verification Strategy

  • Unit-test service-worker route policy and excluded request types.
  • Unit-test POS hub/register route rendering from local entry context when live reads are absent.
  • Characterize register reload behavior around staff authority, drawer state, and local command gates.
  • Add a browser regression that blocks network and hard reloads /pos/register.
  • Validate the HTML review artifact with scripts/validate-plan-html.ts.

The plan intentionally does not implement code in this planning pass.
