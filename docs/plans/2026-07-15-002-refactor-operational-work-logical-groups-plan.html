Project logical Open Work groups across operations

Problem and outcome

Open Work currently groups synced-sale inventory reviews by SKU only in React. The other operations surfaces count or persist raw per-sale rows, and grouped resolution runs as sequential single-row commands. The result is cross-surface count drift, post-cap grouping, and a partial-write risk.

Target outcome: operators see one SKU-scoped unit everywhere and acknowledge it once, while Athena retains the exact sale-level members and their source validation. EOD freezes that membership; Opening never expands it from current work.

Scope and invariants

Logical groupingOnly current synced-sale inventory reviews with the same store and canonical SKU consolidate.
Raw authorityEvery sale-linked work item, source mapping, proof check, terminal status, and event remains authoritative.
Frozen handoffEOD persists the complete raw member set; Opening requires one key for that frozen set.
Truthful limitsIncomplete raw reads produce a logical lower bound and fail closed for resolution and EOD completion.
Safe compatibilityNew snapshots are versioned; legacy history renders exactly as stored as singleton items.
UI continuityExisting layouts and selected-filter heading copy remain; unrelated work types are unchanged.

Explicit exclusions

  • No write-time merge into one mutable SKU record.
  • No grouping by title, receipt, product name, or other weak metadata.
  • No new Daily Operations resolution authority.
  • No consolidation of approvals, pending-checkout work, or unrelated work types.
  • No destructive migration of operational-work or event history.
  • No retrospective regrouping or backfill of legacy EOD snapshots.
  • No broad workspace redesign or agent-owned manual browser signoff.

Technical shape

The projection is server-owned and pure. Synced-sale aliases stay attached to one logical sale member; unrelated work types pass through unchanged. Source completeness owns exact versus lower-bound counts, while resolution availability distinguishes normal, oversized, remediation, and incomplete states. No signed snapshot token is required.

Implementation units

U1 · Canonical logical-work projection

Foundation · no dependencies

Create orthogonal exhaustive source-completeness and resolution-availability unions for available, oversized, remediation, and incomplete states.

U2 · Queue DTO and Open Work consumption

Depends on U1

Project before logical caps, return sanitized member evidence, remove React grouping, and preserve header/filter/mix/pagination behavior.

U3 · Atomic grouped resolution

Depends on U1 and U2

Re-derive the complete current group and require exact submitted membership equality before transitioning at most 50 rows. Any concurrent change writes nothing and asks the operator to review again; larger groups use the internal repair runbook.

U4 · Daily Operations parity

Depends on U1

Use logical units for the Open Work lane and attention feed, preserve routing ownership, and expose lower-bound counts when incomplete.

U5 · EOD freeze and immutable history

Depends on U1

Show one carry-forward group, persist every representative/alias ID and versioned display snapshot, block incomplete freezes, and leave legacy snapshots exactly as stored.

U6 · Opening acknowledgement

Depends on U1 and U5

Reconstruct only frozen versioned groups, keep legacy items singleton, bind one key to complete membership, and make missing evidence non-bypassable across every start path.

U7 · Cross-surface contract and sensors

Depends on U2–U6

Prove parity with shared fixtures, add the durable solution note, keep validation routing current, refresh Graphify, and satisfy merge-grade repository gates.

Primary risks

Group exceeds Convex transaction limitsCap operator transactions at 50 rows; route row 51 to a support-authorized repair with frozen membership, bounded scheduled batches, pause/resume, and one projected remediation state.
Legacy history is falsely regroupedVersion new snapshots only; render old snapshots exactly as stored and perform no inferred backfill.
Raw IDs leak through broad payloadsUse typed membership fields, explicit redactors, and negative payload tests.
Current rows mutate completed historyPersist display aggregates and membership at close time; render persisted evidence.
Terminal or missing members shrink evidenceTrack frozen and unresolved membership separately; missing evidence blocks normal, manager, and automated starts.
Forged, partial, or stale client membershipRe-derive the complete current group inside the mutation and require exact representative/alias equality before writes.
Duplicate aliases resurfaceCount once for display but retain every alias in completeness, freeze, cap, and atomic terminal/supersession evidence.
Event dedupe scans become unboundedInsert after terminal preconditions or use an indexed business-event key; prove deep existing history does not change 50-row cost.
Repair authorization or actor provenance is weakUse internal mutations behind a deploy-credential runbook and persist immutable initiator, reason, ticket, organization, and store evidence.
Alias arrives during repairRecheck before batches/finalization, pause and audit-amend membership, retain completed evidence, and require a final no-alias proof.

Verification strategy

  • Projection and queue backend tests
  • Atomic resolution and no-partial-write tests
  • Exact-current-membership tests for added, removed, terminalized, and alias rows
  • Normalized zero-write “review again” conflict and refreshed UI tests
  • 50-row success and 51-row zero-write boundary tests
  • Oversized support repair pause/resume/event-idempotency tests
  • Repair internal-only authorization and immutable actor/ticket evidence tests
  • Repair alias-before/between/finalization concurrency tests
  • Duplicate-alias resolution, EOD, and Opening lifecycle tests
  • Daily Operations count/attention parity
  • EOD live/completion/history/redaction tests
  • Opening acknowledgement and evidence tests
  • Open Work, Daily Operations, EOD, History, and Opening component tests
  • Legacy singleton and new versioned-snapshot compatibility fixtures
  • Incomplete source and oversized group failures
  • Trusted versus broad-reader payload checks
  • Route/browser-boundary coverage
  • Convex audit and changed lint
  • Frontend changed lint, typecheck, and build
  • Graphify rebuild and freshness
  • Compound/documentation and landed-report gates when triggered
  • Merge-grade pr:athena

Manual browser validation remains delegated to the user. The implementation plan requires automated behavior and boundary evidence before that handoff.
