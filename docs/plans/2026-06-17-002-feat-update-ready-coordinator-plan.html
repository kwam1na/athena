Summary

Build a shared Athena webapp update coordinator that separates new-build detection, background asset staging, surface-owned reload/apply safety, and controlled reload.

The active slice replaces direct auto-reload with a visible Update Ready affordance everywhere, then proves the foundation with POS register and Inventory Import. POS is not waiting off an automatic reload; it only reports when manual update apply would be unsafe.

Requirements

  • Detect a newer build without immediately reloading the current tab.
  • Stage static shell assets when possible while keeping business data out of Cache Storage.
  • Render a persistent Update Ready affordance until the update is applied or the app is current.
  • Let surfaces register unsafe-to-reload reasons through a shared apply-blocker registry.
  • Apply the update through a controlled reload/restart boundary.
  • Use manual apply through Update Ready for the first release; future auto-apply needs its own explicit safety policy.
  • Coordinate update and apply-blocker state across same-origin tabs.
  • Use calm Athena product copy and avoid implementation terms in operator-facing UI.

Key Decisions

  • Coordinator over route predicates: safety is state-owned by surfaces, not hard-coded into paths.
  • Detection separate from apply: build checks emit state; they do not reload directly.
  • Manual v1 apply: all detected updates render Update Ready; reload only happens from the local tab after local and remote blockers clear.
  • Best-effort staging: asset warming helps the next reload but does not prove workflow safety.
  • Reload remains the apply boundary: the production React module graph is not hot-swapped.
  • Cross-tab leases: tabs share update and blocker snapshots, but remote messages cannot clear local blockers or force reload.
  • Service-worker lifecycle must be explicit: staging and control/apply cannot collapse into the same event.

Design Shape

Detect

Use deploy metadata where available and keep HTML/script comparison as fallback.

Stage

Warm entry-linked static app assets without caching API, Convex, payment, customer, or workflow payloads.

Block Apply

Allow POS, Inventory Import, and future workflows to register active apply blockers when manual refresh would be unsafe.

Apply

Reload once through a controlled, idempotent coordinator action when no local or unexpired remote blocker remains.

Implementation Units

  1. U1. Create the update coordinator state model - state machine, provider, apply-blocker registry, cross-tab leases, message validation, and apply latch.
  2. U2. Convert version detection into coordinator events - reuse deploy metadata and HTML/script fallback without direct reload, with an explicit detector-to-stager contract.
  3. U3. Add a static asset staging adapter - warm entry-linked static assets while preserving cache exclusions.
  4. U4. Separate service-worker staging from apply/control - decide activation/cache policy early and keep old/pending caches safe for blocked clients.
  5. U5. Render the global Update Ready affordance - persistent, accessible UI with no apply action while blockers are active.
  6. U6. Add surface opt-in apply-blocker hooks - generic hook, explicit priority model, and presentation helpers for current blockers.
  7. U7. Adopt apply safety in POS register - POS-owned safety signal derived from workflow truth, generic registry payload.
  8. U8. Adopt apply safety in Inventory Import - non-POS proof using autosave and review-version state.
  9. U9. Add browser and validation coverage - production-like two-build update-ready flows and harness routing.

Risks

Risk Mitigation
Service-worker activation bypasses blocked apply semantics. Choose the activation/cache policy early and test old/pending cache coexistence plus staging versus apply/control.
Blockers get stuck after route changes or clear too early. Use workflow source-of-truth for POS/import blockers and add stale cross-tab expiry plus remount tests.
Update Ready becomes noisy. Use a compact persistent affordance and restrained copy instead of modal interruption.
Static staging caches sensitive data. Reuse and test the existing app-shell exclusions for API and business payloads.

Verification Strategy

  • Unit-test coordinator state transitions, blocker lifecycle, staging results, and version detection.
  • Component-test the Update Ready banner and first adopter blockers.
  • Browser-test a production-like v1-to-v2 fixture for detection, blocked apply, and one-time controlled reload.
  • Keep existing POS offline route access and sales continuity tests passing if the service worker changes.
  • Prove cross-tab leases and message validation at unit/integration level, with browser coverage if the harness can support it cleanly.
  • Update validation routing so future app-update and app-shell edits run the right focused tests.
The plan leaves Daily Close, cycle count, support visibility, and emergency forced-update policy for follow-up work after the foundation is proven.
