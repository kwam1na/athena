refactor: Add app-wide message action foundation

Date: 2026-06-20 Type: refactor Status: active Markdown: source
Summary. Create an app-wide messaging/action foundation that owns placement, priority, and action-scoped blockers, then rebuild app update as the first implementation on top of it. App update keeps detection, staging, runtime evidence, and reload safety as adapter-owned behavior.

Core Decisions

Foundation First

The platform layer is app-wide messages/actions, not app update. App update is the first adapter.

Action-Scoped Blockers

Blockers attach to an action such as app-update.apply, preventing unrelated actions from being suppressed.

Provider Ownership

AppMessagesProvider mounts above app update; app update consumes action state while preserving pending-build scoped leases.

Adapter Execution

The foundation gates and renders actions. App update still owns reload/apply execution.

Surface Truth

POS and Inventory Import decide when they block; the foundation does not compute domain state.

Implementation Units

U1. Create app-wide message/action foundation

Add only the neutral action-blocker mechanics app update needs now: action-scoped registration, selected-blocker resolution, provider/context, and tests.

U2. Move update communication preference into messaging

Convert banner/toast placement from app-update-specific preference into app-message placement policy while preserving current behavior.

U3. Adapt app update to publish through app messages

Publish the update-ready message and app-update.apply action through the foundation while preserving update coordinator reload safety.

U4. Migrate POS and Inventory blockers to the foundation

Move first adopter opt-ins from app-update hook imports to the action-scoped messaging contract.

U5. Verify terminal update adapter compatibility

Use POS local runtime, Remote Assist runtime host, and terminal runtime tests as a compatibility gate; modify terminal files only if the app-update adapter API actually changes.

U6. Document the new boundary and validation routing

Update solution docs, validation routing if needed, and Graphify after code changes.

Verification Strategy

  • Foundation tests for action-scoped blockers and unrelated action availability.
  • App-update coordinator and Update Ready UI tests for behavior parity.
  • POS and Inventory component tests for adopter opt-in behavior.
  • Terminal runtime/recovery tests if app-update snapshot mapping changes.
  • Typecheck, build, graphify rebuild, and the Athena PR gate before merge.
