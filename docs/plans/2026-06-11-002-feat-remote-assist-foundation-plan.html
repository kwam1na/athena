feat: Add Athena Remote Assist Foundation

feat Status: active Date: 2026-06-11 Markdown: 2026-06-11-002-feat-remote-assist-foundation-plan.md

Summary

Add a generic Remote Assist foundation for remotely viewing and controlling enrolled Athena browser clients, then connect POS terminals as the first adapter. Unattended sessions use sanitized in-app co-browsing instead of permission-gated screen capture; attended sessions can optionally upgrade to browser screen share.

Problem Frame

Athena can already diagnose and remotely recover some POS terminal-health conditions, but support still cannot operate the affected browser session when a field terminal needs hands-on guidance. The first case is the M Supplies terminal, but the durable primitive is broader: an authenticated Athena browser client should be able to enroll as remote-assistable, accept an attended or unattended support session under policy, and provide bounded view/control without bypassing the business authority gates of the surface it is running.

Core Requirements

  • Model remote-assistable Athena browser clients independently of POS terminals.
  • Support unattended-by-default policy for enrolled clients, with attended and disabled modes available.
  • Issue short-lived transport tokens only after role, policy, enrollment, presence, and runtime checks pass.
  • Provide unattended-safe in-app co-browsing plus bounded low-latency control messages through a provider-neutral adapter.
  • Mask or block sensitive inputs and secrets including PINs, recovery codes, payment credentials, sync secrets, and staff proof material.
  • Audit every session, policy decision, lifecycle event, and privileged recovery action.
  • Use POS terminals as the first adapter while preserving POS authority gates and runtime-check-in verification.

Scope Boundaries

Remote Assist controls the Athena browser app session. It is not arbitrary desktop administration, does not expose devtools or local storage editing, and does not bypass drawer, staff, terminal-integrity, Cash Controls, Operations, sale, payment, inventory, or closeout authority.
  • No direct browser IndexedDB/local storage mutation by support controls.
  • No screen-share-only unattended mode; browser capture requires local permission.
  • No custom realtime/media stack unless provider research proves it necessary.
  • No non-POS adapters in v1.
  • No fleet-wide automatic remediation in this first implementation.

Design Shape

Support user -> Remote Assist API -> Policy and audit -> Transport provider
                                      |                      |
                                      v                      v
                              Enrolled browser client -> Runtime adapter
                                                             |
                                                             v
                                                    POS terminal recovery

The platform owns policy, sessions, audit, enrollment, co-browsing transport, optional attended screen-share upgrade, and provider tokens. The POS adapter contributes terminal identity, terminal-sensitive regions, Terminal Health launch, and existing recovery commands.

Key Decisions

  • Build the foundation around generic remote-assistable Athena browser clients, not POS terminals.
  • Use sanitized in-app co-browsing for unattended default; use browser screen-share only as an attended upgrade.
  • Use a provider-neutral transport adapter and prefer a vetted realtime provider or rrweb-style event stream over hand-rolled signaling/media infrastructure.
  • Make unattended access a policy result for enrolled clients, not an unconditional support power.
  • Keep support control input bounded to application events and adapter-approved actions.
  • Keep sensitive-entry masking in the runtime adapter because each surface knows its own sensitive UI.
  • Keep POS terminal recovery verification based on existing terminal runtime check-ins.

Implementation Units

U1. Remote Assist domain model

Add schemas and repositories for clients, sessions, lifecycle events, capabilities, and redacted audit fields.

U2. Policy and session lifecycle

Evaluate role, store/platform policy, enrollment, presence, mode, reason, and TTL before session start or claim.

U3. Provider-neutral co-browsing and transport adapter

Define server/client room, token, sanitized DOM co-browsing, optional attended screen-share, and bounded control-message contracts around a vetted provider.

U4. Browser runtime shell

Mount runtime presence, connected banner, attended approval, local disconnect, sensitive masking, and control gates.

U5. POS terminal adapter

Enroll active terminals, mark POS-sensitive regions, launch from Terminal Health, and preserve recovery verification.

U6. Support UI

Add Remote Assist session controls, status, reason capture, and audit snippets to Terminal Health detail.

U7. Audit, docs, validation

Add structured audit coverage, harness mapping, a solution note, rollout controls, and operational docs.

Validation Strategy

  • Convex schema, policy, session, transport adapter, and audit tests.
  • React runtime provider, sensitive-region, banner, attended approval, and support UI tests.
  • POS terminal adapter tests covering M Supplies-shaped recovery handoff and authority preservation.
  • Harness validation-map coverage for Remote Assist backend, runtime, transport, and Terminal Health surfaces.
  • Solution documentation for the generic browser-client foundation, provider-neutral transport, unattended co-browsing boundary, sensitive-control restrictions, and POS authority invariants.
  • Repo gates after implementation: focused package tests, Convex audit/lint, TypeScript, build, graphify rebuild, harness review, inferential review, and bun run pr:athena.

Risks

RiskMitigation
Remote control becomes an unsafe admin bypass.Use explicit Remote Assist permission, bounded input events, sensitive masking, recovery command boundaries, and no direct storage mutation.
Provider lock-in leaks through app code.Isolate provider SDK/API use behind transport adapter interfaces.
Browser screen capture blocks unattended sessions.Make unattended mode use sanitized app-DOM co-browsing; reserve screen share for attended upgrade.
Unattended default creates trust concerns.Require enrollment, visible banner, local disconnect, reason, TTL, policy override, and audit.
Sensitive values leak through stream or audit.Mask sensitive UI, pause control during sensitive entry, reject secret-like audit payloads, and test PIN/proof/payment redaction.
