feat: Add Athena Automation Foundation

Type: feat Status: active Date: 2026-06-08 Source: markdown plan

Summary

Add a guarded automation foundation that lets Athena act as a first-class operational actor across product workflows. Daily Operations is the first implementation: clean Opening Handoffs can be auto-started, EOD Review can be prepared for human review, and the manager-approval boundary for EOD completion stays intact.

Scope

In Scope

  • Reusable Athena automation actor identity.
  • Domain-neutral action registry, policy, idempotency, and run ledger.
  • Operational event attribution for automation decisions.
  • Clean Opening Handoff auto-start.
  • EOD Review preparation and routing.
  • Daily Operations visibility for decisions and skips.

Out Of Scope

  • No silent EOD Review completion.
  • No auto-acknowledgement of variances, voids, or carry-forward work.
  • No drawer, register, payment, inventory, or approval-proof creation.
  • No Daily Operations-only foundation abstraction.
  • No global action-mode rollout by default.

Key Decisions

Explicit Automation Identity

Athena is represented through automation metadata, not a fake staff profile.

Foundation First

Shared automation primitives are built before Daily Operations consumes them.

Deterministic Policy

Decisions come from store-day snapshots, policy mode, source subjects, and a versioned policy.

Opening First

Opening has the safest mutation boundary because clean days require no human acknowledgement.

EOD Preparation

EOD automation records readiness and review routing while preserving manager approval.

Design Shape

The automation foundation owns scheduling, registered actions, store-scoped policy, idempotency, durable run history, and Athena actor attribution. Daily Operations supplies the first domain adapter, which reads Opening and EOD snapshots, records automation decisions through the shared ledger, and applies only eligible clean Opening actions. EOD outcomes become prepared or review-needed evidence, not completed close records.

Layer Automation Outcome Boundary
Automation foundation Registers action, evaluates policy, records run, attributes Athena actor Domain-neutral; no Daily Operations imports
Daily Operations adapter Supplies Opening and EOD snapshots, source subjects, and action handlers First implementation on top of foundation
Clean Opening Athena starts Opening Handoff Mutation allowed only in store-scoped action mode
Opening with review or carry-forward Skipped with human-action reason No mutation
EOD Review Prepared blocker, review, or clean-ready evidence No close completion in this plan

Implementation Units

U1. Add Automation Foundation Primitives

BackendFoundationAdd domain-neutral action definitions, policy/run schemas, idempotency, index coverage, and optional automation actor metadata on operational events.

U2. Build Shared Automation Evaluation And Execution Boundary

BackendPolicyEvaluate registered actions into disabled, dry-run, skipped, prepared, eligible, applied, or failed outcomes before domain handlers mutate state.

U3. Add Daily Operations Automation Adapter

BackendOperationsRegister Opening auto-start and EOD preparation as the first domain actions built on top of the foundation.

U4. Surface Daily Operations Automation Status

FrontendOperationsShow what Athena did, prepared, skipped, or failed in the relevant Operations surfaces.

U5. Document And Validate The Automation Boundary

DocsValidationAdd reusable solution guidance for the foundation and first Daily Operations implementation, then rebuild graphify after code changes.

Risks

Risk Mitigation
Athena appears to impersonate staff Use explicit automation actor metadata and keep human actor fields unchanged.
Foundation becomes Daily Operations-only Keep shared automation modules domain-neutral and add tests that action definitions/runs do not require Daily Operations imports.
Cron duplicates lifecycle records Use store/date/action idempotency and event dedupe.
EOD automation weakens review discipline Keep EOD preparation-only and test manager approval proof preservation.
Global rollout acts unexpectedly Default to disabled/dry-run and require store-scoped action enablement.

Verification Strategy

  • Foundation tests cover action registry, automation policy, run ledger, idempotency, and event actor metadata without depending on Daily Operations modules.
  • Daily Operations adapter tests cover Opening action mode, dry-run mode, disabled mode, and EOD preparation mode.
  • Regression tests prove manual Opening still requires human actor authorization and EOD completion still requires manager approval proof.
  • Frontend tests cover Daily Operations, Opening, and EOD Review automation status copy and links.
  • Graphify is rebuilt after implementation changes.
