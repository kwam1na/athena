Decision

The authority channel fixes the split between Convex presentation evidence and IndexedDB command authority. The remaining problem was legacy local mapping inference. That work is removed.

Approved destructive posture: business events have already projected to the server and remaining local review items are stale. Every non-seed local business event may be deleted.

The one-time IndexedDB transaction removes business events, register-session mappings, and drawer authority. It preserves terminal seed and integrity, cashier presence, staff authority, readiness, catalogs, and sequence counters.

Fleet scope

Dev — 7 terminals

Arc, Codex, Codex Authority Test, Codex Cash Controls Test, Codex Cash Controls Headless, Codex II, and Olorin run the reset automatically on register startup.

Production — 6 terminals

Mission Control, M Supplies, Wigshop, iPhone, Codex, and Olorin receive the same deterministic one-time cutover.

Identity invariant: terminal seed, sync secret, cloud terminal ID, register number, and cashier presence are unchanged. Neither environment can gain a duplicate terminal.

Implementation units

U1 + U8 — Characterize and preflight

Lock the old-ID reuse regression and prove the exact terminal-authenticated query in read-only shadow mode.

U2 + U3 — Server ordering and query

Add lifecycle/mapping revisions and return exact bounded authority without heartbeat coupling or latest-session guesses.

U4 + U5 — Atomic local persistence and runtime

Persist versioned authority before refresh. New candidates come only from post-reset active/pending drawers and new mappings; legacy inference is removed.

U6 — Sole local decision source

Remove direct cloud operational gating. Exact local closure creates a distinct replacement; non-empty carts remain read-only until explicit clear.

U9 — Apply targeted reset to both fleets

Atomically delete business events, register-session mappings, and drawer authority before projection while preserving terminal/operator state and counters.

U10 — Verify the complete flow

Open, sell, reload, close from Cash Controls, persist the block, open a distinct replacement, sell again, reload, and record representative dev/production evidence.

U7 — Close rollout evidence

Confirm all 13 terminals run supported builds on post-reset identities while legacy directive retirement remains a separate PR.

Rollout

  1. Dev foundation: deploy server then client; verify the automatic reset and complete flow.
  2. Production foundation: deploy identical server contract then supported client; verify build adoption.
  3. Automatic production cutover: each terminal resets once on its next register startup.
  4. Closure: retain the original 7 dev and 6 production terminal IDs, zero duplicates, and no stale business/drawer projection.

No attended site-data clear or reprovisioning is required. A persistent reset failure blocks only the affected terminal.

Verification strategy

  • Seven dev and six production terminals retain their original cloud IDs and register assignments; no duplicate active terminals.
  • Every reset store begins without business events, register-session mappings, or drawer authority while terminal/cashier/staff/catalog state remains.
  • Every terminal creates a new drawer, completes a sale, consumes Cash Controls closure through the dedicated lane, opens a distinct replacement, and completes another sale.
  • Reload preserves new post-reset state and product lookup remains usable.
  • No persistent Drawer status not saved gate, closed local ID reuse, closed cloud-session reuse, or heartbeat dependency.
