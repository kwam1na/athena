Problem and intended outcome

Arc's reactive cloud register query knew the mapped register session had been closed from Cash Controls, so the view model displayed the drawer gate. Heartbeat was paused, so the runtime-status response that normally persists cloud_closed never arrived in IndexedDB. The local command gateway therefore saw the old local drawer as healthy and returned its ID as an idempotent open.

Root defect: presentation and commands made one lifecycle decision from different authority sources.

The redesign replicates the exact cloud lifecycle consequence into durable local authority first. The drawer gate and openDrawer then consume the same local projection. A remote close remains cloud financial truth; the terminal stores only its authority consequence.

Required contract

Local-first remains intact

Drawer and sale commands append locally and never wait for a Convex mutation. Cloud state is replicated evidence, not a command prerequisite.

Exact subject only

Authority is scoped to store, terminal, register, local session, and mapped cloud session. Foreign-scope cases are externally indistinguishable.

Monotonic persistence

A server-owned lexicographic cursor compares the durable mapping epoch first, then lifecycle revision only within the same exact cloud-session subject.

No invented finance

Remote closure does not create local counted cash, variance, approval, closeout, or manager-review events.

Offline is last-known truth

A disconnected or reconnecting terminal uses last-known local truth until the first authoritative snapshot. Intervening local facts stay attached and reconcile to review.

History stays attached

Replacement uses a distinct local ID. Old carts, transactions, mappings, and hard authority evidence remain tied to the original drawer.

Authority flow

Cash Controls / sync projection │ lifecycle mutation + revision ▼ Register session ─── exact mapping ───► Terminal authority query │ versioned exact subject ▼ IndexedDB compare-and-apply │ durable commit ▼ Local register projection ┌───┴────┐ ▼ ▼ Drawer gate Local commands │ ▼ distinct register.opened Runtime heartbeat ─ ─ diagnostics only ─ ─► Terminal Health / recovery

Loading, offline state, transport error, and skipped query input are non-authoritative and perform no local mutation. Persistence precedes read-model refresh, so a crash after commit recovers from IndexedDB on restart.

Implementation units

U1. Characterize the split-authority incident

Dependencies: none

Reproduce the exact heartbeat-disabled, cloud-closed, locally-open drawer flow and protect healthy duplicate-open idempotency.

U8. Validate the query contract in read-only shadow mode

Dependencies: U1

Prove terminal authentication, exact 16-candidate scoping, reactive behavior, and a 40-document-read worst-case budget before durable revision infrastructure lands.

U2. Add monotonic lifecycle and mapping authority revisions

Dependencies: U8 preflight gate

Add lifecycle revision, a durable per-local-drawer mapping-authority revision/tombstone, centralized transition helpers, and a repository-wide writer guard.

U3. Expose an exact terminal lifecycle-authority query

Dependencies: U1, U2

Add a terminal-sync-secret query for at most 16 prioritized local candidates. Foreign-scope probes are indistinguishable; missing event history remains closure-reconcilable but is not reconstructed in this delivery.

U4. Add an atomic IndexedDB authority reducer

Dependencies: U1, U2

Compare mapping-authority epoch first and lifecycle revision only inside the same exact mapping epoch. Repair and deletion preserve tombstones and remain monotonically ordered.

U5. Mount heartbeat-independent inbound replication

Dependencies: U3, U4

Compose a focused runtime under useRegisterLocalRuntime, mounted even when idle or heartbeat-paused. Canary acknowledgements and the authority-persistence failure guard land here before gate cutover.

U6. Make local projection the sole drawer decision source

Dependencies: U5

Remove the direct cloud blocker. Repair states stay actionable; a non-empty old cart remains read-only until explicitly cleared before opening an empty replacement sale.

U7. Roll out with mixed-client compatibility

Dependencies: U6 plus supported-client adoption milestone

Shadow, canary, and expand with legacy delivery retained. U5 already supplies a one-row-per-terminal, cursor-validated, redacted acknowledgement sink independent of heartbeat.

Phased rollout

  1. Read-only preflight: characterize the incident and prove exact query identity/read-cost assumptions without local application.
  2. Additive server foundation: after preflight, add lifecycle/mapping revisions and the versioned query; keep legacy directives.
  3. Shadow and canary client: classify first, then apply for Arc and a small heartbeat-disabled cohort.
  4. Sole local operational projection: remove direct cloud gating after canary proof and expand by supported build.
  5. Compatibility retirement follow-up: separately retire closure directives after adoption proof and active-session directives only after exact-mapping/recovery parity.
Expansion gate: zero closed-ID reuse, zero cross-terminal adoption, durable restart behavior, and a distinct sale-usable cloud mapping for the replacement drawer.

Rollback disables new snapshot application but does not delete already-persisted hard blocks. The additive server query remains deployed through the client rollback horizon, and legacy directives remain available until old-client risk expires.

Verification strategy

  • Exact incident: mapped local drawer, no local closeout, Cash Controls close, heartbeat false, durable local block, distinct replacement ID.
  • Ordering: duplicate, stale, healthy-after-block, block-after-healthy, legacy/new writer arrival in both orders.
  • Identity: wrong store, terminal, register, local ID, cloud ID, duplicate mapping, and mapping-change race.
  • Durability: restart after closure, crash before React refresh, write failure and retry, durable fail-closed mapping repair classifications.
  • Local-first: no Convex mutation in drawer command; outbound sync continues with heartbeat disabled.
  • Projection: replacement maps to a new sale-usable cloud session; closed-target reuse becomes review; retry is idempotent.
  • Reconnect race: a sale before the first snapshot remains on the original drawer and projects through manager review; received authority persists before changing UI or commands.
  • Rollout: redacted heartbeat-independent apply acknowledgements, shadow mismatch metrics, canary convergence latency, supported build adoption, and zero closed-ID reuse.

Merge-grade validation covers focused Vitest suites, Convex audit and lint, frontend lint, webapp typecheck and build, harness review, graphify rebuild after source changes, and the Athena PR gate.
