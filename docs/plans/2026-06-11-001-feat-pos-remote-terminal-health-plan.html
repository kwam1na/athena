Implementation Plan - 2026-06-11

feat: Add POS Remote Terminal Health Recovery

Add a safe recovery flow that previews terminal blockers, resolves cloud-only stale evidence, queues browser-local repair commands, and verifies recovery through fresh runtime check-ins.

type: feat status: active surface: Athena POS depth: standard
Planning premise. The terminal can be active and online but still unable to transact. The plan separates Healthy idle from Able to transact now, and keeps Convex cloud repair distinct from browser-local IndexedDB repair.

Problem Frame

A production terminal audit showed the current gap: terminal health can expose stale terminal-integrity blocks, drawer-authority blocks, sync review items, expired staff authority, and missing open register context, but support has no safe remote workflow to decide which repairs are allowed and verify that the terminal is healthy afterward.

Convex can repair safe cloud evidence, such as stale duplicate drawer-open conflicts, but it cannot directly mutate browser-local authority state. Terminal seed, drawer authority, staff authority, snapshots, and local review state must be repaired by code running on the terminal and confirmed by a fresh runtime check-in.

Readiness States

State Meaning Proof Required
Healthy idle No blocking setup, sync, terminal-integrity, drawer-authority, local-store, or unsafe conflict evidence. Fresh runtime status plus clean cloud evidence.
Able to transact now Healthy idle plus active cashier/session/drawer context and matching transaction capability. Fresh runtime report of active sale authority, not cloud rows alone.
Needs cloud repair Cloud-only stale evidence can be resolved safely. Policy allow-list and matching precondition snapshot.
Needs terminal action Browser-local IndexedDB or runtime state must be repaired by the terminal. Terminal-scoped command execution and fresh runtime verification.
Needs manual review Evidence touches sale, payment, inventory, closeout, variance, or ambiguous authority facts. Existing Cash Controls, Operations, POS Settings, or Open Work path.

Core Decisions

Preview first

Support sees blockers, safe steps, terminal-required steps, manual-review blockers, and expected verification before taking action.

Separate repair types

Cloud repair resolves safe stale evidence. Browser repair changes local terminal state only after terminal-side precondition checks.

Verify through runtime

A command acknowledgement is not enough. Athena waits for a fresh runtime check-in showing the blocker cleared.

Keep sale gates

Local command gateway, terminal integrity, drawer authority, and staff authority remain the authority for sale-affecting commands.

Flow

Terminal detail
  -> Recovery preview
    -> Cloud-safe repair
    -> Terminal command queue
      -> Browser runtime poller
        -> IndexedDB authority and snapshots
        -> Fresh runtime check-in
          -> Healthy idle / Able to transact now
    -> Manual review when facts are unsafe

Safety Rules

Never auto-clear completed local sales, local event history, payment facts, inventory facts, closeout variance, manager-review items affecting cash/payment/inventory/completed-sale facts, staff proof tokens, or terminal sync secrets.

Automatic repair starts with stale duplicate drawer-open conflicts and terminal-local refresh actions that have explicit precondition checks and runtime verification.
