feat: Build Athena intelligence layer

Type: feat Status: active Date: 2026-06-21 Markdown: 2026-06-21-002-feat-athena-intelligence-layer-plan.md
Build Athena's intelligence layer as a Convex-owned, capability-gated platform that uses Convex's AI strengths for durable state, reactivity, scheduling, retrieval, and workflow orchestration, with one TanStack-backed text adapter as the first SDK path below Athena contracts. The first delivery creates durable run/artifact state, a typed provider adapter, permission-aware context snapshots, and evidence-backed store/user insight artifacts. Image generation remains deferred as a later media adapter.

Requirements

  • Model intelligence as durable Athena-owned state in Convex.
  • Use capability-gated provider selection, with one TanStack-backed structured text adapter in v1.
  • Use Convex AI-native strengths intentionally: reactive subscriptions, scheduled/internal actions, Workpool/Workflow, Agent threads, RAG/vector search, and file support.
  • Capture permission-aware context snapshots.
  • Enforce read-time authorization on run and artifact queries using viewer role, visibility mode, and source refs.
  • Treat snapshot/RAG/customer/store text as untrusted model context.
  • Persist structured outputs, source references, normalized errors, provider metadata, and cost/usage evidence.
  • Reuse command results, approvals, operational events, work items, automation policy, automation runs, and workflow traces.
  • Keep v1 read/propose only; work-item promotion and apply paths are follow-up work.
  • Migrate existing store/user insights away from prompt JSON parsing.
  • Defer image generation while reserving future artifact kinds.

Key Decisions

  • Athena owns the run state machine; TanStack AI is an adapter/orchestration dependency.
  • Convex is the primary intelligence runtime substrate; Agent/RAG/Workflow components are optional future helpers, not the business ledger.
  • Convex RAG is reserved for document-like knowledge retrieval; deterministic operational snapshots come from internal queries first.
  • Provider-native payloads stay out of domain tables, and raw provider payloads are omitted by default.
  • V1 installs no Convex Agent, RAG, or Durable Agents components.
  • First delivery handles text/JSON artifacts and recommendation fields only.
  • Client-facing generation starts through a Convex mutation that records intent and schedules/internalizes work.
  • TanStack tool approval is a runtime pause signal; Athena approval remains the authority model.
  • Operational events record visible intelligence decisions; low-level provider events stay on intelligence run records.

V1 Operator Outcome

The first visible outcome is that an owner can open existing store/user insight panels and review one evidence-backed storefront/customer recommendation, understand why Athena thinks it matters, dismiss it, or rerun it against fresh context. Recommendations without evidence, with stale context, or generated from data the current viewer cannot read must be suppressed or visibly marked as limited or stale.

Design Shape

The architecture keeps Convex as the durable state machine: intent, snapshot, run, provider invocation, artifact, review, and future apply boundaries are all Athena-owned.

Implementation Units

Prerequisite Gate. TanStack runtime spike

SpikePin the v1 package set, likely @tanstack/ai plus one text provider adapter, and prove non-streaming structured output from a server-only Convex action behind Athena's provider interface.

Primary files: convex/schemas/intelligence.ts, shared/intelligence/*, convex/schema.ts.

U2. Create capability registry and provider contract

ProviderRegister capabilities separately from providers; implement one TanStack-backed v1 text adapter while keeping Convex Agent/RAG/Workflow documented as future component choices.

Primary files: convex/intelligence/capabilityRegistry.ts, convex/intelligence/providers/*.

U3. Implement run lifecycle, idempotency, and snapshot capture

DurabilityRecord intent, capture actor-visible context, schedule provider work, expose reactive run progress, persist attempts, and normalize completion/failure.

U4. Add read-only capability tools and structured artifact generation

Primary files: src/components/intelligence/*, StoreInsights.tsx, UserInsightsSection.tsx.

U7. Document platform contract, validation ladder, and deferred adapters

DocsDocument Athena-owned state, provider boundaries, approval/apply rules, and future media adapter constraints.

Primary files: packages/athena-webapp/docs/agent/intelligence.md, docs/solutions/architecture/athena-intelligence-layer-foundation-2026-06-21.md.

System-Wide Impact

  • Interaction graph: New intelligence Convex modules read analytics/user activity and write intelligence tables; existing insight components shift to persisted artifact queries. Convex Agent/RAG/Workflow components can be added later as selected runtime helpers, but are not installed in v1.
  • Error propagation: Provider/config/schema failures normalize into run/artifact state and safe command-result copy; raw provider errors are omitted from UI/logs.
  • State lifecycle: Duplicate generation, stale recommendations, partial output, and context drift are handled by idempotency, snapshot hashes, and artifact states.
  • Unchanged invariants: Domain records stay owned by their current modules; automation runs and workflow traces keep their existing meanings.
