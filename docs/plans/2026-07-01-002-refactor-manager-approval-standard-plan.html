refactor: Standardize manager approval identity

Deep plan for making manager approval a foundational, server-owned authority contract across Athena while separating app-account authorization, staff actor attribution, manager credentials, elevation, async decisions, and automation evidence.

type: refactor status: active date: 2026-07-01 deepened: 2026-07-01 scope: planning only

Plan Summary

The current code does not appear to auto-link the signed-in Athena user to the staff profile that approves a request. The validated gap is semantic drift: ordinary actor attribution, manager credential authentication, manager elevation, async approval decisions, automation evidence, and proof-backed approval authority all exist, but consumers name and persist them differently.

The remedy is to define one authority vocabulary and shared server-side helper contract, then migrate the high-risk approval consumers to record requester, approver, reviewer, applier, proof, request, and automation evidence without overloading actorStaffProfileId.

Core Requirements

  1. Manager approval authority comes from consumed approvalProof, with approver identity from approvedByStaffProfileId.
  2. The signed-in Athena user remains session/account authorization and submitter/reviewer attribution, not implicit approver identity.
  3. Ordinary staff actor attribution remains separate from proof-backed manager approval.
  4. Manager elevation remains route/surface capability and never satisfies command approval.
  5. Async decisions distinguish reviewer account, decision proof, approving manager, and original domain request.
  6. Domain events use consistent requestedBy*, approvedBy*, reviewedBy*, and appliedBy* semantics.
  7. Protected commands retry the same command with approvalProofId; no parallel manager-only mutations.
  8. Automation policy evidence stays separate from human manager approval.
  9. Financial, POS, Cash Controls, and Daily Operations audit records carry typed approval evidence where operators and auditors read from.

Durable Evidence Contract

Untyped metadata can carry supporting context, but it cannot be the only durable location for approval authority. Async approval requests should add top-level decisionApprovalProofId and decisionApprovedByStaffProfileId. reviewedByUserId remains the account reviewer, while reviewedByStaffProfileId is only a legacy-compatible mirror during migration.

Cash Controls, POS, and Daily Operations records must either persist approvalProofId and approvedByStaffProfileId directly where auditors read, persist a typed approval evidence object, or include the proof id needed to join to the canonical approvalProof record.

The implementation should add or consolidate helpers such as resolveOrdinaryActorStaffProfile, consumeManagerApprovalAuthority, recordApprovalAuthorityAudit, and resolveAsyncApprovalDecisionAuthority. Exact names can change, but the contract should not.

Implementation Units

U1. Shared Authority Contract

Define the typed proof authority result and central requester/action/subject validation around existing approvalProofs, approvalActions, staffCredentials, and approval audit helpers.

U2. Async Approval Requests

Add top-level decisionApprovalProofId and decisionApprovedByStaffProfileId, while keeping reviewer account and legacy staff fields distinct in approvalRequests and Operations Queue.

U3. Cash Controls

Migrate closeouts, deposits, register-session sync reviews, and Register Session UI to the shared authority standard after the async decision contract exists.

U4. POS Commands

Audit payment corrections, voids, item adjustments, transaction public APIs, and Transaction UI for proof/requester/approver clarity.

U5. Daily Ops and Elevation

Separate manual Daily Close approval, Daily Opening continuity, manager elevation, and automation policy evidence.

U6. UI Runners

Keep inline approval on the retry-with-proof path and make expired, consumed, mismatched, and stale approval states consistent.

U7. Docs and Validation

Update package agent docs and add a solution note, then run focused Vitest, Convex, TypeScript, and graphify validation after code changes.

Key Decisions

  • Do not add or infer approvedByUserId.
  • Keep Operations Queue full_admin reviewer gating unless product/security explicitly changes it later.
  • Store async decision proof in top-level decisionApprovalProofId when it is bound to approval_request.
  • Move deterministic validation before proof consumption by default and persist named failure states for remaining races.
  • Preserve current self-approval policy and test it per action.
  • Require requester identity for POS and cash-control approvals where staff attribution is available.

Primary Risks

  • Schema churn could expand scope; start with helper contracts and metadata naming.
  • Operations Queue access semantics could change unintentionally; keep the existing reviewer gate and test it.
  • Proofs can be consumed before stale application failures; validate earlier where possible and standardize retry/terminal metadata.
  • Automation records could look like human approval; keep policy evidence in its own lane.

Success Criteria

  • Staff A can request while Staff B approves, with Staff B recorded only as approver and no staff/user link mutation.
  • No protected command accepts managerElevationId or raw client staffProfileId as approval.
  • Async decisions preserve reviewer, approving manager, decision proof, and original domain request separately.
  • Cash Controls, POS, Daily Operations, and Operations Queue tests share one authority vocabulary.
  • Agent docs and solution docs make the standard durable for future work.
