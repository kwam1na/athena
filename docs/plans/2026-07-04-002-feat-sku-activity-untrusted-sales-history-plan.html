V1 Contract

These are the implementation-shaping constraints that reviewers aligned on after the planning loop.

  • The workspace uses a master-detail pattern: source summaries stay visible while selected-source transaction history opens in detail.
  • Source summaries are cursor-paginated independently from selected-source transaction history using indexed source-of-truth evidence.
  • Selected-source history supports load more until hasMore is false; long history is not replaced by a truncation warning.
  • Source-list reads use evidence indexes on both source tables; if nested evidence cannot be indexed directly, add only minimal source-row mirror fields.
  • Source-list pagination uses per-source/status cursors, reads pageSize + 1 per bucket, merges deterministically, and returns participating cursor payloads.
  • No materialized projection or scheduled rebuild is part of v1; freshness is favored over derived-state optimization.
  • The Convex query derives auth server-side, validates organization/store access, and returns safe not-found states for missing or other-store source ids.
  • Missing source ids, zero matched transactions, and cross-store parent mismatches return safe empty or diagnostic states.

Deferred Optimization

The heavier projection path stays out of v1 unless production data proves the indexed read path is too expensive.

  • Do not build a materialized SKU Activity projection or scheduled rebuild job in this slice.
  • Revisit projection only with query-cost metrics from realistic store data.
  • A later projection plan must define freshness guarantees, drift diagnostics, repair behavior, and scheduled-run evidence before implementation.

Transaction Evidence

The history view separates historical facts from current effective circulation, so operators can trust what the page is saying.

  • Source rows show aggregate sale evidence for triage.
  • Detail rows are transaction-backed whenever matching transaction items exist.
  • History includes sale identity, time, historical gross quantity, effective/net quantity, transaction status, refunds, adjustments, and register/session context when available.
  • Voided or adjusted completed transactions remain visible with status labels.
  • Fully voided, fully refunded, or net-zero corrected quantities do not inflate active circulation totals.
  • Summary/history mismatches render calm diagnostics instead of fabricated rows.

DTO Allowlist

The evidence query is intentionally narrow: enough for operator review, not a raw transaction or audit export.

  • Source rows expose source type, store-scoped source id, operator display label, status label, review bucket, aggregate quantities, sale count, last sold time, route availability, and non-sensitive diagnostics.
  • Transaction rows expose transaction id, display reference, completed time, status label, gross/effective quantities, operator-visible totals, refund quantity, adjustment summary, safe register/session label, and route targets.
  • The untrusted evidence query excludes raw customer/staff ids, contact details, notes, local event ids, payload fingerprints, approval ids, register session ids, terminal ids, provenance internals, and backend enum strings.

Risks

The plan treats performance, trust boundaries, and metadata exposure as first-order implementation risks.
