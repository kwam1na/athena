Problem Frame

POS catalog lookup has a durable local metadata snapshot, but availability still depends on a bounded live Convex query. Offline search can therefore show product rows with unknown availability, which tempts the UI into fake stock fallbacks. This plan makes inventory readiness explicit by caching a complete, hold-aware sellable availability snapshot for provisioned POS terminals.

Requirements

  • Persist a store-scoped availability snapshot for every sellable SKU in the register catalog.
  • Use hold-aware sellable availability, not raw SKU quantity.
  • Keep bounded live availability for online displayed or exact-match rows.
  • Use local availability offline and expose readiness gaps when the snapshot is missing.
  • Decrement terminal-local availability during disconnected local selling.
  • Preserve completed local sales and route cloud inventory conflicts to manager review.
  • Keep POS as the only offline-first Athena workflow in this release.
  • Do not make expense sessions offline-first; only preserve shared product-entry presentation parity.

Key Technical Decisions

DecisionRationale
Separate metadata and availability snapshotsPreserves the hot-path split that prevents full-catalog reactivity from cart edits.
Snapshot sellable availabilityOperators need the same hold-aware concept POS command boundaries use.
Refresh full availability separately from bounded live overlayOffline readiness gets full coverage while online search stays fast.
Treat missing snapshot as not offline-readyA missing local snapshot is not proof of inventory and must not become fake stock.
Decrement terminal-local availabilityPrevents one terminal from repeatedly selling the same last-known unit while disconnected.
Keep cloud reconciliation authoritativeOffline sales are preserved; conflicts become manager review work.

Implementation Units

U1. Add a full-store register availability snapshot query

Create a server read model for hold-aware sellable availability across the register catalog scope while preserving the existing bounded query.

Primary files: convex/pos/application/queries/listRegisterCatalog.ts, convex/pos/public/catalog.ts, DTO and port files.

U2. Persist local register availability snapshots

Add complete store-scoped snapshot storage with atomic replace semantics and explicit missing/stale/error states.

Primary files: src/lib/pos/infrastructure/local/posLocalStore.ts, new local snapshot helper and tests.

U3. Add local availability refresh and gateway overlay

Refresh full snapshots opportunistically online, use them offline, and keep bounded live rows authoritative when available.

Primary files: src/lib/pos/infrastructure/convex/catalogGateway.ts and gateway tests.

U4. Render offline availability readiness in product entry

Keep catalog results visible offline while blocking selection and auto-add when trusted availability is missing or zero. Expense remains online-shaped; only shared search presentation stays aligned.

Primary files: register and expense view-model tests plus shared product-entry presentation.

U5. Apply terminal-local availability decrements and sync evidence

Consume local availability on add/complete, restore uncompleted cart releases, and preserve cloud reconciliation authority.

Primary files: local command gateway, register read model, sync contract, ingestion, and projection tests.

U6. Document the offline inventory snapshot contract

Record the invariant in docs/solutions/ and refresh graphify artifacts after implementation.

Risks & Mitigations

RiskMitigation
Full snapshot is expensive for large storesKeep it off the active search hot path and retain bounded visible-SKU reads.
Stale local availability causes cross-terminal oversellsAccept as the offline tradeoff and preserve completed sales with manager conflicts.
Missing snapshot blocks offline sellingTreat as correct readiness behavior and surface setup/sync guidance.
Expense entry inherits POS session behaviorShare search presentation only; keep mutation behavior separate.

Verification Strategy

  • Server tests for catalog-scope, hold-aware full availability snapshots.
  • Local store tests for atomic complete snapshot persistence.
  • Gateway tests for online bounded, offline snapshot, and missing snapshot behavior.
  • Register and expense view-model tests for selectability, exact-match auto-add, and readiness gaps.
  • Local command/read-model tests for decrement, restore, and completed-sale preservation.
  • Sync tests proving cloud inventory conflict creation preserves offline sales.
