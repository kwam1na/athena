fix: Stabilize POS local/cloud sync contract

Stabilize Athena POS drawer lifecycle and sync settlement in two stacked delivery PRs. PR1 tightens drawer authority, replacement freshness, and local/cloud register identity; PR2 builds on that contract to repair broader sync settlement, cursor, reopen, closeout, and Cash Controls presentation behavior.

Type: fix Status: active Date: 2026-06-27 Source: Markdown plan

Delivery Shape

PR1

Drawer lifecycle authority, replacement freshness, local/cloud drawer identity, duplicate register opens, and closeout readiness.

PR2

Sync cursor identity, rejected/held/conflicted settlement, reopen policy, closeout failure truthfulness, and Cash Controls presentation.

Execution Constraint

All work happens in delivery worktrees. Root is reserved for final fast-forward alignment after merge.

Requirements Digest

  • Replacement drawers cannot become sale-usable from uploaded needs_review opens.
  • Drawer authority and closeout settlement must compare local and cloud register-session identity consistently.
  • Duplicate register_opened reuse must conflict unless it is a true idempotent retry.
  • POS and expense sync cursors must batch independently and in upload-sequence order.
  • projected, conflicted, held, and rejected must remain distinct local settlement states.
  • register_reopened is deferred from uploadable local sync until a manager-review replay contract exists; existing uploaded review rows stay visible/rejectable and are not retried into projection.
  • Closeout and Cash Controls copy must present pending/review state truthfully.
  • Manager/runtime repair must be freshness-checked, idempotent, and unable to override current drawer authority.
  • Repair/review authorization must never persist raw PINs or proof tokens; server mutations authorize fresh active same-store manager proof at action time.
  • Old open browsers and queued offline events must remain safe through Convex-first rollout.

U2. PR1 Browser-Local Authority and Readiness

Separate POS/expense cursors and keep rejected, held, conflicted, and projected outcomes visible. Public sync result changes must be additive and old-client safe for open browsers and queued offline events.

U5. PR2 Reopen, Closeout, and Cash Controls Truthfulness

Defer register_reopened cloud upload, keep existing uploaded reopen review rows manager-visible/rejectable, enforce fresh same-store manager proof without persisting raw PINs or proof tokens, and make closeout failure presentation and Cash Controls target context truthful.

Runtime directives require issued-at, directive id/nonce, max age, terminal/store/register scope, and drawer-authority freshness. Expired, replayed, or stale directives fail before local event append at server issue/claim/ack, public terminal mutation, local terminal recovery execution, sync runtime, and command-gateway boundaries.

U6. Integration, Tracking, Review, Merge, and Deploy

Track Linear tickets, execute two stacked PRs, run reviewer loops, merge, align root, and deploy Convex then Athena. Include the Markdown and HTML plan artifacts in final delivery.

Risks

  • PR2 could accidentally redefine drawer authority; mitigate by landing PR1 first and reviewing PR2 against shared policy only.
  • Freshness checks can compare unrelated sequence streams; mitigate with closeout/review boundary evidence and scoped identity tests.
  • Rejected outcomes can disappear as synced; mitigate by replacing current test expectations with visible failed/review semantics.
  • Deploy order can strand clients; mitigate with old-client-safe Convex responses, then deploy Convex before Athena webapp from clean root.
