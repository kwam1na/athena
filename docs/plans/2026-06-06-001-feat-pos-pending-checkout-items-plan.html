Plan: POS pending checkout items

type: feat status: active date: 2026-06-06 source: markdown plan
Summary Requirements Decisions Design Flow Implementation Units Risks

Summary

Add a low-friction POS recovery path for products that are not yet in the catalog. Cashiers can add or reuse a pending checkout item for the current sale online or offline, while Athena keeps permanent catalog truth behind owner review.

Core trust boundary: cashiers record sale evidence; owners decide catalog truth. Quantity means quantity sold in the current transaction, not available inventory.

Requirements

  • Cashiers can add missing items to sales without manager approval in normal cases.
  • Cashier-entered quantity never creates trusted available inventory.
  • Pending items can be reused in later sales while still pending review.
  • Reuse accumulates review evidence across transactions, cashiers, quantities, prices, and timestamps.
  • Owner/admin review can approve, attach, edit and approve, reject, or flag follow-up.
  • Audit events plainly identify who sold what, at what price, in what quantity, and in which sale context.
  • Provisioned terminals can add or reuse pending checkout items offline and sync review evidence later.

Key Decisions

  • Distinct pending checkout domain: do not overload true quick-add catalog creation.
  • Reuse quick-add UX: keep the cashier recovery motion familiar and compact.
  • Structured pending record plus work item: one pending case can collect evidence from many sales.
  • Hidden provisional SKU anchors: satisfy current POS item references without creating trusted stock.
  • Operational events as audit rail: creation, reuse, sale linkage, and review decisions are timeline-visible.
  • Narrow hard stops: unsafe states block; risk signals raise priority instead of stopping ordinary sales.
  • Confirm weak matches: exact barcode reuse can be direct, but name-only reuse should require cashier selection to avoid false merges.
  • Offline local events: offline pending checkout uses terminal-local IDs, append-only POS events, and idempotent sync mapping into cloud review cases.

Design Flow

1. Cashier scans or searches item.
2. Approved catalog match adds normally.
3. No match opens pending checkout recovery.
4. Existing pending match is reused, or a new pending case is created locally or in the cloud.
5. Sale-line quantity is added to the cart.
6. Completed offline sale stays local until sync can mint/reuse the cloud pending case.
7. Completed sale records pending metadata and audit evidence.
8. Owner approves, attaches, edits, rejects, or flags follow-up.

Implementation Units

U1. Model pending checkout review state

Add posPendingCheckoutItem, pending references on POS session/transaction items, and hidden compatibility status on product/SKU records.

U2. Add pending checkout command and audit boundary

Create or reuse pending cases by barcode/name, create hidden provisional anchors, open review work items, and record operational events.

U3. Reuse quick-add POS UX

Adapt the quick-add dialog/search-not-found flow for sale-scoped pending checkout items and pending-review copy.

U4. Preserve online sale facts without trusted inventory mutation

Carry pending metadata through online cart/session/transaction completion and bypass trusted stock decrement for pending lines.

U7. Support offline pending checkout creation and sync

Record local pending checkout events with terminal-local IDs, complete sales offline, then sync and map them into cloud pending review cases.

U5. Add owner review and resolution actions

Surface pending cases in Operations Queue and support approve, attach, edit and approve, reject, and follow-up actions.

U6. Harden policy, copy, and regression coverage

Keep the flow low-friction while making risk signals visible and preserving existing quick-add/catalog behavior.

Risks

Risk Mitigation
Pending provisional SKUs leak into normal catalog or storefront views. Keep records hidden with explicit pending status and add search/projection tests.
Cashier quantity is treated as trusted stock. Carry pending metadata through transaction boundaries and test zero-stock pending sale completion.
Repeated use creates duplicate review cases or false name-only merges. Reuse exact barcode matches directly; present name-only matches as suggestions with duplicate and false-merge tests.
The workflow becomes too restrictive. Hard stops remain rare; most risk signals affect review priority rather than checkout.
Offline local creation creates mapping complexity. Use append-only local POS events with terminal-local IDs and idempotent local-to-cloud sync mappings.

Offline conflict policy: if sync later discovers duplicate, rejected, or conflicting pending evidence, preserve the completed sale and route the uncertainty to review instead of rewriting the receipt.
