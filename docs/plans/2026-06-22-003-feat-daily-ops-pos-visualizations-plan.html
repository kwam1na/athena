Summary

This plan brings the POS hub's sales trend chart, top-items list, and payment-mix visualization into Daily Operations by extracting the POS pulse presentation into a shared store-operations component and feeding it from the Daily Operations read model.

The opening review panel moves into the desktop right rail above the store-day timeline, with a compact rail layout that preserves Opening Handoff as the owning workflow.

Scope

In scope

  • Reuse POS pulse chart, top items, payment mix, metrics, skeletons, and empty states.
  • Extend Daily Operations snapshot with selected store-day pulse data.
  • Move opening review into the right rail above the store-day timeline on desktop with a compact 320px rail variant.
  • Preserve server-side financial redaction and existing workflow ownership.

Out of scope

  • No new analytics route or charting library.
  • No client-only aggregation from raw transaction rows.
  • No change to manual Opening Handoff strictness.
  • No broad Daily Operations or POS hub redesign.

Key Decisions

  • Extract shared store-operations pulse presentation instead of importing POS-owned UI into Daily Operations.
  • Feed Daily Operations from its aggregate snapshot, not a second POS hub query.
  • Use centralized net payment allocation for shared payment mix, including cash-with-change and legacy fallback rows.
  • Make Daily Operations pulse window state explicit and server-scoped.
  • Keep POS hub behavior unchanged through the same shared presentation.
  • Use a compact right-rail Opening review panel as evidence and navigation, not a resolver.
  • Use existing `PageWorkspaceGrid` behavior for desktop rail placement and stacked mobile layout.

Implementation Units

U1. Extract shared store pulse presentation

Move reusable chart, metrics, top-items, payment-mix, skeleton, and empty-state rendering into a shared store-operations component while keeping POS hub behavior stable.

Primary files: src/components/pos/sales-pulse/POSSalesPulseView.tsx, src/components/operations/store-pulse/StorePulseView.tsx, POS pulse tests.

U2. Add Daily Operations POS pulse data to the snapshot

Extend the Daily Operations server snapshot with selected operating-date pulse data, centralized net payment allocation, and Convex-boundary redaction.

Primary files: convex/operations/dailyOperations.ts, convex/pos/application/queries/getTransactions.ts, related Convex tests.

U3. Render shared pulse visualizations on Daily Operations

Add the shared pulse section to the main Daily Operations workspace using snapshot data, explicit pulse-window state, and existing date/search helpers.

Primary files: src/components/operations/DailyOperationsView.tsx, DailyOperationsView.test.tsx.

U4. Move opening review into the desktop right rail

Place Opening review above Store-day timeline in the rail at desktop width, with a single-column panel, two count chips, up to three preview rows, overflow copy, and a full-width Review all action into Opening Handoff.

Primary files: src/components/operations/DailyOperationsView.tsx, DailyOperationsView.test.tsx.

U5. Validate, document, and visually verify

Run focused frontend and Convex tests, changed lints, TypeScript/build gates, Graphify rebuild after code edits, and one visual pass across desktop and mobile.

Risks

Risk Mitigation
POS hub behavior changes during extraction. Use POS tests as characterization and keep wrapper exports if useful.
Financial pulse data leaks to unauthorized users. Add backend redaction tests; never rely on React hiding alone.
Historical Daily Operations view accidentally shows today's pulse. Derive pulse data from Daily Operations operating-date ranges.
Opening review becomes cramped in the 320px rail. Use compact counts, short preview rows, and a clear Review all link.

Verification Strategy

  • POS pulse and POS hub integration tests keep existing behavior stable.
  • Daily Operations UI tests cover pulse rendering, rail ordering, empty/loading states, and timeline overflow.
  • Convex tests cover snapshot pulse data, selected operating dates, server-scoped pulse windows, net payment allocation, legacy payment fallback, and redaction.
  • Run changed lints, TypeScript, build, relevant harness review, and Graphify rebuild after code changes.
  • Capture Daily Operations desktop with opening review above timeline, Daily Operations mobile stacking, and POS hub after shared pulse extraction.
  • Inspect chart labels, rail overflow, and Review all CTA visibility during visual verification.
