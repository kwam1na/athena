feat: Add Completed Transaction Void Workflow

Add a manager-approved workflow for voiding completed POS transactions from transaction detail. The plan replaces the current legacy direct void path with an audited reversal command that preserves original sale facts and records payment, inventory, cash-control, and operational evidence.

Type: feat Status: active Date: 2026-05-21 Deepened: 2026-05-21 Source: markdown plan
Scope Key Decisions Flow Implementation Units Risks Verification

Scope

The workflow covers full void of a completed POS transaction when the current operating-day and register state can safely accept the reversal. Partial refunds, exchanges, external gateway refunds, and post-close historical refunds are deliberately deferred.

Boundary: Original sale rows, transaction number, receipt history, cashier attribution, and completed-sale item rows remain audit facts. The void is additive reversal evidence plus terminal status.

Key Decisions

  • Replace the exposed void path with a browser-safe CommandResult workflow.
  • Require manager approval for completed-sale voids through the existing approval proof system.
  • Use existing ledgers: paymentAllocation, inventoryMovement, SKU activity, operationalEvent, and daily close review items.
  • Block direct voids for transactions with pending/applied item adjustments and transactions inside completed store operating-day closes.
  • Expose the action inside the existing transaction update panel, with reason capture and durable inline errors.

Intended Flow

This is directional guidance for review, not an implementation specification.

1. Operator chooses Void sale, enters a reason, and authenticates staff identity.
2. Transaction detail calls the void command without approval proof and receives approval_required.
3. Manager approval mints an action/store/subject/role-bound proof.
4. The command validates transaction status, adjustment state, register identity, daily close state, and proof.
5. The command records payment allocation reversal, cash drawer reversal, inventory movements, SKU activity, and operational event.
6. The transaction is patched to status: "void"; the UI renders void metadata and disables further updates.

Implementation Units

U1. Command Contract

Add the completed-sale void approval action and public mutation command-result contract. Tests cover approval-required, proof retry, and safe user errors.

U2. Ledger Reversal

Record outbound payment allocations, register cash reversal, inventory movements, SKU activity, and void status with validation-before-write ordering and idempotent retry behavior.

U3. Daily Close Boundaries

Block voids for completed daily closes and item-adjusted transactions; preserve existing daily close void review items.

U4. Operational History

Record void operational events and return void metadata in transaction read models for detail and history surfaces.

U5. Transaction UI

Add the guarded destructive action to TransactionView, reusing staff sign-in, manager approval, and inline command errors.

U6. Validation Docs

Update harness and testing docs so future POS void work maps to command approval, cash-control, daily-close, and UI validation slices.

Risks

RiskMitigation
Closed operating-day reports are corrupted.Block direct voids inside completed daily closes using store operating-day range semantics; use reopen EOD or a future refund workflow.
Inventory restoration lacks audit evidence.Record committed inventory movements and SKU activity for each restored SKU.
Cash drawer and payment allocation diverge.Reuse existing retail void payment allocation and register-session void helpers with focused tests.
Partial reversal state appears after helper-level writes.Validate blockers before writes, patch transaction status last, and require source-scoped idempotency or command-owned consolidation.
Adjusted transactions are reversed incorrectly.Block pending/applied item-adjusted transactions in v1.

Verification Strategy

  • Backend command tests for approval, validation, payment allocation, inventory movement, register cash, daily close blocks, and idempotent retry behavior.
  • Read-model tests for void metadata and transaction history.
  • Transaction UI tests for reason capture, approval retry, inline errors, success state, and already-void state.
  • Harness documentation and validation map updates, followed by package validation during implementation.
Implementation note: after code changes, rebuild graphify and regenerate any generated harness docs instead of editing generated files by hand.
