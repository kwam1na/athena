Summary

POS local-sync projection currently embeds catalog lineage, inventory-mutation, review, and blocking-conflict policy directly inside projectLocalEvents.ts. The plan extracts those decisions into packages/athena-webapp/convex/pos/application/sync/projectionPolicies.ts, then adds the finalized provisional import invariant.

Invariant: when a queued offline sale references a finalized or finalization-closed provisional import row for the same store/product/SKU, projection treats that row as trusted sale lineage behind the scenes. Cash Controls manager approval is not required; Operations inventory review is created if trusted stock preconditions fail or if temporal finalization safety makes stock mutation unsafe.

Problem Frame

The production receipt was uploaded after its provisional import row had already been finalized into trusted inventory. Current projection treats the finalized/hidden row as stale provisional inventory, blocks projection, and leaves Cash Controls showing a manager action that cannot settle the sale because replay hits the same blocking policy.

The correct boundary is server-side projection policy plus Operations inventory review. Cash Controls should remain a drawer/session command boundary, not a SKU workspace or generic sync cleanup queue.

Requirements Digest

  • Extract inline projection policies into a pure module and preserve current behavior before changing semantics.
  • Keep repository reads, writes, conflicts, transaction persistence, inventory movements, and work-item mutations outside the policy module.
  • Accept finalized or finalization-closed provisional lineage only when row, store, product, SKU, and trusted SKU validity match.
  • Project finalized matching lineage without Cash Controls manager approval.
  • Create or reuse synced_sale_inventory_review when trusted stock preconditions fail or stock mutation is unsafe because the sale occurred before finalization.
  • Keep active provisional behavior unchanged and keep invalid lineage blocking.
  • Preserve lineage on sale records without mutating finalized provisional sale evidence.
  • Use a source-aware trusted-demand view for validation, movement/review metadata, and quantity aggregation while preserving persisted provenance.
  • Constrain the first-sync inventory-review exception to accepted finalized lineage; plain trusted stock shortfalls keep existing reviewed-projection gates.
  • Correct affected production conflict rows after deploy through a scoped internal/system remediation runbook with before/after verification.
  • Keep V26-966 as a linked contract single-sourcing follow-up, not part of this batch.

Scope Boundaries

  • No broader V26-966 contract consolidation.
  • No Cash Controls UI redesign or SKU workspace ownership.
  • No conversion of active provisional rows into trusted inventory sales.
  • No auto-acceptance of rejected, non-finalization-closed, missing, wrong-store, invalid trusted SKU/store, or mismatched lineage.
  • No historical transaction rewrite or migration unless implementation proves retry cannot settle existing conflicts.

Implementation Units

U1. Characterize Current Inline Projection Policy

Strengthen tests for active provisional, hidden/inactive provisional, mixed same-SKU, reviewed inventory projection, and Cash Controls replay before moving code.

U2. Extract Projection Policy Decisions Into a Pure Module

Create projectionPolicies.ts and pure policy tests, then have projectLocalEvents.ts convert policy decisions into existing side effects.

U3. Expose Finalized Provisional Lineage Facts to Projection

Widen or add a narrow repository lookup so projection can distinguish finalized trusted lineage from invalid stale provisional rows.

U4. Add the Finalized-Lineage Trusted Inventory Invariant

Accept finalized matching lineage, aggregate it through a source-aware trusted-demand view, preserve persisted lineage, apply the temporal stock-mutation guard, skip finalized evidence mutation, and keep all hard failures blocking.

U5. Settle Existing Conflicts Without Cash Controls Manager Approval

Use or add a narrow internal/system-only idempotent retry/reprojection path for existing finalized-lineage conflicts and suppress the false manager-approval route without minting or consuming manager proof.

U6. Correct Affected Production Conflicts

Preflight the affected production conflict set, reproject only eligible finalized-lineage rows through the internal/system path, record skip reasons, and verify transaction, payment, movement, and inventory-review counts.

U7. Document, Validate, and Link the Follow-Up Contract Work

Add a durable solution note, link V26-966 as adjacent contract work, run focused validation, and regenerate Graphify after code edits.

Risks

Main risks: behavior drift during extraction, accidentally trusting rejected or mismatched lineage, double stock mutation after finalization, broadening the first-sync review exception, duplicate retry side effects, mutating finalized audit evidence, leaving prod conflicts stranded, and leaving Cash Controls showing the wrong manager action.

Mitigation: characterization-first sequencing, pure policy tests, explicit negative cases, temporal stock-mutation guard, accepted-lineage-only review exception, internal retry boundary, production remediation runbook, idempotent mapping reuse, no finalized evidence patch, and Cash Controls regression tests.
