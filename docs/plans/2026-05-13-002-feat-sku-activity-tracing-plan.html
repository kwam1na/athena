Athena implementation plan

Definitive SKU Activity Tracing

Add a source-aware SKU activity ledger and inspection surface so Athena can explain why a SKU is reserved, how availability changed, and which workflow caused each SKU-affecting event.

Type: feature Status: active Date: 2026-05-13 Markdown sibling: 2026-05-13-002-feat-sku-activity-tracing-plan.md
Problem Decisions Design Units Risks Success

Problem Frame

Production symptom

SKU KK38-X3C-MQE had a traceable POS hold/release history, but the current inventoryCount and quantityAvailable gap still had no definitive source.

Target behavior

A store-scoped SKU query should return the current reservation explanation, historical timeline, linked source records, and a clear diagnostic when the remaining gap is unexplained.

Requirements Digest

Trace reservations

Record POS holds, checkout reservations, releases, expirations, and consumption with source context.

Keep ledgers focused

inventoryMovement stays the committed stock-delta ledger; SKU activity covers both transient and committed facts.

Expose answers

Provide a backend query and small inspection surface that can answer future support/operator questions directly.

Key Technical Decisions

Decision Why
Add skuActivityEvent Reservations and releases are SKU facts but not always committed stock deltas, so they do not belong only in inventoryMovement.
Link to source records The answer should point to POS sessions, checkout sessions, holds, orders, movements, operational events, or traces instead of standalone prose.
Instrument mutation boundaries The ledger must be written where reservation or inventory state changes, otherwise cleanup and deletion will keep erasing evidence.
Trace checkout before changing checkout Checkout currently reserves through quantityAvailable. The first release should explain that model without changing its semantics.

System Shape

Existing workflow-specific records remain authoritative. The new SKU activity layer records SKU-scoped facts and joins them into one timeline.

SourcesPOS, checkout, stock ops, orders, service usage
LedgerskuActivityEvent rows with source links
Read modelTimeline plus current reservation explanation
UseOperator/debug inspection and production diagnosis

Risks & Mitigations

Risk Mitigation
Duplicate rows under retry or cron replay Source-scoped idempotency keys and replay tests.
Checkout availability behavior changes accidentally Characterize existing checkout create/update/expiry behavior before adding activity writes.
POS hold release regresses into over-availability Keep expired-ledger tests proving release/expiry does not restore SKU availability.
Historical inferred data appears authoritative Mark backfilled events as inferred and keep unexplained-gap diagnostics explicit.

Success Metrics

  • A SKU activity query returns current reservation summary, timeline, linked source records, and diagnostics.
  • POS and checkout reservations produce activity for acquire, adjust, release, expire, and consume behavior.
  • Committed stock changes produce linked inventory movement and SKU activity rows.
  • The system distinguishes reserved by POS, reserved by checkout, not currently reserved, and unexplained availability gap.
