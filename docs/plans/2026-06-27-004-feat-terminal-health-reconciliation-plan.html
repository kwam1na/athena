Summary

Add a reusable explanation model for terminal health that reconciles sale readiness, support recovery, sync review backlog, and next action lanes from TerminalOperationalState. The plan keeps the motivating incident as a representative fixture, not a special-case implementation.

Problem

Terminal Health can show a fresh online runtime and sale-ready evidence while still marking a terminal as needing attention because sync review or cloud conflict evidence remains unresolved.

Today those facts are split across attention reasons, support recovery, and conflict lists. The plan adds one server-owned explanation of the relationship between them.

Safety Boundary

Business facts remain manual. Sale, payment, inventory, closeout, variance, customer, staff proof, and unknown conflicts must never become automatic repair candidates.

Requirements Digest

  • Derive a health explanation with lane, next step, sale impact, support action, and bounded evidence.
  • Keep terminalHealth, salesReadiness, and supportRecovery distinct.
  • Preserve manual review and safe cloud repair boundaries.
  • Use capped summaries or pagination for large review/conflict backlogs.
  • Make public contract fields, literals, access boundaries, and sync-secret separation explicit.
  • Keep roster/detail explanations additive and keep recovery preview behaviorally consistent without changing its shape.
  • Use calm operational copy and avoid raw backend wording or internal ids.

Key Decisions

  • Extend TerminalOperationalState instead of adding UI-only inference.
  • Use a review-first roster headline plus compact sale-continuity copy when sales can continue.
  • Manual review remains primary in mixed manual-review plus safe-repair states.
  • Generic local review becomes terminal-local retry or diagnostics, not manual business review.
  • Unsafe cloud repair skips stay visible as manual-review or diagnostic evidence.
  • App update remains outside operationalExplanation.lane.
  • Health summaries return bounded evidence and sampling metadata, not unbounded lists.

Lane Matrix

Lane Meaning Owner
sale_ready_with_review_backlog Support review needed; cashier can continue. Existing owner resolved from evidence.
needs_manual_review Business or review facts need a human workflow. Cash Controls or Operations/Open Work.
needs_terminal_action Matching terminal must repair local state or retry sync. Terminal command/runtime.
needs_cloud_repair Safe cloud repair policy can resolve stale duplicate lifecycle evidence. Support cloud repair.
stale_runtime Runtime evidence is stale or absent. Diagnostic / wait for check-in.

Public Contract

operationalExplanation is additive on terminal health list/detail responses. It keeps existing fields intact and uses fixed public literals for lane, severity, blocking domain, sale impact, support action, owner, evidence references, secondary actions, and sampling metadata.

  • previewTerminalRecovery keeps returning only TerminalRecoveryPreview; tests prove recovery-preview consistency with list/detail.
  • Required access stays staff/user-role based; terminal sync secrets never authorize support-facing reads.
  • Evidence references are display-safe only and exclude raw payloads, secrets, payment/customer data, and staff proof/PIN material.
  • Operations/Open Work ownership is definitive only when resolved through bounded indexed reads; otherwise targetResolutionIncomplete is true.

Implementation Units

U1. Characterize Current Terminal Health Cases Lock in sale readiness, support recovery, and health distinctions before policy changes.
U2. Add Bounded Sync Review Evidence Summary Group large review backlogs by type, owner, and actionability with named caps and indexed reads.
U3. Add Operational Explanation Policy Derive lane, sale impact, next step, support action, severity, and bounded evidence while resolving generic local-review ambiguity.
U4. Project And Validate The Public Contract Expose the explanation through terminal queries, Convex validators, and frontend types.
U5. Update Roster And Detail Presentation Use the same explanation in roster and detail, including required sale-continuity copy for sale-ready review backlog.
U6. Strengthen Regression Coverage And Documentation Document the lane boundary and harden cloud repair/manual review tests.

Primary Risks

  • Hiding a true sale blocker behind review copy.
  • Encouraging unsafe cleanup of business facts.
  • Understating huge backlogs when summaries are bounded.
  • Drifting public validators from query return shape.
  • Letting roster and detail copy diverge.
  • Letting sync-secret terminal proof become an accidental read credential.
