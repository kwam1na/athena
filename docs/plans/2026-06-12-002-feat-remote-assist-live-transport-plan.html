Summary

Complete Remote Assist by wiring a real provider-backed live transport into the provider-neutral Remote Assist contract. LiveKit is the production provider of record, while Athena remains authoritative for lifecycle, policy, audit, runtime proof, redaction, and bounded control.

The delivery turns current session orchestration into a usable support experience: Terminal Health opens a live viewer/control console, the POS runtime publishes sanitized co-browse frames, support sends bounded app-surface intents, and either side can end or disconnect safely.

Boundary

LiveKit is the transport provider, not the authority boundary. Convex owns session state and audit; runtime code owns redaction and final control enforcement.

Requirements Digest

  • LiveKit is reached only through an Athena provider adapter.
  • Server-side code mints short-lived, role-scoped credentials.
  • Runtime co-browse data is sanitized before it leaves the browser.
  • Support sees a live viewer/control console in Terminal Health.
  • Control intents are bounded and validated at send and runtime-apply time.
  • Remote Assist does not grant POS terminal, drawer, staff, sale, payment, inventory, closeout, Cash Controls, or Operations authority.

Key Decisions

Decision Rationale
LiveKit through adapter Preserves provider-neutral Remote Assist domain and avoids SDK coupling in POS or Terminal Health.
Convex remains authoritative Provider room membership is live transport state, not permission, lifecycle, or audit source of truth.
rrweb-shaped unattended co-browse Unattended sessions cannot rely on browser screen-capture prompts and must redact before transport.
Data packets are non-durable Control commands need app-level ids, runtime ack/reject, timeout handling, and Convex audit for important outcomes.

Delivery Flow

1. Start or hydrate
Support opens Terminal Health and uses existing session state.
2. Runtime proves identity
POS check-in with sync-secret proof claims the session.
3. Tokens issued
Convex mints short-lived role-scoped transport credentials.
4. Live co-browse
Runtime publishes sanitized app frames over LiveKit.
5. Bounded control
Support sends safe intents; runtime validates and applies.
6. End safely
Support end or runtime disconnect updates Convex authority state.

Implementation Units

U1. Provider adapter and credential issuance

Add the Athena transport provider interface, LiveKit adapter, server-side room/token issuance, and token audit.

U2. Transport lifecycle state and audit API

Represent support/runtime joins, disconnects, provider failures, and rejected controls in Convex state and audit.

U3. Runtime co-browse publisher

Connect the POS runtime, record sanitized rrweb-shaped events, publish frames/state, and disconnect cleanly.

U4. Terminal Health live support console

Add the support-side viewer/control console with reconnect, end, freshness, and provider-error states.

U5. Bounded control channel

Send support control intents over transport and enforce runtime-side guardrails, idempotency, and rejection audit.

U6. Browser validation harness and dev proof

Prove support and POS runtime can complete live assist in dev, using fake transport tests plus real provider validation when configured.

U7. Docs, operational notes, and generated artifacts

Refresh solution docs, Convex/generated artifacts, validation map, graphify, and deployment notes.

Risks

Risk Mitigation
Provider SDK coupling leaks into POS or Terminal Health Keep provider code behind adapter modules and test the public API through fake providers.
Co-browse stream leaks sensitive data Redact before transport and cover PIN/payment/customer/sync-secret fixtures in tests.
Data packets are treated as durable commands Use idempotency, runtime acknowledgements, timeouts, and Convex audit for important outcomes.
Support control bypasses POS authority Runtime applies only bounded visible-surface input; POS authority tests remain part of validation.
Provider-backed browser validation is required before claiming delivery. Missing LiveKit credentials are an environment blocker, not a reason to claim fake-transport parity as full delivery.

Validation Strategy

  • Focused backend tests for session service, repository, public API, POS runtime claim, transport credential issuance, and control-rejection audit.
  • Focused frontend tests for runtime transport, recorder redaction, support console, viewer, control toolbar, runtime shell, Terminal Health integration, and POS authority invariants.
  • Browser proof with support Terminal Health and POS runtime contexts: start, claim, live frame, bounded control result, end, and runtime disconnect.
  • Generated artifact refresh for Convex API, harness validation docs, and graphify.
  • Final repository gate: bun run pr:athena.
  • Final delivery proof: in-app browser/computer-use validation in dev, then local-build production deploy.
