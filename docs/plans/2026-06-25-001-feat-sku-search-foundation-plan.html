Build Generic SKU Search Foundation

Build a reusable backend SKU search foundation for Athena admin surfaces. The foundation returns SKU-first, consumer-neutral search results for every SKU in a store while leaving product page and stock-adjustment integrations as follow-up work.

Requirements

  • Provide a reusable Convex-owned SKU search foundation under inventory/catalog ownership.
  • Return SKU-first generic results for all SKUs, including draft, archived, hidden, zero-price, and out-of-stock rows.
  • Return relevant source SKU fields, joined product/taxonomy/color context, and match metadata.
  • Resolve exact SKU, barcode, and product SKU id matches before text search.
  • Support backend-backed text search without local-first/browser-index requirements.
  • Keep product page and stock-adjustment integration out of active scope.
  • Lock the public Convex args/result contract with named return validators and executable contract proof.
  • Provide idempotent repair/backfill as a deployment gate before follow-up consumers depend on text search.

Scope

The active plan builds the backend search foundation only. Products, stock adjustments, procurement, quick-add, activity lookup, homepage picker, and POS remain unchanged until follow-up work.

Important: This is not local-first work. Future frontend callers may debounce backend calls, but the shared contract lives in Convex.

Key Technical Decisions

  • Use a SKU search sidecar/read-model table rather than a required search field on productSku.
  • Hydrate current source documents after search so the sidecar is lookup infrastructure, not catalog authority.
  • Use exact indexed lookup before full-text search.
  • Expose lifecycle and visibility fields instead of filtering them.
  • Return rich but bounded result rows.
  • Use a fixed v1 envelope with results, limit, truncated, and candidateOverflow; missing joined context returns null.
  • Hydrate only after deduping and slicing candidates to the effective limit.

V1 Contract

The foundation exposes one bounded backend query with storeId, query, and optional limit. V1 does not include pagination args.

  • Default result limit: 25; maximum result limit: 50.
  • Exact match read limit: 10; text candidate read limit: 61 before dedupe.
  • Ordering: product SKU id, exact SKU, exact barcode, then text matches.
  • Named return validators are mandatory, with assertConformsToExportedReturns coverage for rich and sparse envelopes.
  • Joined product, category, subcategory, and color fields are explicit and nullable, not full document passthrough.

High-Level Design

Future consumer surfaces call an inventory SKU search query. The query checks exact indexed paths, queries a store-scoped search sidecar for text results, dedupes candidates with exact matches first, hydrates current SKU/product/taxonomy/color data, and returns generic ProductSkuSearchResult rows.

Implementation Units

U1. Search Read Model Schema

Add a store-scoped SKU search sidecar with a required search text field, search index, deterministic lookup indexes, and a hard projection-size contract.

U2. Search Projection Maintenance

Add reusable helpers to build, update, repair, and backfill sidecar rows from SKU, product, category, subcategory, color, and scalar attribute metadata. Repair/backfill is idempotent, cursor-safe, and reports detailed counts.

U3. Generic Backend Search Query

Add the bounded public query with multi-match exact lookup, search-index-only text matching, dedupe, pre-hydration slicing, hydration, and rich generic result rows.

U4. Contract Tests And Consumer Handoff Notes

Lock the result contract with tests and document how follow-up consumers should adapt results without coupling the foundation to a UI.
