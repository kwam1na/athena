Summary

Reduce the June 27 production Convex read spike by turning the hottest reactive queries into bounded, intent-specific read models. The plan starts with measurement, then narrows register sync-review reads, splits Daily Operations analytics from default posture, separates terminal roster from terminal detail evidence, and moves POS catalog metadata away from full live reactivity.

Production signal: June 27 showed 68.05 GB of prod reads. The main hot functions were getDailyOperationsSnapshot, getState, getTerminalHealthDetail, listRegisterCatalogSnapshot, and getRegisterSessionSnapshot.

Requirements

  • Reduce reads for the five hot functions without removing operational behavior.
  • Preserve POS local-first continuity and keep sale readiness distinct from terminal health and review backlog.
  • Keep Daily Operations as an overview of source workflows.
  • Keep sync-review evidence current and targetable.
  • Keep POS catalog visibility enforcement server-side.
  • Add high-cardinality characterization and read-boundary coverage before changing each hot path.
  • Keep Convex queries read-only.
  • Verify production impact through Convex dashboard and Insights.

Key Decisions

Implementation Unit Digest

U1. Measure Hot Read Sets

Add high-cardinality characterization coverage and a production comparison checklist for the hot functions.

Key files: dailyOperations, deposits, register, terminals, catalog tests. Assertions should prove bounded or proportional reads.

U2. Target Register Sync Review

Split targeted session evidence from store backlog evidence and update register/cash-control callers.

Primary risk: losing derived semantics for inventory open work, non-sale mappings, stale resolved conflicts, or rejected evidence.

U3. Split Daily Operations Reads

Keep default posture lightweight and load analytics, full timeline, and scheduled diagnostics through companion queries.

Primary risk: losing manager context.

U4. Split Terminal Roster and Detail

Return a roster contract with sale impact, lane, runtime age, owner/action summary, backlog, and incomplete flags; optimize the named detail function separately while preserving full support evidence.

Primary risk: under-reporting support work or returning falsely clean roster rows.

U5. Stabilize POS Catalog Metadata

Replace full live metadata subscription with an authenticated point-in-time refresh contract and bounded live availability.

Primary risk: stale metadata at the wrong moment.

U6. Consolidate Subscriptions

Align React hooks with the new backend boundaries and remove duplicate full-snapshot subscriptions.

Primary risk: recreating backend cost through nested rerenders or cart-edit remounts.

U7. Rollout and Verify

Run as an interleaved gate after U2, U3, U4, and U5/U6 using normalized read bytes per call and comparable traffic windows before the next slice proceeds.

Primary risk: misreading lower traffic or subscriber changes as improvement.

Risks

  • Review evidence omission: mitigated by over-cap targeted review tests.
  • Terminal support drift: mitigated by keeping TerminalOperationalState authoritative.
  • Catalog stale reads: mitigated by explicit refresh triggers, checkout command validation, and bounded availability subscriptions.
  • Digest drift: mitigated by conditional mutation maintenance, backfill, fallback, and query-write boundary proof.

Verification Strategy

  • Focused read-boundary tests fail if broad store-wide scans return to target paths or high-cardinality fixtures scale with old read sets.
  • Behavior tests preserve cashier readiness, Daily Operations posture, terminal detail evidence, and catalog visibility.
  • After each slice, compare Convex dashboard Database I/O, read bytes per call, comparable traffic windows, and npx convex insights --prod --details for the relevant hot functions, including terminal roster and detail separately.
  • Capture final boundaries in docs/solutions/performance/athena-convex-read-amplification-2026-06-29.md.

Reviewer alignment target: unanimous approval across performance, POS continuity, operations/cash-control, and rollout/test lenses.
