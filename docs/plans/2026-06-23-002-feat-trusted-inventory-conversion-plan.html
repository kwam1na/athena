Summary

The plan adds a product-edit finalization path for draft legacy-import SKUs. Staff make a SKU visible, review stock, quantity, price, and cost, then invoke a dedicated server-owned conversion command that finalizes the active provisional row and returns the SKU to normal trusted inventory behavior.

Main boundary: conversion is not ordinary SKU save. It must update trusted SKU fields, provisional lifecycle state, POS availability behavior, stock-adjustment blockers, SKU activity/inventory movement evidence, and operational evidence together.

Explicit assumptions

  • Reviewed stock values are final trusted on-hand counts, not imported baseline counts before provisional sales.
  • Finalization submits reviewed fields atomically; staff should not need a separate Save first.
  • Finalization uses the stricter inventory import permission model.
  • SKU trust can be finalized without automatically publishing incomplete product merchandising.
  • Freshness checks cover both provisional sale evidence and the persisted trusted SKU baseline.

Implementation Units

U1. Expose product-page provisional binding data

Add active provisional row lookup state for product edit: none, unique, or ambiguous, including import metadata, review version context, a sale evidence fingerprint, and a trusted SKU baseline fingerprint.

U2. Add atomic trusted inventory finalization command

Add finalization idempotency fields and a by_storeId_finalizationConversionRequestId index on the provisional row schema, check the key after auth and before active-row lookup, validate authority, re-read product/SKU/provisional row/sale evidence/reservations, require exact sale and trusted-SKU freshness, patch trusted SKU values, finalize and hide the provisional row, always record SKU activity evidence, and record inventory movement only for a real stock delta.

U3. Model product edit finalization state in UI

Replace the disabled CTA with stateful make-visible, finalize, and blocked states that respect reservations, validation, and ambiguous provenance.

U4. Wire action, refresh, and handoff

Submit only the active SKU's reviewed values and conversion expectations, show pending state, handle command-result failures, use an explicit merge helper or per-field dirty tracking after success, update only the finalized SKU baseline, and preserve unrelated dirty product edits for the normal Save flow.

U5. Add integration coverage

Prove downstream behavior changes from provisional to trusted across stock adjustment, POS catalog lookup/search/snapshot, reservation race, trusted SKU baseline freshness, idempotency, zero-delta SKU activity, and dirty-state merge boundaries.

Verification Focus

  • Backend conversion mutates trusted SKU and provisional row together.
  • Failed validation leaves SKU fields, product totals, provisional row, ledger evidence, and operational events unchanged.
  • Trusted SKU baseline changes block finalization unless current fields already match the submitted reviewed payload.
  • Active provisional SKUs block stock adjustments; finalized SKUs do not.
  • Active provisional POS policy disappears after finalization across barcode/SKU lookup, search, register snapshot, and local snapshot surfaces.
  • Conversion appears in SKU activity/support review, including zero-delta finalizations.
  • UI enables finalization only for visible, valid, uniquely bound, unreserved SKUs.
  • Finalize uses an explicit merge contract that preserves unrelated dirty product, image, category, attribute, and variant edits and disables during global save/context refresh.
  • Existing draft status, review row styling, stock/quantity mirroring, and legacy NULL normalization remain intact.
