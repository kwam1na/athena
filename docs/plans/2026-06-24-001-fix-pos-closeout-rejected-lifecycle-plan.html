Problem

Rejected variance closeouts currently have a path that patches the register session back to active, clearing counted-cash evidence and making the drawer look usable in POS.

Scope note: this plan supersedes the earlier no-new-status assumption. It deliberately does not implement settlement commands; closeout_rejected blocks Daily Close until explicit follow-up resolves it.

Requirements

  • Rejected variance closeouts enter closeout_rejected, not active.
  • Review-only sessions are never POS-sale-usable and cannot receive ordinary sale/session bindings.
  • Evidence remains visible in Cash Controls, Daily Close, Daily Operations, workflow traces, and POS review gates.
  • POS offers opening a replacement register session after closeout submission, including variance closeouts still in review; the old register-session id remains non-reusable for sales.
  • Schema, DTO, public API, local read-model, and terminal-runtime compatibility ships before mutations write the new state.
  • Historical repair starts as a read-only dry-run audit, not an automatic write.

Key Decisions

Status Name

Use closeout_rejected because it matches prior session context and existing trace language.

Sale Policy

Only open and active remain sale-usable. Reviewed replay cannot create, bind, update, or cash-record sales against closed or closeout_rejected.

Replacement Drawer

A submitted closeout in closing or a rejected closeout in closeout_rejected cannot sell, but a new register session may open for the same terminal/register.

Evidence

Canonical evidence spans session status, approval decision, sync conflict/event, operational event, and trace context.

Synced Rejection

General active -> closeout_rejected is invalid; the synced path requires scoped reviewed local closeout evidence.

Daily Close

closeout_rejected is an unresolved blocker, not active work or closed financial evidence.

Implementation Units

U1. Define Closeout-Rejected Status Policy

Add status policy plus schema, DTO, local read-model, public terminal/register, and terminal-runtime compatibility before writes, with tests proving submitted closing closeouts and closeout_rejected do not block replacement.

U2. Persist Rejected Variance Closeouts As Review-Only

Move rejection paths to closeout_rejected atomically while preserving approval, sync, operational, and trace evidence.

U3. Expose Review-Only Sessions In Cash Controls, Daily Close, And Daily Operations

Show rejected closeouts as unresolved review work and Daily Close blockers, not active drawers or settled closed sessions.

U4. Make POS Gate Review-Only Sessions And Offer Replacement Opening

Select submitted closing and rejected review-only sessions for drawer gate context while sale ids attach only to the replacement sale-usable session.

U5. Harden Sale Attachment And Sync Projection Boundaries

Route direct, stale, and local/offline sale paths through shared POS usability policy, including pending checkout, payment correction, and reviewed replay against closed/rejected sessions.

U6. Address Historical Repair And Operational Handoff

Document forward-only behavior or produce a read-only repair audit before any migration decision; rebuild Graphify after code changes.

System Impact

  • Status policy affects schema validators, Cash Controls, POS register state, local sync projection, sale commands, Daily Close, Daily Operations, terminal runtime status, and traces.
  • Manager rejection must settle source sync event, conflict row, session status, operational event, and trace consistently.
  • Runtime status remains evidence, not authority.
