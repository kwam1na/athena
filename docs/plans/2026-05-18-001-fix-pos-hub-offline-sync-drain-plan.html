Rebuild POS hub offline sync drain

Plan to make offline POS history drain automatically from the POS hub once connection is available, without requiring the original cashier to sign in again.

Date: 2026-05-18 Type: fix Status: active Origin: docs/brainstorms/2026-05-13-pos-local-first-register-requirements.md

Core Problem

The browser currently computes upload order from events that are syncable at that moment. Since syncability requires a transient original-cashier proof, an earlier cashier's event can disappear from the upload set and a later cashier's event can take its sequence number.

The backend then correctly rejects or holds the original earlier event as out of order because the cursor has already advanced. Waiting and retrying cannot fix a shifted sequence.

Decisions

Durable sequence at write time

Every uploadable local POS event gets an immutable upload sequence when it is appended to IndexedDB. Upload sequence must never be derived from whichever staff proof is currently available.

Actor and submitter are separate

The event keeps actor staff attribution for cashier, audit, receipt, cash-control, and trace records. The hub or terminal supplies submission authority for upload.

Hub owns upload

The POS hub drains pending local history. The register flow appends events and shows status, but it does not run the upload scheduler.

Permission drift becomes review

When cloud policy no longer validates an offline actor, the event should be accepted into ordered sync history and projected to conflict or review state where needed, not stranded before upload.

Flow

RegisterCashier performs a local POS action.
Local storeAppend event with actor staff id and durable upload sequence.
POS hubRead contiguous pending upload rows when online.
Cloud syncValidate terminal submission authority and preserve actor attribution.
ProjectionMark projected, conflicted, held, or rejected and return mappings.

Non-Goals

  • No product-wide offline sync platform.
  • No v1 dependency on service worker Background Sync.
  • No repair for already-stranded development events.
  • No removal of cashier attribution.
  • No offline-only payment confirmation states.

Risk Notes

Accountability risk: terminal-level upload authority can look like it weakens staff accountability. Preserve actor staff id in all projected business records.

Permission risk: manager-only actions need review-safe projection. Accept ordered history, but do not apply privileged side effects when authority cannot be validated.

Sequence risk: any remaining computed sequence path can recreate the original failure. Tests must assert stored sequence is used.
