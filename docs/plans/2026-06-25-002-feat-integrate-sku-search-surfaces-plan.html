Integrate SKU Search Foundation Across Admin Surfaces

Integrate the generic SKU search foundation from PR #564 into Athena's non-POS admin search surfaces by first hardening access and projection freshness across canonical SKU writers, then adapting results into stock adjustments, products, quick-add recovery, admin product pickers, procurement, and SKU activity.

Scope

Assumption to review: generic SKU search should enforce store-scoped authorization before browser consumers adopt it. Stock adjustments and products are first, while procurement, quick-add recovery, homepage pickers, and SKU activity follow as relevant non-POS surfaces.

Requirements Digest

  • Preserve PR #564's generic contract and adapt locally in each consumer.
  • Wire authorization, projection refresh across canonical SKU discovery-field writers, protected repair/cleanup entry points, and stale drift handling before relying on text search.
  • Make stock adjustments and products globally searchable without widening capped snapshot queries.
  • Extend quick-add, homepage picker, promo-code selected-products, complimentary-product, procurement, and SKU activity lookup where they currently depend on local/domain-specific result sets.
  • Exclude POS register/local catalog search and local-index expense flows.

Out of Scope

  • Changing `searchProductSkus` response envelope, ordering, or limits.
  • Encoding stock, homepage, storefront, sellability, or procurement policy in the foundation.
  • Backend pagination in SKU search v1.
  • Inventory import review search unless a later decision makes it an interactive catalog lookup surface.

Context

Foundation

packages/athena-webapp/convex/inventory/skuSearch.ts owns the generic query, validators, exact-first ordering, canonical hydration, repair/backfill, stale cleanup, and projection helpers.

Primary Consumers

StockAdjustmentWorkspace.tsx and Products.tsx still filter loaded rows locally, which is exactly the limitation PR #564 set up the foundation to remove.

Secondary Consumers

Quick-add recovery, homepage product pickers, procurement, and SKU activity lookup need generic identity search but must preserve their own eligibility and actionability rules.

Prior Sessions

Prior work rejected widening product/stock queries and rejected coupling the foundation to products UI. Reviewers required performance, correctness, data integrity, and API-contract alignment.

Key Decisions

DecisionWhy
Authorize search and protect repair/cleanup entry points.Generic search and store-scoped repair operations can expose or mutate catalog search metadata if store id is the only guard.
Add adapters, not consumer fields, to the foundation.The PR #564 contract is intentionally SKU-first and consumer-neutral.
Projection refresh for discovery fields is enabling work across canonical SKU writers.Search results are only useful when sidecar lookup/search data stays fresh; POS and expense search consumers stay excluded, but their canonical SKU writes can still affect admin search. Products search groups by product and orders groups by best matched SKU rank.The products workspace is product-shaped even though the foundation is SKU-shaped, and grouping must not erase exact-first relevance.

Implementation Units

U1. Backend Safety and Projection Freshness

Enforce access, protect repair/cleanup entry points, wire refresh helpers into all canonical SKU write paths that affect sidecar discovery, and preserve repair/stale cleanup as rollout gates.

Key tests: unauthorized search and repair, create/update/archive/delete/barcode refresh, taxonomy refresh, POS-origin SKU creation/barcode refresh, quantity-only canonical hydration proof, deleted-source stale row suppression, and drifted searchable-source refresh.

U2. Shared Consumer Adapter Layer

Add narrow mapping helpers for product grouping, stock candidates, quick-add options, homepage options, and procurement/activity summaries.

Key tests: rich rows, sparse nullable rows, multi-SKU product grouping, truncation/refinement hints, browser import boundary.

U3. Stock Adjustment Global Search

Use generic search for global candidate lookup, then hydrate stock-owned reservation/blocker state, restore exact-first match order, and preserve URL/detail/filter behavior.

Key tests: outside-page match, exact barcode ordering after hydration, outside-category recovery, blocker rows, auth failure, blank-query pagination regression.

U4. Products Workspace Global Search

Search all store SKUs, group results by product, order groups by best matched SKU rank, preserve matched SKU context, and keep products filters/quick-add behavior intact.

Key tests: unloaded SKU match, exact SKU/barcode grouped product ordering, multiple matched SKUs, category exclusion, lifecycle filters, failure states.

U5. Quick-Add Recovery and Admin Product Pickers

Add async search for unbarcoded SKU recovery, homepage placement pickers, promo-code selected-products, and complimentary-product selectors while enforcing consumer eligibility.

Key tests: attach unknown barcode to unbarcoded SKU, already-barcoded ineligible result, no-match vs no-eligible states, homepage eligible search, promo-code and complimentary selector search, hidden/draft/archive handling.

U6. Procurement and SKU Activity Lookup Parity

Resolve catalog identity through generic search while keeping procurement actionability and activity routing domain-owned.

Key tests: recommendation match, catalog match without action, route state preservation, SKU activity navigation, stale/unauthorized result handling.

U7. Rollout Validation and Harness Updates

Refresh generated artifacts, graphify outputs, harness coverage, and rollout notes after implementation.

Key tests: generated Convex API refs, validation-map coverage, cross-surface SKU/barcode parity excluding POS.

System Impact

  • Interaction graph: one generic foundation feeds several adapters; policy remains consumer-owned.
  • Error propagation: no matches, no eligible matches, auth failures, loading, and truncated results need distinct states in every policy-filtered consumer.
  • Lifecycle risk: deleted-source stale rows and drifted existing-source discovery fields are mitigated through write refresh, repair, and stale cleanup across canonical SKU writers, including POS-origin writers.
  • Parity: exact SKU, barcode, and productSkuId queries should resolve recognizable identity across all non-POS consumers.
  • Unchanged invariant: POS stays local-first and sellability-aware.
