feat: Make POS entry and readiness local-first

Type: feat Status: active Date: 2026-05-14 Source: Markdown plan
Make the path from /pos into /pos/register use the landed POS local-first runtime instead of waiting on live Convex reads. A provisioned terminal should be able to launch POS from the route, resolve store and terminal authority from local state, and pass the register readiness gate when its local store-day state allows selling.
Contents Problem Requirements Scope Decisions Design Implementation Units Risks

Problem Frame

The landed POS local-first work gives the cashier path durable local events, a local command gateway, a local register read model, and background sync. The remaining gap is before those pieces can help: /pos currently blanks behind live analytics/store reads, and POSRegisterOpeningGuard waits on live daily-opening and daily-close snapshots before rendering the register.

This plan complements docs/plans/2026-05-14-001-feat-pos-always-local-first-flow-plan.md; it does not re-plan the already-landed command-gateway inversion. It focuses on entry, local context, and readiness gates.

Requirements

  • A provisioned POS terminal must render the POS entry action from /pos without live analytics, summary, organization, or store reads.
  • /pos must treat POS as the offline-first launcher while non-POS cards and summaries may remain unavailable or stale.
  • /pos/register must evaluate POS readiness from local terminal and store-day state when live snapshots are unavailable.
  • Online daily-opening, daily-close, store, organization, and terminal reads may refresh local POS state, but must not be required after provisioning.
  • If no provisioned local POS seed exists, offline entry must fail closed with setup guidance instead of a blank route.
  • Local readiness must preserve the distinction between store-day state and register/drawer state.
  • Existing local command gateway, register read model, terminal seed, sync runtime, and sync projection behavior must be reused.
  • Offline-first behavior must not expand into non-POS Athena workspaces.

Scope Boundaries

In Scope

  • POS launcher rendering from local context.
  • Local POS entry context using route params plus terminal seed.
  • Local store-day readiness consumed by the register guard.
  • Tests and harness metadata for the route-entry contract.

Out of Scope

  • Offline first-time provisioning or new business creation.
  • Full service-worker app-shell caching.
  • Offline analytics, procurement, admin, cash-controls review, or expense sessions.
  • Payment, receipt messaging, sync projection, or reconciliation changes.

Key Technical Decisions

  • Use the landed local runtime as foundation. Consume posLocalStore, terminal seed fallback, local register projection, local command gateway, and sync status rather than re-planning them.
  • Separate navigation identity from POS authority. Route params provide org/store slugs for links; the provisioned seed provides local store and terminal authority.
  • Add a POS-local entry/readiness helper. Route components should not own direct IndexedDB orchestration.
  • Refresh local readiness opportunistically. Live daily-opening and daily-close snapshots update local readiness when available.
  • Fail closed when authority is absent or unsafe. Missing seed, mismatched store, local closed day, or unsupported schema renders blocked guidance.

High-Level Design

This illustrates the intended approach and is directional guidance for review, not implementation specification.

/pos route params + provisioned terminal seed + optional live org/store reads | v POS local entry context | +--> render POS launcher +--> degrade non-POS cards and summaries | v /pos/register guard + local POS readiness + local register read model + opportunistic live snapshot refresh | +--> render register +--> blocked/setup guidance
Mode /pos behavior /pos/register behavior
Online with live data Render full landing and refresh local context Use fresh snapshots and update local readiness
Offline with provisioned seed and open local day Render POS launcher without analytics blocking Allow register entry from local readiness
Offline without provisioned seed Show setup-required POS guidance Block register entry with setup guidance
Local day closed or local closeout blocking sales POS launcher remains visible Block sales until permitted reopen/readiness change

Implementation Units

U1. Add POS local entry context

Goal: Provide a local-first context API that can resolve route slugs, store id, terminal seed, terminal identity, and local availability without live Convex reads.

Primary files: packages/athena-webapp/src/lib/pos/infrastructure/local/localPosEntryContext.ts, localPosEntryContext.test.ts, and focused updates around posLocalStore.ts or useGetTerminal.ts if needed.

Verification: POS route consumers can derive a register link and local store/terminal authority without Convex resolving first.

U2. Make /pos a local-first launcher

Goal: Ensure the POS landing page renders the POS entry action when local POS context is available, even if analytics, today summary, active organization, or active store queries are unavailable.

Primary files: packages/athena-webapp/src/components/pos/PointOfSaleView.tsx and PointOfSaleView.test.tsx.

Verification: /pos no longer returns blank solely because live analytics or summary queries are unresolved.

U3. Persist and read local POS readiness

Goal: Give POS a local store-day readiness source that can represent started, not-started, closed, reopened, unknown, and stale states without live daily-operation snapshots.

Primary files: packages/athena-webapp/src/lib/pos/infrastructure/local/localPosReadiness.ts, localPosReadiness.test.ts, and focused local store/read-model updates.

Verification: POS readiness can be evaluated from local state and refreshed from live daily-operation snapshots.

U4. Make POSRegisterOpeningGuard local-first

Goal: Replace the live-snapshot render gate with a local-first readiness decision that still honors daily opening, daily close, local closeout, and setup-required states.

Primary files: packages/athena-webapp/src/components/pos/register/POSRegisterOpeningGuard.tsx and POSRegisterOpeningGuard.test.tsx.

Verification: The guard no longer requires live Convex snapshots to render a locally authorized register.

U5. Update validation coverage and planning docs

Goal: Make the new POS entry/readiness contract visible in tests, harness metadata, and durable local-first learnings.

Primary files: packages/athena-webapp/docs/agent/testing.md, packages/athena-webapp/docs/agent/validation-map.json, scripts/harness-app-registry.ts if needed, and a new solution note.

Verification: Harness metadata routes future touched POS entry/readiness files to the focused POS local/register validation surface.

Risks & Mitigations

Risk Mitigation
Local readiness lets a cashier sell after the store day was closed online. Refresh readiness from live snapshots when available and fail closed for stale or unknown closed-day states.
Route params and terminal seed disagree about store identity. Validate seed and store scope explicitly, then render setup-required guidance on mismatch.
POS landing becomes a general offline dashboard. Keep /pos as a launcher and degrade non-POS cards instead of making them offline-capable.
Harness misses the new route-entry contract. Add validation-map coverage and focused tests for POS landing and opening guard.
Planning boundary: this plan assumes the local cashier command path has landed. It only calls for checking narrow residual online-only actions if they directly affect POS entry or readiness.
