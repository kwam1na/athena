Refactor PR Athena Proof Ladder

Split the Athena PR gate into explicit prepare, validate, and proof-recording commands so agents validate the exact tree they intend to commit.

Problem

Recent delivery sessions showed bun run pr:athena can pass the heavy ladder while intended new files remain untracked. The proof then fails to record, so git push repeats expensive pre-push validation.

Core Requirements

  • Keep pr:athena as the safe composed default.
  • Add independent pr:athena:prepare, pr:athena:validate, and pr:athena:record-proof commands.
  • Block before heavy validation when unstaged or untracked files would prevent proof reuse.
  • Never auto-stage arbitrary untracked files.
  • Preserve fail-closed pre-push proof evaluation.

Implementation Units

U1. Split Commands And Add Prepare Guard

Update root command wiring and add a testable prepare guard around generated-artifact repair and mixed-tree detection.

Primary files: package.json, scripts/pre-push-validation-proof.ts, optional scripts/pr-athena-prepare.ts, related script tests.

Posture: test-first.

U2. Refresh Harness Documentation And Durable Learning

Update harness docs, add a solution note, and rebuild graphify so future agents follow the new ladder.

Primary files: docs/harness.md, README.md, docs/solutions/harness/pr-athena-prepare-validate-proof-2026-06-13.md, graphify-out/.

Posture: sensor-only.

Key Decisions

  • Prepare should stage tracked files only and list untracked files for explicit staging or removal.
  • Validation can run independently, but the default pr:athena path remains prepare, validate, record proof.
  • Proof reuse remains fingerprint-based and fail-closed.

Risks

Risk Mitigation
Prepare blocks a manual exploratory validation run. Keep pr:athena:validate available for advanced manual use.
Command split drifts from docs. Add package-script tests, update docs/harness.md, and add a solution note.
Proof reuse becomes too permissive. Do not weaken existing pre-push proof evaluation semantics.

Verification Strategy

Run focused script tests around preparation and proof recording first, then regenerate graphify and run the repo PR gate from the fully staged intended tree.

The final proof should be reusable by pre-push only when the pushed commit has the same validated tree.
