Summary

Add an additive typed homepage snapshot contract for storefront first-render content while preserving existing public endpoints. The work is a projection/API migration over existing store config, banner, best seller, and featured item data, not a storage migration.

Scope

In Scope

  • Typed public homepage snapshot DTO and route.
  • Storefront loader and homepage rendering migration.
  • Bootstrap-compatible snapshot route and admin draft/public active banner split.
  • Homepage placement visibility and ownership hardening.
  • Generated artifacts, Graphify, reviewer loop, and browser validation.

Out of Scope

  • No new homepage storage table or data backfill.
  • No removal of existing public routes.
  • No engagement prompt, upsell, analytics, or footer hydration migration.
  • No production deploy.

Key Decisions

Decision Area Chosen Direction Rationale
API rollout Additive snapshot route Keeps existing routes stable for current wrappers and unknown consumers.
Contract ownership Dedicated HomepageSnapshotV1 DTO Prevents raw Convex document shape and ambiguous money fields from becoming public contract.
Banner semantics Admin draft read plus public active read Supports admin editing without leaking inactive banners to shoppers.
Store bootstrap Snapshot route owns bootstrap-compatible cookie behavior The homepage has one authoritative first-render contract while downstream storefront requests keep store context.

Implementation Units

U1. Typed snapshot presenter and contract

Create the read-only projection over store config, banner, best seller, featured, products, and SKUs. Prove active/draft banner behavior, read-time expiry, visibility filtering, reserved taxonomy exclusion, bounded section sizes, and minor-unit money naming.

Primary files: convex/storeFront/homepageSnapshot.ts, convex/inventory/bannerMessage.ts, convex/inventory/bestSeller.ts, convex/inventory/featuredItem.ts.

U2. Additive public route and bootstrap-safe boundary

Expose the snapshot through a customer-channel route while keeping legacy endpoints unchanged and owning bootstrap-compatible store context cookie behavior.

Primary files: convex/http/domains/customerChannel/routes/homepageSnapshot.ts, convex/http.ts.

U3. Storefront loader and rendering migration

Add the storefront snapshot API/query wrapper and move -homePageLoader, HomePage, and content derivation onto snapshot data while replacing the separate store-plus-merchandising fan-out and keeping the ready shell stable.

Primary files: packages/storefront-webapp/src/api/homepageSnapshot.ts, routes/-homePageLoader.ts, components/HomePage.tsx.

U4. Admin homepage workspace reliability

Preserve the moved admin UI improvements while making single-slot shop-look behavior, banner draft semantics, product picker states, accessibility, and same-store placement integrity explicit requirements.

Primary files: packages/athena-webapp/src/components/homepage/*.

U5. Harness, generated artifacts, review, and browser validation

Refresh generated Convex and Graphify artifacts, update storefront first-load fixtures if needed, loop reviewer agents to unanimous approval, validate admin and storefront in browser, then run the heavy Athena gate.

Primary files: convex/_generated/api.d.ts, scripts/harness-behavior-fixtures/storefront-runtime-api.ts, graphify-out/*.

Risks

Risk Mitigation
Inactive banner draft leaks publicly Split draft/admin reads from active public snapshot and test both consumers.
Store bootstrap cookies disappear Make the snapshot route bootstrap-compatible and validate store context cookies.
Hidden POS-only taxonomy appears on homepage Apply storefront-hidden filtering and add reserved taxonomy regression tests.
Generated artifacts drift Run generated-artifact repair and Graphify rebuild after code edits.
Public response leaks hidden/internal content Expose only customer-safe empty/null sections and keep detailed omission reasons server-side or admin-only.

Validation Strategy

Run focused Convex, route, storefront loader/content, and admin homepage tests first. After reviewer agents unanimously approve, validate the authenticated admin homepage and storefront homepage in browser, then run bun run pr:athena before merge readiness.
