feat: Add POS recovery-code login for field operators

Date: 2026-06-03 Status: draft Scope: Wigclub POS account recovery
Summary Requirements Decisions Alternatives Units Sign-off

Summary

Add a tightly scoped recovery-code login path for the shared pos@wigclub.store POS app account. Field operators can sign in on a fresh browser when the normal OTP flow is blocked by lack of inbox access.

The recommended path is a static-until-rotated POS recovery code stored server-side as a hash, validated through a dedicated Convex Auth credentials provider, and managed by full admins.

Problem Frame

The POS field account occasionally needs to sign in again. The current login path sends an OTP to the email address on the app account, but operators using the POS do not have inbox access.

  • pos@wigclub.store loads the POS web app.
  • Staff PIN and terminal-scoped credentials identify the cashier/operator.
  • Terminal trust, drawer lifecycle, and command checks still decide whether a sale can happen.

Requirements

  • Fresh browsers can sign in as pos@wigclub.store without email OTP access.
  • Recovery is restricted to the configured POS account and rejects admin or non-pos_only accounts.
  • Successful recovery creates a normal authenticated app session and reuses Athena auth sync.
  • Staff PIN remains the cashier/operator identity layer.
  • Recovery code storage is server-side hash only.
  • Failed attempts are rate-limited, locked, and audited.
  • Full admins can create, rotate, lock, unlock, or revoke the code.
  • Codes are generated server-side with enough entropy for shared-credential use.

High-Level Flow

Operator -> Login UI: Enter pos@wigclub.store recovery code
Login UI -> POS recovery provider: signIn(provider, email, code, store)
Provider -> Provider: Validate account scope, code hash, lockout
Provider -> Convex Auth: Return configured auth user id
Convex Auth -> Login UI: Auth session established
Login UI -> Athena auth sync: Trigger existing pending auth sync
Athena auth sync -> POS runtime: Store Athena user id and POS app account id
POS runtime -> POS runtime: Staff PIN and drawer gates still apply

Implementation Units

U1. Add POS recovery credential storage

Add a server-side posRecoveryCredential table scoped to organization, store, and POS app account. Store only hashes, status, counters, lockout metadata, and rotation audit fields.

U2. Add a dedicated Convex Auth recovery provider

Register a ConvexCredentials provider that validates the configured account, pos_only membership, recovery code hash, and lockout before returning the real auth user id.

U3. Add a recovery-code login UI

Keep OTP as the default. Add a secondary POS recovery-code form that uses the same pending-auth-sync handoff as OTP success.

U4. Add full-admin recovery-code management

Let full admins create, rotate, unlock, and revoke the code. Show plaintext once on create or rotate only.

U5. Preserve POS authority boundaries and auditability

Confirm recovery login never substitutes for staff proof. POS actions should still attribute cashier/operator activity through staff profiles.

U6. Validate, rollout, and document operations

Run targeted provider, UI, settings, staff-authority, Convex, typecheck, build, browser, and graphify validation before shipping.

Security Notes

  • Keep validation server-side; UI hiding is not security.
  • Do not log submitted code, code hash, salt, or pepper.
  • Use generic failure copy for invalid, locked, missing, or disallowed states.
  • Rotate after staff turnover or suspected exposure.
ThreatMitigation

Denial-of-service by repeated wrong attemptsCredential lockout telemetry, full-admin unlock, clear admin status, and rollout guidance for support.
Using the code for an admin accountProvider hard-checks configured email, store, and active pos_only membership.
Secret leakageAudit events and metadata exclude code, hash, salt, and pepper.
Staff accountability lossRecovery login audit stays separate from POS action audit.

Final Sign-Off Needed

  1. The v1 recovery account is exactly pos@wigclub.store for Wigclub.
  2. The recovery code remains valid until a full admin rotates or revokes it.
  3. Full admins are the only users who can manage the recovery code.
  4. Failed attempts lock the recovery code globally after a small number of failures, with full-admin unlock.
  5. Staff PIN remains mandatory for cashier/operator identity.
No implementation should start until these policy choices are confirmed.
