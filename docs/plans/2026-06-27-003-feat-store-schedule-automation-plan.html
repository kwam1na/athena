feat: Add foundational store schedule domain

Type: feat Status: active Date: 2026-06-27 Source: Markdown plan

Summary

Build an effective-dated canonical store schedule domain that can serve every store-local time use case in Athena. Opening and EOD automation are the first consumers because they exposed the boundary problem, but the schedule model, resolver, and read APIs should be reusable by Daily Operations, POS, appointments, storefront display, analytics, intelligence context, support diagnostics, and future workflows.

Problem Frame

Daily Operations currently can show pre-window EOD automation checks as “Athena checked EOD Review. No change was made.” That copy is scheduler noise, not an operator action. The symptom points at the broader missing primitive: Athena does not have one canonical domain for store-local business time. The long-term fix is to make store schedule own business time, let consumers adapt schedule context to their workflows, keep automation policy focused on automation judgment, and read raw ledger outcomes through operator-facing state buckets.

Requirements

  • Canonical store schedule must cover timezone, weekly hours, closed days, exceptions, and overnight windows.
  • Store schedule must provide reusable store-day context for consumers to derive timing windows, not rely on browser timezone or action-specific static offsets.
  • Automation policy keeps mode, blocker handling, EOD thresholds, pause state, versions, and automation-specific grace settings.
  • Routine pre-window scheduler wakeups must not surface as operator-facing “Athena checked…” rows.
  • EOD preparation and completion remain separate audit concepts.
  • Migration preserves current behavior while treating existing timezone offsets as hints, not canonical truth.
  • Schedule changes are versioned so completed closes and automation evidence are not reinterpreted by later edits.
  • Schedule APIs and resolver outputs stay consumer-neutral: store-local operating date, schedule phase, open/closed state, effective version, current/next windows, and derived UTC timestamps.

Key Decisions

  • First-class effective-dated schedule table: schedule is indexed, validated, versioned, cross-domain business truth.
  • Consumer-neutral core vocabulary: the schedule schema stores business hours, closed days, exceptions, effective versions, and timezone, not EOD, Opening, grace periods, or automation thresholds.
  • Policy owns judgment: automation policy is not the canonical source of store hours.
  • IANA timezone with native Intl: static offsets are migration hints only, and resolver tests cover DST and overnight windows without adding a package dependency.
  • Scheduled-later is quiet context: pre-window checks do not become automation cards.
  • Daily Close guard stays strict: schedule-derived eligibility must still revalidate at mutation time.

Implementation Units

U1. Create the canonical store schedule domain

Add consumer-neutral schedule version schema/table, effective-range indexes, overlap invariants, native Intl resolver, reusable store-day context APIs, and tests for timezone, DST, weekly windows, closed days, exceptions, overnight windows, phase/window output, and missing-schedule fallback. Consumers decide how to persist resolved schedule versions in their own evidence.

U2. Expose store-hours administration

Add a full-admin store-hours editor under store configuration with schedule-specific Convex commands, getStoreDayContext, and useStoreScheduleUpdate; do not write schedule data through store.config. Define read-only permissions, candidate/admin-confirmed states, save/cancel/dirty states, responsive layout, accessibility behavior, POS timing summary, and a foundational “Store Hours define this store’s local business day” readout.

U3. Adopt store schedule in Daily Operations automation

Refactor Daily Operations automation as the first consumer of the foundational schedule domain. Automation adapters convert generic store-day context into Opening and EOD evaluation windows, then persist resolved schedule version and derived timestamp evidence while preserving policy and ledger responsibilities.

U4. Clean up automation read model and operator status

Introduce schedule-aware status buckets with explicit precedence, copy, CTA, source-link behavior, and redaction. Suppress routine no-op scheduler rows from Daily Operations and EOD UI.

U5. Migrate existing automation timing safely

Backfill candidate schedules through an idempotent, bounded, dry-run-capable migration. Preserve current timing behavior, skip admin-confirmed schedules, require deterministic IANA timezone or admin confirmation before canonical use, and keep compatibility fallback otherwise.

U6. Refresh generated artifacts and validation coverage

Refresh Convex generated refs, harness metadata, Graphify output, and solution documentation after implementation.

Verification Strategy

  • Focused Convex tests for schedule resolver, automation timing, Daily Close guard parity, migration fallback, and read-model status semantics.
  • Component tests for store-hours admin permissions, responsive/accessibility behavior, POS settings summary, Daily Operations, Daily Close, and Opening status copy.
  • Convex artifact refresh through bunx convex dev --once when functions change.
  • Repo validation after implementation: focused Vitest slices, Convex audit/lint, TypeScript, build, Graphify rebuild, and the Athena PR gate.

Reviewer Focus

The main architectural decision to review is the first-class, consumer-neutral schedule domain. Reject it only if store schedule should intentionally remain presentation config; otherwise it gives the cleanest boundary for cross-domain business time with automation as the first consumer.
