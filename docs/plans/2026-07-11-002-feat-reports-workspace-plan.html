Trust states the UI must preserve

Complete or valid zero

Known values are usable; zero activity is not confused with missing data.

Cost-partial

Revenue and units remain visible while unsupported profit or valuation is withheld.

Stale or unsynchronized

Last-good data carries refresh context and names affected sources, dates, or metrics.

Pre-cutover

Historical reporting is unavailable; Athena does not present an empty business period.

Processing or rebuilding

Last verified generation remains readable while replacement work proceeds.

Failed or unavailable

Failure is explicit and scoped; safe metrics or prior data remain only when contracts permit.

Verification strategy

Requirements proof

Every origin acceptance example AE1–AE10 maps to named backend, component, or integrated scenarios.

Access proof

Full-admin success and denial for pos-only, manager elevation, unauthenticated access, cursors, detail, and store switch.

Performance proof

Numeric gates: 500-document Overview, 350-document Items/Inventory pages, 100-row pages, 100-document custom-range batches, and a 200-document intraday remainder.

Accessibility and responsive proof. Sort direction, coverage, limitations, progress, and attention state use text and accessible names—not color alone. Dense tables retain a clear narrow-screen information hierarchy, keyboard semantics, live async status, and focus restoration after detail navigation.

Storefront timing: financial period state is preserved for return navigation, but period controls are hidden on Storefront and the page announces its independent live-engagement window.

The canonical implementation detail, file list, test matrix, system impact, and origin traceability live in the Markdown plan.

Scope boundary: no organization rollups, accounting system, service profitability, speculative forecasting, configurable thresholds, or operational mutations inside Reports.
