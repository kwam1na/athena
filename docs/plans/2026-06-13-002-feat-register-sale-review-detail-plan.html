feat: Add Sale-Level Register Sync Review Details

Add an in-flow sale-level detail panel to Cash Controls register-session sync review so a manager can understand the synced sale being rejected or reviewed without leaving the drawer workspace.

Requirements Key Decisions Implementation Units Impact Risks

Problem Frame

The register session correctly owns sync review decisions, but the current review banner explains the action more than the sale. Managers need sale reference, amount, payment, cashier, time, and consequence in the register session before taking action.

Scope shape: inline sale-level review details inside the existing review banner. No new route, no item/payment drilldown, and no correction workflow in this first pass.

Requirements

  • Managers stay on the Cash Controls register session while inspecting synced sale review context.
  • The review banner exposes sale reference, amount, item count, payment method, cashier, completed/offline time, queue number, status, and reported time when available.
  • Unsupported or reject-only states explain why the sale cannot be applied and what rejecting does.
  • Multiple review items render as compact sale-level rows with independent expansion.
  • Closeout review behavior remains value-first and unchanged.
  • Raw sync internals and sensitive proof or payment material stay out of the primary UI.
  • Fallback-only records continue to render useful reason/type/queue evidence.

Key Technical Decisions

  • Extend the existing reconciliation item contract instead of adding a separate review-detail query.
  • Enrich sale context at projection/query boundaries from safe sync conflict details.
  • Render detail inline in the existing review banner so the register session remains the workspace.
  • Keep closeout review presentation separate from sale-level review detail.
  • Treat missing sale-level fields as normal and preserve fallback evidence.

Intended Flow

Local sale sync conflict
  -> safe sale summary in conflict details
  -> register-session reconciliation item
  -> register snapshot
  -> compact synced sale row
  -> inline sale detail expansion
  -> existing approve/reject action area

Implementation Units

U1. Extend Sale-Level Review Evidence Contract

Add optional sale-summary fields to PosReconciliationItem and matching Convex/domain return types. Keep the contract additive so old review records render cleanly.

Primary files: sync status presentation, register session sync review builder, POS domain types, RegisterSessionView tests.

U2. Populate Safe Sale Context From Sync Projection Evidence

Preserve safe sale context when sale sync conflicts are created and summarized: receipt/reference, total, item count, payment label, completed time, and staff/cashier context where safe.

Primary files: local event projection, register session sync review builder, Cash Controls snapshot query, Convex tests.

U3. Render Inline Sale-Level Review Details

Render compact sale rows and inline expansion inside RegisterSessionSyncNotice, while preserving closeout review and existing manager sign-in flows.

Primary files: RegisterSessionView, sync status presentation, component tests.

U4. Validate Browser Flow, Copy, And Guardrails

Verify the actual register-session page at desktop and mobile widths, check copy against Athena tone guidance, and update solution docs only if a reusable pattern emerges.

Primary files: RegisterSessionView tests, deposits tests, optional docs/solutions note.

System-Wide Impact

Interaction Graph

Local sync projection creates conflict evidence, the register snapshot exposes reconciliation items, and Cash Controls renders the review packet before manager action.

State Lifecycle

Older conflict rows may lack sale-level fields; the UI must degrade to existing evidence without blocking review.

Unchanged Invariants

Projection settlement, manager authorization, linked transaction navigation, closeout review, and terminal recovery routing remain unchanged.

Risks And Mitigations

RiskMitigation
Projection does not populate enough data.Add projection and snapshot tests before UI-only assertions.
UI exposes raw sync internals or sensitive proof data.Keep the contract display-safe and add negative tests.
Closeout review regresses.Preserve closeout-specific tests and keep sale detail out of the closeout branch.
Multiple review items crowd the banner.Use compact rows and independent expansion with actions after evidence.
Old review rows lack sale fields.Treat fields as optional and preserve fallback evidence.
Completed/offline time is not present on older conflict rows.Copy it at projection time or deliberately join the source sync event when building review status.

Verification Strategy

  • Focused component coverage in RegisterSessionView.test.tsx.
  • Convex projection and snapshot coverage in sync and deposits tests.
  • Browser verification of the register-session review banner in flow.
  • Graphify rebuild after code changes during implementation.
