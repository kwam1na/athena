Make POS Flow Traces Remote-Monitorable

This plan strengthens Athena POS evidence so remote operators can monitor store sales from cloud surfaces without needing the cashier browser. It aligns online commands and offline sync projection across POS session traces, register session traces, operational events, and Daily Operations timeline links.

Problem Frame

POS is already local-first and can continue selling when cloud validation is unavailable, but the remote-monitoring story is uneven. Some online actions write rich workflow and operational evidence, some synced offline actions write partial evidence, and some trace identifiers are computed but not surfaced.

Planning assumption: Daily Operations remains a committed-milestone feed. Full in-progress cart reconstruction belongs in POS session traces and support surfaces.

Requirements

  • Remote operators can monitor committed POS activity from cloud surfaces.
  • Online and synced offline sales produce comparable POS session trace, register session trace, and operational event evidence.
  • Register session traces distinguish sale total, cash delta, tender summary, and transaction identity.
  • Synced offline closeout and reopen/review outcomes produce operator-visible evidence equivalent to online cash-control actions.
  • Daily Operations links committed sale, register, review, and sync-conflict milestones to the relevant transaction, register, or trace surface.
  • POS register and Cash Controls expose trace links where trace ids exist.

Key Technical Decisions

  • Reuse existing evidence rails instead of creating a new monitoring subsystem.
  • Keep Daily Operations milestone-oriented and keep detailed cart reconstruction in workflow traces.
  • Normalize sale evidence at server command and sync projection boundaries.
  • Enrich register-session trace inputs with transaction and tender context.
  • Write operational events during offline closeout projection instead of relying only on read-model synthesis.
  • Keep local-only cart events local unless implementation proves completed-sale payloads are insufficient.

Design Shape

POS register view
  -> local POS event log
  -> online POS commands

local POS event log -> local sync projection
online POS commands -> POS trace + register trace + operational events
local sync projection -> POS trace + register trace + operational events

operational events -> Daily Operations timeline
register traces -> Cash Controls register detail
POS traces -> POS transaction and support trace surfaces
local backlog/conflicts -> Terminal Health diagnostics

Implementation Units

U1. Define POS Monitoring Evidence Contract

Extend trace and event input shapes so sale total, cash delta, tender summary, payment count, transaction identity, actor, and sync origin can be represented consistently.

Primary files: registerSessionTracing.ts, posSessionTracing.ts, projectLocalEvents.ts, sync/types.ts.

U2. Align Online Sale Completion Evidence

Thread transaction context into online register sale traces and ensure completed online sales produce monitorable operational evidence.

Primary files: completeTransaction.ts, registerSessions.ts.

U3. Align Offline Sync Projection Evidence

Make accepted offline sale and closeout projection emit POS traces, register traces, and operational events equivalent to online actions.

Primary files: projectLocalEvents.ts, localSyncRepository.ts.

U4. Surface Remote Monitoring Links

Render transaction, register, and trace links across Daily Operations, POS register controls, Cash Controls, and transaction surfaces.

Primary files: dailyOperations.ts, DailyOperationsView.tsx, SessionManager.tsx.

U5. Make Review Evidence Monitorable

Ensure sync conflicts and manager-review outcomes are visible through operational events and linked review surfaces.

Primary files: projectLocalEvents.ts, deposits.ts, closeouts.ts, syncStatusPresentation.ts.

U6. Validation, Documentation, Delivery Readiness

Run focused parity tests, update durable solution documentation, and refresh generated artifacts when Convex changes require it.

Primary files: focused Convex tests, React tests, and a new docs/solutions note.

Risks

Risk Mitigation
Trace noise overwhelms Daily Operations. Limit Daily Operations to committed milestones and review states.
Online and offline evidence drift again. Centralize evidence shaping where practical and add parity tests.
Sync retries duplicate evidence. Guard event writes with local event ids, mappings, or projection idempotency checks.
Non-cash drawer trace copy becomes misleading. Separate sale total from cash delta and test non-cash copy explicitly.

Validation Strategy

  • Projection tests for offline sale, closeout, retry, and review evidence.
  • Online command tests for sale completion trace and operational event parity.
  • Register trace tests for cash and non-cash sale copy.
  • Daily Operations read-model and React tests for transaction/register/trace links.
  • POS register tests for active sale trace link rendering.
  • Cash Controls tests for synced closeout and review trace visibility.
