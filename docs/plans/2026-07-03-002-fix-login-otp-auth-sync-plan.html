Summary

After a valid OTP, Athena can briefly route through guards before Convex Auth and the Athena-user sync have settled. The plan keeps the handoff on /login until sync writes the local account ids, then sends the user into the authenticated app.

The scope preserves the existing Convex Auth OTP provider, POS recovery-code handoff, and server-owned Athena user sync. Pending handoff state is bounded metadata only; it does not change staff, terminal, drawer, POS, remote-assist, or manager authority.

Key Decisions

  • Keep sync ownership in LoginLayout. The login route already owns retries, local id storage, and safe sync failures.
  • Stop premature root navigation. The OTP form should start pending auth sync, then let the login layout navigate only after sync succeeds.
  • Apply the same rule to POS recovery. POS recovery should store a safe redirect target in handoff metadata and let LoginLayout navigate after sync.
  • Use pending auth sync as a recovery signal. Guards should not treat an explicit login handoff as final sign-out.
  • Fail closed. Pending state must carry bounded metadata, expire or clear on no-session settlement, and never act as auth proof.
  • Keep server authority. syncAuthenticatedAthenaUser remains the only sync/create boundary for Athena users.
