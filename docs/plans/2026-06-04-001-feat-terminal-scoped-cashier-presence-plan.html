Terminal-scoped cashier store-day presence

Persist the active POS cashier through reloads on the same terminal and store day without weakening terminal repair, drawer authority, local staff proof, or manager approval boundaries.

Feature plan POS register Local-first Terminal scoped Store-day scoped

What Changes

New local presence record

Add a POS local store record for the active cashier on one terminal and operating date.

Fast reload restore

Restore staffProfileId, proof token, roles, and cashier display state before showing the sign-in gate.

Quiet validation

Validate existing unexpired proof online without success toasts; proof renewal stays out of the MVP.

Boundary Model

Record Answers Must Not Do
staffAuthority Can this staff member sign in offline with username and PIN? Represent the active signed-in cashier after reload.
cashierPresence Who is currently signed in on this terminal for this store day? Follow to another terminal or approve manager-only actions.
drawerAuthority Can this drawer/register session sell right now? Trust cashier presence as drawer authority.
terminalIntegrity Is this terminal trusted to keep appending local sale events? Let a restored cashier override terminal repair/reset blocks.

Reload Lifecycle

1. Sign in Cashier authenticates online or through valid offline authority.
2. Persist Write active cashier presence with terminal, store, day, role, credential, and proof scope.
3. Reload Register boot reads same-terminal, same-day presence.
4. Validate Resolve local presence to sign-in guidance without granting cashier authority from storage.
5. Validate When online, confirm staff, terminal, credential, and role state.
6. Sell safely Local sale events still pass drawer and terminal gates.

Store-Day And Repair Rules

State Cashier Identity Sellability
Same operating date reload Restore if proof and freshness are valid Allowed only after drawer, terminal, and proof gates pass
Ordinary drawer close Preserved Blocked until drawer opens
Non-destructive terminal repair Preserved only after online validation Blocked until repair and validation pass
Terminal reset or re-provision Cleared Blocked until sign-in
Register or store-day close Cleared Blocked until a new ready store day/register

Implementation Units

  1. Define and expose the POS store-day identity.
  2. Add cashier presence storage, migration, proof wrapping, and redaction tests.
  3. Write presence on sign-in and clear it on identity-clearing transitions.
  4. Rehydrate presence during register boot before the sign-in gate opens.
  5. Add online validation for existing unexpired active presence.
  6. Update POS UI state, copy, accessibility, and local diagnostics.
  7. Add targeted validation, graph maintenance, and a solution note.

Safety Checks

Presence can restore identity, not sellability

Terminal repair, drawer authority, closeout, and local review states still block sale-affecting commands.

Proof freshness is checked at append

Restore-time validity is not enough. Sale start, cart changes, payment updates, and completion must fail closed if proof expires.

No plaintext proof

Persisted POS local staff proof material must be wrapped or encrypted. Raw bearer tokens cannot appear in local serialization.

No manager approval shortcut

A restored manager cashier still needs command-boundary approval for protected actions.

Success Criteria

  • Same-terminal reload resolves presence to validation-pending sign-in guidance for the current store day.
  • Different terminal, store, organization, or date cannot restore the cashier.
  • Explicit sign-out, register closeout, store-day close, terminal reset, and credential changes clear presence.
  • Ordinary drawer close preserves cashier continuity context but blocks sale commands until drawer authority returns.
  • Proof expiry mid-sale preserves the draft and blocks further sale-affecting commands.
  • Warm offline hard reload reaches validation-pending sign-in guidance when the app shell is already available.
  • Offline PIN sign-in still fails closed when local proof truly needs refresh.
  • No success toasts are shown for restore, validation, or non-destructive repair.
