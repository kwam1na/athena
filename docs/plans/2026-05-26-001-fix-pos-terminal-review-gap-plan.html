fix: Close POS terminal review evidence gaps

Type: fix Status: active Date: 2026-05-26 Source: Markdown plan
Summary Requirements Design Implementation Units Risks Verification

Summary

Align POS terminal health with the evidence operators can actually see: backend health summaries will return explicit attention reasons, the terminal detail page will explain local runtime review state beside cloud sync evidence, and the local sync runtime will stop leaving already-resolved review events stuck on the terminal.

Production Gap

Terminal M Supplies was labeled Needs review because its runtime check-in reported sync.status: needs_review and reviewEventCount: 1. The visible detail page emphasized cloud evidence, which showed projected events and a resolved conflict, so the badge looked unsupported.

Requirements

  • Expose explicit backend attention reasons for terminal health classifications.
  • Show local runtime review state when it drives the terminal detail badge.
  • Clear local review counts after server-side projection or intentional settlement.
  • Keep Cash Controls and Operations as the manager-review action surfaces.
  • Keep operator copy clear and avoid exposing raw local payloads or secrets.
  • Cover the production mismatch with regression tests.

High-Level Design

The backend should answer why a terminal needs attention. The frontend should render that explanation alongside cloud evidence, rather than recalculating and implying that cloud conflicts are the only possible cause.

Browser local event log
  -> Runtime status builder
  -> Terminal runtime check-in
  -> Terminal health summary + attention reasons
  -> Terminal list/detail/register diagnostics
Local runtime evidence

Review count, oldest event age, next upload sequence, last trigger, local store health.

Cloud sync evidence

Accepted cursor, projected/held/rejected/conflicted counts, latest event sample.

Review ownership

Terminal health explains support evidence; Cash Controls and Approvals retain actions.

Risks & Mitigations

RiskMitigation
Clearing local review state too early hides real work.Only clear after server-projected or intentionally settled evidence.
Reason logic drifts again between backend and frontend.Backend returns reason payloads; frontend renders instead of rediscovering.
Copy implies terminal health owns approval actions.Route actions back to Cash Controls and Operations approval flows.
Older deployments lack the new reason field.Make frontend handling optional with fallback rendering.

Verification Strategy

  • Backend tests prove runtime review + clear cloud conflicts returns a local runtime attention reason.
  • Detail view tests prove the production mismatch is explained without relying on the register debug strip.
  • Local sync runtime tests prove projected reviewed closeout events clear from local review counts.
  • Terminal list/detail/register diagnostics tests prove status wording stays aligned.
