Implementation plan · May 10, 2026 · Status: draft for signoff

Manager elevation access for store-day surfaces

Build temporary manager elevation as a store-scoped operating context for POS terminals. The signed-in Athena account remains visible in the global header, and the active elevated manager state appears directly next to that account email. Elevated navigation can unlock selected store-day surfaces, while protected mutations continue to enforce server-side command approval.

Origin and Scope

Approved direction

Use temporary manager elevation rather than broadening pos_only or requiring full account switching on the terminal.

The elevated state must be visible in the global header next to the signed-in account email.

Source artifact

This plan follows docs/brainstorms/2026-05-10-access-management-manager-elevation-signoff.html.

The signoff brief frames access as terminal-level manager presence, not as a change to the underlying Athena account role.

Technical Thesis

Athena should introduce a first-class manager-elevation session rather than overloading existing approval proofs. Approval proofs are intentionally action-bound and one-use. Manager elevation is different: it is a short-lived operating context that affects which store-day surfaces the terminal can see.

The implementation should split "can see store-day operations" from "is a full admin." The current UI has places where canAccessOperations() effectively means full-admin access and is reused for cash controls, operations, and procurement navigation. Manager elevation should unlock only the selected store-day capabilities, not procurement administration or organization configuration by accident.

Key Decisions

Persist elevation server-side

Store manager elevation in Convex with expiry, store, terminal, Athena account, and manager staff identity. Do not rely on local browser state as the authority.

Show elevation in the global header

Add an elevation chip beside the signed-in account email in src/routes/_authed.tsx. The chip should show who elevated the terminal and include a clear end-elevation action.

Use capability gates

Replace broad role checks in the affected shell surfaces with capability helpers such as store-day operations access, procurement admin access, and full-admin access.

Rejected Alternatives

Broaden pos_only

Rejected because it permanently weakens POS account boundaries and makes ordinary terminal use too privileged.

Reuse approval proofs as elevation

Rejected because approval proofs are intentionally bound to one action and subject. Elevation needs session semantics and visible lifecycle controls.

Client-only elevation

Rejected because route and command behavior would be spoofable or inconsistent after refresh, tab changes, or stale terminal state.

Implementation Units

IU-1

Server-side elevation model and API

Add a Convex-backed manager elevation record with store, organization, terminal, authenticated Athena user, manager staff profile, credential, creation time, expiry, ended time, and end reason. Add mutations to start and end elevation and a query to read the active elevation for the current store/terminal/account.

Primary files: packages/athena-webapp/convex/schema.ts, packages/athena-webapp/convex/operations/managerElevations.ts, packages/athena-webapp/convex/operations/staffCredentials.ts, packages/athena-webapp/convex/operations/operationalEvents.ts.

Tests: packages/athena-webapp/convex/operations/managerElevations.test.ts, packages/athena-webapp/convex/operations/staffCredentials.test.ts.

IU-2

Client elevation provider and hook

Add a provider under the authenticated shell that knows the active store and terminal, subscribes to the active elevation query, and exposes start, end, loading, expired, and active-manager state. Keep this separate from permissions so existing account-role checks remain understandable.

Primary files: packages/athena-webapp/src/contexts/ManagerElevationContext.tsx, packages/athena-webapp/src/hooks/useManagerElevation.ts, packages/athena-webapp/src/routes/_authed.tsx.

Tests: packages/athena-webapp/src/contexts/ManagerElevationContext.test.tsx, packages/athena-webapp/src/routes/_authed.test.tsx.

IU-3

Global header indicator and elevation control

Update the top bar so the signed-in account email remains visible and the elevated manager state appears immediately beside it. The default state should show only the account email. The elevated state should show a compact "Elevated by [manager]" indicator, expiry hint, and end-elevation control.

Primary files: packages/athena-webapp/src/routes/_authed.tsx, packages/athena-webapp/src/components/staff-auth/StaffAuthenticationDialog.tsx, optional packages/athena-webapp/src/components/operations/ManagerElevationDialog.tsx.

Tests: packages/athena-webapp/src/routes/_authed.test.tsx, packages/athena-webapp/src/components/operations/ManagerElevationDialog.test.tsx.

IU-4

Capability-aware permissions and route gates

Extend the permission layer with capability helpers that combine account membership and active manager elevation. Do not make elevation equivalent to full_admin. Replace affected shell checks with capability-specific checks.

Primary files: packages/athena-webapp/src/contexts/PermissionsContext.tsx, packages/athena-webapp/src/hooks/usePermissions.ts, packages/athena-webapp/src/components/PermissionGate.tsx, packages/athena-webapp/src/components/ProtectedRoute.tsx, packages/athena-webapp/src/components/app-sidebar.tsx.

Tests: packages/athena-webapp/src/hooks/usePermissions.test.tsx, packages/athena-webapp/src/components/PermissionGate.test.tsx, packages/athena-webapp/src/components/app-sidebar.test.tsx.

IU-5

First-release surface wiring

Wire manager elevation into the first set of store-day surfaces: cash controls, daily operations, open work, approvals, and stock adjustments where existing commands already enforce approval or role boundaries. Keep procurement, analytics, configuration, members, storefront admin, and organization settings full-admin only unless separately approved.

Primary files: packages/athena-webapp/src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/cash-controls.index.tsx, packages/athena-webapp/src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/operations.index.tsx, packages/athena-webapp/src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/operations/approvals.tsx, packages/athena-webapp/src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/operations/stock-adjustments.tsx.

Tests: affected route auth tests plus packages/athena-webapp/src/components/operations/OperationsQueueView.auth.test.tsx and cash-control route/view tests.

IU-6

Audit, expiry, and cleanup behavior

Record operational events when elevation starts, ends, expires, or is rejected. Ensure active elevation cannot cross store, terminal, or signed-in account boundaries. Expired elevation should stop granting capabilities immediately, and the header should fall back to the base signed-in account state.

Primary files: packages/athena-webapp/convex/operations/managerElevations.ts, packages/athena-webapp/convex/operations/operationalEvents.ts, packages/athena-webapp/convex/operations/helpers/eventBuilders.ts.

Tests: packages/athena-webapp/convex/operations/managerElevations.test.ts, with explicit coverage for expired, ended, wrong-store, wrong-terminal, and wrong-account records.

Sequencing

  1. Add the server-side elevation model, start/end API, active query, and backend tests.
  2. Add the client provider and header read path with no surface access changes yet.
  3. Add start/end elevation UI in the global header area and verify the account email plus elevated manager indicator layout.
  4. Split permission helpers into capability-level checks and update the sidebar without widening admin-only surfaces.
  5. Wire the first store-day surfaces and add route/view tests for base POS, elevated manager, and full-admin accounts.
  6. Run focused tests, Convex audit, TypeScript, browser verification, and graph rebuild.

Risks and Mitigations

Risk: accidental admin widening

Mitigation: introduce capability helpers instead of treating manager elevation as full_admin. Add tests proving procurement, configuration, and members stay admin-only.

Risk: stale elevation on shared terminals

Mitigation: server-side expiry, visible header state, explicit end action, store/terminal/account scoping, and immediate capability revocation when the active query no longer returns a valid record.

Risk: command approval boundary drift

Mitigation: keep action-level approval proof validation unchanged. Elevation can expose a surface, but commands still decide whether approval is required.

Risk: header clutter

Mitigation: use a compact chip next to the account email, collapse details into the account/elevation menu, and verify at desktop and tablet widths.

Open Items Before Execution

The plan is ready to execute with one default assumption: first release unlocks cash controls and operations-related store-day surfaces only. Procurement, analytics, configuration, and member management remain full-admin only.

If analytics or order-management read access should be part of the first release, decide that before implementation starts so route gates and tests can cover it deliberately.
