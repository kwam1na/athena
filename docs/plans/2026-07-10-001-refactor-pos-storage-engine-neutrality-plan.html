Refactor plan · Active · 10 July 2026

Make POS Local Storage Engine-Neutral

Move Athena POS and expense flows behind a high-level storage contract and one asynchronous runtime. IndexedDB remains the production engine, while future engines retain freedom over their schema, indexes, transactions, concurrency, and execution topology.

The refreshed plan assumes #642 is the terminal architecture going forward. Within #642, only its enduring IndexedDB-facing contracts and consumers are in scope—durable authority apply, mapping and authority record types, local coordinator composition, scoped access, and portable migration semantics—while the broader engine-neutral foundation remains unchanged.

Problem and scope

The event ledger is already the first durable cashier record, but IndexedDB selection and availability leak through register, expense, catalog, login, provisioning, readiness, sync, recovery, settings, and terminal lookup. Storage health, schema evolution, bounded operations, maintenance, and cross-engine migration are incomplete.

Invariant: A sale or expense succeeds only after the selected durable engine commits. Convex remains downstream of cashier success.
Engine freedom: The contract defines POS outcomes. It does not define object stores, SQL tables, indexes, transactions, workers, RPC, VFS, batching, or lock primitives.
Failure isolation: Storage initialization never blocks login or unrelated routes; only POS operations that require durable state fail when the selected engine is unavailable.
Deliberate deferral: This delivery does not ship SQLite and adds no retention-driven event deletion. Compaction still waits for an approved offline history contract.

Design shape

POS and expense flows→ Storage runtime→ Application capability ports→ IndexedDB now / future engine later

Health and maintenance sit beside the store contract. They publish redacted evidence, coordinate exclusive lifecycle work, and verify logical migration without deciding engine-internal representation.

Implementation units

U1 Contract→ U2 Runtime→ U3 Critical paths + U4 Consumers→ U5 Health + U6 Versions + U7 Bounded operations→ U8 Lifecycle proof

U1 · Contract and conformance

Name narrow POS-domain capabilities and separate lifecycle ports. Prove storage semantics with IndexedDB and an independent high-level memory test engine.

U2 · Shared runtime

Provide one app-root asynchronous runtime for login and authenticated routes while keeping pure application functions explicitly injected.

U3 · Critical command paths

Move register, expense, authentication, and provisioning writes; remove volatile fallback and preflight durability.

U4 · Remaining consumers

Move catalog, entry, login, settings, terminal lookup, sync/recovery, and the dedicated authority coordinator onto narrow runtime capabilities.

U5 · Durability health

Add persistence posture, quota pressure, structured failures, restricted support detail, and calm operator states. Diagnostics warn; actual commit outcomes retain authority.

U6 · Version evolution

Separate version domains and characterize the current #642 logical records stored in IndexedDB layout v9.

U7 · Bounded ledger policy

Replace hot full scans with scoped operations and define retention pressure while keeping deletion disabled.

U8 · Maintenance and cutover

Fence authority apply as well as commands/sync, and migrate exact authority subjects, two-part revisions, and channel separation. Engine mechanics remain native.

Verification strategy

  • Run one conformance suite against IndexedDB and an independently implemented high-level memory test engine; defer worker/RPC topology proof.
  • Characterize every register and expense success boundary before replacing construction.
  • Prove offline seed, catalog, availability, staff authority, drawer lifecycle, closeout, receipt, sync, activity, and recovery parity.
  • Inject quota, corruption, schema, migration, maintenance, partial-provision, and transaction failures.
  • Prove cross-engine logical completeness and equivalent register/expense projections before activation.
  • Keep migration data, secrets, proofs, and raw payloads absent from diagnostics at every layer.
  • Keep migration inside the maintenance runtime: no downloadable export, no production activation or source-destruction capability, and attempt-scoped cleanup of any protected staging.
  • Treat integrity as corruption detection only; existing trusted terminal, staff-authority, proof, and cloud-validation boundaries remain authoritative.
  • Prove denied persistence and estimated pressure remain diagnostic, actual commit/quota failures block only the affected command, and operator copy hides engine internals.
  • Prove both authority revision orders, exact mapping races, offline continuity, readiness timeout, persistence retry, old-cart replacement behavior, and semantic effect stability.
  • Verify cloud closure with an old cart remains read-only/clear-only and opens a distinct replacement only after durable clear.
  • Finish with focused Vitest, offline browser coverage, production-backend POS smoke, typecheck/build, Convex audit, harness review, Graphify rebuild, and pr:athena.
