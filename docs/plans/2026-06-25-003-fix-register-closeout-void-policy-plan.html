fix: Allow approved sale voids during register closeout review

Type: fix Status: active Date: 2026-06-25 Source: markdown plan
Summary Requirements Decisions Units Risks Verification

Summary

Approved completed-sale voids should be able to apply while the original register session is in closing, so Cash Controls can update expected cash and variance before final closure.

Closeout submission stays available. Final register closure waits while pending pos_transaction_void approvals exist.

Requirements

Lifecycle
Allow void application for open, active, and closing; keep closed blocked.
Closeout
Do not block cashier submission or counting; defer final closure until void approvals settle.
Cash Controls
Show a minimal pending void count, route managers to /operations/approvals, and expose authorized Finalize closeout after review clears.
Policy
Move void applicability into a named shared lifecycle policy instead of reusing sale usability.

Key Decisions

  • Add a named void-application policy in shared/registerSessionLifecyclePolicy.ts.
  • Allow closing only for approved void application, not for new sale creation.
  • Defer final closure, not closeout submission, when pending void approvals exist.
  • Start with the existing approvalRequest.by_registerSessionId index for pending-void lookup.
  • Extend the closeout result contract so submitted-but-unresolved does not masquerade as closed.
  • Finalize resolved closeouts through Cash Controls with Finalize closeout instead of hiding closure inside void approval resolution.
  • Use a shared closeout-finalization guard for every path that can mark a register closed, including async variance approval and local sync closeout projection.
  • Recompute variance at finalization after voids settle, and keep finalization server-authorized.

Implementation Units

Goal: Add and test a named policy for void applicability across register-session statuses.

Files: shared/registerSessionLifecyclePolicy.ts, shared/registerSessionLifecyclePolicy.test.ts

U2. Apply queued void approvals through the void lifecycle policy

Goal: Allow queued approvals to apply in closing while preserving existing blockers.

U3. Expose pending void approvals in Cash Controls read models

Goal: Project pending void review counts into dashboard and register-session snapshots with minimal, server-filtered review data.

U4. Defer final closeout closure while void approvals are pending

Goal: Begin closeout normally, then leave the register in closing until void approvals settle; gate every closure path, recompute variance, and provide manager/full-admin Finalize closeout once review clears.

Files: convex/cashControls/closeouts.ts, convex/pos/application/sync/projectLocalEvents.ts, closeout, local sync, and trace lifecycle tests.

U5. Surface pending void review in Cash Controls UI

Goal: Show managers why the register remains unresolved, link to /operations/approvals with return context, and expose Finalize closeout once pending void review clears.

Files: CashControlsDashboard.tsx, RegisterSessionView.tsx, optional approval queue label tests.

Risks

RiskMitigation

Pending void approvals stay invisibleAdd dashboard and detail-page read-model coverage.
Final closure remains blocked after reviewFilter only pending approvals and cover resolved states.
A close path bypasses pending-void reviewUse one shared closeout-finalization guard before any register status changes to closed.
Finalization closes under stale variance policyRecompute variance after voids settle and require approval when the new variance crosses policy.
Deferred finalization bypasses approval controlsRequire server-side store and manager/full-admin checks, plus proof lifecycle tests.

Verification Strategy

  • Shared lifecycle tests enumerate all register-session statuses.
  • Command tests prove queued void approval succeeds for closing and fails for closed.
  • Cash Controls backend tests prove pending voids defer final closure but not submission.
  • Closeout tests cover submit, inline manager approval, async variance approval, and finalization paths.
  • UI tests prove pending void review copy and approval routing appear without hiding closeout submission.
  • After implementation, run the focused webapp tests, TypeScript, Convex audit, graph rebuild, and diff check.
