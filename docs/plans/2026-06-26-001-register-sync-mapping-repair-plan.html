fix: Repair missing register-session mappings for completed POS sales

Type: fix Status: active Date: 2026-06-26 Source: markdown plan
Summary Requirements Design Units Risks Verification

Summary

Completed local POS sales that sync without a usable register-session mapping should be repairable, not reject-only. A manager-approved repair creates or reuses the missing local-to-cloud register-session mapping, replays the original sale idempotently, and keeps inventory, payment, and closeout consequences explicit.

Field example: register session th7dggy861qj04qm0rbk3xk635898wcv shows completed receipts under review, but the current backend classifies the missing mapping as generic review and leaves rejection as the available action.

Requirements

Review kind
Classify missing register-session mappings as first-class repair review, including ungrouped conflicts that validate against the current register session.
Evidence
Show full receipt, cashier, tender, item, local event, and target-drawer facts only to store-scoped managers.
Repair
Require manager staff proof, selected conflict ids, a freshness token, repair mapping reuse, sale replay, and exact conflict settlement.
Closeout
Use a generic closeout hold helper so pending void approvals and repairable sale-mapping conflicts both block final closure without blocking submitted counts.

Key Design

  • Add missing_register_session_mapping review kind and repair-aware action policy.
  • Validate the explicit current Cash Controls register session and detect the finalization hold; v1 does not add target search or Operations UI.
  • Evaluate ungrouped store/terminal missing-mapping conflicts against the current register session so repairable sales do not depend on a mapping before they appear.
  • Create or reuse an audited repair mapping with same-target reuse across source event ids, then reuse projectLocalSyncEvent.
  • Require a freshness token covering conflicts, source events, local register session, target session, Daily Close state, mapping state, and eligibility facts.
  • Use existing reviewed-sale inventory and payment semantics to retain the sale while routing unsafe stock correction to Operations.
  • Generalize the pending-void closeout guard into typed hold providers, then add repairable sale-mapping conflicts as another provider.
  • Allow reviewed sale repair while an unresolved closeout is held, refresh expected cash and variance evidence, and never reopen finalized drawers.
  • Keep Terminal Health explanatory only for these business-fact conflicts.

Implementation Units

U1. Validate target and finalization hold

Validate only the current register session and detect the closeout finalization hold from persisted store, terminal, register, Daily Close date/status, mapping, lifecycle, and ungrouped missing-mapping conflict facts.

U2. Classify and enrich review evidence

Add missing_register_session_mapping, attach completed-sale evidence, target validation facts, and the freshness token.

U3. Add manager-approved mapping repair mutation

Require linked manager staff proof, recompute persisted preconditions, create or reuse the repair mapping, replay the sale, audit the action, and preserve rejection as explicit intent.

U4. Replay repaired sale events

Use existing sale projection with focused natural-key recovery for the repaired sale_completed path and settle exact selected conflicts only.

U5. Generalize closeout holds

Extract the pending-void finalization guard into a shared POS sync/review policy helper with typed hold providers, add repairable sale-mapping conflicts as a provider, guard every final-close and deposit boundary, and refresh closeout evidence after repair.

U6. Update Cash Controls surface

Show Repair and apply sale activity for scoped repairable rows, including submitted-closeout hold states.

U7. Validate the implementation slice

Run focused backend and Cash Controls UI tests, then graphify after implementation code changes.
