Plan

refactor: Make Athena delivery loops proof-aware and measurable

Date: 2026-06-18 · Status: active · Type: refactor

Summary

Make delivery work cheaper and easier to reason about by pruning duplicate local gate work, making $execute and reviewer loops proof-aware, and adding baseline capture plus a durable delivery-run ledger. Metrics support the audit findings; they are not the whole project.

Audit Findings

  • Agents often rerun full bun run pr:athena during reviewer loops instead of using focused tests between small fixes.
  • pr:athena:validate currently runs harness:check directly, then calls harness:review, which runs harness:check internally.
  • The full local gate runs both test:coverage and harness:test, while both currently target root scripts/*.test.ts tests.
  • Root test:coverage runs @athena/webapp test:coverage, while harness:review can still select @athena/webapp test or focused webapp Vitest commands in the same local gate.
  • Existing proof reuse is strong, but handoffs do not always separate validation success from reusable proof success.
  • Completed subagents can stay open across reviewer waves unless cleanup is made explicit.
  • Historical metrics exist for harness runtime trends and scorecards, but not for delivery runs, reviewer waves, proof reuse, and duplicate validation work.

Scope Guard

The first package is local-only. CI duplicate pruning and CI provider-semantics changes are deferred unless dedicated workflow-provider tests are added. Existing CI provider flags stay frozen.

Requirements

R1
Capture a pre-pruning baseline, then record durable delivery-run metrics.
R2
Prune duplicate local gate work after equivalent coverage is proven.
R3
Use narrow capability-based provider evidence for the local duplicate targets, including webapp Vitest duplication.
R4
Update $execute guidance for proof-aware reviewer loops.
R5
Make subagent cleanup sequence explicit before reviewer waves.
R6
Report validation status and proof status separately in ledger and handoffs.

Provider Contract

The provider contract covers only the audited local duplicate targets in this package. Provider evidence must include schema version, provider, base/head/tree identifiers, validated time, capabilities, and command evidence.

athena-webapp-vitest@athena/webapp test:coverageCharacterization proves coverage-mode Vitest execution covers the full or focused package test command selected by harness:review.

Run Ledger States

StateMeaning
startedA delivery run or gate phase began with base, head, and tree identifiers.
validatedA validation command or capability completed with status and duration.
provider_skippedA validation was skipped with status: "covered_by_provider" and same-tree evidence.
proof_recordedThe gate recorded reusable proof.
proof_not_recordedValidation passed, but proof was degraded and pre-push will validate normally.
prepush_reusedPre-push reused proof and avoided the expensive suite.
blockedA run stopped before a valid delivery state.
interruptedA run ended before completion with the last completed state recorded.

Implementation Units

U0. Baseline Capture Before Pruning

Capture the current expanded gate chain and either run an observe-only baseline or import audit findings into an ignored baseline artifact before pruning begins, including duplicate webapp package test execution.

U1. Delivery-Run Ledger, Wrapper, and Scorecard Visibility

Create scripts/pr-athena-delivery-run.ts as the concrete local gate instrumentation point, plus ledger storage, tests, scorecard integration, and docs.

U2. Capability Provider Contract and Harness Review Dedupe

Add local provider evidence for harness-doc-freshness, root-script-tests, and athena-webapp-vitest. Evidence is written only after provider commands pass; missing, dirty, stale, or incomplete evidence runs validation instead of skipping.

U3. Prune Local pr:athena Duplicate Work

Remove direct duplicate harness:check, duplicate root script test execution, and duplicate webapp Vitest execution only after policy, regression, parity, and package characterization tests prove coverage.

U4. Pre-Push and Proof-Status Instrumentation

Emit structured proof events and require a machine-readable handoff summary that distinguishes validation success from reusable proof success.

U5. Proof-Aware $execute and Reviewer Guidance

Update $execute first, and only directly referenced companion guidance if needed, after script states and commands are settled.

U6. Docs and Post-Implementation Learning

Document the final contract, keep the HTML artifact alongside markdown, add the solution note after tests prove behavior, and rebuild Graphify after code changes.
