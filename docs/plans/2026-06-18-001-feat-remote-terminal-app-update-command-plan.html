Scope

The active scope is the bridge between two existing foundations: typed terminal recovery commands and the update-ready coordinator. Support can queue the request any time; the terminal evaluates local coordinator state and blockers before any reload.

Boundary: this is not remote control and not a forced refresh. Command acknowledgement is action lifecycle only; fresh command-correlated runtime evidence is the update truth.

Requirements Digest

  • Full-admin support can send an allow-listed update_app command to active terminals.
  • The terminal handles current, update-ready, blocked, applying, detector-failed, ready-unstaged, stale, and unknown states locally.
  • The command queue enforces single-consumer execution with a claim token or execution id.
  • Runtime status publishes sanitized, code-based app-update evidence for Terminal Health.
  • Coordinator blockers and cross-tab blockers remain authoritative.
  • No secret, staff proof, payment, customer, or raw local payload data leaks through command or runtime evidence.

Key Decisions

  • Add update_app to the existing terminal command queue instead of adding a new support channel.
  • Make the command an anytime intent; the terminal applies the latest pending build at execution time.
  • Publish structured app-update evidence instead of relying only on ad hoc build-string comparison.
  • Acknowledge before reload because applyUpdate() can reload synchronously.
  • Use server-side claim semantics as the authoritative exactly-once guard; browser locks are only defense in depth.
  • Require command-correlated runtime evidence so replayed pre-command status cannot verify a new command.
  • Model blocked and no-op outcomes as completed local evaluations, not support-recovery failures.
  • Keep Terminal Health state derived from fresh runtime evidence, with command status shown as lifecycle context.

Canonical App-Update Status Model

Terminal Health and command acknowledgement should use the same normalized vocabulary even if the coordinator keeps its own internal state names.

StatusMeaningCommand behaviorTerminal Health behavior
currentNo pending update, or already on the latest known build.Acknowledge completed no-op.Show current evidence; keep the action available for future retry.
update_readyPending update exists and coordinator says apply is safe.Acknowledge applying, then invoke the reload latch.Show update ready until post-reload check-in verifies current.
update_ready_unstagedUpdate exists, but assets are not ready or canApply is false for staging reasons.Acknowledge completed evaluation unless canApply is true at execution time.Show update available but not ready to refresh.
blockedLocal work, sync review, payment/sale risk, or cross-tab blocker prevents refresh.Acknowledge blocked evaluation; do not reload.Show operational blocker guidance without marking terminal failed.
applyingThe terminal accepted apply and acknowledged before reload.Wait for fresh command-correlated runtime evidence.Show waiting for check-in, not verified.
detector_failedThe update detector errored or cannot produce a trustworthy snapshot.Acknowledge unable-to-determine evaluation.Show unable to determine, with retry available.
staleRuntime evidence is older than the freshness window.Do not use stale evidence to verify a command.Show stale/unknown, not current or outdated certainty.
unknownOlder client, missing adapter, or missing app-update evidence.Acknowledge unsupported/unable-to-determine when claimed by such a client.Show not reported/unknown and keep active-terminal action scoped by role and command lifecycle.

Implementation Units

U1. Extend the terminal command contract with Update app

Add update_app through backend/frontend command type unions, validators, expected evidence, command issue tests, dedupe, redaction, and server-side single-consumer claim semantics.

Primary files: terminal recovery types/schema/service, public terminal functions, terminal health types, command service/public tests.

U2. Publish structured app-update runtime evidence

Thread current build, pending build, canonical coordinator status code, apply availability, staging state, selected blocker code, observed timestamp, and optional command execution id/nonce through runtime status.

Primary files: runtime status schema, public terminal input stripping, terminal commands, local runtime builder, runtime tests.

U3. Execute Update app through the coordinator

Add an optional runtime adapter and two-phase executor: evaluate, persist acknowledgement, then run the post-ack effect such as applyUpdate(). Server claim tokens remain the authoritative duplicate-tab guard.

Primary files: local recovery command executor, POS local sync runtime, update coordinator/provider tests.

U4. Derive Terminal Health update state from runtime evidence

Normalize fresh, command-correlated app-update evidence into the canonical states without treating update readiness as a health blocker or letting stale command state overrule runtime truth.

Primary files: terminal queries, terminal health types/presentation, terminal detail/roster UI tests.

U5. Wire Terminal Health Update app action

Expose the anytime Update app support action for active terminals with explicit disabled states, duplicate protection, accessible status announcements, and calm operational copy.

Primary files: terminal detail, terminal health roster, presentation helpers, component tests.

U6. Update validation routing and durable docs

Update solution notes and, only if current routing is missing coverage, harness routing so future changes to this boundary run both app-update and terminal-health validation slices.

Primary files: solution notes, validation guide/map, harness registry and tests.

Verification Strategy

Implementation should combine the existing app-update readiness/apply-safety suite with the POS terminal health diagnostics suite.

  • Command service/public tests for update_app, dedupe, auth, and redaction.
  • Concurrent claim and stale-token tests for server-side single-consumer execution.
  • Runtime status tests for app-update evidence and publish-loop stability.
  • Local runtime tests for ack-before-reload, blocked/no-op/applying outcomes, replayed evidence rejection, and duplicate-tab safety.
  • Terminal Health query/presentation/component tests for fresh, stale, blocked, current, unknown, accessibility, and command/runtime lane separation.
  • At least one manual or Playwright multi-tab gate before production rollout.
  • Typecheck, build, Convex audit/lint, graphify rebuild after code changes, and harness review when validation routing changes.
