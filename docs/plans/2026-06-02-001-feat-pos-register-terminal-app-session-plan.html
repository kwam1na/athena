Keep POS hub app sessions recoverable

Markdown source: 2026-06-02-001-feat-pos-register-terminal-app-session-plan.md Scope: protected POS hub route family
Summary Requirements Design Units Risks Rollout

Summary

Registered POS terminals should keep the POS hub available when the app-level Athena login drifts into a recoverable signed-out state. A static deploy is treated as a forced app reload and auth rehydration event, not an intentional logout. The plan adds a POS-hub recovery path that validates terminal, store, and app-account authority on the server while preserving the existing local-first POS sale gates.

Chosen approach

Terminal-scoped app-session continuity for the protected POS hub route family.

Not chosen

No global infinite app session, no stored app-account credentials, and no admin/operations continuity yet.

Sale authority remains separate

Terminal integrity, drawer authority, local command invariants, and staff proof continue to decide whether the terminal can sell.

Requirements

  • POS hub routes must not redirect to /login for recoverable app-login drift when same-store local terminal context exists.
  • Non-POS routes must keep current login redirect behavior.
  • Recovery must validate terminal, store, app account, and account capability server-side.
  • Recovery must not persist reusable app credentials, staff proof, raw tokens, or OTP material.
  • Unrecoverable states render POS operational gates with safe reasons and a recovery path.
  • Terminal health must expose app-session recovery status without leaking sensitive data.
  • Any fallback POS-scoped assertion must be short-lived, same-store, same-terminal, route-scoped, and abuse-resistant.
  • Static deploy reloads must not clear POS continuity or navigate the POS hub to /login during transient auth rehydration.

Design Shape

The app shell needs route-scoped auth classification instead of a single user === null login redirect decision. POS hub routes can enter a recoverable terminal state after deploy reload or other auth rehydration events; other routes still go to login.

State Meaning Route behavior Sale behavior
Loading Convex Auth, token recovery, or Athena user sync is unresolved. Keep existing loading behavior. No new authority granted.
Recoverable POS terminal App user is missing, but local POS terminal context is valid for the route store. Mount POS hub shell and start background recovery. Allowed only if existing POS local gates say sellable.
Blocked POS terminal Terminal seed is missing, mismatched, revoked, invalid, or rejected. Show POS setup or support gate. Block sale-affecting commands.
Signed out Non-POS route or no recoverable terminal context. Redirect to /login. Not applicable.

Risk Analysis

Risk Mitigation
Auth bypass expands beyond POS hub. Route-match continuity explicitly and test non-POS redirect parity.
Local terminal seed is stale or stolen. Validate terminal status, store scope, revocation, and proof on the server.
Recovery grants admin or operations access. Keep POS app-account capability POS-only and route-scoped.
Recovery endpoint can be probed for terminal or app-account existence. Require terminal proof, return safe reasons, and throttle or rate-limit repeated invalid attempts.
Static deploy reload is mistaken for user logout. Treat reload as POS auth rehydration, preserve terminal continuity while recovery settles, and require explicit sign-out or server rejection before blocking.
Recovery hides real sale blockers. Keep app-session recovery separate from terminal integrity, drawer authority, staff proof, and command gates.
Convex Auth does not support silent full session restoration. Use supported Auth APIs if available; otherwise keep a POS-scoped terminal app-session projection for POS hub sales/runtime behavior.
Fallback assertion becomes a durable credential. Keep assertions short-lived, same-store, same-terminal, route-scoped, and non-durable.

Rollout and Verification

  • Phase 1: route-safe continuity and non-POS negative coverage.
  • Phase 2: server-validated recovery and bounded client repair.
  • Phase 3: terminal diagnostics, POS gate integration, and regression coverage.
  • Regression coverage must simulate a static deploy reload followed by delayed Convex/Auth rehydration.
  • Visual validation remains user-owned per request; implementation should still preserve unit and integration verification.
