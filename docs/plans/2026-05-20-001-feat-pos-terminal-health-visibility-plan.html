Add POS Terminal Health Visibility

Type: feat Status: active Date: 2026-05-20 Source: Markdown plan
Summary Scope Key Decisions Implementation Units System Impact Risks Verification

Summary

Add a production support layer around the existing local-first POS runtime. Terminals will publish safe runtime check-ins to Convex, support users will get a terminal health console and detail drill-in, and existing register/cash-controls views will link cloud sync evidence to local browser evidence.

Execution boundary

Check-ins are visibility only. Cashier success remains tied to browser-local durable events.

Support outcome

Support can distinguish local-only, uploaded, projected, conflicted, stale, and unknown terminal states.

Privacy boundary

No PINs, verifier metadata, proof tokens, raw local event payloads, payment details, or customer details enter terminal health snapshots.

Scope

In Scope

  • Server-side terminal runtime status record and store-scoped support queries.
  • Browser runtime check-in publisher and redacted copy-diagnostics payload.
  • Dedicated POS terminal health console and terminal detail view.
  • Cash-controls and register links to terminal evidence without changing review ownership.
  • Validation-map, tests, and solution documentation for the new visibility boundary.

Out of Scope

  • No changes to cashier checkout, local event durability, sync ordering, idempotency, inventory reconciliation, payment semantics, or closeout policy.
  • No automatic repair, alerting, or conflict resolution in this first pass.
  • No organization-wide fleet dashboard; this plan stays store-scoped.

Key Decisions

DecisionRationale
Dedicated runtime status recordRegistration identity changes slowly; runtime health changes often and should be replaceable and timestamped.
Best-effort check-insA missed check-in must never block offline POS or cloud sync.
Summary-only diagnosticsSupport needs counts, timestamps, sequences, local ids, cloud ids, and statuses, not sensitive payloads.
Dedicated POS terminal health routeTerminal troubleshooting is operational support, not current-device setup. Settings can link to health but should not host the command center.
Cash Controls remains review ownerTerminal health explains state; it does not create a parallel reconciliation queue.

System-Wide Impact

  • Interaction graph: POS hub/register publishes health; the POS Terminal Health console and terminal detail consume health; POS Settings links to health while remaining setup-focused; cash-controls links terminal evidence.
  • Error propagation: Check-in failures stay diagnostic and do not escape as cashier workflow failures.
  • State lifecycle: Check-ins may be stale; local events remain authoritative until uploaded.
  • API parity: Public terminal queries must continue hiding syncSecretHash.
  • Unchanged invariants: POS commands remain local-first; sync ordering remains per local register session; cash-controls remains review owner.

Risks

RiskMitigation
Check-ins become an execution dependencyKeep calls best-effort and test failed check-ins during normal register operation.
Sensitive local data leaksUse strict validators, central redaction, and security-focused tests.
Stale cloud data is misreadDistinguish not reported, local-only, uploaded, projected, conflicted, and stale states.
Terminal console becomes a second review queueSummarize and link to Cash Controls rather than resolving conflicts there.
Validation misses new surfacesAdd registry and generated validation-map coverage in the delivery unit.

Verification Strategy

  • Convex tests for runtime status mutation/query authorization, redaction, stale/no-check-in terminals, and sync evidence joins.
  • Browser local runtime tests for non-blocking check-ins, local-store schema failures, staff authority readiness, and redacted diagnostics.
  • POS Terminal Health tests for roster status, stale posture, and no sync-secret exposure; POS Settings tests for setup focus and health handoff link.
  • Terminal detail tests for local-only, uploaded, projected, conflicted, and stale evidence states.
  • Cash-controls/register tests for terminal links, pending sync vs. needs-review separation, and support diagnostics availability.
  • Harness registry tests and generated validation-map review for the new POS terminal health surface.
