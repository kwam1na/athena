Summary

Extract POS terminal health and recovery read semantics into one computed server aggregate, TerminalOperationalState. The normalized ledgers remain separate, while Terminal Health, terminal detail, previewTerminalRecovery, and future diagnostics classify terminal state from one policy output.

Problem Frame

Terminal truth is currently assembled inline inside the terminal query path from runtime status, sync ledgers, active register-session evidence, command status, cloud repair classification, attention reasons, and public return shaping. That join is becoming the hidden source of truth.

The plan addresses that split before any self-heal behavior is implemented.

Key Decisions

Compute, Do Not Persist

The aggregate is computed per query for now. Materialization is deferred until invalidation and race behavior are explicitly designed.

Preserve Ledgers

Sync events, cursors, mappings, conflicts, runtime status, register sessions, and recovery commands remain distinct durable ledgers.

Separate Readiness Concepts

Sales readiness, support recovery, and diagnostic evidence are explicit outputs. healthy_idle and able_to_transact_now stay distinct.

No Self-Heal Yet

Diagnostic classifications are non-actuating. This plan does not issue commands or trigger repairs automatically.

Sales Readiness

State Meaning Key proof
healthy_idle No current support blocker, not proven to be transacting Active terminal, fresh enough runtime, no support/manual blockers
drawer_open Register/drawer evidence exists, but sale authority is not proven Cloud lifecycle allows drawer, sale authority not ready
able_to_transact_now Fresh evidence says the terminal can transact now Fresh runtime, ready sale and staff authority, usable active/open register

Support Recovery / Review Action

State Meaning Key proof
needs_cloud_repair Server-side safe repair candidate exists Safe cloud repair policy and fresh precondition facts
needs_terminal_action Matching terminal must perform local recovery Current terminal action candidate or active command requirement
needs_manual_review Business/review facts need human workflow Payment, inventory, closeout, variance, customer, sale, or manager-review facts

Implementation Units

U1. Characterize Current Terminal Read Semantics

Add parity and return-shape characterization before extraction.

U2. Create Query-Safe Terminal Operational Facts Boundary

Gather normalized facts in one query-safe, fact-oriented boundary without policy or writes.

U3. Build TerminalOperationalState Aggregate and Policy

Classify sales readiness, support recovery, diagnostic evidence, provenance, and freshness.

U4. Project Terminal Health and Recovery Preview From Aggregate

Make existing query outputs projections of the aggregate.

U5. Preserve Public Contracts and Validator Parity

Keep public args, aliases, validators, browser types, POS-authorized viewing, full-admin repair issuance, and terminal sync-secret check-in/list/claim/ack APIs aligned.

U6. Route Frontend Presentation Through Aggregate-Derived Semantics

Keep roster/detail support presentation preview-first and parity-aligned.

U7. Document the Boundary and Run Plan-Specific Validation

Capture the aggregate as the future diagnostics comparison boundary.

Risks

Risk Mitigation
Readiness behavior drifts during extraction Characterization-first tests and roster/detail/preview parity fixtures.
Aggregate becomes write-capable Keep it query-safe and fact-oriented; writes stay in existing mutations.
Public validators drift Dedicated validator parity unit before frontend reliance; U4/U5 land together if public return fields change.
Runtime evidence becomes sale authority Cloud lifecycle policy remains the drawer usability guardrail.

Validation Posture
