Block stale POS terminals from local sales

Markdown source: 2026-05-29-001-fix-pos-stale-terminal-sale-block-plan.md Origin: POS local-first register requirements
Summary Decisions Implementation Units Impact Risks Verification

Summary

Preserve POS local-first checkout for normal offline or pending sync, but pause selling when cloud explicitly rejects the terminal's sync/check-in authorization or when a mapped cloud drawer is closed/rejected while IndexedDB still projects it open. The fix adds persistent terminal-integrity and drawer-lifecycle authority state, threads both through local projection and command boundaries, self-heals eligible sync-secret drift, and falls back to setup or drawer-authority recovery only when repair cannot be trusted.

Allowed

Offline, pending upload, ordinary syncing, and normal local-first checkout for provisioned terminals.

Blocked

New sale-affecting events after terminal sync/check-in returns authorization_failed, or after cloud drawer lifecycle contradicts local open state.

Self-heal first

Use the authenticated session and terminal fingerprint to repair eligible sync-secret drift, preserving stranded local event diagnostics.

Problem Frame

Arc completed local-only sales from stale IndexedDB state even though Convex had no active register session for that Arc terminal and rejected its local sync token. Separately, the Codex browser showed a successful terminal check-in while local IndexedDB still projected an open drawer whose mapped cloud register session was closed. In both cases the browser failed to convert explicit cloud authority into a local sale block.

Scope

  • Do not change POS into cloud-confirm-before-sale.
  • Do not auto-delete local IndexedDB events or stranded sales.
  • Do not move manager reconciliation out of Cash Controls / Operations.
  • Do not add payment-provider-specific offline authorization or cross-terminal offline coordination.

Key Decisions

  • Terminal auth rejection becomes terminal-integrity state. This separates stale-terminal rejection from normal pending sync.
  • Persist the block locally. Reloads and route entry must not resume selling from stale projections.
  • Enforce at command boundaries. UI gates help operators, but local command writes must also reject blocked sale-affecting events.
  • Cloud drawer lifecycle is authority after mapping. A mapped cloud register session that is closed, rejected, missing, or not current overrides stale local open/reopen events.
  • Self-healing is the first recovery path. Eligible sync-secret drift can repair itself after cloud re-attestation and durable local seed persistence.
  • Operator recovery is fallback. Revoked terminals, ownership conflicts, register conflicts, missing access, and local seed write failures remain blocked.

State Model

Sellable
healthy local terminal
Pending Sync
offline or upload pending
Repairing
auth rejected, re-attesting
Lifecycle Blocked
drawer authority rejected or closed
Repaired
fresh seed persisted

Pending sync can remain sellable. Authorization failure pauses new sale-affecting commands while safe repair runs; mapped closed/rejected drawer authority blocks until a valid drawer authority is established.

Implementation Units

U1Persist terminal and drawer authority state

Add safe local metadata for terminal-integrity and drawer-lifecycle authority status, then expose it through POS local storage and runtime diagnostics.

Primary files: posLocalStore.ts, terminalRuntimeStatus.ts, and their tests.

U2Classify sync and check-in authorization failures

Convert `authorization_failed` responses from sync upload and runtime check-in into the persistent terminal-integrity block.

Primary files: usePosLocalSyncRuntime.ts, convex/pos/public/sync.ts, convex/pos/public/terminals.ts.

U7Self-heal terminal re-attestation

Automatically repair eligible sync-secret drift using the authenticated Athena session, same terminal fingerprint, same store/register scope, and durable local seed write.

Primary files: registerAndProvisionPosTerminal.ts, usePosLocalSyncRuntime.ts, convex/pos/application/commands/terminals.ts.

U3Block local projection and commands

Thread terminal-integrity, uploaded lifecycle review, and mapped cloud drawer closed/rejected authority through `canSell` and the local command gateway so stale clients cannot append new sale events or reuse a stale local drawer.

Primary files: registerReadModel.ts, localRegisterReader.ts, localCommandGateway.ts, syncStatus.ts.

U4Add POS repair/recovery gate

Render automatic repair progress, drawer authority recovery, or setup recovery in the register, and disable checkout and sale controls while blocked.

Primary files: useRegisterViewModel.ts, POSRegisterView.tsx, syncStatusPresentation.ts.

U5Align setup and health evidence

Keep POS Settings, terminal health/detail, and register diagnostics consistent about stale authorization and stale drawer authority without creating new reconciliation ownership.

Primary files: POSSettingsView.tsx, POSTerminalDetailView.tsx, terminalHealthPresentation.ts, convex/pos/application/queries/terminals.ts.

U6Validation, graph, and docs

Update harness coverage, solution docs, and graphify artifacts so stale-terminal sale blocking remains protected.

Primary files: scripts/harness-app-registry.ts, generated validation docs, solution docs, and graphify output.

Risks

RiskMitigation
Over-blocking breaks local-first POSOnly explicit auth rejection, lifecycle-authority review, or mapped cloud drawer closure/rejection block; offline/pending remains sellable.
UI-only blocking misses bypassesEnforce in local command gateway and read model.
Stale local drawer keeps bypassing cloud closeoutCheck mapped cloud drawer authority before `openDrawer` can return idempotent local success.
Self-healing hides a real access problemRepair only when cloud verifies the same active terminal, scope, and allowed user policy.
Recovery deletes evidenceClear the block and update seed without deleting local events by default.
Cloud repair succeeds but local seed write failsKeep POS blocked until seed persistence succeeds.
Support copy leaks sensitive dataReuse runtime redaction and avoid raw payloads/secrets.

  • Run the focused POS local sync/register validation slice, Convex audits, typecheck, build, harness review, and graphify rebuild during implementation.
