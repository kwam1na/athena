# Athena Webapp Architecture

The app is a TanStack Router browser client in [src/main.tsx](../../src/main.tsx). The generated route tree in [src/routeTree.gen.ts](../../src/routeTree.gen.ts) fans into public/login routes plus the authenticated shell rooted at [src/routes/\_authed.tsx](../../src/routes/_authed.tsx). Most user-facing work starts by figuring out whether the change belongs in route composition, a feature component under `src/components`, or a data hook under `src/hooks`.

Backend and integration boundaries sit in Convex. The top-level HTTP entrypoint [convex/http.ts](../../convex/http.ts) mounts Hono routes for owner/admin core flows, storefront commerce, guest checkout, rewards, and payment webhooks. Core route composition is grouped in [convex/http/domains/core/routes/index.ts](../../convex/http/domains/core/routes/index.ts), customer-channel HTTP surfaces are grouped in [convex/http/domains/customerChannel/routes/index.ts](../../convex/http/domains/customerChannel/routes/index.ts), and payment collection/webhook routes live under [convex/http/domains/moneyMovement/routes/index.ts](../../convex/http/domains/moneyMovement/routes/index.ts). Shared persistence shape lives in [convex/schema.ts](../../convex/schema.ts).

AI, LLM, recommendation, provider, retrieval, and future media-generation work belongs in the Athena intelligence layer documented in [intelligence.md](./intelligence.md). Do not add new durable behavior to the old `convex/llm` seam except as a temporary compatibility wrapper during migration. Athena-owned Convex tables, run/artifact state, context snapshots, provider invocations, review operations, authorization checks, and future apply boundaries remain the source of truth; TanStack AI, model providers, and optional Convex Agent/RAG/Durable Agents components stay adapter or runtime helpers below that contract.

Client/server command error handling now has an explicit shared foundation. Use [shared/commandResult.ts](../../shared/commandResult.ts) for browser-safe `ok` and `user_error` results that both Convex and React code can import without crossing the browser/server boundary. Approval-sensitive commands opt into [shared/approvalPolicy.ts](../../shared/approvalPolicy.ts) and return `approval_required` through the same command result rail rather than burying approval in success payloads. Browser code should normalize command execution through [src/lib/errors/runCommand.ts](../../src/lib/errors/runCommand.ts), surface safe `user_error` copy inline when the action has a durable error region, use toast only as a fallback for toast-only actions through [src/lib/errors/presentCommandToast.ts](../../src/lib/errors/presentCommandToast.ts), route legacy toast-only admin quick actions through [src/lib/errors/presentUnexpectedErrorToast.ts](../../src/lib/errors/presentUnexpectedErrorToast.ts), and let [src/components/auth/DefaultCatchBoundary.tsx](../../src/components/auth/DefaultCatchBoundary.tsx) stay the generic route/render backstop for escaped faults. Raw thrown server text is never acceptable user-facing copy, and [src/routeTree.browser-boundary.test.ts](../../src/routeTree.browser-boundary.test.ts) guards the browser import boundary for shared modules.

The command approval policy layer is domain-owned and command-enforced. Domain commands decide whether an action is allowed, denied, unsupported, requires inline manager proof, or requires async approval; shared types only define the browser/server contract. Inline manager approval must mint a short-lived, one-use, action/store/subject/role-bound proof through [convex/operations/staffCredentials.ts](../../convex/operations/staffCredentials.ts) and [convex/operations/approvalProofs.ts](../../convex/operations/approvalProofs.ts), then the protected command must consume that proof before mutation. Do not treat a client-supplied `staffProfileId` as approval. Shared React presentation lives in [src/components/operations/CommandApprovalDialog.tsx](../../src/components/operations/CommandApprovalDialog.tsx). Approval observability goes through [convex/operations/approvalAuditEvents.ts](../../convex/operations/approvalAuditEvents.ts) and [convex/operations/operationalEvents.ts](../../convex/operations/operationalEvents.ts); only domains with an existing trace lifecycle, such as register closeout variance review through [convex/operations/registerSessionTracing.ts](../../convex/operations/registerSessionTracing.ts), should add workflow trace milestones.

POS local-sync inventory authority lives in [convex/pos/application/sync/projectionPolicies.ts](../../convex/pos/application/sync/projectionPolicies.ts). Active provisional imports remain evidence-only, finalized provisional lineage becomes trusted sale demand, and stock precondition failures create `synced_sale_inventory_review` Operations work instead of requiring Cash Controls manager approval. Use the production repair runbook at [docs/operations/pos-sync-finalized-lineage-remediation-2026-07-06.md](../../../../docs/operations/pos-sync-finalized-lineage-remediation-2026-07-06.md) when correcting already-conflicted finalized-lineage sync events.

Manager approval authority must stay separate from the signed-in Athena session. Treat requester fields (`requestedByUserId`, `requestedByStaffProfileId`), reviewer-session fields (`reviewedByUserId`, `reviewedByStaffProfileId`), and decision-authority fields (`decisionApprovalProofId`, `decisionApprovedByStaffProfileId`) as different lanes. Domain records that apply an approved command should persist the proof lane explicitly, such as `voidDecisionApprovalProofId`, `completionApprovalProofId`, or `reopenApprovalProofId`, while keeping legacy compatibility fields only for existing readers. Automation evidence remains its own lane and should not backfill human approval proof fields. The durable pattern note is [Athena Manager Approval Authority Is Decision Evidence](../../../../docs/solutions/architecture/athena-manager-approval-authority-standard-2026-07-01.md).

Inventory auth and admin configuration mutations now sit on that same contract in [convex/inventory/auth.ts](../../convex/inventory/auth.ts) and [convex/inventory/stores.ts](../../convex/inventory/stores.ts). The login recovery loop in [src/routes/login/_layout.tsx](../../src/routes/login/_layout.tsx) treats auth-sync failures as command data, retries only the explicit retryable sign-in case, and otherwise renders safe inline fallback copy. Store-configuration screens under [src/components/store-configuration](../../src/components/store-configuration) share [src/components/store-configuration/hooks/useStoreConfigUpdate.ts](../../src/components/store-configuration/hooks/useStoreConfigUpdate.ts) so fulfillment, maintenance, tax, contact, fees, and MTN MoMo saves all pass through the same normalized success and toast-fallback path.

A smaller legacy-admin slice still resolves through toast-only UX. When actions in [src/components/add-product](../../src/components/add-product), [src/components/assets](../../src/components/assets), [src/components/join-team](../../src/components/join-team), [src/components/organization-members](../../src/components/organization-members), [src/components/promo-codes](../../src/components/promo-codes), or the settings views under [src/settings](../../src/settings) fail unexpectedly, route them through [src/lib/errors/presentUnexpectedErrorToast.ts](../../src/lib/errors/presentUnexpectedErrorToast.ts) so the title can stay task-specific while the description collapses to the shared generic fallback copy. Preserve explicit business-validation messages when those surfaces return them intentionally.

Service-management work now spans two Convex layers. Intake capture and shared operational rails start in [convex/operations/serviceIntake.ts](../../convex/operations/serviceIntake.ts), while the first-class service domain lives in [convex/serviceOps/catalog.ts](../../convex/serviceOps/catalog.ts), [convex/serviceOps/appointments.ts](../../convex/serviceOps/appointments.ts), and [convex/serviceOps/serviceCases.ts](../../convex/serviceOps/serviceCases.ts). Those surfaces reuse shared inventory movement, payment allocation, approval, and operational work-item helpers so service cases stay aligned with the manager queue instead of inventing a parallel workflow stack.

Stock operations now have a first-class Convex namespace in [convex/stockOps](../../convex/stockOps). Use [convex/stockOps/adjustments.ts](../../convex/stockOps/adjustments.ts) for inventory correction batches and approval-ready stock moves, then [convex/stockOps/purchaseOrders.ts](../../convex/stockOps/purchaseOrders.ts), [convex/stockOps/receiving.ts](../../convex/stockOps/receiving.ts), [convex/stockOps/replenishment.ts](../../convex/stockOps/replenishment.ts), and [convex/stockOps/vendors.ts](../../convex/stockOps/vendors.ts) for procurement planning, inbound receiving, and vendor context. [convex/operations/approvalRequests.ts](../../convex/operations/approvalRequests.ts), [convex/stockOps/adjustments.ts](../../convex/stockOps/adjustments.ts), and [convex/stockOps/receiving.ts](../../convex/stockOps/receiving.ts) now return command-result payloads for the operator-facing queue, adjustment workspace, and receiving flows. [convex/stockOps/access.ts](../../convex/stockOps/access.ts) is the server-side guardrail for store-scoped full-admin procurement access, so treat it as the boundary when procurement data should stay hidden from non-admin operators.

Operational staff identity now lives in [convex/operations/staffProfiles.ts](../../convex/operations/staffProfiles.ts) and [convex/operations/staffCredentials.ts](../../convex/operations/staffCredentials.ts). Staff creation now requires first name, last name, username, and role, while start date stays optional. The credential is provisioned as pending first and finished later with PIN setup or reset from staff management. Use `staffProfile` as the shared actor record for POS, expense, service, and cash-control flows, and `staffCredential` as the subsystem sign-in boundary. The legacy `cashier` table and API have been removed, so any new operator-auth or attribution work should start from `staffProfileId` rather than introducing a POS-only identity model.

The authenticated service-management UI hangs off the store shell through [src/components/app-sidebar.tsx](../../src/components/app-sidebar.tsx) and the routed views under [src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/services](../../src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/services). Use [src/components/services/ServiceIntakeView.tsx](../../src/components/services/ServiceIntakeView.tsx), [src/components/services/ServiceAppointmentsView.tsx](../../src/components/services/ServiceAppointmentsView.tsx), [src/components/services/ServiceCasesView.tsx](../../src/components/services/ServiceCasesView.tsx), and [src/components/services/ServiceCatalogView.tsx](../../src/components/services/ServiceCatalogView.tsx) when you need the feature entry points for intake, scheduling, active-case execution, and catalog administration.

The authenticated stock-ops UI splits across the manager queue and the procurement workspace. Use [src/components/operations/OperationsQueueView.tsx](../../src/components/operations/OperationsQueueView.tsx) and [src/components/operations/StockAdjustmentWorkspace.tsx](../../src/components/operations/StockAdjustmentWorkspace.tsx) for adjustment review and approval flows, then [src/components/procurement/ProcurementView.tsx](../../src/components/procurement/ProcurementView.tsx) and [src/components/procurement/ReceivingView.tsx](../../src/components/procurement/ReceivingView.tsx) for replenishment planning and purchase-order receiving. Those screens hang off [src/components/app-sidebar.tsx](../../src/components/app-sidebar.tsx) and the store routes under [src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/operations](../../src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/operations) plus [src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/procurement.index.tsx](../../src/routes/_authed/$orgUrlSlug/store/$storeUrlSlug/procurement.index.tsx).

Production readiness for Athena belongs on the Hono boundary in [convex/http.ts](../../convex/http.ts), where the shallow `/health` route can answer without depending on the browser router or authenticated shell composition.

For production monitor inventory, Cloudflare thresholds, Checkly login reachability expectations, and Slack routing, use the repo runbook in [docs/operations/production-observability-v1.md](../../../../docs/operations/production-observability-v1.md) instead of recreating the contract locally in package docs.

Storefront reporting boundaries also live in Convex. Business-facing analytics defaults flow through [convex/storeFront/analytics.ts](../../convex/storeFront/analytics.ts), which excludes the reserved `origin=synthetic_monitor` monitor traffic by default, while operator diagnostics stay inclusive through [convex/storeFront/storefrontObservabilityReport.ts](../../convex/storeFront/storefrontObservabilityReport.ts) and [convex/storeFront/customerObservabilityTimelineData.ts](../../convex/storeFront/customerObservabilityTimelineData.ts).

Storefront intelligence context is separate from those reporting surfaces. The public `/tracking-events` route records registered storefront context events into [convex/contextTracking/contextEvents.ts](../../convex/contextTracking/contextEvents.ts), and store/customer readouts compile from [convex/contextTracking/contextBundles.ts](../../convex/contextTracking/contextBundles.ts). Active readout paths must not query `analytics` or call legacy storefront analytics helpers. Historical storefront analytics contributes to readouts only after a bounded import writes safe, allowlisted evidence into `contextEvent` rows; quarantined, rejected, synthetic, or non-compilable context events stay out of business/customer bundles.

Omnichannel order operations also stay on those shared storefront rails now. Return and exchange planning plus execution live in [convex/storeFront/helpers/returnExchangeOperations.ts](../../convex/storeFront/helpers/returnExchangeOperations.ts) and [convex/storeFront/onlineOrder.ts](../../convex/storeFront/onlineOrder.ts), while order updates, refund initiation, review moderation, and order-email follow-up now normalize through [convex/storeFront/onlineOrder.ts](../../convex/storeFront/onlineOrder.ts), [convex/storeFront/payment.ts](../../convex/storeFront/payment.ts), [convex/storeFront/reviews.ts](../../convex/storeFront/reviews.ts), and [convex/storeFront/onlineOrderUtilFns.ts](../../convex/storeFront/onlineOrderUtilFns.ts). Loyalty and follow-up milestones continue to flow through the shared customer profile helpers in [convex/storeFront/helpers/customerEngagementEvents.ts](../../convex/storeFront/helpers/customerEngagementEvents.ts), [convex/storeFront/rewards.ts](../../convex/storeFront/rewards.ts), [convex/storeFront/offers.ts](../../convex/storeFront/offers.ts), and [convex/storeFront/customerBehaviorTimeline.ts](../../convex/storeFront/customerBehaviorTimeline.ts). The authenticated operator UI for those flows sits in [src/components/orders/OrderView.tsx](../../src/components/orders/OrderView.tsx), [src/components/orders/OrderDetailsView.tsx](../../src/components/orders/OrderDetailsView.tsx), [src/components/orders/OrderItemsView.tsx](../../src/components/orders/OrderItemsView.tsx), [src/components/orders/EmailStatusView.tsx](../../src/components/orders/EmailStatusView.tsx), [src/components/orders/RefundsView.tsx](../../src/components/orders/RefundsView.tsx), [src/components/orders/ReturnExchangeView.tsx](../../src/components/orders/ReturnExchangeView.tsx), [src/components/reviews/ReviewsView.tsx](../../src/components/reviews/ReviewsView.tsx), [src/components/users/CustomerBehaviorTimeline.tsx](../../src/components/users/CustomerBehaviorTimeline.tsx), and [src/components/users/TimelineEventCard.tsx](../../src/components/users/TimelineEventCard.tsx). Those surfaces now drive expected command failures through `runCommand` plus `presentCommandToast` instead of surfacing raw thrown backend text.

For any UI amount rendering, use the shared minor-unit-to-display conversion pipeline (`toDisplayAmount`) before formatting with `Intl.NumberFormat`, through [src/lib/pos/displayAmounts.ts](../../src/lib/pos/displayAmounts.ts), rather than formatting stored integer amounts directly. Storefront uses the same stored-money contract through `packages/storefront-webapp/src/lib/currency.ts`. For any UI amount entry that persists money, parse display input through `parseDisplayAmountInput` before mutation payload construction; percentage fields remain raw percentages. Storefront checkout money is server-owned: client totals are advisory, while Convex recomputes SKU prices, delivery fees, discounts, and refund caps at the command boundary. The durable bug-pattern note is [Athena Money Inputs Must Persist Minor Units](../../../../docs/solutions/logic-errors/athena-money-inputs-minor-units-2026-04-23.md).

When you need to move logic across the browser/server boundary, prefer the Convex guidance in [convex/README.md](../../convex/README.md): keep public functions thin, treat Hono routes as explicit public boundaries, and push reusable server logic behind internal helpers instead of duplicating fetch-time behavior in React code.

The first-pass architecture lint check protects the authenticated admin/store entrypoints in [src/routes/\_authed.tsx](../../src/routes/_authed.tsx) plus the shell-only files it composes: [src/components/app-sidebar.tsx](../../src/components/app-sidebar.tsx), [src/contexts/PermissionsContext.tsx](../../src/contexts/PermissionsContext.tsx), [src/components/ui/modals/store-modal.tsx](../../src/components/ui/modals/store-modal.tsx), and [src/components/ui/modals/organization-modal.tsx](../../src/components/ui/modals/organization-modal.tsx). Lower-layer files under `src/components`, `src/hooks`, `src/contexts`, `src/lib`, `src/settings`, and `src/stores` may be imported by those entrypoints, but they must not import back upward into them.
